인증(Authentication) & JWT 토큰

백엔드 면접에서 자주 나오는 인증/인가 핵심 Q&A · Session · Token · JWT · OAuth 2.0 · Security

인증(Authentication)과 인가(Authorization)의 차이를 설명해주세요.

비유로 설명하면, 인증(Authentication)은 건물 입구에서 "당신이 누구인지" 신분증을 확인하는 것이고, 인가(Authorization)는 신분증이 확인된 뒤에 "몇 층까지 출입 가능한지" 권한을 확인하는 것입니다. 즉 인증은 "너 누구야?", 인가는 "너 이거 해도 돼?"의 문제입니다.

백엔드에서 인증은 로그인(ID/PW, OAuth, 생체인증 등)으로 사용자를 식별하는 과정이고, 인가는 인증된 사용자에게 특정 리소스(API, 페이지, 데이터)에 대한 접근 권한을 부여하는 과정입니다. 예를 들어 일반 사용자는 자기 프로필만 볼 수 있지만, 관리자는 모든 사용자의 프로필을 볼 수 있는 것이 인가의 영역입니다. Spring Security에서는 AuthenticationFilter가 인증을, @PreAuthorize나 Role 기반 설정이 인가를 담당합니다.

꼬리질문 대비: "인증 없이 인가가 가능한가요?" — 일반적으로는 불가능합니다. 누구인지 모르는데 권한을 판단할 수 없기 때문입니다. 다만 익명 사용자에게 공개 리소스 접근을 허용하는 것은 "인증 없이 기본 인가 정책을 적용하는" 설계로 볼 수 있습니다.
세션 기반 인증과 토큰 기반 인증의 차이를 비교해주세요.

세션 기반 인증은 서버가 사용자의 로그인 상태를 서버 메모리(또는 Redis 등)에 저장하고, 클라이언트에게는 Session ID만 쿠키로 내려주는 방식입니다. 클라이언트가 요청할 때마다 쿠키에 담긴 Session ID를 서버가 받아서, 저장소에서 해당 세션 정보를 조회합니다. 반면 토큰 기반 인증은 서버가 인증 정보를 담은 토큰을 발급하고, 서버는 상태를 저장하지 않습니다(Stateless). 클라이언트가 매 요청마다 토큰을 보내면 서버는 토큰 자체를 검증만 합니다.

비교 항목세션 기반토큰 기반 (JWT)
상태 관리Stateful (서버에 세션 저장)Stateless (서버 저장 없음)
저장 위치서버 메모리/Redis + 클라이언트 쿠키클라이언트 (localStorage/쿠키)
확장성서버 확장 시 세션 공유 문제서버 확장에 유리
보안CSRF 취약 / 세션 하이재킹XSS 취약 / 토큰 탈취
무효화서버에서 즉시 세션 삭제 가능만료 전 강제 무효화 어려움
용도전통적 웹 애플리케이션SPA, 모바일, MSA

핵심 trade-off를 요약하면, 세션은 서버 측 제어력이 강하지만 확장에 불리하고, 토큰은 확장에 유리하지만 즉시 무효화가 어렵다는 점입니다. 실무에서는 MSA 환경이나 모바일 앱 대응을 위해 토큰(JWT) 방식을 많이 쓰고, 즉시 무효화가 필요한 경우 Redis에 블랙리스트를 관리하는 하이브리드 방식을 취합니다.

쿠키(Cookie)와 세션(Session)의 관계를 설명해주세요.

쿠키는 운반 수단이고, 세션은 서버에 저장된 상태 데이터입니다. 비유하면 쿠키는 "주차 티켓"이고, 세션은 주차장에 주차된 "실제 차량"입니다. 클라이언트는 쿠키(티켓)만 갖고 있고, 실제 정보(차량)는 서버(주차장)에 있습니다.

동작 흐름을 보면, 사용자가 로그인하면 서버가 세션을 생성하고 고유한 Session ID를 발급합니다. 이 Session ID를 Set-Cookie 헤더로 클라이언트에 전달하고, 이후 클라이언트는 매 요청마다 쿠키에 Session ID를 실어 보냅니다. 서버는 이 ID로 세션 저장소를 조회해서 사용자 정보를 꺼냅니다. 여기서 중요한 점은, 쿠키 자체는 단순한 키-값 저장소일 뿐이고 세션뿐 아니라 다른 목적(장바구니, 추적 등)으로도 사용됩니다. 반대로 세션은 쿠키가 아닌 URL 파라미터로도 전달할 수 있지만, 보안상 쿠키를 사용하는 것이 표준입니다.

꼬리질문 대비: "쿠키에 HttpOnly, Secure, SameSite 속성이 각각 무엇인가요?" — HttpOnly는 JavaScript에서 쿠키 접근을 차단하여 XSS 공격을 방어합니다. Secure는 HTTPS에서만 쿠키를 전송하게 합니다. SameSite는 크로스 사이트 요청 시 쿠키 전송을 제한하여 CSRF를 방어합니다. Strict/Lax/None 세 가지 옵션이 있습니다.
JWT(JSON Web Token)란 무엇이고, 왜 사용하나요?

JWT는 JSON 형태의 클레임(claim) 정보를 자체적으로 서명(self-contained, signed)하여 안전하게 전달하기 위한 토큰 표준(RFC 7519)입니다. 비유하면, 세션 ID가 "은행 번호표"라면 JWT는 "여권"과 같습니다. 번호표는 은행 시스템에서 조회해야 정보를 알 수 있지만, 여권은 그 자체에 이름, 국적, 유효기간 등 정보가 다 들어있고 위변조 방지 장치(서명)도 있습니다.

JWT를 사용하는 가장 큰 이유는 Stateless 인증이 가능하기 때문입니다. 서버가 세션 저장소를 관리할 필요 없이 토큰의 서명만 검증하면 되므로, 수평 확장(Scale-out)에 매우 유리합니다. 특히 MSA 환경에서 인증 서버가 발급한 토큰을 각 마이크로서비스가 독립적으로 검증할 수 있어, 서비스 간 세션 공유 문제가 사라집니다. 또한 모바일 앱이나 SPA처럼 쿠키 사용이 제한되는 환경에서 Authorization: Bearer <token> 헤더로 간편하게 전달할 수 있다는 장점이 있습니다.

JWT의 구조(Header, Payload, Signature)를 설명해주세요.

JWT는 점(.)으로 구분된 세 부분, Header.Payload.Signature로 구성됩니다.

eyJhbGciOi....eyJzdWIiOi....SflKxwRJSM...
Header
알고리즘 + 토큰 타입
Payload
클레임 (사용자 정보)
Signature
위변조 검증용 서명

Header에는 서명 알고리즘(HS256, RS256 등)과 토큰 타입(JWT)이 들어갑니다. Payload에는 실제 데이터인 클레임(claim)이 담기는데, 표준 클레임으로 sub(subject), exp(만료시간), iat(발급시간) 등이 있고 커스텀 클레임(userId, role 등)도 넣을 수 있습니다. 여기서 중요한 점은, Payload는 Base64Url 인코딩된 것이지 암호화된 것이 아니므로 누구나 디코딩해서 내용을 볼 수 있습니다. 따라서 비밀번호나 민감한 개인정보는 절대 넣으면 안 됩니다.

Signature는 Header와 Payload를 합친 문자열을 비밀키(Secret Key)로 서명한 값입니다. 서버는 토큰을 받으면 같은 방식으로 서명을 다시 계산해서, 토큰에 포함된 Signature와 비교합니다. 하나라도 변조되었으면 서명이 불일치하여 토큰이 거부됩니다. 이것이 JWT의 무결성(Integrity)을 보장하는 핵심 메커니즘입니다.

JWT 서명 알고리즘 HS256과 RS256의 차이점은 무엇인가요?

HS256(HMAC-SHA256)은 대칭키 알고리즘으로, 토큰 발급(서명)과 검증에 동일한 비밀키(Secret Key)를 사용합니다. 반면 RS256(RSA-SHA256)은 비대칭키 알고리즘으로, 발급 시에는 개인키(Private Key)로 서명하고 검증 시에는 공개키(Public Key)로 검증합니다.

비교 항목HS256 (대칭키)RS256 (비대칭키)
키 관리하나의 Secret Key 공유Private/Public Key 쌍
발급과 검증같은 키로 둘 다 수행Private Key로 발급, Public Key로 검증
성능빠름상대적으로 느림
MSA 적합성모든 서비스가 Secret 알아야 함Public Key만 배포하면 됨
대표 사용처단일 서버, 내부 서비스MSA, OAuth, OIDC

실무에서 MSA 환경이라면 RS256이 훨씬 적합합니다. 인증 서버만 Private Key를 보유하고 토큰을 발급하면, 나머지 마이크로서비스들은 Public Key만 갖고도 토큰을 검증할 수 있습니다. HS256처럼 모든 서비스에 Secret Key를 배포할 필요가 없어 키 유출 리스크가 훨씬 줄어듭니다. 반면 단일 서버 환경에서는 HS256이 간단하고 성능도 좋으므로 적절한 선택입니다.

Access Token과 Refresh Token은 왜 분리하나요?

비유하면, Access Token은 건물의 "임시 출입증"이고 Refresh Token은 "출입증 재발급 카드"입니다. 임시 출입증은 유효기간이 짧아(보통 15분~1시간) 분실해도 피해가 제한적이고, 재발급 카드는 보안이 더 강하게 관리되며 유효기간이 깁니다(7일~30일).

클라이언트 → Access Token으로 API 요청 → 리소스 서버
Access Token 만료 시 ↓
클라이언트 → Refresh Token 전송 → 인증 서버 → 새 Access Token 발급

분리하는 핵심 이유는 보안과 사용자 경험의 균형입니다. Access Token의 유효기간을 짧게 두면 탈취되어도 피해 시간이 제한됩니다. 하지만 유효기간이 너무 짧으면 사용자가 계속 재로그인해야 하므로 UX가 나빠집니다. 이때 Refresh Token이 백그라운드에서 자동으로 새 Access Token을 발급받아 주는 역할을 합니다. Refresh Token은 서버 DB나 Redis에 저장하여, 필요시 즉시 무효화(revoke)할 수 있게 관리하는 것이 일반적입니다.

꼬리질문 대비: "Refresh Token도 탈취되면 어떻게 하나요?" — 대표적인 방어 전략으로 Refresh Token Rotation이 있습니다. Refresh Token을 사용할 때마다 기존 토큰을 폐기하고 새 Refresh Token을 함께 발급합니다. 만약 이미 사용된(폐기된) Refresh Token이 다시 사용되면 모든 토큰을 즉시 무효화하여 탈취를 탐지합니다.
JWT의 단점과 한계는 무엇인가요?

JWT는 만능이 아니며, 면접에서 장점만 말하면 깊이가 부족해 보입니다. 가장 큰 한계는 발급 후 즉시 무효화가 어렵다는 점입니다. 세션 방식은 서버에서 세션을 삭제하면 끝이지만, JWT는 서명이 유효한 한 만료 시간까지 살아 있습니다. 사용자가 로그아웃했거나 비밀번호를 변경했더라도 기존 토큰이 유효할 수 있습니다. 이를 해결하려면 Redis에 블랙리스트를 두거나, 토큰 버전 번호를 DB에 관리하는 등 결국 서버 측 상태가 필요해져서 Stateless라는 장점이 약해집니다.

두 번째 한계는 토큰 크기입니다. Session ID는 수십 바이트에 불과하지만, JWT는 Payload에 정보를 담기 때문에 수백 바이트~수 KB까지 커질 수 있습니다. 매 API 요청마다 이 토큰이 전송되므로 네트워크 오버헤드가 생깁니다. 세 번째로, Payload가 암호화가 아닌 인코딩이므로 민감 정보를 넣으면 노출됩니다. 이런 한계들을 인지한 위에서 상황에 맞게 세션/토큰/하이브리드를 선택하는 것이 좋은 설계 판단입니다.

JWT를 어디에 저장해야 하나요? (localStorage vs 쿠키)

이 질문은 면접에서 보안 인식을 테스트할 때 자주 나옵니다. 결론부터 말하면, HttpOnly 쿠키에 저장하는 것이 가장 안전한 방식으로 권장됩니다.

저장 위치XSS 방어CSRF 방어비고
localStorage취약안전JavaScript로 직접 접근 가능
일반 쿠키취약취약양쪽 다 취약
HttpOnly 쿠키안전대비 필요+ SameSite + CSRF 토큰

localStorage에 저장하면 JavaScript로 자유롭게 접근할 수 있어 편리하지만, XSS 공격으로 악성 스크립트가 실행되면 토큰이 그대로 탈취됩니다. 반면 HttpOnly 쿠키는 JavaScript에서 접근 자체가 불가능하므로 XSS에 안전합니다. 다만 쿠키는 자동으로 전송되기 때문에 CSRF 공격에는 별도 대비(SameSite 속성, CSRF 토큰)가 필요합니다. 실무에서는 Access Token을 HttpOnly 쿠키에, Refresh Token도 HttpOnly + Secure + SameSite=Strict 쿠키에 저장하는 패턴이 일반적입니다.

OAuth 2.0이 무엇이고, 어떤 흐름(Flow)으로 동작하나요?

OAuth 2.0은 인가(Authorization) 프레임워크로, 사용자가 자신의 비밀번호를 제3자 앱에 넘기지 않고도 제3자 앱이 사용자의 리소스에 접근할 수 있게 해줍니다. 비유하면, 호텔에 묵을 때 프론트 데스크에서 "이 손님의 짐을 객실로 옮겨주세요"라고 벨보이에게 제한된 권한을 위임하는 것과 같습니다. 마스터키를 주는 것이 아니라 특정 객실에만 접근 가능한 임시 카드를 주는 것입니다.

가장 널리 사용되는 Authorization Code Flow의 흐름은 다음과 같습니다.

① 사용자 → 클라이언트 앱: "구글로 로그인" 클릭
② 클라이언트 → 인가 서버(Google): 사용자를 Google 로그인 페이지로 리다이렉트
③ 사용자 → 인가 서버: Google에서 직접 로그인 + 권한 동의
④ 인가 서버 → 클라이언트: Authorization Code를 redirect URI로 전달
⑤ 클라이언트 서버 → 인가 서버: Code + Client Secret으로 Access Token 교환 (서버-서버 통신)
⑥ 클라이언트 서버 → 리소스 서버: Access Token으로 사용자 정보 요청

핵심 포인트는 ⑤단계에서 Authorization Code를 Access Token으로 교환하는 과정이 서버 간 통신으로 이루어진다는 것입니다. Access Token이 브라우저에 노출되지 않아 보안이 강합니다. SPA에서는 PKCE(Proof Key for Code Exchange) 확장을 사용하여 Code 가로채기 공격을 방어합니다.

OAuth 2.0과 OIDC(OpenID Connect)의 차이는 무엇인가요?

OAuth 2.0은 원래 인가(Authorization)를 위한 프레임워크입니다. "이 앱이 내 Google Drive에 접근해도 됩니다"처럼 권한 위임이 목적이지, "이 사람이 누구인지" 확인하는 인증이 주 목적이 아닙니다. 그런데 실무에서 많은 개발자들이 OAuth 2.0을 인증 용도로 사용하기 시작했고, 이 과정에서 표준화되지 않은 방식들이 난립했습니다.

OIDC(OpenID Connect)는 OAuth 2.0 위에 인증 계층을 표준화한 프로토콜입니다. 가장 큰 차이는 OIDC가 Access Token 외에 ID Token(JWT 형식)을 추가로 발급한다는 점입니다. ID Token에는 사용자 식별 정보(sub, email, name 등)가 표준화된 클레임으로 담겨 있어, "이 사용자가 누구인지"를 안전하고 일관된 방식으로 확인할 수 있습니다. 요약하면, OAuth 2.0은 "이 앱이 무엇을 할 수 있는가"를, OIDC는 "이 사용자가 누구인가"를 다룹니다. 소셜 로그인을 구현할 때는 실질적으로 OIDC를 사용하는 것입니다.

MSA 환경에서 인증/인가는 어떻게 설계하나요?

MSA 환경에서 가장 일반적인 패턴은 API Gateway + 중앙 인증 서버 구조입니다. 비유하면, 대형 쇼핑몰에서 정문 경비실(Gateway)에서 한 번 신분 확인을 하고 출입증(JWT)을 발급하면, 몰 내 각 매장(서비스)에서는 그 출입증만 보고 입장을 허용하는 구조입니다.

클라이언트 (SPA / Mobile) ↓ JWT API Gateway (인증 검증 + 라우팅) ↓ 검증된 요청 전달
주문 서비스 상품 서비스 결제 서비스
인증 서버 (토큰 발급/갱신) — RS256 Private Key 보유

구체적으로, 인증 서버가 로그인을 처리하고 JWT(RS256)를 발급합니다. API Gateway는 모든 요청을 받아 JWT 서명을 Public Key로 검증하고, 검증이 통과하면 각 마이크로서비스로 요청을 라우팅합니다. 각 서비스는 JWT의 Payload에서 userId, role 등을 꺼내 인가 처리를 합니다. 이 방식의 장점은 각 서비스가 인증 서버를 매번 호출할 필요 없이 토큰 자체만으로 검증이 가능하다는 것입니다. 서비스 간 통신이 필요한 경우에는 내부 전용 서비스 토큰을 별도로 관리하기도 합니다.

JWT가 탈취당했을 때 어떻게 대응하나요?

JWT는 기본적으로 Stateless이므로 발급된 토큰을 "취소"하는 내장 메커니즘이 없습니다. 따라서 다중 방어 전략이 필요합니다.

첫 번째로 Access Token 만료 시간을 짧게(15분 이하) 설정하여 탈취되어도 피해 시간을 최소화합니다. 두 번째로 토큰 블랙리스트를 Redis에 관리합니다. 로그아웃하거나 비밀번호를 변경하면 해당 토큰의 jti(JWT ID)를 Redis에 추가하고, 검증 시 블랙리스트 여부를 확인합니다. TTL을 토큰 만료 시간과 맞추면 자동으로 정리됩니다. 세 번째로 Refresh Token Rotation으로, Refresh Token 사용 시마다 새로 발급하고 이전 것을 폐기합니다. 이미 폐기된 토큰이 사용되면 해당 사용자의 모든 토큰을 강제 무효화합니다.

추가로, 실무에서는 토큰에 사용자의 IP나 User-Agent 해시를 클레임에 포함시켜, 다른 환경에서 사용되면 거부하는 디바이스 바인딩 전략도 고려합니다. 물론 이는 IP가 바뀌는 모바일 환경에서는 주의가 필요합니다.

"JWT는 암호화된 건가요?" 라고 물어보면 어떻게 답하나요?

아닙니다. 많은 분들이 혼동하시는데, 일반적인 JWT(JWS, JSON Web Signature)는 서명(Sign)은 하지만 암호화(Encrypt)는 하지 않습니다. Header와 Payload는 단순히 Base64Url 인코딩된 것이므로, 누구나 디코딩하여 내용을 읽을 수 있습니다. jwt.io 같은 사이트에서 토큰을 붙여넣으면 바로 내용이 보이는 것이 그 증거입니다.

서명이 보장하는 것은 무결성(Integrity)입니다. 즉 "이 토큰의 내용이 변조되지 않았다"는 것을 보장하지, "이 내용을 아무도 못 읽는다"는 기밀성(Confidentiality)은 보장하지 않습니다. 만약 Payload의 기밀성까지 필요하다면 JWE(JSON Web Encryption)를 사용해야 합니다. JWE는 Payload를 실제로 암호화하여 복호화 키 없이는 내용을 읽을 수 없게 합니다. 하지만 대부분의 실무에서는 JWS + HTTPS 조합으로 충분하고, Payload에 민감 정보를 넣지 않는 것이 올바른 설계입니다.

세션 클러스터링 문제와 JWT의 관계를 설명해주세요.

서버가 한 대일 때는 세션 방식에 문제가 없지만, 트래픽이 늘어 서버를 여러 대로 스케일 아웃하면 문제가 발생합니다. 사용자 A가 서버 1에서 로그인하여 세션이 서버 1 메모리에 저장되었는데, 다음 요청이 로드밸런서에 의해 서버 2로 라우팅되면 세션을 찾지 못해 로그인이 풀리게 됩니다.

이 문제의 전통적 해결책으로는 Sticky Session(특정 사용자를 항상 같은 서버로 라우팅), Session Replication(모든 서버 간 세션 복제), 외부 세션 저장소(Redis 등)가 있습니다. 하지만 Sticky Session은 서버 장애 시 세션 유실 위험이 있고, Session Replication은 네트워크 부하와 동기화 지연 문제가 있으며, Redis는 추가 인프라 비용이 발생합니다.

JWT는 이 문제를 근본적으로 해결합니다. 서버에 상태를 저장하지 않기 때문에 어떤 서버가 요청을 받든 토큰의 서명만 검증하면 됩니다. 서버 간 세션 공유 자체가 필요 없어지는 것입니다. 이것이 JWT가 MSA와 클라우드 환경에서 선호되는 가장 실무적인 이유입니다.

CSRF와 XSS 공격이 인증에 미치는 영향을 설명해주세요.

CSRF(Cross-Site Request Forgery)는 사용자가 로그인된 상태에서, 공격자가 만든 페이지를 방문하면 사용자의 의도 없이 인증된 요청이 전송되는 공격입니다. 예를 들어 은행 사이트에 로그인한 상태에서 악성 링크를 클릭하면, 브라우저가 자동으로 쿠키를 보내기 때문에 공격자가 의도한 송금 요청이 정상 인증된 요청으로 처리될 수 있습니다. 쿠키 기반 세션 인증이 이 공격에 특히 취약합니다. 방어 방법으로는 SameSite 쿠키, CSRF 토큰, Referer/Origin 헤더 검증이 있습니다.

XSS(Cross-Site Scripting)는 공격자가 악성 JavaScript를 웹 페이지에 삽입하는 공격입니다. 이 스크립트가 실행되면 localStorage의 JWT를 탈취하거나, DOM을 조작하여 사용자의 입력을 가로챌 수 있습니다. JWT를 localStorage에 저장하면 XSS에 직접 노출되고, HttpOnly 쿠키에 저장하면 JavaScript 접근이 차단되므로 XSS로부터 토큰을 보호할 수 있습니다. 결국 쿠키는 CSRF에 취약하고, localStorage는 XSS에 취약하므로, HttpOnly 쿠키 + SameSite + CSRF 토큰 조합이 가장 견고한 방어 전략입니다.

Spring Security + JWT 인증 필터를 간단히 구현해보세요.

면접에서 화이트보드에 쓸 수 있는 수준의 핵심 코드입니다. JWT 인증 필터는 OncePerRequestFilter를 상속받아, 매 요청마다 Authorization 헤더에서 토큰을 추출하고 검증하는 역할을 합니다.

public class JwtAuthFilter extends OncePerRequestFilter {

    private final JwtTokenProvider tokenProvider;

    @Override
    protected void doFilterInternal(HttpServletRequest request,
            HttpServletResponse response, FilterChain chain)
            throws ServletException, IOException {

        // 1. 헤더에서 토큰 추출
        String token = resolveToken(request);

        // 2. 토큰 유효성 검증
        if (token != null && tokenProvider.validateToken(token)) {
            // 3. 토큰에서 인증 정보 추출 → SecurityContext에 저장
            Authentication auth = tokenProvider.getAuthentication(token);
            SecurityContextHolder.getContext().setAuthentication(auth);
        }

        // 4. 다음 필터로 전달
        chain.doFilter(request, response);
    }

    private String resolveToken(HttpServletRequest request) {
        String bearer = request.getHeader("Authorization");
        if (bearer != null && bearer.startsWith("Bearer ")) {
            return bearer.substring(7);
        }
        return null;
    }
}

동작 흐름은 이렇습니다. 요청이 들어오면 Authorization: Bearer xxx 헤더에서 토큰을 추출합니다. validateToken()에서 서명 검증과 만료 시간을 확인하고, 유효하면 토큰의 Payload에서 사용자 정보를 꺼내 SecurityContext에 저장합니다. 이후 @PreAuthorize나 hasRole() 등의 인가 로직이 이 SecurityContext를 기반으로 동작합니다. 이 필터는 Security 설정에서 UsernamePasswordAuthenticationFilter 앞에 등록합니다.

JWT 토큰 발급과 검증 로직의 핵심 코드를 보여주세요.

Java에서 io.jsonwebtoken (jjwt) 라이브러리를 사용한 핵심 코드입니다.

@Component
public class JwtTokenProvider {

    @Value("${jwt.secret}")
    private String secretKey;
    private final long ACCESS_TOKEN_VALIDITY = 15 * 60 * 1000; // 15분

    // ---- 토큰 발급 ----
    public String createToken(String userId, String role) {
        Claims claims = Jwts.claims().setSubject(userId);
        claims.put("role", role);

        Date now = new Date();
        Date expiry = new Date(now.getTime() + ACCESS_TOKEN_VALIDITY);

        return Jwts.builder()
                .setClaims(claims)
                .setIssuedAt(now)
                .setExpiration(expiry)
                .signWith(SignatureAlgorithm.HS256, secretKey)
                .compact();
    }

    // ---- 토큰 검증 ----
    public boolean validateToken(String token) {
        try {
            Jwts.parser().setSigningKey(secretKey)
                .parseClaimsJws(token);
            return true;
        } catch (ExpiredJwtException e) {
            // 토큰 만료
        } catch (JwtException e) {
            // 서명 불일치 등
        }
        return false;
    }

    // ---- 인증 객체 생성 ----
    public Authentication getAuthentication(String token) {
        Claims claims = Jwts.parser().setSigningKey(secretKey)
                .parseClaimsJws(token).getBody();
        String userId = claims.getSubject();
        String role = claims.get("role", String.class);

        UserDetails userDetails = new User(userId, "",
                List.of(new SimpleGrantedAuthority("ROLE_" + role)));
        return new UsernamePasswordAuthenticationToken(
                userDetails, "", userDetails.getAuthorities());
    }
}

면접에서 이 코드를 설명할 때 강조할 포인트는, signWith()로 서명하고 parseClaimsJws()로 서명을 검증하는 대칭 구조입니다. 실무에서는 Secret Key를 환경변수나 Vault에서 관리하고, RS256으로 전환할 경우 KeyPair를 사용합니다. 또한 jjwt 0.12+ 버전에서는 API가 변경되어 Jwts.builder().subject(userId).signWith(key) 형식으로 사용합니다.

[설계] "강제 로그아웃" 기능을 JWT 기반 시스템에서 어떻게 구현하겠습니까?

이 질문은 JWT의 Stateless 특성과 현실적 요구사항의 충돌을 어떻게 해결하는가를 테스트합니다. 가장 실용적인 접근 방식 세 가지를 비교해보겠습니다.

전략원리장점단점
토큰 블랙리스트 (Redis) 로그아웃된 토큰의 jti를 Redis에 저장, 검증 시 체크 즉시 무효화, 구현 간단 매 요청마다 Redis 조회, 순수 Stateless 아님
토큰 버전 관리 User 테이블에 tokenVersion 필드, 토큰 claim에 포함 사용자 단위 전체 무효화 용이 매 요청마다 DB/캐시 조회
짧은 만료 + Refresh Token 삭제 Access Token을 5~15분, 서버 측 Refresh Token 삭제 Stateless 유지, 간단 최대 15분간 기존 토큰 유효

실무에서는 대부분 Redis 블랙리스트 + 짧은 만료 시간을 조합합니다. 비밀번호 변경, 계정 탈취 의심 등 긴급 상황에서는 블랙리스트로 즉시 무효화하고, 일반 로그아웃에서는 Refresh Token만 삭제하여 자연 만료를 기다리는 식입니다. Redis에 저장할 때 TTL을 토큰 남은 만료 시간과 동일하게 설정하면 자동 정리되므로 메모리 부담도 크지 않습니다. 면접에서는 이렇게 "상황에 따라 전략을 나눈다"는 접근을 보여주면 좋습니다.
