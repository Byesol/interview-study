Replication & Sharding

DB 확장 전략 — 복제, 샤딩, 파티셔닝, 읽기/쓰기 분리

🔄 Part 1. Replication — 복제 구조, 동기/비동기, Lag, HA
DB Replication이란 무엇이고, 왜 필요한가요?

Replication(복제)은 하나의 DB 서버(Primary/Master)의 데이터를 하나 이상의 다른 서버(Replica/Slave)에 자동으로 복사하는 기술입니다.

필요한 이유:

목적설명
고가용성 (HA) Primary 장애 시 Replica가 Primary로 승격되어 서비스 중단 최소화
읽기 성능 확장 SELECT 쿼리를 여러 Replica에 분산시켜 부하 감소
데이터 백업 Primary에 영향 없이 Replica에서 백업 수행
지리적 분산 사용자와 가까운 지역에 Replica를 두어 지연 감소
핵심: Replication은 수평 확장(Scale-Out) 중 읽기(Read) 트래픽 분산에 초점을 맞춥니다. 쓰기(Write)는 여전히 하나의 Primary에서만 처리합니다 (단방향 복제 기준).
Master-Slave(Primary-Replica) 구조를 설명해주세요. MySQL은 내부적으로 어떻게 복제하나요?

MySQL Replication은 Binary Log(binlog) 기반으로 동작합니다.

Primary                          Replica
┌─────────────────┐              ┌─────────────────┐
│  쓰기 요청 처리  │──binlog 전송→ │  IO Thread      │
│  Binary Log 기록 │              │  (binlog 수신)  │
└─────────────────┘              │  Relay Log 저장  │
                                 │  SQL Thread     │
                                 │  (Relay Log 실행)│
                                 └─────────────────┘

복제 흐름:

① Primary에서 데이터 변경(INSERT/UPDATE/DELETE)이 발생하면 Binary Log에 기록합니다.

② Replica의 IO Thread가 Primary에 접속하여 Binary Log를 가져와 Relay Log에 저장합니다.

③ Replica의 SQL Thread가 Relay Log를 읽어 순서대로 실행하여 데이터를 동기화합니다.

Binary Log 포맷 3가지:

포맷저장 내용장점단점
Statement SQL 문장 그대로 저장 용량 작음 비결정적 함수(NOW(), RAND()) 불일치 가능
Row 변경된 행 데이터 정확한 복제 보장 binlog 크기 큼
Mixed 기본 Statement, 필요시 Row 균형 예측이 어려울 수 있음
MySQL 5.7.7+ 기본값은 Row 포맷입니다. 정확성이 보장되므로 권장됩니다.
동기 복제 vs 비동기 복제 vs 반동기 복제를 비교해주세요.

비동기 복제 (Asynchronous Replication) — MySQL 기본값

Primary는 binlog를 기록하고 즉시 클라이언트에 커밋 완료를 응답합니다. Replica가 실제로 반영했는지 확인하지 않습니다. Primary 장애 시 아직 Replica에 전달되지 않은 트랜잭션이 유실될 수 있습니다.

동기 복제 (Synchronous Replication)

Primary는 모든 Replica가 데이터를 적용했음을 확인한 뒤 커밋 응답을 보냅니다. 데이터 손실이 없지만 모든 Replica의 응답을 기다려야 하므로 쓰기 지연이 매우 큽니다. 네트워크 장애 시 Primary까지 멈출 수 있습니다.

반동기 복제 (Semi-Synchronous Replication) — 실무에서 많이 사용

Primary는 최소 하나의 Replica가 binlog를 수신(Relay Log에 기록)했음을 확인한 뒤 커밋 응답합니다. SQL Thread 실행까지 기다리지 않습니다. 비동기보다 안전하고 동기보다 빠릅니다. 타임아웃 발생 시 자동으로 비동기로 전환됩니다.

방식성능데이터 안전성실무 사용
비동기 높음 낮음 (유실 가능) 읽기 분산 목적 Replica
반동기 중간 중간 (최소 1개 확인) 금융, 중요 데이터 서비스
동기 낮음 높음 (완전 보장) MySQL Group Replication, Galera
꼬리 질문: MySQL Group Replication이란?
→ Paxos 기반 분산 합의 프로토콜로 여러 노드 간 동기 복제를 구현합니다. 최소 3대 이상이 필요하며, 쓰기 시 과반수(Quorum) 확인을 거칩니다. AWS Aurora MySQL도 유사한 방식입니다.
복제 지연(Replication Lag) 문제와 어떻게 대응하나요?

복제 지연은 Primary에서 커밋된 변경이 Replica에 아직 반영되지 않은 시간 차이입니다. 비동기 복제의 구조적 한계입니다.

SHOW REPLICA STATUS\G의 Seconds_Behind_Source 값으로 확인합니다.

지연 발생 원인:

  • Primary의 대용량 배치 쓰기 (단일 SQL Thread가 순차 처리)
  • Replica 서버 스펙 부족 (CPU, I/O)
  • 네트워크 대역폭 부족
  • Replica에서 대용량 읽기 쿼리와 SQL Thread 간 Lock 경합

대응 방법:

방법설명
병렬 복제 (MTS) MySQL 5.6+에서 Multithreaded Slave(MTS) 설정으로 여러 SQL Thread 병렬 실행. replica_parallel_workers 설정
Replica 스펙 향상 I/O 병목이면 SSD, CPU 병목이면 코어 증설
읽기 일관성 보장 쓰기 직후 읽기는 Primary에서 (Session stickiness). 또는 반동기 복제 도입
Read-Your-Writes 사용자가 방금 쓴 데이터를 읽을 때는 Primary로 라우팅
실무 주의: "방금 저장했는데 조회가 안 된다"는 버그의 상당수가 Replication Lag 때문입니다. 쓰기 직후 조회는 Primary에서 읽거나, 클라이언트 캐시로 보완해야 합니다.
읽기/쓰기 분리(Read/Write Splitting)란? Spring에서 어떻게 구현하나요?

읽기/쓰기 분리는 쓰기 쿼리(INSERT/UPDATE/DELETE)는 Primary로, 읽기 쿼리(SELECT)는 Replica로 라우팅하여 Primary 부하를 줄이는 패턴입니다.

Spring에서 AbstractRoutingDataSource 활용:

@Configuration
public class DataSourceConfig {

    @Bean
    public DataSource routingDataSource(
            @Qualifier("primaryDataSource") DataSource primary,
            @Qualifier("replicaDataSource") DataSource replica) {

        RoutingDataSource routing = new RoutingDataSource();
        Map<Object, Object> sources = new HashMap<>();
        sources.put("primary", primary);
        sources.put("replica", replica);
        routing.setTargetDataSources(sources);
        routing.setDefaultTargetDataSource(primary);
        return routing;
    }
}

public class RoutingDataSource extends AbstractRoutingDataSource {
    @Override
    protected Object determineCurrentLookupKey() {
        // @Transactional(readOnly=true)이면 replica, 아니면 primary
        return TransactionSynchronizationManager.isCurrentTransactionReadOnly()
               ? "replica" : "primary";
    }
}

위 구조에서 @Transactional(readOnly = true)가 붙은 메서드는 자동으로 Replica DataSource를 사용합니다.

주의: LazyConnectionDataSourceProxy를 함께 사용해야 합니다. 이것 없이는 트랜잭션 시작 시점에 DataSource가 결정되어, readOnly 설정이 반영되기 전에 커넥션을 획득해버립니다. 
@Bean
public DataSource dataSource() {
    // LazyConnection으로 감싸야 readOnly 판단 후 라우팅됨
    return new LazyConnectionDataSourceProxy(routingDataSource());
}
꼬리 질문: 여러 Replica 중 하나를 선택하는 로드밸런싱은?
→ RoutingDataSource에서 Replica 목록을 List로 관리하고, Round-Robin 또는 랜덤으로 선택합니다. 또는 ProxySQL, MaxScale 같은 DB Proxy를 앞에 두면 애플리케이션 코드 없이 라우팅할 수 있습니다.
Failover와 고가용성(HA)을 어떻게 구성하나요?

Failover는 Primary 장애 시 Replica 중 하나를 자동으로 새 Primary로 승격시켜 서비스를 계속하는 과정입니다.

자동 Failover 구성 방법:

솔루션설명사용 환경
MySQL InnoDB Cluster Group Replication + MySQL Router + MySQL Shell. 공식 HA 솔루션 온프레미스, 자체 구성
MHA (Master HA) Primary 장애 감지 후 가장 최신 Replica를 승격. 수십 초 내 Failover 온프레미스 레거시
AWS RDS Multi-AZ 다른 가용 영역에 동기 복제 Standby. 장애 시 60~120초 내 자동 전환 AWS 클라우드
AWS Aurora 스토리지 레이어 공유 방식. 30초 내 Failover. Replica가 바로 Primary 가능 AWS 클라우드 고성능
Orchestrator 오픈소스 Failover/토폴로지 관리 도구. GitHub이 오픈소스로 공개 대규모 온프레미스

Failover 시 주의사항:

  • Split-Brain: 두 노드가 동시에 Primary로 동작하는 상황 → 과반수(Quorum) 기반으로 방지
  • 애플리케이션이 새 Primary 주소를 알아야 함 → VIP(Virtual IP) 또는 DNS TTL 단축으로 해결
  • 비동기 복제에서 미전달 트랜잭션 유실 가능 → GTID 기반 복제로 추적
GTID(Global Transaction ID): 각 트랜잭션에 전역 고유 ID를 부여합니다. Failover 후 새 Replica가 어디까지 받았는지 정확히 파악하여 갭 없이 동기화할 수 있습니다.
🔀 Part 2. Sharding — 수평/수직 분할, Shard Key, Consistent Hashing
Sharding이란 무엇이고, Replication과 어떻게 다른가요?

Sharding은 하나의 데이터베이스/테이블을 여러 서버에 나누어 저장하는 수평 파티셔닝 기법입니다. 각 서버(Shard)는 전체 데이터의 일부분만 보유합니다.

구분ReplicationSharding
목적 가용성 향상, 읽기 분산 데이터 분산, 쓰기·저장 용량 확장
데이터 각 노드가 전체 데이터 보유 (복사본) 각 노드가 데이터의 일부만 보유
쓰기 확장 불가 (Primary 단일 지점) 가능 (각 Shard에 쓰기 분산)
저장 용량 확장 어려움 (전체 복사) Shard 추가로 선형 확장
복잡도 상대적으로 낮음 높음 (라우팅, 트랜잭션 등)
실무에서는 Replication + Sharding을 함께 사용합니다. 각 Shard 안에서 Primary-Replica 구성으로 고가용성과 읽기 분산을 동시에 달성합니다.
수평 분할(Horizontal Sharding) vs 수직 분할(Vertical Sharding)이란?

수평 분할 (Horizontal Sharding)

같은 스키마를 가진 테이블을 행(Row) 기준으로 여러 DB 서버에 나눕니다. 가장 일반적인 샤딩 방식입니다.

users 테이블
Shard 0: user_id % 3 == 0 → [0, 3, 6, 9, ...]
Shard 1: user_id % 3 == 1 → [1, 4, 7, 10, ...]
Shard 2: user_id % 3 == 2 → [2, 5, 8, 11, ...]

장점: 각 서버의 데이터 양이 균일하게 분산. 단점: Shard 간 JOIN 불가, 트랜잭션 복잡.

수직 분할 (Vertical Sharding)

테이블 또는 기능 단위로 다른 DB 서버에 나눕니다. 마이크로서비스의 DB 분리와 유사합니다.

DB 서버 A: users, profiles, follows (사용자 관련)
DB 서버 B: posts, comments, likes (컨텐츠 관련)
DB 서버 C: orders, payments (주문/결제 관련)

장점: 도메인 독립성. 단점: 서비스 간 JOIN 불가, 데이터 증가 시 한계.

일반적 전략: 먼저 수직 분할(도메인 분리)로 성장하고, 특정 도메인의 데이터가 커지면 수평 분할(샤딩) 적용. 수평 분할만으로 시작하면 관리가 복잡해집니다.
Shard Key는 어떤 기준으로 선택해야 하나요?

Shard Key는 데이터를 어느 Shard에 저장할지 결정하는 기준 컬럼입니다. 잘못 선택하면 특정 Shard에 부하가 몰리는 Hot Shard 문제가 발생합니다.

좋은 Shard Key의 조건:

조건설명나쁜 예시
높은 카디널리티 값의 종류가 다양해야 균등 분산 가능 성별(M/F), 상태코드(3~5가지)
균등 분포 특정 값에 데이터가 쏠리지 않아야 함 생성일(특정 날짜에 집중), 국가코드
쿼리 패턴 일치 대부분의 쿼리가 Shard Key로 필터링 Shard Key 없이 전체 Shard 조회
변경 불가 한번 정해지면 변경 시 리밸런싱 필요 이메일(변경 가능), 전화번호

좋은 Shard Key 예시:

  • user_id: 사용자 기반 서비스에서 가장 흔한 선택. 균등 분포, 높은 카디널리티
  • tenant_id: SaaS 서비스에서 고객사 기준 분리
  • UUID v4: 완전 랜덤이라 균등 분산에 유리
Hot Shard 주의: 인플루언서처럼 특정 user_id에 트래픽이 집중되는 경우, 그 user가 속한 Shard만 과부하가 됩니다. 이를 Hot Shard 또는 Hot Spot이라 합니다. 해결책으로 해당 엔티티만 별도 Shard로 분리하거나, 복합 Shard Key를 사용합니다.
Hash Sharding vs Range Sharding을 비교해주세요.

Hash Sharding

Shard Key에 해시 함수를 적용하여 Shard를 결정합니다. shard_id = hash(key) % N

user_id = 12345
shard_id = 12345 % 4 = 1  → Shard 1에 저장

장점: 균등 분산 보장. 단점: 범위 쿼리(2024년 1~3월 데이터) 시 모든 Shard 조회 필요. Shard 수가 바뀌면 거의 모든 데이터를 재분배해야 합니다.

Range Sharding

Shard Key의 값 범위로 Shard를 결정합니다.

user_id 1~1,000,000     → Shard 0
user_id 1,000,001~2,000,000 → Shard 1
user_id 2,000,001~3,000,000 → Shard 2

장점: 범위 쿼리 효율적 (특정 Shard만 조회). 단점: 데이터가 순차적으로 쌓이면 최신 Shard에만 쓰기 집중 (Hot Write).

항목Hash ShardingRange Sharding
데이터 분산균등불균등할 수 있음
범위 쿼리비효율 (전체 Shard)효율적
Shard 추가대규모 재분배 필요새 범위 추가만
Hot Spot낮음시계열 데이터에서 발생
Consistent Hashing이란 무엇이고, 왜 필요한가요?

일반 Hash Sharding에서 서버(Shard)를 추가/제거하면 hash(key) % N의 N이 바뀌어 거의 모든 키의 Shard가 변경됩니다. 이는 대규모 데이터 재분배(Resharding)를 유발합니다.

Consistent Hashing은 이 문제를 해결합니다. 해시 공간을 0~2³²-1의 원형 링(Ring)으로 구성하고, 서버와 데이터 모두 같은 링 위에 배치합니다.

         0
      /     \
 270 ─       ─ 90   ← 해시 링 (0~2³²-1)
      \     /
        180

서버 A: hash("ServerA") = 90 위치
서버 B: hash("ServerB") = 180 위치
서버 C: hash("ServerC") = 270 위치

키 K: hash(K) = 120 → 시계 방향으로 가장 가까운 서버 B에 저장

서버 추가/제거 시:

서버 D를 링의 150 위치에 추가하면, 120~150 구간의 키만 D로 이동합니다. 나머지 대부분의 키는 그대로 유지됩니다. 이론적으로 K/N개 키만 재분배됩니다 (K: 총 키 수, N: 서버 수).

Virtual Node(가상 노드):

서버마다 여러 개의 가상 위치를 링에 배치하여 더 균등한 분산을 달성합니다. 서버 스펙에 따라 가상 노드 수를 조절하여 가중치 부여도 가능합니다.

사용 사례: Amazon DynamoDB, Apache Cassandra, Redis Cluster가 Consistent Hashing 기반입니다. 분산 캐시(Memcached 클러스터)에서도 많이 사용합니다.
샤딩의 문제점들을 설명해주세요. (JOIN, 트랜잭션, 리밸런싱)

① Cross-Shard JOIN 불가

서로 다른 Shard에 있는 데이터는 DB 레벨에서 JOIN할 수 없습니다. 애플리케이션에서 여러 Shard에 쿼리를 보내고 결과를 합쳐야 합니다(Fan-out Query). 성능 저하와 코드 복잡도가 증가합니다.

// 애플리케이션 레벨 Fan-out Join (의사코드)
List<Order> all = new ArrayList<>();
for (Shard shard : allShards) {
    all.addAll(shard.query("SELECT * FROM orders WHERE ..."));
}
// 결과 병합 및 정렬을 애플리케이션에서 처리

② 분산 트랜잭션 어려움

여러 Shard에 걸친 트랜잭션은 ACID를 보장하기 어렵습니다. 2PC(Two-Phase Commit)를 쓸 수 있지만 성능이 나쁘고 구현이 복잡합니다. SAGA 패턴이나 Eventual Consistency로 타협하는 경우가 많습니다.

③ 데이터 리밸런싱 (Resharding)

Shard를 추가하거나 줄일 때 기존 데이터를 재분배해야 합니다. 운영 중인 시스템에서 리밸런싱은 서비스 중단 없이 하기 매우 어렵습니다. Consistent Hashing, 더블 라이팅 전략 등으로 완화합니다.

④ 전역 고유 ID 생성 어려움

각 Shard가 독립적인 AUTO_INCREMENT를 쓰면 ID가 충돌합니다. Twitter Snowflake(타임스탬프 + 머신ID + 시퀀스), UUID, DB Sequence 서버 별도 운용 등의 해결책이 필요합니다.

⑤ 운영 복잡도 증가

배포, 스키마 마이그레이션, 모니터링, 백업 모두 Shard 수만큼 작업이 증가합니다.

꼬리 질문: Snowflake ID란?
→ Twitter가 오픈소스로 공개한 분산 환경 ID 생성 알고리즘. 64비트로 구성: 41비트(밀리초 타임스탬프) + 10비트(데이터센터+머신ID) + 12비트(시퀀스). 시간 순서를 보장하면서 충돌 없이 분산 생성 가능합니다.
실무에서의 샤딩 사례를 알고 계신 것이 있나요?

카카오 / 네이버 — user_id 기반 Hash Sharding

메시지, 피드, 친구 데이터를 user_id로 해싱하여 Shard에 분산합니다. 동일 사용자의 데이터는 같은 Shard에 모여 Single-Shard 트랜잭션을 유지합니다.

Pinterest — 복합 Shard Key

shard_id(16bit) + entity_type(10bit) + local_id(38bit)를 합친 64비트 ID를 사용합니다. ID만 보고 어느 Shard에 있는지 즉시 파악 가능합니다.

GitHub — Vitess

MySQL 위에서 투명한 샤딩을 제공하는 미들웨어인 Vitess를 사용합니다. 애플리케이션은 샤딩을 의식하지 않고 일반 MySQL처럼 쿼리를 보내면 Vitess가 라우팅합니다. YouTube도 동일 솔루션 사용.

MongoDB — Sharded Cluster

MongoDB는 내장 샤딩을 지원합니다. Config Server(메타데이터) + Router(mongos, 라우팅) + Shard(데이터 저장) 구조. Chunk 단위로 데이터를 분산하고 자동 Balancer가 균등 유지합니다.

샤딩 도입 판단 기준: 단일 MySQL 인스턴스가 100~200GB를 초과하거나, 쓰기 처리량이 Primary 한계에 도달했을 때. 그 전까지는 Replication + 인덱스 최적화 + 캐시(Redis)로 버티는 것이 현명합니다.
📦 Part 3. Partitioning — 파티션 종류, Pruning, 샤딩과 차이
Partitioning과 Sharding의 차이는 무엇인가요?

두 개념 모두 데이터를 나눈다는 공통점이 있지만 핵심 차이가 있습니다.

구분PartitioningSharding
위치 하나의 DB 서버 안에서 분리 여러 DB 서버에 분리
서버 단일 DB 인스턴스 복수 DB 인스턴스
목적 쿼리 성능 향상 (파티션 프루닝), 관리 효율화 저장 용량·쓰기 부하 분산 (스케일아웃)
JOIN DB 내 파티션 간 JOIN 가능 Shard 간 JOIN 불가 (애플리케이션 처리)
투명성 애플리케이션이 파티션 구조 몰라도 됨 라우팅 로직이 애플리케이션 또는 미들웨어에 필요
용량 한계 단일 서버 한계 있음 서버 추가로 무제한 확장 이론상 가능
비유: Partitioning = 하나의 큰 창고를 구역 A, B, C로 나누어 정리 (창고는 하나). Sharding = 창고를 여러 건물로 쪼개어 각각 운영 (건물이 여러 개).
Partitioning의 종류(Range, List, Hash, Key)를 설명해주세요.

① Range Partitioning

파티션 키의 값 범위에 따라 파티션을 나눕니다. 시계열 데이터(날짜, 연도)에 가장 많이 사용합니다.

CREATE TABLE orders (
    id INT, created_at DATE, amount DECIMAL
) PARTITION BY RANGE (YEAR(created_at)) (
    PARTITION p2022 VALUES LESS THAN (2023),
    PARTITION p2023 VALUES LESS THAN (2024),
    PARTITION p2024 VALUES LESS THAN (2025),
    PARTITION p_future VALUES LESS THAN MAXVALUE
);

장점: 오래된 파티션 한 번에 DROP 가능 (대용량 삭제 빠름). 단점: 최신 파티션 Hot Spot.

② List Partitioning

파티션 키의 특정 값 목록으로 나눕니다. 지역, 카테고리 등 열거 값에 적합합니다.

PARTITION BY LIST (region) (
    PARTITION p_korea VALUES IN ('KR', 'KR-SEO'),
    PARTITION p_us    VALUES IN ('US', 'US-NY'),
    PARTITION p_eu    VALUES IN ('DE', 'FR', 'GB')
);

③ Hash Partitioning

파티션 키에 해시 함수를 적용하여 N개 파티션에 균등 분산합니다. 특정 범위 조회보다는 균등 분산이 목적일 때 사용합니다.

PARTITION BY HASH(user_id)
PARTITIONS 8;

④ Key Partitioning

Hash와 유사하지만 MySQL 내장 해시 함수를 사용합니다. 파티션 키가 PK 또는 Unique Key여야 합니다. 문자열 컬럼도 사용 가능합니다.

PARTITION BY KEY(email)
PARTITIONS 4;
종류적합한 컬럼범위 쿼리데이터 관리
Range날짜, 순번효율적파티션 DROP 용이
List지역, 카테고리제한적명시적 값 관리
Hashuser_id 등 숫자비효율균등 분산
KeyPK/Unique (문자 포함)비효율균등 분산
파티션 프루닝(Partition Pruning)이란 무엇인가요?

파티션 프루닝은 쿼리의 WHERE 조건을 분석하여 관련 없는 파티션을 조회에서 제외하는 최적화입니다. 마치 필요한 구역만 열어서 찾는 것과 같습니다.

-- 2024년 데이터만 있는 p2024 파티션만 읽음 (다른 파티션 스킵)
SELECT * FROM orders
WHERE created_at >= '2024-01-01' AND created_at < '2025-01-01';
-- EXPLAIN PARTITIONS로 확인
EXPLAIN PARTITIONS SELECT * FROM orders
WHERE created_at >= '2024-01-01';
-- partitions: p2024  ← 이 파티션만 조회됨

프루닝이 동작하지 않는 경우:

  • 파티션 키 컬럼에 함수를 사용: WHERE YEAR(created_at) = 2024 → 인덱스 미사용과 같은 이유로 프루닝도 안 됨
  • 서브쿼리로 파티션 키를 비교하는 경우
  • 파티션 키가 WHERE 조건에 포함되지 않은 경우 → 전체 파티션 스캔
설계 원칙: 파티션 키는 쿼리의 WHERE 조건에 항상 포함되어야 합니다. 그렇지 않으면 전체 파티션을 다 스캔해서 오히려 성능이 더 나빠질 수 있습니다.
언제 파티셔닝을 쓰고, 언제 샤딩을 써야 하나요?

데이터 규모와 서비스 요구사항에 따라 다릅니다.

파티셔닝을 먼저 고려하는 경우:

  • 단일 서버로도 처리량은 충분하지만, 특정 테이블이 수억 건을 넘어 관리가 어려운 경우
  • 시계열 로그 데이터처럼 오래된 데이터를 주기적으로 삭제/아카이빙해야 하는 경우 (파티션 DROP이 DELETE보다 수백 배 빠름)
  • 날짜 범위 쿼리가 많아 특정 기간 데이터만 빠르게 조회해야 하는 경우
  • 애플리케이션 코드 변경 없이 DB 레벨에서만 최적화하고 싶은 경우

샤딩을 고려하는 경우:

  • 단일 서버의 쓰기 처리량(TPS)이 한계에 도달한 경우
  • 데이터 총량이 단일 서버의 물리적 한계(수 TB)를 초과하는 경우
  • 특정 서버 장애가 전체 서비스에 영향을 주는 것을 막고 싶은 경우
  • 글로벌 서비스처럼 지리적으로 분산된 데이터 저장이 필요한 경우
단계전략적용 시점
1단계 인덱스 최적화 + 쿼리 튜닝 항상 먼저
2단계 Replication (읽기 분산) 읽기 트래픽 과부하
3단계 캐싱 (Redis, Memcached) 동일 데이터 반복 조회
4단계 Partitioning 대용량 단일 테이블 관리
5단계 Vertical Sharding (도메인 분리) 서비스 도메인별 독립 필요
6단계 Horizontal Sharding 쓰기 한계 도달, 수십억 행 이상
조기 샤딩의 위험: 샤딩은 개발·운영 복잡도가 크게 증가합니다. "언젠가 필요할 것 같아서" 조기에 도입하면 팀 생산성이 크게 저하됩니다. 실제 병목이 측정된 이후에 도입하는 것이 원칙입니다.
꼬리 질문: MySQL 파티셔닝의 제약사항은?
→ ① 파티션 키는 테이블의 모든 Unique Key에 포함되어야 합니다. ② 외래키(FK) 사용 불가. ③ 전체 파티션 수는 최대 8192개 (MySQL 8.0). ④ 서브파티셔닝은 Range+Hash, Range+Key, List+Hash, List+Key 조합만 가능합니다.
✦ 면접 한 줄 정리
Replication = 데이터 복제로 가용성 + 읽기 분산. Primary에만 쓰기. 비동기가 기본, 반동기로 안전성 향상.
Replication Lag = 비동기 복제의 구조적 한계. 쓰기 직후 읽기는 Primary로. 병렬 복제(MTS)로 완화.
Sharding = 여러 DB 서버에 데이터 분산. 쓰기 확장 가능. Cross-Shard JOIN/트랜잭션이 어려워 복잡도 폭증.
Consistent Hashing = Shard 추가 시 최소 데이터만 재분배. K/N개만 이동. Virtual Node로 균등 분산.
Partitioning = 단일 서버 내 논리적 분리. 파티션 프루닝으로 성능 향상. Range 파티션 + 날짜 조건이 가장 흔한 사례.
확장 순서: 인덱스 → 캐시 → Replication → Partitioning → Vertical Sharding → Horizontal Sharding. 조기 샤딩은 독.
