JOIN & 정규화 & 쿼리 최적화

JOIN 종류, 정규화/반정규화, 실행 계획, 쿼리 튜닝까지

🔗 Part 1. JOIN — 종류, 내부 동작, 인덱스 영향, ON vs WHERE
JOIN이란 무엇이고, 종류별로 설명해주세요.

JOIN은 두 개 이상의 테이블을 특정 조건으로 결합하여 하나의 결과 집합으로 반환하는 연산입니다. 관계형 데이터베이스는 데이터를 테이블 단위로 분리해서 저장하므로, JOIN이 없으면 여러 테이블의 데이터를 연결해서 조회할 수 없습니다.

종류설명결과
INNER JOIN 양쪽 테이블 모두에 매칭되는 행만 반환 교집합
LEFT (OUTER) JOIN 왼쪽 테이블 전체 + 오른쪽 매칭 행. 오른쪽에 없으면 NULL 왼쪽 기준 합집합
RIGHT (OUTER) JOIN 오른쪽 테이블 전체 + 왼쪽 매칭 행. 왼쪽에 없으면 NULL 오른쪽 기준 합집합
FULL OUTER JOIN 양쪽 테이블 모두 반환. 매칭 없으면 상대쪽 NULL (MySQL 미지원 → UNION으로 구현) 합집합
CROSS JOIN 두 테이블의 모든 행을 곱집합(카티션 곱)으로 결합. 조건 없음 M × N 행
SELF JOIN 같은 테이블을 두 번 참조. 계층 구조(상위-하위 관계) 조회 등 자기 자신과 결합 
-- SELF JOIN 예시: 직원의 매니저 이름 조회
SELECT e.name AS employee, m.name AS manager
FROM employees e
LEFT JOIN employees m ON e.manager_id = m.id;
꼬리 질문: FULL OUTER JOIN을 MySQL에서 어떻게 구현하나요?
→ LEFT JOIN UNION RIGHT JOIN으로 구현합니다. UNION은 중복을 제거하므로 양쪽 결과를 합쳐도 중복 행이 생기지 않습니다.
INNER JOIN vs LEFT JOIN — 차이와 언제 어떤 걸 써야 하나요?

INNER JOIN은 양쪽 테이블에 모두 매칭되는 행만 결과에 포함됩니다. 한쪽 테이블에 매칭 데이터가 없으면 해당 행은 결과에서 완전히 제외됩니다.

LEFT JOIN은 왼쪽(FROM 뒤) 테이블의 모든 행을 항상 유지합니다. 오른쪽 테이블에 매칭 데이터가 없으면 오른쪽 컬럼은 NULL로 채워집니다.

상황추천 JOIN이유
주문이 있는 고객만 조회 INNER JOIN 양쪽 데이터 모두 존재해야 의미 있음
주문 여부와 상관없이 모든 고객 조회 LEFT JOIN 고객 기준으로 주문 정보는 옵션
주문이 없는 고객만 찾기 LEFT JOIN + WHERE IS NULL NULL인 행 = 오른쪽에 없는 데이터 
-- 주문이 없는 고객 찾기
SELECT c.id, c.name
FROM customers c
LEFT JOIN orders o ON c.id = o.customer_id
WHERE o.id IS NULL;
실무 팁: 명확히 양쪽 데이터가 있다고 보장되는 경우 INNER JOIN이 쿼리 옵티마이저에게 더 많은 최적화 여지를 줍니다. "혹시 모르니까" LEFT JOIN을 남용하면 의도치 않게 NULL 행이 섞여들어 버그 원인이 됩니다.
JOIN의 내부 동작 방식 3가지(Nested Loop, Hash, Sort Merge)를 설명해주세요.

데이터베이스 옵티마이저는 테이블 크기, 인덱스 존재 여부, 조인 조건에 따라 아래 3가지 알고리즘 중 하나를 선택합니다.

Nested Loop Join (NLJ)

외부 테이블(Driving Table)을 한 행씩 순회하면서, 각 행마다 내부 테이블(Inner Table)에서 매칭 행을 찾습니다. 내부 테이블에 인덱스가 있으면 탐색이 O(log N)으로 빠릅니다. 작은 결과 집합 + 인덱스가 있는 경우에 최적입니다.

for each row in outer_table:
    for each row in inner_table where condition matches:
        output row

시간 복잡도: O(M × log N) — M: 외부 테이블 행 수, N: 내부 테이블 행 수

Hash Join

작은 테이블을 메모리에 해시 테이블로 구축하고, 큰 테이블을 스캔하면서 해시 테이블을 조회합니다. 인덱스가 없고 두 테이블이 모두 클 때 유리합니다. MySQL 8.0+에서 지원합니다.

Build: hash_table = { key: row for row in small_table }
Probe: for each row in large_table:
           match = hash_table.get(row.key)
           if match: output

메모리 사용이 크고, 등치 조인(=)에만 사용할 수 있습니다.

Sort Merge Join

두 테이블을 조인 키 기준으로 정렬한 뒤, 두 포인터를 동시에 이동하며 병합합니다. 두 테이블이 이미 정렬되어 있거나(인덱스 순서와 일치), 범위 조건 조인에 유리합니다.

알고리즘인덱스 필요메모리적합한 상황
Nested LoopInner 테이블에 필요낮음소량 데이터, 인덱스 있음
Hash Join불필요높음대량 데이터, 인덱스 없음, 등치 조인
Sort Merge불필요(정렬 비용 있음)중간이미 정렬됨, 범위 조인
꼬리 질문: MySQL에서 Hash Join은 언제부터 지원됐나요?
→ MySQL 8.0.18부터 등치 조인(equi-join)에 Hash Join을 지원합니다. 그 이전 버전은 항상 Nested Loop를 사용했습니다.
JOIN 시 인덱스가 미치는 영향은?

JOIN 성능에서 인덱스는 핵심입니다. Nested Loop Join을 예로 들면, Inner 테이블의 조인 조건 컬럼에 인덱스가 있으면 각 행 탐색이 O(log N)이 되어 전체 비용이 O(M × log N)입니다. 인덱스가 없으면 Inner 테이블을 매번 풀 스캔하여 O(M × N)이 됩니다.

-- orders.customer_id에 인덱스가 있어야 빠름
SELECT c.name, o.amount
FROM customers c
INNER JOIN orders o ON c.id = o.customer_id;
-- 인덱스 없으면 customers 한 행당 orders 전체 스캔

Driving Table(드라이빙 테이블) 선택도 중요합니다. 옵티마이저는 일반적으로 결과 행 수가 적은 테이블을 Driving Table로 선택합니다. WHERE 조건으로 필터링된 후 행 수가 적은 테이블이 외부 루프가 되면 전체 반복 횟수가 줄어듭니다.

실무 체크리스트:
1. 조인 조건 컬럼(ON 절)에 인덱스가 있는가?
2. FK(외래키)는 자동으로 인덱스가 생성되지 않는다 — 명시적으로 추가해야 한다.
3. 조인 후 WHERE/ORDER BY 컬럼도 인덱스 대상으로 고려한다.
꼬리 질문: FK를 걸면 자동으로 인덱스가 생기나요?
→ MySQL InnoDB는 FK 컬럼에 자동으로 인덱스를 생성합니다. 하지만 PostgreSQL은 FK 인덱스를 자동 생성하지 않으므로 직접 추가해야 합니다.
JOIN에서 ON 절과 WHERE 절의 차이는?

INNER JOIN에서는 ON과 WHERE 모두 동일한 결과를 냅니다. 하지만 OUTER JOIN(LEFT/RIGHT JOIN)에서는 결과가 달라집니다.

ON 절 — 조인 조건입니다. 이 조건으로 두 테이블을 결합하며, LEFT JOIN에서 오른쪽 테이블에 매칭이 없어도 왼쪽 행은 유지하고 오른쪽 컬럼을 NULL로 채웁니다.

WHERE 절 — 조인이 완료된 결과 집합에서 필터링합니다. LEFT JOIN 후 WHERE에 오른쪽 테이블 조건을 걸면, NULL인 행이 제거되어 사실상 INNER JOIN과 같아집니다.

-- 1. ON에 조건: 2021년 주문이 없는 고객도 결과에 포함 (NULL)
SELECT c.name, o.amount
FROM customers c
LEFT JOIN orders o ON c.id = o.customer_id AND o.year = 2021;

-- 2. WHERE에 조건: 2021년 주문이 없는 고객은 제거됨 (INNER JOIN 효과)
SELECT c.name, o.amount
FROM customers c
LEFT JOIN orders o ON c.id = o.customer_id
WHERE o.year = 2021;
함정 주의: "LEFT JOIN을 썼는데 INNER JOIN처럼 동작한다"는 버그 리포트의 상당수가 WHERE 절에 오른쪽 테이블 조건을 잘못 넣은 경우입니다. 오른쪽 테이블에 대한 필터 조건은 ON 절에 넣어야 LEFT JOIN 의미가 유지됩니다.
서브쿼리 vs JOIN — 성능 비교와 언제 무엇을 쓰나요?

현대 옵티마이저(MySQL 8.0+, PostgreSQL)는 많은 서브쿼리를 내부적으로 JOIN으로 변환(쿼리 재작성)합니다. 따라서 단순한 경우에는 성능 차이가 없습니다.

서브쿼리가 성능상 불리한 경우

상관 서브쿼리(Correlated Subquery)는 외부 쿼리의 각 행마다 서브쿼리가 반복 실행됩니다. 이는 옵티마이저가 항상 최적화하지는 못합니다.

-- 상관 서브쿼리: customers 행마다 orders 서브쿼리 실행 (느릴 수 있음)
SELECT c.name,
       (SELECT COUNT(*) FROM orders o WHERE o.customer_id = c.id) AS cnt
FROM customers c;

-- JOIN으로 변환: 훨씬 효율적
SELECT c.name, COUNT(o.id) AS cnt
FROM customers c
LEFT JOIN orders o ON c.id = o.customer_id
GROUP BY c.id, c.name;

서브쿼리가 유리한 경우

  • EXISTS 서브쿼리: 존재 여부만 확인하면 되어 첫 번째 매칭 시점에 실행을 멈춤 (short-circuit)
  • 복잡한 집계가 필요한 인라인 뷰(FROM 절 서브쿼리)
  • 가독성이 훨씬 높은 경우 — 쿼리 유지보수성도 중요
결론: 현대 DB에서 단순 서브쿼리는 성능 차이 거의 없음. 상관 서브쿼리는 JOIN으로 변환 고려. EXISTS는 IN보다 성능 유리한 경우 많음.
📐 Part 2. 정규화 — 1NF~BCNF, 장단점, 반정규화
정규화란 무엇이고, 왜 필요한가요?

정규화(Normalization)는 데이터의 중복을 최소화하고, 삽입·수정·삭제 이상(Anomaly)을 제거하기 위해 테이블을 올바른 구조로 분해하는 과정입니다.

정규화가 없으면 다음 이상 현상이 발생합니다.

이상 현상설명예시
삽입 이상 원치 않는 데이터 없이 삽입 불가 수강생 없이 강의 정보만 등록 불가
갱신 이상 일부만 수정되어 불일치 발생 강사 전화번호가 여러 행에 중복 → 일부만 수정됨
삭제 이상 원치 않는 데이터까지 삭제됨 마지막 수강생 삭제 시 강의 정보도 사라짐
정규화의 목표: 각 데이터는 한 곳에만 존재하며, 테이블은 하나의 주제(Entity)만 다루어야 한다.
1NF, 2NF, 3NF, BCNF를 각각 예시와 함께 설명해주세요.

1NF (제1정규형) — 원자값(Atomic Value)

모든 컬럼의 값이 더 이상 분해할 수 없는 원자 단위여야 합니다. 반복 그룹(배열, 쉼표로 구분된 값)이 없어야 합니다.

위반 (1NF 전)
학번이름수강과목
001홍길동수학, 영어, 국어
준수 (1NF 후)
학번이름수강과목
001홍길동수학
001홍길동영어
001홍길동국어

2NF (제2정규형) — 부분 함수 종속 제거

1NF를 만족하고, 복합 기본키의 일부에만 종속되는 컬럼(부분 함수 종속)을 제거합니다. 기본키가 단일 컬럼이면 자동으로 2NF를 만족합니다.

학번과목코드성적과목명
001CS101A자료구조

여기서 기본키는 (학번, 과목코드)인데, 과목명은 과목코드에만 종속됩니다 → 부분 함수 종속 위반

분리 후: 수강(학번, 과목코드, 성적) + 과목(과목코드, 과목명)

3NF (제3정규형) — 이행 함수 종속 제거

2NF를 만족하고, 기본키가 아닌 컬럼이 다른 비키 컬럼에 종속(이행 종속)되는 것을 제거합니다.

학번 (PK)학과코드학과명
001CS컴퓨터공학과

학번 → 학과코드 → 학과명 (이행 종속) → 학과명 분리

분리 후: 학생(학번, 학과코드) + 학과(학과코드, 학과명)

BCNF (Boyce-Codd 정규형) — 모든 결정자가 후보키

3NF보다 엄격합니다. 모든 함수 종속에서 결정자가 반드시 후보키여야 합니다. 복수의 후보키가 겹칠 때 3NF는 만족하지만 BCNF를 위반하는 경우가 있습니다.

학생과목교수
홍길동데이터베이스김교수

조건: 한 교수는 한 과목만 담당, 한 학생은 한 과목에 한 명의 교수. 후보키: (학생, 과목), (학생, 교수)

교수 → 과목 종속이 있는데 교수는 후보키가 아님 → BCNF 위반

분리 후: 수강(학생, 교수) + 교수담당(교수, 과목)

암기법: 1NF(원자값) → 2NF(부분종속 제거) → 3NF(이행종속 제거) → BCNF(결정자=후보키)
정규화의 장단점은?
장점단점
데이터 중복 최소화 → 저장 공간 절약 테이블 수 증가 → JOIN 필요
삽입·갱신·삭제 이상 제거 복잡한 JOIN으로 쿼리 성능 저하 가능
데이터 일관성 보장 (한 곳만 수정하면 됨) 개발 복잡도 증가
쓰기(INSERT/UPDATE/DELETE) 성능 향상 읽기(SELECT) 성능은 상대적으로 낮을 수 있음
트레이드오프: 정규화는 쓰기 최적화, 반정규화는 읽기 최적화입니다. 서비스 특성(읽기 중심 vs 쓰기 중심)에 따라 적절한 수준을 선택해야 합니다.
반정규화(역정규화)란 무엇이고, 언제 사용하나요?

반정규화(Denormalization)는 정규화된 테이블에 의도적으로 중복을 추가하거나 테이블을 합쳐서 조회 성능을 높이는 기법입니다. 정규화가 과도하면 복잡한 다중 JOIN이 필요해져 읽기 성능이 떨어지는데, 이를 완화하기 위해 사용합니다.

반정규화를 고려해야 하는 시점:

  • 조회가 쓰기보다 압도적으로 많은 서비스 (읽기 중심)
  • 여러 테이블을 JOIN해야 하는 쿼리가 성능 병목으로 확인된 경우
  • EXPLAIN으로 실행 계획 확인 후 인덱스로도 해결 안 될 때
  • 리포트, 통계, 분석 쿼리처럼 대량 집계가 잦은 경우
주의: 반정규화는 데이터 일관성 관리 책임을 개발자가 져야 합니다. 중복 데이터를 여러 곳에 유지해야 하므로, 갱신 시 모든 곳을 동기화해야 합니다. 인덱스 최적화나 캐싱으로 해결 가능한지 먼저 검토해야 합니다.
반정규화 기법들을 설명해주세요.

① 중복 컬럼 추가 (Redundant Column)

자주 JOIN되는 테이블의 컬럼을 현재 테이블에 복사합니다.

-- 원래: orders JOIN customers로 고객명 가져옴
-- 반정규화: orders 테이블에 customer_name 컬럼 추가
ALTER TABLE orders ADD COLUMN customer_name VARCHAR(100);
-- 단점: customers.name 변경 시 orders.customer_name도 동기화 필요

② 파생 컬럼 추가 (Derived Column)

자주 계산되는 집계값을 컬럼으로 저장합니다.

-- 원래: SELECT COUNT(*) FROM orders WHERE customer_id = ?
-- 반정규화: customers 테이블에 order_count 컬럼 추가
ALTER TABLE customers ADD COLUMN order_count INT DEFAULT 0;
-- 주문 추가/삭제 시 order_count를 함께 업데이트해야 함

③ 테이블 병합 (Table Merge)

항상 함께 조회되는 1:1 관계 테이블을 하나로 합칩니다.

-- users + user_profiles (항상 JOIN해서 쓰는 경우) → 하나로 합침

④ 테이블 분리 (Horizontal/Vertical Partitioning)

하나의 큰 테이블을 자주 쓰는 컬럼과 그렇지 않은 컬럼으로 분리하거나(수직), 행을 기준으로 분리(수평)합니다. 이는 정규화 기준과는 별개의 물리적 최적화입니다.

⑤ 요약 테이블 (Summary Table)

집계 결과를 별도 테이블에 주기적으로 저장합니다. 대시보드, 통계 페이지에 유용합니다.

-- 매일 집계를 배치로 계산해 daily_stats 테이블에 저장
INSERT INTO daily_stats (date, total_sales, order_count)
SELECT DATE(created_at), SUM(amount), COUNT(*)
FROM orders WHERE created_at >= CURDATE() - INTERVAL 1 DAY
GROUP BY DATE(created_at);
꼬리 질문: 반정규화 없이 성능을 높일 수 있는 방법은 무엇이 있나요?
→ 복합 인덱스 최적화, 커버링 인덱스, 캐시(Redis), 읽기 전용 레플리카 활용, 쿼리 자체 최적화(실행 계획 개선) 등을 먼저 검토합니다.
⚡ Part 3. 쿼리 최적화 — 실행 계획, 인덱스, 페이징, N+1
EXPLAIN(실행 계획)을 어떻게 읽나요? 주요 컬럼 설명해주세요.

MySQL의 EXPLAIN은 옵티마이저가 쿼리를 어떻게 실행할 계획인지 보여줍니다. 성능 튜닝의 출발점입니다.

EXPLAIN SELECT * FROM orders o
JOIN customers c ON o.customer_id = c.id
WHERE o.status = 'PAID';

꼬리 질문: EXPLAIN ANALYZE와 EXPLAIN의 차이는?
→ EXPLAIN은 실행 계획만 보여주지만, EXPLAIN ANALYZE(MySQL 8.0.18+, PostgreSQL)는 실제로 쿼리를 실행하고 각 단계의 실제 소요 시간과 처리 행 수를 함께 보여줍니다.
풀 테이블 스캔(Full Table Scan)이 일어나는 경우는 어떤 때인가요?

풀 테이블 스캔은 인덱스를 사용하지 않고 테이블의 모든 행을 순차 읽는 것입니다. EXPLAIN의 type: ALL로 확인됩니다.

원인예시
조건 컬럼에 인덱스 없음 WHERE email = 'x' — email 인덱스 없음
인덱스가 있어도 선택도(Selectivity)가 낮음 WHERE status = 'ACTIVE' — 90%가 ACTIVE이면 풀 스캔이 더 빠름
테이블이 매우 작음 행 수가 적으면 옵티마이저가 풀 스캔 선택
인덱스를 못 타는 쿼리 작성 함수 적용, 타입 불일치, LIKE '%앞' 등
옵티마이저 통계 오류 ANALYZE TABLE로 통계 갱신 필요
선택도(Selectivity): 인덱스가 있어도 옵티마이저는 비용 기반으로 판단합니다. 전체 행의 20~30% 이상을 가져와야 한다면 풀 테이블 스캔이 랜덤 I/O보다 순차 I/O가 빠르다고 판단해 풀 스캔을 선택할 수 있습니다.
인덱스를 안 타는 경우들을 설명해주세요.

인덱스가 있어도 쿼리 작성 방식에 따라 인덱스를 사용하지 못하는 경우가 있습니다.

① 인덱스 컬럼에 함수 또는 연산 적용

-- 인덱스 미사용: YEAR() 함수로 인덱스 컬럼 변환
WHERE YEAR(created_at) = 2024

-- 인덱스 사용: 범위 조건으로 변환
WHERE created_at >= '2024-01-01' AND created_at < '2025-01-01'

② 타입 불일치 (묵시적 형 변환)

-- phone이 VARCHAR인데 숫자로 비교 → 모든 행 형변환 후 비교 → 풀 스캔
WHERE phone = 01012345678   -- 따옴표 없음

-- 올바른 방법
WHERE phone = '01012345678'

③ LIKE 패턴이 '%'로 시작

-- 인덱스 미사용: 접두 와일드카드
WHERE name LIKE '%홍'

-- 인덱스 사용 가능: 접미 와일드카드
WHERE name LIKE '홍%'

④ OR 조건으로 인덱스 분산

-- a에 인덱스 있어도 b에 없으면 전체를 풀 스캔하는 경우 있음
WHERE a = 1 OR b = 2

-- UNION ALL로 분리 (각각 인덱스 활용)
SELECT * FROM t WHERE a = 1
UNION ALL
SELECT * FROM t WHERE b = 2 AND a != 1

⑤ 복합 인덱스 선두 컬럼 미사용

-- 인덱스: (a, b, c)
WHERE b = 1 AND c = 2   -- a가 없으면 인덱스 미사용
WHERE a = 1 AND c = 2   -- a만 인덱스 사용, c는 미사용
WHERE a = 1 AND b = 2   -- (a,b) 인덱스 사용

⑥ NOT IN, NOT EXISTS, != , <>

부정 조건은 일반적으로 인덱스를 탈 수 없습니다. 단, MySQL 8.0에서는 일부 상황에서 range 스캔을 사용하기도 합니다.

핵심 원칙: 인덱스 컬럼은 변형하지 말고 비교 대상을 변형하라. YEAR(col) = 2024 대신 col BETWEEN '2024-01-01' AND '2024-12-31'.
커버링 인덱스(Covering Index)란 무엇이고, 어떻게 활용하나요?

커버링 인덱스는 쿼리가 필요로 하는 모든 컬럼이 인덱스 안에 포함되어 있어서, 실제 테이블 행을 읽지 않아도 되는 경우입니다. EXPLAIN의 Extra에 Using index로 표시됩니다.

InnoDB에서 일반 인덱스는 인덱스 키 + PK를 함께 저장합니다. SELECT 컬럼이 인덱스에 포함되어 있으면 클러스터드 인덱스(실제 테이블)를 찾아가는 I/O를 생략할 수 있어 매우 빠릅니다.

-- 인덱스: (status, created_at, amount) 가정
SELECT status, created_at, amount   -- ← 모두 인덱스에 있음
FROM orders
WHERE status = 'PAID'
ORDER BY created_at DESC;
-- Extra: Using index (테이블 접근 없이 인덱스만으로 결과 반환)

페이징에서의 커버링 인덱스 활용 (Deferred Join)

-- OFFSET이 크면 느린 쿼리 (대량 행 읽고 버림)
SELECT * FROM orders ORDER BY id LIMIT 10 OFFSET 100000;

-- 커버링 인덱스로 id만 먼저 찾고, 그것만 JOIN
SELECT o.*
FROM orders o
JOIN (
  SELECT id FROM orders ORDER BY id LIMIT 10 OFFSET 100000
) t ON o.id = t.id;
커버링 인덱스 설계 팁: SELECT 컬럼 + WHERE 조건 컬럼 + ORDER BY 컬럼을 복합 인덱스로 구성하면 커버링 인덱스가 됩니다. 단, 인덱스 크기가 커지면 쓰기 오버헤드도 증가합니다.
꼬리 질문: SELECT *를 쓰면 커버링 인덱스가 안 되는 이유는?
→ 인덱스에 없는 컬럼이 포함되면 결국 테이블을 읽어야 하므로 커버링 인덱스 효과가 없어집니다. 필요한 컬럼만 명시적으로 SELECT해야 합니다.
OFFSET 기반 페이징의 문제와 커서 기반 페이징을 설명해주세요.

OFFSET 방식의 문제

LIMIT N OFFSET M은 M+N개의 행을 읽은 뒤 앞의 M개를 버립니다. 페이지가 뒤로 갈수록 읽는 데이터가 기하급수적으로 증가합니다. 100만 번째 페이지에 가려면 100만 행을 읽고 버려야 합니다.

-- 10만 번째 offset: 내부적으로 100,010행을 읽음
SELECT * FROM products ORDER BY id LIMIT 10 OFFSET 100000;

커서 기반 페이징 (Cursor / Keyset Pagination)

마지막으로 읽은 행의 기준값(커서)을 이용해 WHERE 조건으로 다음 페이지를 가져옵니다. 항상 인덱스를 타기 때문에 어느 페이지든 동일하게 빠릅니다.

-- 첫 페이지
SELECT * FROM products ORDER BY id LIMIT 10;

-- 다음 페이지: 마지막으로 받은 id = 100
SELECT * FROM products
WHERE id > 100
ORDER BY id LIMIT 10;
방식성능임의 페이지 이동정렬 기준적합한 케이스
OFFSET 페이지 뒤로 갈수록 느림 가능 자유 관리자 화면, 소량 데이터
커서 항상 O(log N) 불가 (순차만) 커서 컬럼 인덱스 필요 무한 스크롤, 피드, 대량 데이터
커서 기반의 제약: "5페이지로 바로 이동" 같은 임의 페이지 점프가 불가능합니다. 무한 스크롤(SNS 피드, 커머스 목록)에 적합하고, 번호 기반 페이지네이션에는 부적합합니다.
N+1 쿼리 문제란 무엇이고, 어떻게 해결하나요?

N+1 쿼리는 1개의 쿼리로 N개의 레코드를 가져온 뒤, 각 레코드마다 추가 쿼리 N개를 실행하는 현상입니다. 총 N+1개의 쿼리가 발생합니다. JPA/Hibernate의 지연 로딩(Lazy Loading)과 함께 자주 발생합니다.

// N+1 문제 예시 (JPA Lazy Loading)
List<Order> orders = orderRepository.findAll(); // 쿼리 1번: N개 조회
for (Order order : orders) {
    order.getCustomer().getName();  // 쿼리 N번: 각 주문마다 customer 조회
}

해결 방법 1: Fetch Join (JOIN FETCH)

// JPQL Fetch Join: 한 번의 쿼리로 연관 엔티티까지 조회
@Query("SELECT o FROM Order o JOIN FETCH o.customer")
List<Order> findAllWithCustomer();

해결 방법 2: EntityGraph

@EntityGraph(attributePaths = {"customer"})
List<Order> findAll();

해결 방법 3: Batch Size (N+1 → 1+1)

# application.yml: Hibernate가 IN 절로 묶어서 조회
spring.jpa.properties.hibernate.default_batch_fetch_size: 100
// 또는 @BatchSize(size = 100) 엔티티 위에 선언
// 결과: N개 개별 쿼리 → IN (id1, id2, ...) 배치 쿼리로 변환

해결 방법 4: 일반 JOIN + DTO 조회 (MyBatis, QueryDSL)

-- 처음부터 JOIN으로 필요한 컬럼만 가져오기
SELECT o.id, o.amount, c.name
FROM orders o
JOIN customers c ON o.customer_id = c.id;
방법특징컬렉션(OneToMany) 처리
Fetch Join한 쿼리에 모두 처리컬렉션은 중복 주의 (DISTINCT 필요). Pagination 불가
Batch SizeIN절로 배치 조회컬렉션도 안전. Pagination 가능
DTO 직접 조회가장 성능 좋음직접 매핑. 코드 복잡
컬렉션 Fetch Join + Pagination 금지: OneToMany Fetch Join과 LIMIT을 함께 쓰면 Hibernate가 메모리에 전체 데이터를 로드한 뒤 페이징합니다 (HHH90003004 경고). 컬렉션 페이징은 Batch Size를 사용하세요.
꼬리 질문: N+1 문제는 Eager Loading으로 해결되나요?
→ Eager Loading을 쓰면 항상 JOIN해서 가져오지만, 사용하지 않는 연관 데이터까지 매번 로딩해 오히려 성능이 나빠질 수 있습니다. 또한 예상치 못한 쿼리가 나가는 문제가 있어 JPA 표준은 기본을 Lazy로 권장합니다.
✦ 면접 한 줄 정리
INNER JOIN = 교집합. LEFT JOIN = 왼쪽 전체 + 오른쪽 NULL 가능. ON vs WHERE는 OUTER JOIN에서 결정적 차이.
정규화 = 중복·이상 제거. 1NF(원자값) → 2NF(부분종속↓) → 3NF(이행종속↓) → BCNF(결정자=후보키).
반정규화 = 읽기 성능을 위해 의도적 중복 도입. 인덱스·캐시 먼저 검토 후 최후 수단.
EXPLAIN type: ALL = 풀 테이블 스캔 위험. Using index = 커버링 인덱스로 최고 효율.
커서 페이징 = OFFSET 대비 항상 O(log N). 무한 스크롤에 필수. 임의 페이지 이동 불가.
N+1 = Lazy Loading 함정. Fetch Join 또는 Batch Size(100)로 해결. 컬렉션 + Pagination은 Batch Size.
