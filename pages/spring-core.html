Part 1 — IoC & DI · 스프링의 근본 철학인 제어의 역전과 의존성 주입의 원리를 이해합니다.
IoC(제어의 역전)란 무엇인가요?

IoC(Inversion of Control, 제어의 역전)는 객체의 생성과 의존성 관리의 제어권을 개발자가 아닌 프레임워크(컨테이너)에게 위임하는 설계 원칙입니다.

전통적인 방식에서는 개발자가 직접 객체를 생성하고, 의존하는 객체도 직접 생성해서 연결합니다.

// 전통적 방식 — 개발자가 제어
class OrderService {
    private OrderRepository repo = new OrderRepositoryImpl(); // 직접 생성

    public void order() { repo.save(...); }
}

// IoC 방식 — 컨테이너가 제어
class OrderService {
    private final OrderRepository repo; // 선언만 함

    public OrderService(OrderRepository repo) {
        this.repo = repo; // Spring이 주입해줌
    }
}

IoC의 핵심 이점은 관심사의 분리입니다. 비즈니스 로직을 담은 객체가 "어떤 구현체를 사용할지"를 신경 쓰지 않아도 됩니다. 이는 낮은 결합도(Low Coupling)와 높은 응집도(High Cohesion)로 이어집니다. IoC는 개념이고, 이를 구현하는 구체적인 패턴이 DI(의존성 주입)입니다.

꼬리질문 대비

"IoC 컨테이너와 DI 컨테이너는 같은 말인가요?" — 엄밀히는 다르지만 실무에서 혼용됩니다. IoC 컨테이너는 더 넓은 개념으로 객체 생명주기 전반을 관리하고, DI 컨테이너는 그 중 의존성 주입에 초점을 둔 표현입니다. Spring의 ApplicationContext는 둘 다입니다.

DI(의존성 주입)의 3가지 방법과 왜 생성자 주입을 권장하는지 설명해주세요.

Spring에서 의존성을 주입하는 방법은 세 가지입니다.

// 1. 생성자 주입 (Constructor Injection) — 권장
@Service
public class OrderService {
    private final OrderRepository orderRepository;
    private final PaymentService paymentService;

    // Lombok @RequiredArgsConstructor로 대체 가능
    public OrderService(OrderRepository orderRepository,
                        PaymentService paymentService) {
        this.orderRepository = orderRepository;
        this.paymentService = paymentService;
    }
}

// 2. 세터 주입 (Setter Injection) — 선택적 의존성에만 사용
@Service
public class OrderService {
    private OrderRepository orderRepository;

    @Autowired
    public void setOrderRepository(OrderRepository repo) {
        this.orderRepository = repo;
    }
}

// 3. 필드 주입 (Field Injection) — 실무에서 지양
@Service
public class OrderService {
    @Autowired
    private OrderRepository orderRepository; // 간결하지만 문제 있음
}
방식장점단점
생성자 주입불변성(final), 필수 의존성 명확, 테스트 용이, 순환 참조 조기 감지파라미터 많으면 코드 길어짐 (Lombok으로 해결)
세터 주입선택적 의존성 표현 가능불변성 없음, 런타임 NPE 위험
필드 주입코드 간결테스트 시 Spring 없이 주입 불가, 불변성 없음, 순환 참조 감지 늦음

생성자 주입을 권장하는 이유: ① final로 선언하여 불변성 보장. ② 객체 생성 시점에 모든 의존성이 주입되므로 완전한 객체 보장. ③ 순수 자바로 테스트 가능 (Spring 컨텍스트 없이 new로 직접 주입). ④ 순환 참조 발생 시 애플리케이션 시작 시점에 즉시 예외가 발생하여 조기 감지.

@Autowired의 동작 원리를 설명해주세요.

@Autowired는 Spring이 의존성을 자동으로 주입하게 하는 어노테이션입니다. 주입 우선순위는 타입(Type) → 이름(Name) 순서입니다.

동작 과정: ① Spring 컨테이너가 @Autowired가 붙은 지점을 스캔합니다. ② 주입 대상의 타입으로 컨테이너에서 빈을 조회합니다. ③ 해당 타입의 빈이 정확히 하나면 주입합니다. ④ 빈이 없으면 NoSuchBeanDefinitionException (required=false로 선택적 처리 가능). ⑤ 빈이 여러 개면 NoUniqueBeanDefinitionException — 이때 @Qualifier나 @Primary로 해결합니다.

// 타입으로 여러 빈이 있을 때 해결 방법
@Autowired
private PaymentService kakaoPaymentService; // 필드명 = 빈 이름으로 매칭

@Autowired
@Qualifier("kakaoPaymentService")
private PaymentService paymentService; // 명시적 이름 지정

내부적으로 Spring은 AutowiredAnnotationBeanPostProcessor라는 BeanPostProcessor를 통해 빈 생성 후 @Autowired 처리를 수행합니다. 이것이 바로 Spring AOP, 트랜잭션 등 다양한 기능도 BeanPostProcessor 패턴으로 동작하는 이유입니다.

@Qualifier와 @Primary의 차이와 사용법을 설명해주세요.

같은 타입의 빈이 여러 개일 때 어떤 빈을 주입할지 결정하는 두 가지 방법입니다.

@Component
@Primary // 같은 타입에서 기본으로 선택될 빈
public class KakaoPaymentService implements PaymentService { ... }

@Component
public class TossPaymentService implements PaymentService { ... }

// @Primary — 특별히 지정 안 하면 KakaoPaymentService가 주입됨
@Autowired
private PaymentService paymentService; // → KakaoPaymentService

// @Qualifier — 명시적으로 특정 빈 지정 (@Primary보다 우선순위 높음)
@Autowired
@Qualifier("tossPaymentService")
private PaymentService paymentService; // → TossPaymentService
구분@Primary@Qualifier
선언 위치빈 클래스/메서드에 선언주입받는 곳에 선언
범위해당 타입의 전역 기본값특정 주입 지점에만 적용
우선순위낮음 (기본값 역할)높음 (@Primary보다 우선)
사용 상황주로 사용될 구현체 지정특정 위치에서 다른 구현체 사용
의존성 주입 시 인터페이스를 사용하는 이유는 무엇인가요?

인터페이스로 의존성을 선언하면 DIP(의존 역전 원칙, Dependency Inversion Principle)을 만족합니다. 고수준 모듈이 저수준 구현에 직접 의존하지 않고, 둘 다 추상화(인터페이스)에 의존합니다.

// 구현체에 의존 — 나쁜 예
private MySQLOrderRepository repository; // MySQL로 고정됨

// 인터페이스에 의존 — 좋은 예
private OrderRepository repository; // 구현체 교체에 자유로움

인터페이스를 사용하는 구체적 이점: ① OCP(개방-폐쇄 원칙): 새 구현체를 추가해도 사용하는 코드를 수정하지 않아도 됩니다. ② 테스트 용이성: 테스트 시 Mock 객체로 교체가 쉽습니다. ③ Spring AOP 적용: Spring은 인터페이스가 있을 때 JDK Dynamic Proxy를 사용하여 프록시를 생성합니다(트랜잭션, AOP 등이 동작하려면 프록시가 필요).

실무 팁: 구현체가 하나뿐이어서 인터페이스가 과하다고 느낄 수 있지만, 테스트 가능성과 AOP 호환성 때문에 Service 레이어에는 인터페이스를 두는 것이 관례입니다. 다만 Repository는 Spring Data JPA 인터페이스 자체가 추상화 역할을 하므로 별도 인터페이스가 필요하지 않습니다.
Part 2 — Bean · Spring Bean의 등록 방법, 생명주기, 스코프, 주의사항을 이해합니다.
Spring Bean이란 무엇인가요?

Spring Bean은 Spring IoC 컨테이너가 관리하는 자바 객체입니다. 일반 자바 객체(new ClassName()로 생성)와 달리, 스프링 빈은 컨테이너가 생성, 의존성 주입, 초기화, 소멸 전 과정을 관리합니다.

비유하면, 일반 객체는 직접 채용한 프리랜서이고, 스프링 빈은 회사(컨테이너)가 정규직으로 채용하고 관리하는 직원입니다. 회사가 출퇴근, 업무 지시, 퇴직 절차를 모두 관리합니다.

스프링 빈으로 관리되면 얻는 혜택: ① 자동 의존성 주입(@Autowired). ② AOP 적용 (트랜잭션, 로깅 등). ③ 생명주기 콜백 (@PostConstruct, @PreDestroy). ④ 스코프 관리 (싱글톤, 프로토타입 등).

Bean 등록 방법 — @Component, @Bean, @Configuration의 차이를 설명해주세요.
// 1. @Component 계열 — 클래스에 직접 붙여 자동 등록 (컴포넌트 스캔)
@Component   // 범용
@Service     // 비즈니스 로직 레이어 (의미상 구분)
@Repository // 데이터 접근 레이어 + 예외 변환
@Controller // 웹 레이어
@RestController // @Controller + @ResponseBody

@Service
public class OrderService { ... } // 자동으로 빈 등록됨

// 2. @Bean — @Configuration 클래스 내 메서드에 붙여 수동 등록
@Configuration
public class AppConfig {

    @Bean
    public ObjectMapper objectMapper() {
        return new ObjectMapper()
            .configure(DeserializationFeature.FAIL_ON_UNKNOWN_PROPERTIES, false);
    }

    @Bean
    public RestTemplate restTemplate() {
        return new RestTemplate();
    }
}
구분@Component@Bean
선언 위치클래스에 직접@Configuration 클래스 내 메서드
빈 생성 제어제한적 (클래스 생성자만)자유롭게 커스터마이징 가능
외부 라이브러리불가 (소스 수정 불가)가능 (ObjectMapper, RestTemplate 등)
적합한 상황직접 만든 클래스외부 라이브러리, 복잡한 초기화 필요 시

Spring Bean 생명주기
@Component
public class DatabaseConnectionPool {

    @Autowired
    private DataSourceProperties props;

    @PostConstruct // 의존성 주입 완료 후 호출 — 초기화 로직
    public void init() {
        // props가 이미 주입되어 있어 안전하게 사용 가능
        connectToDatabase(props.getUrl());
    }

    @PreDestroy // 컨테이너 종료 직전 호출 — 자원 해제 로직
    public void destroy() {
        closeAllConnections();
    }
}

@PostConstruct와 @PreDestroy는 JSR-250 표준 어노테이션이므로 Spring에 종속되지 않아 권장됩니다. 이외에 InitializingBean 인터페이스의 afterPropertiesSet()이나 @Bean의 initMethod/destroyMethod 속성도 사용 가능하지만, @PostConstruct가 가장 일반적입니다.

prototype 주의점: prototype 빈은 Spring 컨테이너가 생성과 주입만 담당하고, 이후 관리는 하지 않습니다. @PreDestroy가 호출되지 않습니다. 자원 해제가 필요하다면 직접 처리해야 합니다.
Singleton Bean에서 상태를 가지면 안 되는 이유는 무엇인가요?

Singleton 빈은 컨테이너당 하나의 인스턴스가 모든 스레드에서 공유됩니다. 이 인스턴스가 상태(instance variable)를 가지면 다음과 같은 동시성 문제가 발생합니다.

// 위험한 코드 — 상태를 가진 싱글톤 빈
@Service
public class OrderService {
    private User currentUser; // 위험! 모든 요청이 공유

    public void process(User user) {
        this.currentUser = user; // 스레드A가 설정
        // 스레드B가 다른 user로 덮어쓸 수 있음!
        doProcess(currentUser);
    }
}

// 올바른 코드 — 로컬 변수 사용
@Service
public class OrderService {
    public void process(User user) {
        // user는 스택(메서드 파라미터)에 저장 → 스레드 독립적
        doProcess(user);
    }
}

규칙: Singleton 빈의 필드는 무상태(stateless)이거나 불변(final, 읽기 전용)이어야 합니다. 요청별 상태가 필요하다면 메서드 파라미터나 로컬 변수를 사용하고, 부득이한 경우 ThreadLocal을 활용합니다.

Part 3 — AOP · 관점 지향 프로그래밍의 개념, Spring AOP의 프록시 방식, 실무 활용을 이해합니다.
AOP(관점 지향 프로그래밍)란 무엇인가요?

AOP(Aspect-Oriented Programming)는 횡단 관심사(Cross-Cutting Concern)를 핵심 비즈니스 로직에서 분리하는 프로그래밍 패러다임입니다.

횡단 관심사란 로깅, 트랜잭션, 보안 인증, 성능 측정처럼 여러 비즈니스 메서드에 공통으로 반복 적용되는 코드입니다. AOP가 없다면 이 코드를 모든 메서드에 직접 삽입해야 합니다.

AOP — 횡단 관심사를 Aspect로 분리하여 각 서비스에 적용

AOP를 사용하면 비즈니스 로직 코드에 공통 코드가 없어지므로 단일 책임 원칙(SRP)을 더 잘 지킬 수 있고, 공통 로직의 변경이 한 곳(Aspect)에서만 이루어집니다.

AOP 핵심 개념 — Aspect, Advice, Pointcut, JoinPoint, Weaving을 설명해주세요.
용어의미비유
Aspect횡단 관심사를 모듈화한 것. Advice + Pointcut의 묶음보안 요원 부서 전체
Advice실제 실행할 부가 로직 (언제 무엇을 할지)보안 요원이 하는 행동
PointcutAdvice를 적용할 지점을 정의하는 표현식보안 검사 대상 구역 선택
JoinPointAdvice가 적용될 수 있는 모든 지점 (메서드 실행, 예외 발생 등)보안 검사가 가능한 모든 출입구
WeavingAspect를 대상 객체에 적용하는 과정보안 시스템 설치
TargetAdvice가 적용되는 실제 객체보안 검사 대상
@Aspect
@Component
public class LoggingAspect {

    // Pointcut: service 패키지의 모든 public 메서드
    @Pointcut("execution(public * com.example.service.*.*(..))")
    public void serviceLayer() {}

    // Advice: 메서드 실행 전후
    @Around("serviceLayer()")
    public Object logExecutionTime(ProceedingJoinPoint pjp) throws Throwable {
        long start = System.currentTimeMillis();
        Object result = pjp.proceed(); // 실제 메서드 실행 (JoinPoint)
        long elapsed = System.currentTimeMillis() - start;
        log.info("{} executed in {}ms", pjp.getSignature(), elapsed);
        return result;
    }
}
Advice 종류(@Before, @After, @Around 등)를 각각 설명해주세요.
Advice실행 시점주요 용도
@Before메서드 실행 전파라미터 검증, 인증 확인
@After메서드 실행 후 (성공/예외 무관)자원 해제, 로그 기록
@AfterReturning메서드 정상 반환 후반환값 로깅, 결과 캐싱
@AfterThrowing예외 발생 후예외 로깅, 알림 발송
@Around전체 (Before + After + 예외 모두 제어)성능 측정, 트랜잭션, 재시도
@Before("execution(* com.example.service.*.*(..))")
public void checkAuth(JoinPoint jp) {
    // jp.getArgs()로 파라미터 접근 가능
    if (!isAuthenticated()) throw new UnauthorizedException();
}

@AfterReturning(pointcut = "execution(* getUser(..))", returning = "result")
public void logResult(JoinPoint jp, Object result) {
    log.info("Result: {}", result); // 반환값 접근 가능
}

@AfterThrowing(pointcut = "execution(* pay(..))", throwing = "ex")
public void handleException(JoinPoint jp, Exception ex) {
    alertTeam(ex); // 예외 정보 접근 가능
}

@Around가 가장 강력하며, pjp.proceed()를 직접 호출하므로 실행 자체를 막거나 반환값을 수정할 수도 있습니다. 단, proceed()를 잊으면 실제 메서드가 실행되지 않으므로 주의해야 합니다.

AOP 실무 활용 사례를 설명해주세요.

// 1. 트랜잭션 — @Transactional의 내부 구현이 AOP
@Transactional
public void transfer(Long from, Long to, BigDecimal amount) {
    // AOP가 메서드 전: beginTransaction()
    // AOP가 메서드 후: commit() 또는 rollback()
}

// 2. 커스텀 로깅 Aspect — 모든 API 요청 자동 로깅
@Around("@annotation(org.springframework.web.bind.annotation.GetMapping)")
public Object logApiCall(ProceedingJoinPoint pjp) throws Throwable {
    log.info("API: {}", pjp.getSignature());
    return pjp.proceed();
}

// 3. 커스텀 어노테이션으로 권한 확인
@Target(ElementType.METHOD)
@Retention(RetentionPolicy.RUNTIME)
public @interface AdminOnly {}

@Before("@annotation(AdminOnly)")
public void checkAdmin() {
    if (!SecurityUtils.isAdmin()) throw new ForbiddenException();
}

실무에서 AOP를 사용하면 좋은 영역: ① 성능 모니터링 (메서드 실행 시간 측정). ② 분산 추적 (Trace ID 로그 삽입). ③ 캐싱 (@Cacheable 내부도 AOP). ④ 재시도 로직 (@Retryable). ⑤ API 호출 제한 (Rate Limiting).

Part 4 — 심화 · 순환 참조, @Configuration CGLIB, BeanFactory vs ApplicationContext, 자동 설정 원리를 이해합니다.
순환 참조(Circular Dependency)란 무엇이고, 어떻게 해결하나요?

순환 참조는 A → B → A처럼 두 빈이 서로를 의존하는 상황입니다. Spring에서는 이를 어떻게 처리하는지가 주입 방식에 따라 달라집니다.

// 순환 참조 예시
@Service
public class ServiceA {
    private final ServiceB serviceB; // A는 B가 필요
    public ServiceA(ServiceB serviceB) { this.serviceB = serviceB; }
}

@Service
public class ServiceB {
    private final ServiceA serviceA; // B는 A가 필요
    public ServiceB(ServiceA serviceA) { this.serviceA = serviceA; }
}
// Spring Boot 2.6+ 기본: BeanCurrentlyInCreationException 발생!

해결 방법 세 가지:

// 해결 1: 설계 개선 — 공통 의존 추출 (권장)
// ServiceA, ServiceB 모두 의존하는 공통 로직을 ServiceC로 분리

// 해결 2: @Lazy — 지연 주입으로 순환 고리 끊기
@Service
public class ServiceA {
    private final ServiceB serviceB;
    public ServiceA(@Lazy ServiceB serviceB) {
        this.serviceB = serviceB; // 실제 사용 시점에 초기화
    }
}

// 해결 3: 세터/필드 주입으로 변경 (비권장)
// 필드 주입 시 Spring이 "3단계 캐시"로 처리 가능하지만
// 이는 잘못된 설계를 숨기는 것에 불과
근본 해결책: 순환 참조는 대부분 설계 문제의 신호입니다. @Lazy로 숨기기보다 의존성 방향을 재설계하는 것이 최선입니다. 공통 로직을 새 클래스로 추출하거나, 이벤트(ApplicationEvent)로 의존 방향을 역전시키는 방법도 있습니다.
@Configuration의 CGLIB 프록시 동작 원리를 설명해주세요.

@Configuration은 단순한 마커 어노테이션이 아닙니다. Spring은 @Configuration 클래스를 CGLIB로 프록싱하여 @Bean 메서드 간 호출이 항상 싱글톤 빈을 반환하도록 보장합니다.

@Configuration
public class AppConfig {

    @Bean
    public MemberService memberService() {
        return new MemberService(memberRepository()); // 메서드 직접 호출
    }

    @Bean
    public OrderService orderService() {
        return new OrderService(memberRepository()); // 메서드 직접 호출
    }

    @Bean
    public MemberRepository memberRepository() {
        return new MemoryMemberRepository(); // new가 3번 호출될 것 같지만?
    }
}
// 실제로는 memberRepository()가 컨테이너에서 같은 빈을 반환 → 인스턴스 1개만 생성

CGLIB 프록시가 @Bean 메서드 호출을 가로채서, 이미 컨테이너에 등록된 빈이 있으면 그것을 반환합니다. 덕분에 memberRepository()를 여러 번 호출해도 항상 동일한 인스턴스를 사용합니다.

@Configuration vs @Component: @Bean 메서드를 @Component 클래스 안에 정의하면 CGLIB 프록시가 없어서 메서드 직접 호출 시 매번 새 객체가 생성됩니다. 이를 Lite Mode라고 합니다. @Bean 간 의존이 있으면 반드시 @Configuration을 사용해야 합니다.
BeanFactory vs ApplicationContext — 차이점을 설명해주세요.

둘 다 Spring IoC 컨테이너이지만, ApplicationContext가 BeanFactory를 상속하고 더 많은 기능을 제공합니다.

기능BeanFactoryApplicationContext
빈 관리기본 DI 컨테이너BeanFactory 모든 기능 포함
빈 초기화 시점Lazy (getBean() 호출 시)Eager (컨테이너 시작 시 싱글톤 즉시 생성)
이벤트 발행없음ApplicationEvent, ApplicationListener
국제화(i18n)없음MessageSource
환경/프로파일없음Environment, @Profile
AOP제한적완전 지원

실무에서는 항상 ApplicationContext를 사용합니다. BeanFactory는 임베디드 기기 등 리소스가 극히 제한된 환경에서만 고려합니다. Spring Boot에서는 SpringApplication.run()이 내부적으로 AnnotationConfigServletWebServerApplicationContext(또는 Reactive 환경이면 다른 구현체)를 생성합니다.

Spring의 Environment, Profile, Property 관리를 설명해주세요.

Environment는 애플리케이션의 환경 정보(프로파일, 프로퍼티)를 추상화하는 인터페이스입니다.

// 프로파일별 빈 등록
@Configuration
@Profile("prod") // prod 환경에서만 활성화
public class ProdDataSourceConfig {
    @Bean
    public DataSource dataSource() { return createRDSDataSource(); }
}

@Configuration
@Profile("local")
public class LocalDataSourceConfig {
    @Bean
    public DataSource dataSource() { return createH2DataSource(); }
}

// application.yml에서 프로파일 지정
// spring.profiles.active=prod
// 또는 JVM 옵션: -Dspring.profiles.active=prod

// 프로퍼티 읽기
@Value("${app.max-size:100}") // 없으면 기본값 100
private int maxSize;

@ConfigurationProperties("app") // 타입 안전 프로퍼티 바인딩 (권장)
@Component
public class AppProperties {
    private int maxSize;
    private String name;
    // getter/setter
}

프로퍼티 우선순위 (높을수록 우선): 커맨드라인 인수 > 환경 변수 > application-{profile}.yml > application.yml > @PropertySource. 실무에서 민감한 정보(DB 비밀번호, API 키)는 환경 변수나 AWS Secrets Manager 같은 외부 저장소에서 주입하는 것이 보안상 권장됩니다.

Spring Boot 자동 설정(@EnableAutoConfiguration) 원리를 설명해주세요.

Spring Boot의 핵심 마법인 자동 설정(Auto-Configuration)은 클래스패스에 특정 라이브러리가 있으면 관련 빈을 자동으로 등록해주는 메커니즘입니다.

// @SpringBootApplication = @SpringBootConfiguration + @EnableAutoConfiguration + @ComponentScan
@SpringBootApplication
public class Application {
    public static void main(String[] args) {
        SpringApplication.run(Application.class, args);
    }
}

동작 원리 단계별 설명:

① @EnableAutoConfiguration이 AutoConfigurationImportSelector를 활성화합니다.

② AutoConfigurationImportSelector는 클래스패스의 모든 JAR에서 META-INF/spring/org.springframework.boot.autoconfigure.AutoConfiguration.imports 파일(Spring Boot 2.7 이전은 spring.factories)을 읽습니다.

③ 이 파일에 등록된 수백 개의 AutoConfiguration 클래스 목록을 가져옵니다.

④ 각 AutoConfiguration 클래스에 붙은 @ConditionalOn* 어노테이션으로 실제 적용 여부를 결정합니다.

// Spring Boot 내부 DataSourceAutoConfiguration 단순화 예시
@AutoConfiguration
@ConditionalOnClass(DataSource.class) // DataSource 클래스가 클래스패스에 있을 때만
@ConditionalOnMissingBean(DataSource.class) // 개발자가 직접 DataSource 빈을 정의하지 않았을 때만
@EnableConfigurationProperties(DataSourceProperties.class)
public class DataSourceAutoConfiguration {
    @Bean
    public DataSource dataSource(DataSourceProperties properties) {
        return properties.initializeDataSourceBuilder().build();
    }
}

핵심: @ConditionalOnMissingBean 덕분에 개발자가 직접 빈을 정의하면 자동 설정이 비켜갑니다. 이것이 "Spring Boot는 기본값을 제공하되, 개발자가 오버라이드할 수 있다"는 철학의 구현입니다.

꼬리질문 대비

"어떤 자동 설정이 적용되는지 확인하는 방법은?" — application.yml에 logging.level.org.springframework.boot.autoconfigure=DEBUG를 추가하거나, --debug 옵션으로 실행하면 Conditions Evaluation Report가 출력되어 어떤 AutoConfiguration이 적용/제외되었는지 확인할 수 있습니다.

IoC = 제어권을 컨테이너에 위임. DI = 생성자 주입 권장 (불변성 + 테스트 + 조기 오류 감지)
