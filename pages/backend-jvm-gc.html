Part 1 — JVM 기본 구조 · JVM이 무엇이고, 어떤 구조로 동작하는지 면접관에게 설명할 수 있어야 합니다.
JVM이 무엇이고 왜 필요한가요?

JVM(Java Virtual Machine)은 자바 바이트코드를 실행하는 가상 머신입니다. 비유하자면, 자바 소스코드를 한국어로 쓴 책이라고 하면, 컴파일러가 이걸 '국제 공용 번역본(바이트코드)'으로 만들고, JVM은 각 나라(OS)에서 그 번역본을 현지 언어로 읽어주는 통역사입니다.

JVM이 필요한 핵심 이유는 "Write Once, Run Anywhere"입니다. 개발자가 Windows에서 작성한 코드가 Linux, Mac에서도 그대로 실행됩니다. JVM이 OS별 차이를 추상화해주기 때문입니다. 또한 JVM은 메모리 관리(GC), 보안(샌드박스), 최적화(JIT 컴파일)를 자동으로 처리해줍니다.

꼬리질문 대비

"자바가 인터프리터 언어인가요, 컴파일 언어인가요?" — 둘 다입니다. javac가 소스를 바이트코드로 컴파일하고, JVM이 바이트코드를 인터프리트하다가 핫스팟을 발견하면 JIT 컴파일로 네이티브 코드로 변환합니다.

JVM의 메모리 구조를 설명해주세요.

JVM 메모리는 크게 5가지 영역으로 구분됩니다. 비유하자면 JVM은 하나의 사무실 건물이고, 각 영역은 용도가 다른 방입니다.

Method Area는 클래스 메타데이터, static 변수, 상수 풀이 저장되는 공용 창고입니다. Heap은 new 키워드로 생성된 모든 객체가 살아있는 곳이며, GC의 주요 대상입니다. 이 두 영역은 모든 스레드가 공유합니다.

반면 Stack, PC Register, Native Method Stack은 스레드마다 독립적으로 존재합니다. Stack에는 메서드 호출 시 프레임 단위로 지역변수와 연산 결과가 쌓이고, 메서드가 끝나면 프레임이 제거됩니다. 면접관이 "스택 오버플로우가 왜 발생하나요?"라고 물으면, 재귀 호출이 너무 깊어져 스레드의 Stack 크기를 초과할 때 발생한다고 답하면 됩니다.

JIT 컴파일러란 무엇이고, 어떻게 최적화하나요?

JIT(Just-In-Time) 컴파일러는 자주 실행되는 코드(핫스팟)를 런타임에 네이티브 머신 코드로 변환하는 최적화 엔진입니다. 식당에 비유하면, 처음 주문은 레시피를 보고 만들지만(인터프리팅), 같은 메뉴가 계속 나오면 아예 레시피를 외워서 빠르게 만드는 것(JIT 컴파일)과 같습니다.

HotSpot JVM에는 C1(Client) 컴파일러와 C2(Server) 컴파일러가 있습니다. C1은 빠른 컴파일에 초점을 맞추고, C2는 시간이 더 걸리지만 더 공격적인 최적화를 수행합니다. Java 9부터는 Tiered Compilation이 기본으로, C1으로 먼저 컴파일한 뒤 핫한 코드를 C2로 재컴파일하는 전략을 사용합니다.

대표적인 JIT 최적화로는 인라이닝(메서드 호출을 본문으로 대체), 루프 언롤링, 이스케이프 분석(객체가 메서드 밖으로 나가지 않으면 스택에 할당) 등이 있습니다.

클래스 로딩 과정을 설명해주세요.

JVM의 클래스 로더는 .class 파일을 메모리에 올리는 역할을 합니다. 이 과정은 Loading → Linking → Initialization 세 단계로 진행됩니다.

Loading에서는 Bootstrap → Extension → Application 순서의 부모 위임 모델(Parent Delegation Model)로 클래스를 찾습니다. 부모 클래스 로더가 찾지 못하면 자식이 찾는 구조입니다. 이렇게 하는 이유는 핵심 클래스(java.lang.Object 등)가 악의적으로 교체되는 것을 방지하기 위함입니다. Linking에서는 바이트코드 검증, static 변수 메모리 할당, 심볼릭 레퍼런스를 다이렉트 레퍼런스로 교체합니다. Initialization에서는 static 블록과 static 변수의 초기값이 실행됩니다.

Part 2 — JVM 심화
Java 8의 Permanent Generation이 왜 Metaspace로 바뀌었나요?

Java 7까지는 클래스 메타데이터가 PermGen(Permanent Generation)이라는 Heap 내부의 고정 크기 영역에 저장되었습니다. 문제는 이 크기가 고정(-XX:MaxPermSize)이어서, 동적으로 클래스를 많이 생성하는 프레임워크(Spring, Hibernate 등)에서 java.lang.OutOfMemoryError: PermGen space가 빈번하게 발생했다는 것입니다.

Java 8부터는 PermGen을 없애고 Metaspace라는 네이티브 메모리 영역으로 대체했습니다. Metaspace는 OS의 네이티브 메모리를 사용하므로 기본적으로 크기 제한이 없고(물론 -XX:MaxMetaspaceSize로 제한 가능), 필요에 따라 자동으로 늘어납니다. 실무에서는 메모리 누수 방지를 위해 MaxMetaspaceSize를 적절히 설정하는 것이 좋습니다.

스레드 덤프와 힙 덤프의 차이점과 활용법을 설명해주세요.

스레드 덤프는 현재 JVM의 모든 스레드 상태를 스냅샷으로 찍은 것입니다. "지금 각 직원이 뭐 하고 있지?"를 확인하는 것과 같습니다. 데드락, CPU 과점유 스레드, 스레드 블로킹 문제를 진단할 때 사용합니다. jstack 명령어나 kill -3으로 생성할 수 있습니다.

힙 덤프는 Heap 메모리에 존재하는 모든 객체의 스냅샷입니다. "사무실에 어떤 물건들이 얼마나 쌓여있지?"를 확인하는 것과 같습니다. 메모리 누수(OOM)를 진단할 때 사용하며, jmap이나 -XX:+HeapDumpOnOutOfMemoryError 옵션으로 생성합니다. Eclipse MAT이나 VisualVM으로 분석합니다.

Part 1 — GC 기본 개념 · Garbage Collection의 원리와 세대별 구조를 이해합니다.
Garbage Collection이란 무엇이고, 왜 필요한가요?

Garbage Collection(GC)은 JVM이 더 이상 참조되지 않는 객체를 자동으로 찾아서 메모리를 회수하는 메커니즘입니다. C/C++에서는 개발자가 malloc/free를 직접 관리해야 하지만, 자바는 GC가 대신 해줍니다. 비유하면, 사무실에서 사용이 끝난 서류를 자동으로 파쇄해주는 청소 로봇과 같습니다.

GC가 필요한 이유는 메모리 누수 방지와 개발 생산성 향상입니다. 수동 메모리 관리에서 흔한 dangling pointer, double free 같은 버그를 원천 차단합니다. 다만 GC에도 비용이 있어서, GC가 동작할 때 애플리케이션이 잠시 멈추는 Stop-The-World(STW)가 발생합니다. 이 STW 시간을 줄이는 것이 GC 튜닝의 핵심입니다.

GC의 세대별(Generational) 구조를 설명해주세요.

JVM Heap은 Young Generation과 Old Generation으로 나뉩니다. 이런 구조를 사용하는 이유는 "대부분의 객체는 금방 죽는다"는 경험적 관찰(Weak Generational Hypothesis) 때문입니다.

Young Generation
Eden | S0 | S1
→ Promotion →
Old Generation
장기 생존 객체
Generational Heap 구조

Young Generation은 새로 생성된 객체가 들어가는 공간으로, Eden과 두 개의 Survivor(S0, S1) 영역으로 구성됩니다. Eden이 가득 차면 Minor GC가 발생하고, 살아남은 객체는 Survivor로 이동합니다. Survivor에서 일정 횟수(Age Threshold, 기본 15)를 버틴 객체가 Old Generation으로 승격(Promotion)됩니다. Old Generation이 가득 차면 Major GC(Full GC)가 발생하는데, 이것이 STW가 길어지는 주범입니다.

꼬리질문 대비

"Survivor 영역이 왜 두 개인가요?" — Minor GC 시 Eden + 활성 Survivor의 살아남은 객체를 빈 Survivor로 복사하는 방식을 씁니다. 이렇게 하면 메모리 단편화(fragmentation)가 발생하지 않고, 연속된 메모리 공간을 확보할 수 있습니다.

GC가 "살아있는 객체"를 어떻게 판별하나요?

현대 JVM은 Reachability Analysis(도달성 분석)를 사용합니다. GC Root라고 불리는 시작점에서 참조 체인을 따라가며, 도달할 수 있는 객체는 살아있고, 도달할 수 없으면 GC 대상입니다.

GC Root가 될 수 있는 것들은 스레드 스택의 지역변수, static 변수, JNI 참조, 활성 스레드 등입니다. 참조 카운팅(Reference Counting) 방식은 순환 참조 문제가 있어 자바에서는 사용하지 않습니다. 예를 들어 객체 A와 B가 서로만 참조하면 카운트가 0이 안 되지만, Reachability 방식에서는 GC Root에서 도달 불가하므로 정상적으로 수거됩니다.

Minor GC와 Major GC(Full GC)의 차이를 설명해주세요.

Minor GC는 Young Generation만 대상으로 하는 GC입니다. Eden이 가득 차면 트리거되며, 대부분의 객체가 이미 죽어있기 때문에 보통 수 ms 이내로 매우 빠릅니다. 살아남은 소수의 객체만 Survivor로 복사하면 되기 때문입니다.

Major GC(Full GC)는 Old Generation을 대상으로 하며, 경우에 따라 전체 Heap을 스캔합니다. Old Generation에는 오래 살아남은 객체가 밀집해 있어 스캔 범위가 넓고 STW 시간이 길어집니다. 실무에서 GC 튜닝의 핵심은 바로 이 Full GC의 빈도와 시간을 줄이는 것입니다.

Serial, Parallel, CMS, G1, ZGC의 차이를 설명해주세요.

각 GC 수집기를 식당의 청소 방식에 비유하면 이해가 쉽습니다.

Serial은 혼자서 청소하는 것이고, Parallel은 여러 명이 한꺼번에 청소하되 가게 문을 닫습니다. CMS는 영업 중에 청소하지만 쓰레기가 구석구석 흩어져(단편화) 결국 한번은 문을 닫아야 합니다. G1은 구역별로 나눠서 쓰레기가 가장 많은 구역(Garbage-First)부터 청소하니 효율적입니다. ZGC는 아예 손님이 거의 느끼지 못하게 청소합니다.

G1 GC의 동작 원리를 자세히 설명해주세요.

G1(Garbage-First) GC는 Heap을 동일 크기의 Region(기본 1~32MB)으로 나누고, 각 Region에 Eden, Survivor, Old, Humongous 역할을 동적으로 부여합니다. 기존의 물리적으로 고정된 세대 구분 대신, 논리적 세대를 사용하는 것이 핵심 혁신입니다.

G1 GC — Region 기반 Heap 레이아웃

G1의 수집 과정은 크게 Young GC와 Mixed GC로 나뉩니다. Young GC는 Eden Region만 수집하며, Mixed GC는 Young + 쓰레기가 많은 Old Region을 함께 수집합니다. -XX:MaxGCPauseMillis(기본 200ms)로 목표 STW 시간을 설정하면, G1이 수집할 Region 수를 자동으로 조절합니다. 또한 Remembered Set(RSet)으로 Region 간 참조를 추적하여, 전체 Heap을 스캔하지 않고도 특정 Region의 참조 관계를 파악합니다.

ZGC는 어떻게 1ms 이하의 STW를 달성하나요?

ZGC의 핵심 기술은 Colored Pointer(컬러 포인터)와 Load Barrier(로드 배리어)입니다. 64비트 포인터의 상위 비트에 객체의 상태(Marked, Remapped, Finalizable 등)를 인코딩하여, 객체를 읽는 순간에 해당 객체의 GC 상태를 즉시 파악합니다.

ZGC는 객체를 이동시키면서도 애플리케이션을 멈추지 않습니다. 객체가 이동되면 이전 주소로 접근하는 스레드가 Load Barrier를 통해 자동으로 새 주소로 리다이렉트됩니다. 이것은 마치 택배 물류센터가 이사해도, 기존 주소로 보낸 택배가 자동 전달되는 것과 같습니다. STW는 GC Root 스캔에만 발생하며, 이는 매우 짧은 시간(보통 1ms 미만)에 완료됩니다. Java 21 LTS에서 Generational ZGC가 도입되어 Young/Old 세대 구분까지 추가되었습니다.

Part 2 — GC 실무 튜닝
실무에서 GC 튜닝은 어떻게 접근하나요?

GC 튜닝의 첫 번째 원칙은 "측정 먼저, 튜닝은 나중에"입니다. -Xlog:gc* (Java 9+) 또는 -verbose:gc로 GC 로그를 활성화하고, GC Viewer나 GCEasy 같은 도구로 분석합니다.

확인해야 할 핵심 지표는 GC 빈도, STW 시간, Heap 사용률 추이입니다. Old Generation 사용량이 GC 후에도 계속 증가하면 메모리 누수를 의심해야 합니다. 보통은 Heap 크기(-Xms, -Xmx)를 적절히 설정하는 것만으로도 대부분의 문제가 해결됩니다. -Xms와 -Xmx를 동일하게 설정하면 Heap 리사이징에 의한 불필요한 GC를 방지할 수 있습니다.

실무 권장 사항으로는, Java 17+ 환경에서 특별한 이유가 없다면 G1 GC(기본값)를 그대로 사용하고, 초저지연이 필요하면 ZGC를 고려합니다. 무작정 옵션을 많이 건드리기보다, 애플리케이션 특성(처리량 vs 지연시간)에 맞는 GC를 선택하고 최소한의 옵션만 조정하는 것이 좋습니다.

GC가 애플리케이션 성능에 미치는 영향을 어떻게 모니터링하나요?

실무에서는 Prometheus + Grafana 조합으로 JVM 메트릭을 실시간 모니터링합니다. Spring Boot라면 Micrometer가 JVM 메트릭을 자동 수집하고, jvm_gc_pause_seconds, jvm_memory_used_bytes 같은 메트릭으로 GC 상태를 대시보드에서 확인합니다.

주의 깊게 볼 지표는 GC Pause 시간의 P99(99퍼센타일), GC 발생 빈도, Heap 점유율 추이입니다. Full GC가 잦아지거나 STW 시간이 급격히 늘어나면 알람을 설정해두는 것이 좋습니다. 또한 jstat -gcutil 명령어로 각 세대의 사용률을 실시간으로 확인할 수 있습니다.

Part 1 — HTTPS 기본
HTTPS란 무엇이고, HTTP와 무엇이 다른가요?

HTTPS(HTTP Secure)는 HTTP에 TLS(Transport Layer Security) 암호화 계층을 추가한 프로토콜입니다. 비유하면, HTTP가 엽서(내용이 보임)라면, HTTPS는 밀봉된 등기우편(내용을 볼 수 없고, 보낸 사람도 인증됨)입니다.

HTTPS가 보장하는 세 가지는 기밀성(데이터 암호화), 무결성(데이터 변조 방지), 인증(서버가 진짜 그 서버인지 확인)입니다. 포트는 HTTP가 80, HTTPS가 443을 기본으로 사용합니다.

TLS Handshake 과정을 설명해주세요.

TLS Handshake는 클라이언트와 서버가 안전한 통신 채널을 수립하는 협상 과정입니다. TLS 1.2 기준으로 설명하겠습니다.

TLS 1.2 Handshake 흐름

핵심 포인트는 비대칭키로 대칭키를 안전하게 교환한다는 것입니다. 비대칭 암호화(RSA, ECDHE)는 느리기 때문에 키 교환에만 쓰고, 실제 데이터 전송은 빠른 대칭키(AES 등)로 합니다.

꼬리질문 대비

"TLS 1.3에서 달라진 점은?" — Handshake가 1-RTT로 줄었습니다(1.2는 2-RTT). 불안전한 알고리즘(RSA 키 교환, SHA-1 등)을 제거하고, 0-RTT Resumption도 지원하여 이전에 연결했던 서버에는 Handshake 없이 바로 데이터를 보낼 수 있습니다(보안 Trade-off 존재).

인증서(Certificate)는 어떤 역할을 하고, 어떻게 검증하나요?

인증서는 "이 서버가 진짜 naver.com이 맞다"는 것을 보증하는 디지털 신분증입니다. CA(Certificate Authority)라는 신뢰할 수 있는 기관이 발급하며, 인증서에는 서버의 공개키, 도메인 이름, 발급자(CA), 유효기간, CA의 디지털 서명이 포함됩니다.

검증 과정은 인증서 체인(Certificate Chain)을 따라갑니다. 서버 인증서 → 중간 CA 인증서 → Root CA 인증서까지 체인을 따라가며, 각 단계에서 상위 CA의 공개키로 하위 인증서의 서명을 검증합니다. Root CA의 인증서는 브라우저나 OS에 사전 설치(Trust Store)되어 있어, 최종적으로 신뢰의 기반이 됩니다. 실무에서 Let's Encrypt를 많이 사용하는데, 이는 무료 CA로 90일 단위의 자동 갱신 인증서를 제공합니다.

Part 2 — HTTPS 심화
대칭키와 비대칭키 암호화의 차이, 그리고 HTTPS에서의 역할은?
구분대칭키 (AES)비대칭키 (RSA/ECDHE)
키 개수1개 (같은 키로 암/복호화)2개 (공개키 + 개인키)
속도빠름느림 (100~1000배)
문제점키 교환이 어려움성능 부담
HTTPS 용도실제 데이터 암호화세션키(대칭키) 교환

HTTPS는 두 방식의 장점을 결합한 하이브리드 방식입니다. 비대칭키의 보안성으로 안전하게 세션키를 교환하고, 대칭키의 속도로 실제 데이터를 빠르게 주고받습니다. 면접관이 "왜 처음부터 대칭키만 쓰면 안 되나요?"라고 물으면, 대칭키를 안전하게 전달할 방법이 없기 때문이라고 답하면 됩니다. 네트워크 상에서 키를 평문으로 보내면 중간에서 탈취될 수 있습니다.

HTTPS를 쓰면 성능 저하가 있나요? 어떻게 최적화하나요?

TLS Handshake로 인해 초기 연결 시 RTT가 추가됩니다. TLS 1.2는 2-RTT, TLS 1.3은 1-RTT가 필요합니다. 또한 암/복호화 연산에 CPU 리소스가 소모됩니다. 하지만 현대 하드웨어에서는 AES-NI 명령어 지원으로 암호화 오버헤드가 거의 무시할 수준입니다.

최적화 방법으로는 TLS Session Resumption(이전 세션 정보 재사용으로 Handshake 생략), HTTP/2 사용(TLS 위에서 멀티플렉싱으로 성능 향상), OCSP Stapling(인증서 유효성 확인을 서버가 미리 해서 클라이언트 부담 감소), HTTP Keep-Alive(연결 재사용)가 있습니다. 실무에서 HTTPS 성능 문제를 체감하는 경우는 거의 없으며, 보안 이점이 압도적입니다.

Part 1 — Redis 클러스터 기본
Redis 클러스터란 무엇이고, 왜 필요한가요?

Redis 클러스터는 데이터를 여러 Redis 노드에 자동으로 분산 저장하는 수평 확장(Scale-Out) 솔루션입니다. 단일 Redis 인스턴스는 하나의 서버 메모리에 제한되지만, 클러스터를 사용하면 여러 노드의 메모리를 합쳐 더 큰 데이터셋을 처리할 수 있습니다.

비유하면, 단일 Redis는 하나의 금고이고, Redis 클러스터는 여러 금고가 연결된 은행 볼트 시스템입니다. 각 금고(노드)에 특정 번호대의 자산(데이터)을 분산 보관하고, 어떤 금고에 있는지는 시스템이 자동으로 안내해줍니다.

Redis 클러스터의 데이터 분산(Hash Slot) 방식을 설명해주세요.

Redis 클러스터는 16,384개의 해시 슬롯(Hash Slot)을 사용합니다. 키의 해시 슬롯은 CRC16(key) mod 16384로 계산됩니다. 각 마스터 노드가 해시 슬롯의 일부를 담당합니다.

Node A
슬롯 0~5460
Node B
슬롯 5461~10922
Node C
슬롯 10923~16383
3개 마스터 노드의 해시 슬롯 분배

클라이언트가 키에 접근하면, 해시 슬롯을 계산하여 해당 노드로 요청합니다. 만약 잘못된 노드에 요청하면 MOVED 리다이렉션 응답을 받고, 올바른 노드로 재요청합니다. 스마트 클라이언트(Jedis, Lettuce 등)는 슬롯-노드 매핑을 캐싱하여 대부분 한 번에 올바른 노드로 요청합니다.

꼬리질문 대비

"멀티 키 연산은 어떻게 하나요?" — 같은 슬롯에 있는 키만 멀티 키 연산이 가능합니다. 이를 위해 Hash Tag를 사용합니다. 예: {user:1000}.profile, {user:1000}.session처럼 중괄호 안의 부분만 해시 계산에 사용되어 같은 슬롯에 배치됩니다.

Redis 클러스터에서 장애 감지와 페일오버는 어떻게 동작하나요?

Redis 클러스터는 Gossip 프로토콜을 사용하여 노드 간 상태를 주기적으로 교환합니다. 각 노드는 다른 노드에 PING을 보내고, 일정 시간(cluster-node-timeout) 내에 응답이 없으면 해당 노드를 PFAIL(Probable Fail)로 표시합니다.

과반수 이상의 마스터 노드가 특정 노드를 PFAIL로 보고하면, 그 노드는 FAIL 상태가 됩니다. FAIL 상태가 되면 해당 마스터의 레플리카(Slave) 중 하나가 자동으로 마스터로 승격됩니다. 이 과정에서 레플리카는 다른 마스터들에게 투표를 요청하고, 과반 이상의 동의를 받으면 새 마스터가 됩니다. 전체 과정은 보통 수 초 이내에 완료됩니다.

주의할 점은 Redis 클러스터는 비동기 복제를 사용하므로, 마스터가 쓰기를 받은 직후 장애가 나면 레플리카에 아직 복제되지 않은 데이터가 유실될 수 있습니다. 이것이 Redis 클러스터가 강한 일관성(Strong Consistency)을 보장하지 않는 이유입니다.

Sentinel은 "수문장"으로 마스터 장애를 감지하고 슬레이브를 승격시키는 역할에 집중합니다. 데이터는 여전히 단일 마스터에 있으므로 한 서버의 메모리 한계가 곧 데이터 한계입니다. 반면 클러스터는 데이터를 나누어 저장하므로 노드를 추가하면 용량과 처리량이 선형으로 증가합니다.

Redis 클러스터 운영 시 주의사항은?

첫째, 키 설계가 중요합니다. 특정 슬롯에 데이터가 집중되면(Hot Key) 부하가 편중됩니다. Hash Tag를 사용할 때도 지나치게 많은 키를 같은 슬롯에 몰지 않도록 주의해야 합니다.

둘째, 네트워크 파티션에 대비해야 합니다. cluster-node-timeout을 너무 짧게 설정하면 네트워크 순간 지연에도 불필요한 페일오버가 발생합니다. 보통 5~15초가 적절합니다. 셋째, 리샤딩(Resharding) 시 데이터 마이그레이션이 발생하므로 피크 시간을 피해야 합니다. 넷째, 클러스터 모드에서는 SELECT(DB 전환)가 불가하고 DB 0만 사용 가능하며, Lua 스크립트에서 접근하는 모든 키가 같은 슬롯에 있어야 합니다.

Part 1 — Docker 이미지 기본
Docker 이미지란 무엇이고, 컨테이너와 어떤 관계인가요?

Docker 이미지는 애플리케이션 실행에 필요한 모든 것(코드, 런타임, 라이브러리, 환경설정)을 담은 불변의 템플릿입니다. 컨테이너는 이미지를 기반으로 실행된 인스턴스입니다. 비유하면, 이미지는 붕어빵 틀이고, 컨테이너는 그 틀로 구워낸 붕어빵입니다. 하나의 이미지로 여러 컨테이너를 만들 수 있습니다.

이미지는 읽기 전용(Read-Only)이며, 컨테이너가 실행되면 이미지 위에 읽기-쓰기 레이어가 추가됩니다. 컨테이너에서 파일을 수정해도 원본 이미지에는 영향을 주지 않습니다.

Docker 이미지의 레이어(Layer) 구조를 설명해주세요.

Docker 이미지는 여러 개의 읽기 전용 레이어가 쌓인 구조입니다. Dockerfile의 각 명령어(FROM, RUN, COPY 등)가 하나의 레이어를 생성합니다. 이 레이어들은 Union File System(OverlayFS)으로 합쳐져 마치 하나의 파일시스템처럼 보입니다.

Docker 이미지 레이어 구조

레이어 구조의 최대 장점은 캐싱과 공유입니다. 같은 베이스 이미지를 사용하는 여러 이미지가 하위 레이어를 공유하므로 디스크를 절약합니다. 또한 빌드 시 변경되지 않은 레이어는 캐시에서 재사용하여 빌드 시간을 단축합니다.

꼬리질문 대비

"Dockerfile에서 레이어 캐시를 효율적으로 쓰려면?" — 자주 변경되지 않는 명령을 위에, 자주 변경되는 명령을 아래에 배치합니다. 예를 들어, COPY requirements.txt → RUN pip install → COPY . (소스코드) 순서로 하면, 소스코드만 바뀌었을 때 의존성 설치 레이어는 캐시됩니다.

Part 2 — Docker 이미지 최적화
Docker 이미지 크기를 줄이는 방법은?

이미지 크기 최적화는 배포 속도와 보안 공격 표면 감소 모두에 중요합니다.

첫째, 멀티 스테이지 빌드(Multi-Stage Build)를 활용합니다. 빌드에 필요한 도구(JDK, Maven)와 실행에 필요한 환경(JRE)을 분리하여, 최종 이미지에는 실행에 필요한 것만 담습니다. 둘째, 경량 베이스 이미지를 사용합니다. ubuntu 대신 alpine(약 5MB)이나 distroless를 쓰면 이미지 크기를 극적으로 줄일 수 있습니다. 셋째, RUN 명령을 합치고 불필요한 파일을 같은 레이어에서 삭제합니다. RUN apt-get update && apt-get install -y ... && rm -rf /var/lib/apt/lists/*처럼 하나의 RUN에서 설치와 캐시 정리를 함께 합니다.

넷째, .dockerignore 파일로 빌드 컨텍스트에서 불필요한 파일(.git, node_modules, *.log)을 제외합니다.

Docker 이미지 보안은 어떻게 관리하나요?

이미지 보안의 핵심은 최소 권한 원칙과 취약점 스캔입니다. 먼저, 비루트 사용자로 컨테이너를 실행합니다. Dockerfile에 USER 명령어로 비루트 사용자를 지정합니다. 다음으로, Trivy, Snyk, Docker Scout 같은 도구로 이미지의 취약점(CVE)을 정기적으로 스캔합니다.

또한 이미지 태그를 고정해야 합니다. FROM python:latest 대신 FROM python:3.11.7-slim처럼 구체적 버전을 지정해야 빌드 재현성과 보안을 확보할 수 있습니다. 실무에서는 Private Registry(Harbor, ECR, GCR 등)를 운영하고, 이미지 서명과 검증(Docker Content Trust)을 적용하는 것이 권장됩니다.

멀티 스테이지 빌드를 예시와 함께 설명해주세요.

멀티 스테이지 빌드는 빌드 환경과 실행 환경을 분리하여 최종 이미지를 가볍게 만드는 기법입니다. Java 애플리케이션의 예를 들면:

# Stage 1: 빌드
FROM maven:3.9-eclipse-temurin-17 AS builder
WORKDIR /app
COPY pom.xml .
RUN mvn dependency:go-offline
COPY src ./src
RUN mvn package -DskipTests

# Stage 2: 실행 (빌드 도구 없음)
FROM eclipse-temurin:17-jre-alpine
WORKDIR /app
COPY --from=builder /app/target/*.jar app.jar
USER 1001
ENTRYPOINT ["java", "-jar", "app.jar"]

Stage 1(builder)에서는 Maven, JDK 등 빌드 도구가 포함된 큰 이미지를 쓰지만, Stage 2에서는 JRE-alpine만 사용합니다. 최종 이미지에는 빌드 도구가 포함되지 않으므로 이미지 크기가 수백 MB에서 수십 MB로 줄어듭니다.

Part 1 — TCP/UDP 기본
TCP와 UDP의 핵심 차이를 설명해주세요.

TCP(Transmission Control Protocol)는 연결 지향적, 신뢰성 보장 프로토콜이고, UDP(User Datagram Protocol)는 비연결, 최선 노력(Best-Effort) 프로토콜입니다.

비유하면, TCP는 등기우편입니다 — 배달 확인, 순서 보장, 분실 시 재발송. UDP는 전단지 뿌리기 — 빠르지만 받았는지 확인하지 않고, 순서도 보장 안 됩니다.

TCP 3-Way Handshake를 설명해주세요.

TCP 3-Way Handshake는 클라이언트와 서버가 양방향 통신 가능 여부를 확인하고 연결을 수립하는 과정입니다. 전화에 비유하면, "여보세요?" → "네, 잘 들려요!" → "저도 잘 들립니다!" 의 세 단계입니다.

TCP 3-Way Handshake

①번에서 클라이언트가 SYN(Synchronize) 패킷으로 "연결하고 싶다"는 신호를 보냅니다. ②번에서 서버가 SYN+ACK로 "알겠다, 나도 준비됐다"고 응답합니다. ③번에서 클라이언트가 ACK로 "확인했다"고 답하면 연결이 수립됩니다. 이 과정이 3단계인 이유는 양방향 통신이 모두 가능한지 확인하기 위함입니다.

TCP 4-Way Handshake(연결 종료)를 설명해주세요.

연결 종료는 4-Way Handshake로 진행됩니다. 3-Way가 아닌 4-Way인 이유는, 한쪽이 보내기를 끝내도 상대방은 아직 보낼 데이터가 남아있을 수 있기 때문입니다(Half-Close).

①클라이언트 → FIN("나는 보낼 거 다 보냈어") → ②서버 → ACK("알겠어, 근데 나는 아직 보낼 거 있어") → ③서버 → FIN("나도 다 보냈어") → ④클라이언트 → ACK("확인"). 마지막에 클라이언트는 TIME_WAIT 상태로 일정 시간(보통 2×MSL = 60초) 대기합니다.

꼬리질문 대비

"TIME_WAIT는 왜 필요한가요?" — 두 가지 이유입니다. 첫째, 마지막 ACK가 유실됐을 때 서버가 FIN을 재전송하면 응답할 수 있어야 합니다. 둘째, 이전 연결의 지연 패킷이 새 연결에서 오인되는 것을 방지합니다. 서버에 TIME_WAIT가 많이 쌓이면 포트 고갈 문제가 발생할 수 있는데, 이때 SO_REUSEADDR 옵션이나 net.ipv4.tcp_tw_reuse 커널 파라미터로 완화할 수 있습니다.

Part 2 — TCP 심화
TCP의 흐름 제어(Flow Control)와 혼잡 제어(Congestion Control)의 차이는?

흐름 제어는 수신자의 처리 속도에 맞춰 송신 속도를 조절하는 것이고, 혼잡 제어는 네트워크의 혼잡 상태에 맞춰 송신 속도를 조절하는 것입니다. 비유하면, 흐름 제어는 "상대방이 얼마나 빨리 먹을 수 있는지" 보고 음식을 내는 속도를 조절하는 것이고, 혼잡 제어는 "배달 도로가 막히지 않게" 배달량을 조절하는 것입니다.

흐름 제어는 Sliding Window 방식을 사용합니다. 수신자가 TCP 헤더의 Window Size 필드에 자신의 여유 버퍼 크기를 알려주면, 송신자는 그만큼만 데이터를 보냅니다. 혼잡 제어는 Slow Start → Congestion Avoidance → Fast Retransmit → Fast Recovery 단계로 동작합니다. 처음엔 윈도우를 지수적으로 늘리다가(Slow Start), 임계점(ssthresh) 이후 선형으로 늘리고(Congestion Avoidance), 패킷 유실이 감지되면 윈도우를 줄입니다.

UDP를 쓰면서도 신뢰성을 확보하는 방법은?

UDP 자체에는 신뢰성이 없지만, 애플리케이션 레벨에서 신뢰성을 구현할 수 있습니다. 대표적인 프로토콜이 QUIC입니다. QUIC은 Google이 개발한 UDP 기반 프로토콜로, HTTP/3의 전송 계층으로 채택되었습니다.

QUIC은 UDP 위에서 연결 수립, 순서 보장, 재전송, 혼잡 제어를 모두 구현하면서도, TCP의 단점인 Head-of-Line Blocking을 해결합니다. TCP는 하나의 패킷이 유실되면 뒤의 모든 패킷이 대기해야 하지만, QUIC은 스트림 단위로 독립적이어서 한 스트림의 유실이 다른 스트림에 영향을 주지 않습니다. 또한 TLS 1.3을 내장하여 0-RTT 연결 수립이 가능하고, 연결 ID 기반이라 Wi-Fi에서 LTE로 전환할 때도 연결이 유지됩니다.

TCP Keep-Alive와 HTTP Keep-Alive의 차이는?

이름이 같아서 혼동하기 쉽지만, 완전히 다른 계층에서 동작합니다.

TCP Keep-Alive는 운영체제(전송 계층)에서 동작하며, 유휴 상태인 TCP 연결이 아직 살아있는지 확인하는 프로브 패킷을 주기적으로 보냅니다. 기본적으로 2시간 동안 데이터가 없으면 프로브를 보내는데, 이는 주로 죽은 연결을 감지하기 위함입니다.

HTTP Keep-Alive(= Persistent Connection)는 애플리케이션 계층에서 동작하며, 하나의 TCP 연결 위에서 여러 HTTP 요청/응답을 재사용하는 것입니다. HTTP/1.0에서는 매 요청마다 TCP 연결을 새로 맺었지만, HTTP/1.1부터 Keep-Alive가 기본이 되어 연결 수립 오버헤드가 크게 줄었습니다.
