N+1 문제 & Lazy Loading 면접 가이드

JPA/Hibernate 기반 · 코드 레벨 구체적 답변 · 꼬리질문 대비

N+1 문제가 무엇인지 설명해주세요

N+1 문제는 연관된 엔티티를 조회할 때 부모 1건의 쿼리 + 자식 N건의 추가 쿼리가 발생하는 현상입니다. 비유하자면, 학교에서 반(Team) 목록을 조회한 뒤 "각 반의 학생(Member) 명단"을 하나씩 따로 물어보는 셈입니다. 반이 100개면 쿼리가 101번 나갑니다.

JPA에서 가장 흔하게 발생하는 시나리오를 보겠습니다. Team과 Member가 1:N 관계일 때:

@Entity
public class Team {
    @Id @GeneratedValue
    private Long id;
    private String name;

    @OneToMany(mappedBy = "team")  // 기본 전략: LAZY
    private List<Member> members = new ArrayList<>();
}

// 1번 쿼리: SELECT * FROM team
List<Team> teams = teamRepository.findAll();

// N번 쿼리: 각 team마다 SELECT * FROM member WHERE team_id = ?
for (Team team : teams) {
    System.out.println(team.getMembers().size()); // 여기서 LAZY 로딩 발동!
}

핵심은 LAZY든 EAGER든 N+1은 발생할 수 있다는 점입니다. LAZY는 실제 접근 시점에, EAGER는 findAll() 시점에 추가 쿼리가 나갑니다. EAGER로 바꾼다고 해결되는 문제가 아닙니다.

Lazy Loading과 Eager Loading의 차이를 설명해주세요

Lazy Loading은 연관 엔티티를 실제로 접근하는 시점까지 로딩을 미루는 전략이고, Eager Loading은 부모를 조회하는 순간 연관 엔티티도 즉시 함께 가져오는 전략입니다. Hibernate에서 Lazy는 프록시 객체를 넣어두고, getter를 호출하는 순간 실제 쿼리를 날립니다.

구분Lazy LoadingEager Loading
로딩 시점실제 접근(getter) 시점부모 조회 시점 즉시
프록시프록시 객체 사용실제 객체 즉시 할당
JPA 기본값@OneToMany, @ManyToMany@ManyToOne, @OneToOne
N+1 발생 시점루프 등에서 getter 호출 시findAll() 등 JPQL 실행 시
불필요한 쿼리접근 안 하면 쿼리 없음항상 JOIN/추가 쿼리 발생

JPA 실무에서는 기본적으로 모든 연관관계를 LAZY로 설정하고, 필요한 시점에 Fetch Join이나 EntityGraph로 한번에 가져오는 것이 정석입니다. @ManyToOne도 기본이 EAGER이므로, 반드시 fetch = FetchType.LAZY로 변경해야 합니다.

@ManyToOne(fetch = FetchType.LAZY)  // 기본이 EAGER이므로 반드시 LAZY로!
@JoinColumn(name = "team_id")
private Team team;
꼬리질문 대비: "Lazy Loading이면 트랜잭션 밖에서 접근하면 어떻게 되나요?" — LazyInitializationException이 발생합니다. 영속성 컨텍스트가 닫힌 상태에서 프록시를 초기화하려 하기 때문입니다. 이것이 OSIV(Open Session In View) 논쟁으로 이어집니다.
Hibernate의 프록시 객체는 어떻게 동작하나요?

Hibernate는 Lazy 로딩 대상 엔티티에 대해 바이트코드를 조작하여 프록시 클래스를 런타임에 생성합니다. 예를 들어 Team 엔티티에 대해 Team$HibernateProxy$Xyz 같은 서브클래스를 만듭니다. 이 프록시는 내부에 target(실제 엔티티 참조)과 initialized(초기화 여부) 플래그를 가지고 있습니다.

프록시의 어떤 메서드(getId() 제외)를 호출하면, initialized가 false인 경우 영속성 컨텍스트를 통해 실제 DB 쿼리를 실행하고, target에 결과를 채운 뒤 메서드를 위임합니다. getId()는 프록시가 이미 ID를 알고 있으므로 쿼리를 발생시키지 않습니다.

Team team = member.getTeam();       // 프록시 반환, 쿼리 X
Long teamId = team.getId();          // ID는 이미 있으므로 쿼리 X
String name = team.getName();        // 이 순간 SELECT 쿼리 발생!

// 프록시 확인 방법
System.out.println(team.getClass()); // Team$HibernateProxy$... 출력
Hibernate.isInitialized(team);       // false → 아직 초기화 안 됨

면접관이 여기서 더 파고들 수 있는데요, 프록시의 equals/hashCode 이슈입니다. 프록시와 실제 엔티티를 비교할 때 instanceof 대신 getClass()를 쓰면 false가 나올 수 있습니다. 그래서 엔티티의 equals에는 반드시 instanceof를 사용해야 합니다.

EAGER로 바꾸면 N+1이 해결되지 않는 이유를 코드로 설명해주세요

많은 분들이 "EAGER로 바꾸면 한 번에 가져오니까 N+1이 해결되지 않나?"라고 생각하는데, JPQL을 사용하는 순간 EAGER도 N+1이 발생합니다. 이유는 JPQL이 먼저 SQL로 번역된 후, Hibernate가 결과를 보고 "어? 이 엔티티의 연관관계가 EAGER네, 추가 로딩해야지" 하고 사후에 추가 쿼리를 날리기 때문입니다.

// EAGER로 설정된 상태
@OneToMany(mappedBy = "team", fetch = FetchType.EAGER)
private List<Member> members;

// Spring Data JPA의 findAll() → JPQL "SELECT t FROM Team t"
List<Team> teams = teamRepository.findAll();

// 실행되는 쿼리:
// 1) SELECT * FROM team                          -- JPQL이 먼저 실행
// 2) SELECT * FROM member WHERE team_id = 1      -- EAGER이니 즉시 추가
// 3) SELECT * FROM member WHERE team_id = 2      -- ...
// ... N번 반복

반면 em.find(Team.class, 1L)처럼 EntityManager의 find()를 사용하면, Hibernate가 최적화하여 LEFT JOIN으로 한 방 쿼리를 만들어줍니다. 하지만 findAll()이나 커스텀 JPQL에서는 이 최적화가 적용되지 않습니다. 결론적으로 EAGER는 N+1을 해결하지 못할 뿐 아니라, 제어할 수 없는 시점에 쿼리가 나가므로 더 위험합니다.

면접 핵심 포인트: "EAGER는 N+1 해결책이 아닙니다. 오히려 어디서 쿼리가 나가는지 예측할 수 없게 만들어 더 위험합니다."
Fetch Join으로 N+1을 해결하는 코드를 보여주세요

Fetch Join은 JPQL에서 JOIN FETCH 키워드를 사용하여, 연관 엔티티를 한 방 쿼리로 즉시 로딩하는 방법입니다. 일반 JOIN과 달리 SELECT 절에 연관 엔티티 데이터까지 포함시키기 때문에 추가 쿼리가 발생하지 않습니다.

// Repository
public interface TeamRepository extends JpaRepository<Team, Long> {

    @Query("SELECT t FROM Team t JOIN FETCH t.members")
    List<Team> findAllWithMembers();
}

// 실행되는 쿼리: 딱 1번!
// SELECT t.*, m.* FROM team t
// INNER JOIN member m ON t.id = m.team_id

List<Team> teams = teamRepository.findAllWithMembers();
for (Team team : teams) {
    team.getMembers().size(); // 추가 쿼리 없음! 이미 로딩됨
}

하지만 Fetch Join에는 중요한 제약 사항이 있습니다. 컬렉션(1:N) Fetch Join 시 데이터 뻥튀기(카테시안 곱)가 발생합니다. Team 1개에 Member 3명이면 Team 행이 3번 반복됩니다. 이를 해결하려면 DISTINCT를 붙여야 합니다.

@Query("SELECT DISTINCT t FROM Team t JOIN FETCH t.members")
List<Team> findAllWithMembers();

// Hibernate 6(Spring Boot 3)부터는 DISTINCT가 자동 적용됩니다
// Hibernate 5에서는 반드시 명시해야 합니다
꼬리질문 대비: "컬렉션 Fetch Join을 2개 이상 할 수 있나요?" — 안 됩니다. MultipleBagFetchException이 발생합니다. 2개 이상의 컬렉션을 동시에 페치하면 카테시안 곱이 기하급수적으로 커지기 때문입니다. 하나는 Fetch Join, 나머지는 @BatchSize나 default_batch_fetch_size로 해결합니다.
@EntityGraph로 N+1을 해결하는 방법을 설명해주세요

@EntityGraph는 JPQL을 작성하지 않고도 Fetch Join과 같은 효과를 낼 수 있는 방법입니다. 어노테이션 기반이라 가독성이 좋고, Spring Data JPA의 기본 메서드에도 적용할 수 있다는 장점이 있습니다.

// 방법 1: attributePaths로 직접 지정
@EntityGraph(attributePaths = {"members"})
@Query("SELECT t FROM Team t")
List<Team> findAllWithMembers();

// 방법 2: 기본 findAll()에 적용
@EntityGraph(attributePaths = {"members"})
@Override
List<Team> findAll();

// 방법 3: 엔티티에 Named EntityGraph 정의
@NamedEntityGraph(
    name = "Team.withMembers",
    attributeNodes = @NamedAttributeNode("members")
)
@Entity
public class Team { ... }

// Repository에서 참조
@EntityGraph(value = "Team.withMembers")
List<Team> findAll();

EntityGraph와 Fetch Join의 차이점이 있습니다. EntityGraph는 LEFT OUTER JOIN을 사용하고, Fetch Join의 기본 JOIN은 INNER JOIN입니다. 즉 EntityGraph는 멤버가 없는 팀도 결과에 포함하지만, Fetch Join은 기본적으로 멤버가 있는 팀만 반환합니다. 필요에 따라 JOIN FETCH t.members m을 LEFT JOIN FETCH로 변경할 수 있습니다.

Fetch Join과 페이징을 함께 쓸 수 없는 이유는?

컬렉션(1:N) Fetch Join에 Pageable을 적용하면, Hibernate가 경고 로그를 남기고 메모리에서 페이징합니다. "HHH000104: firstResult/maxResults specified with collection fetch; applying in memory!"라는 로그가 나옵니다.

이유는 데이터 뻥튀기 때문입니다. Team 1개에 Member 3명이면 DB에서 3행이 나오는데, 이걸 LIMIT/OFFSET으로 자르면 Team 기준의 페이징이 깨집니다. 그래서 Hibernate는 전체 데이터를 다 가져온 뒤 메모리에서 페이징합니다 — 데이터가 많으면 OOM(OutOfMemory) 위험이 있습니다.

// 위험! 메모리 페이징 발생
@Query("SELECT t FROM Team t JOIN FETCH t.members")
Page<Team> findAllWithMembers(Pageable pageable);

// 올바른 해결법: @BatchSize + 페이징
// 1단계: Team만 페이징 조회 (쿼리 1번)
@Query("SELECT t FROM Team t")
Page<Team> findAllTeams(Pageable pageable);

// 2단계: application.yml에서 batch_size 설정
// spring.jpa.properties.hibernate.default_batch_fetch_size: 100

// 결과: Team 페이징 1번 + Member IN 쿼리 1번 = 총 2번!
핵심: 컬렉션 Fetch Join + 페이징 = 절대 X. 대안은 default_batch_fetch_size입니다.
default_batch_fetch_size가 내부적으로 어떻게 동작하나요?

default_batch_fetch_size는 Lazy 로딩이 발생할 때, 한 건씩 쿼리를 날리는 대신 IN 절을 사용해 한 번에 여러 건을 조회하는 설정입니다. 비유하자면, 편의점에서 물건을 하나씩 계산하지 않고 장바구니에 모아서 한 번에 계산하는 것과 같습니다.

# application.yml
spring:
  jpa:
    properties:
      hibernate:
        default_batch_fetch_size: 100

// batch_size = 100일 때의 쿼리 변화

// Before (N+1): 팀 100개 → 쿼리 101번
SELECT * FROM member WHERE team_id = 1;
SELECT * FROM member WHERE team_id = 2;
...
SELECT * FROM member WHERE team_id = 100;

// After (Batch): 팀 100개 → 쿼리 2번
SELECT * FROM team;
SELECT * FROM member WHERE team_id IN (1,2,3,...,100);

만약 팀이 250개이고 batch_size가 100이면, Member 조회 쿼리가 100+100+50으로 3번만 나갑니다. 보통 100~1000 사이로 설정하며, 너무 크면 IN 절의 파라미터가 많아져 DB 부하가 생길 수 있습니다. 개별 엔티티에만 적용하려면 @BatchSize(size = 100) 어노테이션을 사용할 수도 있습니다.

@BatchSize(size = 100)  // 이 컬렉션만 배치 로딩 적용
@OneToMany(mappedBy = "team")
private List<Member> members;
N+1 해결 전략을 상황별로 정리해주세요

N+1 해결은 "무조건 Fetch Join"이 아니라 상황에 따라 최적의 전략을 선택하는 것이 중요합니다. 아래 표로 정리하겠습니다.

상황추천 전략이유
단일 컬렉션, 페이징 XFetch Join가장 직관적, 한 방 쿼리
단일 컬렉션, 페이징 OBatchSizeFetch Join+페이징은 메모리 페이징 위험
다중 컬렉션 (2개↑)1개 Fetch Join + 나머지 BatchSizeMultipleBagFetchException 방지
@ManyToOne / @OneToOneFetch Join카테시안 곱 없음, 안전하게 여러 개 가능
API 응답에 일부 필드만DTO 직접 조회 (Projection)엔티티 전체 로딩 자체를 안 함
글로벌 디폴트default_batch_fetch_size누락된 N+1도 안전망으로 잡아줌

실무에서 가장 많이 쓰는 조합은 "기본 LAZY + default_batch_fetch_size: 100 + 핫 쿼리에 Fetch Join"입니다. batch_size를 글로벌로 깔아두면 실수로 N+1이 발생해도 최소한의 안전망이 됩니다.

// 실무 최적 조합 예시

// 1. 글로벌 안전망: application.yml
spring.jpa.properties.hibernate.default_batch_fetch_size: 100

// 2. 핫 API에는 Fetch Join 최적화
@Query("SELECT DISTINCT t FROM Team t "
     + "JOIN FETCH t.members "
     + "WHERE t.status = :status")
List<Team> findActiveTeamsWithMembers(@Param("status") Status status);

// 3. DTO Projection으로 필요한 것만
@Query("SELECT new com.example.dto.TeamSummary(t.id, t.name, SIZE(t.members)) "
     + "FROM Team t")
List<TeamSummary> findTeamSummaries();
QueryDSL에서 Fetch Join은 어떻게 작성하나요?

QueryDSL은 타입 안전한 동적 쿼리에 강점이 있고, Fetch Join도 메서드 체이닝으로 직관적으로 작성할 수 있습니다.

public List<Team> findTeamsWithMembers(String teamName) {
    return queryFactory
        .selectFrom(team)
        .leftJoin(team.members, member).fetchJoin()   // 핵심!
        .where(
            teamNameContains(teamName)  // 동적 조건
        )
        .distinct()
        .fetch();
}

// @ManyToOne Fetch Join도 가능
public List<Member> findMembersWithTeam() {
    return queryFactory
        .selectFrom(member)
        .join(member.team, team).fetchJoin()
        .fetch();
}

주의할 점은 QueryDSL에서도 컬렉션 Fetch Join + 페이징은 동일하게 위험하다는 것입니다. .offset()과 .limit()을 Fetch Join과 함께 쓰면 Hibernate가 메모리 페이징합니다. 이때도 마찬가지로 BatchSize 전략을 사용해야 합니다.

운영 중 N+1을 어떻게 탐지하나요?

실무에서 N+1은 개발 단계에서 놓치고 운영에서 터지는 경우가 많습니다. 3가지 레벨에서 탐지할 수 있습니다.

첫째, 개발 시점에는 Hibernate의 SQL 로그를 켜고 쿼리 수를 세는 것이 가장 기본입니다.

# application.yml - 개발 환경용
logging:
  level:
    org.hibernate.SQL: DEBUG               # 실행 쿼리
    org.hibernate.orm.jdbc.bind: TRACE     # 바인딩 파라미터 (Hibernate 6)

둘째, 테스트 코드에서 쿼리 카운트를 검증할 수 있습니다. hibernate-testing 모듈이나 커스텀 유틸을 사용합니다.

// 테스트에서 쿼리 카운트 검증 (spring-data-jpa 기반)
@Test
void N플러스1이_발생하지_않는다() {
    // given
    var statistics = entityManager.unwrap(Session.class)
            .getSessionFactory().getStatistics();
    statistics.setStatisticsEnabled(true);
    statistics.clear();

    // when
    List<Team> teams = teamRepository.findAllWithMembers();
    teams.forEach(t -> t.getMembers().size()); // 강제 접근

    // then
    long queryCount = statistics.getPrepareStatementCount();
    assertThat(queryCount).isLessThanOrEqualTo(2L); // 최대 2번 이내
}

셋째, 운영 환경에서는 p6spy나 spring-data-jpa의 slow query log, 혹은 APM 도구(Datadog, Pinpoint 등)로 특정 API에서 쿼리가 비정상적으로 많이 나가는 것을 모니터링합니다.

LazyInitializationException은 왜 발생하고 어떻게 해결하나요?

LazyInitializationException은 영속성 컨텍스트(Session)가 닫힌 상태에서 Lazy 프록시를 초기화하려 할 때 발생합니다. 가장 흔한 상황은 Service 계층에서 엔티티를 반환하고, Controller나 View에서 Lazy 필드에 접근하는 경우입니다.

// 문제 발생 코드
@Service
@Transactional(readOnly = true)
public class TeamService {
    public Team getTeam(Long id) {
        return teamRepository.findById(id).orElseThrow();
        // 여기서 트랜잭션 종료 → 영속성 컨텍스트 닫힘
    }
}

@RestController
public class TeamController {
    public TeamResponse getTeam(Long id) {
        Team team = teamService.getTeam(id);
        team.getMembers().size(); // LazyInitializationException!
    }
}

해결 방법은 3가지입니다. 첫째, 가장 권장되는 방법은 Service 계층에서 필요한 데이터를 모두 DTO로 변환하여 반환하는 것입니다. 둘째, Fetch Join이나 EntityGraph로 필요한 연관관계를 미리 로딩합니다. 셋째, OSIV(Open Session In View)를 켜면 Controller까지 영속성 컨텍스트가 살아있지만, DB 커넥션을 오래 잡고 있어 실무에서는 비추천입니다.

// 권장: DTO 변환을 Service 안에서
@Transactional(readOnly = true)
public TeamResponse getTeam(Long id) {
    Team team = teamRepository.findTeamWithMembers(id); // Fetch Join
    return TeamResponse.from(team); // 트랜잭션 안에서 DTO 변환
}
OSIV 설정: spring.jpa.open-in-view가 기본 true입니다. Spring Boot 2.0부터 경고 로그가 뜹니다. API 서버에서는 false로 끄는 것이 좋습니다.
@OneToOne 관계에서 Lazy Loading이 안 되는 경우가 있다는데?

이 질문은 JPA 심화를 잘 아는 면접관이 던지는 질문입니다. @OneToOne에서 연관 엔티티의 외래 키가 아닌 쪽(mapped side)은 Lazy Loading이 적용되지 않습니다.

이유는 프록시 생성 조건과 관련됩니다. Hibernate가 프록시를 만들려면 "이 연관관계에 값이 있는지 없는지(null 여부)"를 먼저 알아야 합니다. FK를 가진 쪽은 FK 컬럼이 null인지 확인하면 되지만, FK가 없는 쪽은 실제로 상대 테이블을 조회해봐야 null 여부를 알 수 있어서 결국 쿼리가 나갑니다.

@Entity
public class Member {
    @OneToOne(fetch = FetchType.LAZY)
    @JoinColumn(name = "locker_id")
    private Locker locker;  // FK 보유 → LAZY 정상 작동
}

@Entity
public class Locker {
    @OneToOne(mappedBy = "locker", fetch = FetchType.LAZY)
    private Member member;  // FK 없음 → LAZY 무시, 즉시 쿼리!
}

해결 방법으로는: 양방향 OneToOne을 단방향으로 변경하거나, mappedBy 쪽을 제거하고 필요할 때만 별도 쿼리로 조회하거나, @OneToOne 대신 @ManyToOne으로 모델링(unique 제약조건 추가)하는 방법이 있습니다.

Fetch Join 시 별칭(alias)에 조건을 걸면 안 되는 이유는?

Fetch Join의 목적은 엔티티의 상태를 DB와 일치시키는 것입니다. 만약 JOIN FETCH t.members m WHERE m.age > 20처럼 필터링하면, 영속성 컨텍스트에는 일부 Member만 올라오게 됩니다. 이 상태에서 팀의 members를 수정하면, 누락된 Member가 삭제될 수 있는 데이터 정합성 문제가 발생합니다.

Hibernate 공식 문서에서도 Fetch Join 대상에 별칭 조건을 거는 것을 권장하지 않습니다. 필터링이 필요하면 별도 쿼리로 분리하거나 DTO Projection을 사용하는 것이 안전합니다.

// 위험
"SELECT t FROM Team t JOIN FETCH t.members m WHERE m.age > 20"

// 안전: 별도 쿼리로 분리
"SELECT m FROM Member m JOIN FETCH m.team WHERE m.age > 20"
MultipleBagFetchException이 뭐고, List를 Set으로 바꾸면 해결되나요?

MultipleBagFetchException은 2개 이상의 List 타입 컬렉션을 동시에 Fetch Join할 때 발생합니다. Hibernate에서 List(Bag)는 중복을 허용하는 컬렉션인데, 2개의 Bag을 JOIN하면 카테시안 곱이 발생하고 어떤 행이 어떤 컬렉션에 속하는지 구분할 수 없기 때문입니다.

List를 Set으로 바꾸면 예외는 사라지지만, 권장되지 않습니다. Set으로 바꿔도 카테시안 곱 자체는 발생하기 때문에, 데이터가 기하급수적으로 뻥튀기됩니다. Team에 Member 5명, Project 3개가 있으면 결과가 5×3=15행이 됩니다.

// MultipleBagFetchException
@OneToMany
private List<Member> members;
@OneToMany
private List<Project> projects;
"SELECT t FROM Team t JOIN FETCH t.members JOIN FETCH t.projects"

// 예외는 안 나지만 카테시안 곱 (비추천)
private Set<Member> members;
private Set<Project> projects;

// 권장: 하나만 Fetch Join + 나머지는 BatchSize
"SELECT t FROM Team t JOIN FETCH t.members"
// projects는 default_batch_fetch_size로 IN 쿼리
OSIV를 끄면 실무에서 어떤 점이 달라지나요?

OSIV(Open Session In View)가 켜져 있으면 HTTP 요청의 시작부터 끝까지 영속성 컨텍스트가 살아있습니다. Controller에서 Lazy 로딩이 가능하지만, DB 커넥션도 그만큼 오래 잡고 있습니다. 트래픽이 많은 API 서버에서는 커넥션 풀 고갈로 이어질 수 있습니다.

OSIV를 끄면 @Transactional이 끝나는 시점에 영속성 컨텍스트가 닫히므로, Service 밖에서 Lazy 접근 시 LazyInitializationException이 발생합니다. 하지만 이것이 오히려 "Service에서 필요한 데이터를 확실히 준비한다"는 좋은 아키텍처를 강제합니다.

# application.yml
spring:
  jpa:
    open-in-view: false  # API 서버는 끄는 것 권장

// OSIV OFF 시 Service 계층 패턴
@Service
@Transactional(readOnly = true)
public class TeamService {

    public TeamDetailResponse getTeamDetail(Long id) {
        // Fetch Join으로 필요한 데이터를 한 번에 로딩
        Team team = teamRepository.findTeamWithMembers(id);
        // 트랜잭션 안에서 DTO 변환 완료
        return TeamDetailResponse.from(team);
    }
}
Hibernate의 1차 캐시와 N+1은 어떤 관계가 있나요?

1차 캐시(영속성 컨텍스트)는 같은 트랜잭션 내에서 이미 로딩된 엔티티를 재사용하는 캐시입니다. 하지만 N+1 문제 해결에는 크게 도움이 되지 않습니다. 왜냐하면 N+1은 "Team을 순회하며 각 Team의 Members를 로딩"하는 상황이고, 각 Team의 Members는 서로 다른 엔티티 집합이기 때문에 1차 캐시에 히트하지 않습니다.

다만, @ManyToOne에서는 도움이 됩니다. 예를 들어 100명의 Member가 같은 Team에 속해있으면, 첫 번째 Member 조회 시 Team이 1차 캐시에 올라가고, 나머지 99명은 캐시에서 가져옵니다. 그러나 이는 "같은 엔티티 인스턴스"에 한정된 최적화이며, 근본적인 N+1 해결책은 아닙니다.

Spring Data JPA에서 Projection으로 N+1을 우회하는 방법은?

엔티티를 통째로 조회하지 않고 필요한 컬럼만 DTO로 바로 조회하면, 엔티티 그래프 자체를 타지 않으므로 N+1이 원천적으로 발생하지 않습니다. 이것이 DTO Projection입니다.

// 1. 생성자 표현식 (JPQL)
@Query("SELECT new com.example.dto.TeamMemberDto("
     + "t.name, m.name, m.age) "
     + "FROM Team t JOIN t.members m")
List<TeamMemberDto> findTeamMembers();

// 2. Interface-based Projection
public interface TeamSummary {
    String getName();
    int getMemberCount();
}

@Query("SELECT t.name AS name, SIZE(t.members) AS memberCount FROM Team t")
List<TeamSummary> findTeamSummaries();

// 3. QueryDSL Projection
List<TeamMemberDto> result = queryFactory
    .select(Projections.constructor(TeamMemberDto.class,
        team.name, member.name, member.age))
    .from(team)
    .join(team.members, member)
    .fetch();

Projection의 장점은 N+1 방지뿐 아니라, 불필요한 컬럼 조회와 영속성 컨텍스트 관리 비용을 줄일 수 있다는 점입니다. 조회 전용 API에서는 엔티티 조회보다 Projection이 성능상 유리합니다.
