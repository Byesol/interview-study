좌석 예약 시스템 설계

System Design Interview — 동시성 · 분산락 · 이벤트 소싱 · 장애 대응

좌석 예약 시스템을 설계해주세요. 어디서부터 시작하시겠어요?

먼저 요구사항을 명확히 해야 합니다. 좌석 예약이라고 하면 영화관, 공연장, 비행기, KTX 등 도메인마다 제약 조건이 다릅니다. 면접관에게 "어떤 도메인인지", "동시 사용자 규모는 어느 정도인지", "좌석 선택형인지 자동 배정형인지"를 먼저 확인합니다.

공통적으로 확인할 사항은 이렇습니다. 좌석 수(수백~수만), 피크 트래픽(티켓 오픈 순간 수만 TPS 가능), 결제 연동 여부, 임시 홀딩(TTL) 필요 여부, 그리고 한 사용자가 여러 좌석을 동시에 예약할 수 있는지 여부입니다. 이런 질문을 면접관에게 먼저 던지면 "이 사람은 설계 전에 스코프를 잡을 줄 안다"는 인상을 줍니다.

가정을 정했다면, 고수준 컴포넌트를 잡습니다. 클라이언트 → API Gateway → Reservation Service → DB가 기본 뼈대이고, 여기에 동시성 제어를 위한 분산 락(Redis 등), 결제 처리를 위한 Payment Service, 좌석 상태 조회를 빠르게 하기 위한 Cache Layer가 붙는 구조입니다.

면접관이 기대하는 포인트: 바로 설계에 들어가지 않고 요구사항을 물어보는 것 자체가 플러스입니다. "영화관 기준으로 하겠습니다. 좌석 약 300석, 피크 시 1,000~5,000 TPS 가정하겠습니다" 같이 직접 가정을 세우고 시작해도 좋습니다.
좌석의 상태는 어떻게 모델링하시겠어요?

좌석은 상태 머신(State Machine)으로 모델링하는 것이 가장 명확합니다. 기본적으로 3가지 상태가 있습니다: AVAILABLE(선택 가능), HELD(임시 점유), RESERVED(결제 완료). 여기에 필요하다면 DISABLED(사용 불가)를 추가합니다.

중요한 건 상태 전이 규칙입니다. AVAILABLE → HELD는 사용자가 좌석을 선택할 때, HELD → RESERVED는 결제가 완료될 때, HELD → AVAILABLE은 TTL이 만료되거나 사용자가 취소할 때 발생합니다. RESERVED → AVAILABLE은 환불 시에만 가능합니다. 이 전이 규칙을 DB 레벨에서 강제하지 않으면, 동시성 문제로 하나의 좌석이 두 번 팔리는 double booking이 발생할 수 있습니다.

꼬리질문 대비: "HELD 상태의 TTL은 얼마로 잡으시겠어요?"라는 질문이 자주 옵니다. 보통 5~15분이 일반적입니다. 너무 짧으면 결제 중 만료되고, 너무 길면 다른 사용자가 좌석을 오래 기다려야 합니다. 인터파크나 멜론티켓 기준 약 10분이 표준입니다.
DB 스키마는 어떻게 설계하시겠어요?

핵심 테이블은 크게 4개입니다. events(공연/상영 정보), seats(좌석 마스터), seat_inventory(특정 이벤트의 좌석 상태), reservations(예약 내역)입니다.

seat_inventory가 가장 중요한 테이블인데, 여기에 event_id, seat_id, status, held_by, held_until, version 컬럼을 둡니다. version은 낙관적 락(Optimistic Lock)을 위한 것이고, held_until은 HELD 상태의 TTL 만료 시점입니다. (event_id, seat_id)에 유니크 인덱스를 걸어서 같은 이벤트의 같은 좌석이 두 행으로 존재하는 것을 원천 차단합니다.

reservations 테이블은 reservation_id, user_id, event_id, seat_ids(여러 좌석 가능), status, created_at을 갖습니다. 이 테이블은 예약 이력 추적과 결제 상태 관리에 사용됩니다.

면접관 깊이 파기: "seat_inventory와 reservations를 분리한 이유가 뭔가요?" — seat_inventory는 현재 좌석 상태를 빠르게 조회하기 위한 것이고, reservations는 비즈니스 이벤트 기록입니다. 하나로 합치면 좌석 조회 시 JOIN이 복잡해지고, 취소/환불 이력을 추적하기 어려워집니다. 관심사의 분리(Separation of Concerns)입니다.
전체 예약 플로우를 설명해주세요.

사용자 관점에서 플로우는 이렇습니다. ① 이벤트 선택 → ② 좌석 목록 조회 (AVAILABLE 좌석 확인) → ③ 좌석 선택 (HELD로 전환) → ④ 결제 진행 → ⑤ 결제 완료 시 RESERVED로 전환. 이 5단계가 기본입니다.

서버 관점에서 핵심은 ③번 단계입니다. 좌석을 HELD로 전환하는 순간이 동시성 충돌이 가장 많이 발생하는 지점입니다. 두 사용자가 동시에 같은 좌석을 클릭하면, 한 명만 성공하고 나머지는 "이미 선택된 좌석입니다"를 받아야 합니다. 이걸 보장하지 못하면 double booking이 발생합니다.

④번 결제 단계도 중요한데, 결제는 외부 PG사 연동이므로 실패할 수 있습니다. 결제 실패 시 HELD → AVAILABLE로 즉시 롤백해야 하고, 결제 타임아웃 시에도 HELD의 TTL 만료로 자동 복구되어야 합니다. 이 두 가지 경로를 모두 처리해야 안전한 시스템입니다.

좌석 목록 조회 → 좌석 선택 (HOLD) → 결제 요청 → 결제 완료 (RESERVE)

실패 시: HELD → AVAILABLE 자동 롤백 (TTL 만료 또는 명시적 취소)

두 사람이 동시에 같은 좌석을 선택하면 어떻게 처리하나요?

핵심은 "한 좌석에 대해 하나의 HOLD만 성공하도록" 보장하는 것입니다. 방법은 크게 세 가지가 있고, 각각 trade-off가 다릅니다.

첫째, 낙관적 락(Optimistic Lock)입니다. seat_inventory에 version 컬럼을 두고 UPDATE ... WHERE status='AVAILABLE' AND version=?으로 업데이트합니다. 먼저 성공한 트랜잭션만 version이 맞으므로 1행이 업데이트되고, 나머지는 0행이 업데이트되어 실패를 감지합니다. DB 자체의 row-level lock만 사용하므로 별도 인프라가 불필요하다는 장점이 있습니다.

둘째, 비관적 락(Pessimistic Lock)입니다. SELECT ... FOR UPDATE로 행을 먼저 잠그고 상태를 변경합니다. 동시 요청이 많으면 lock contention이 심해져 응답 시간이 급격히 늘어나는 단점이 있습니다.

셋째, 분산 락(Distributed Lock)입니다. Redis의 SET NX EX를 사용해서 좌석 단위로 락을 잡습니다. DB에 부하를 주지 않으면서도 빠르게 동시성을 제어할 수 있습니다. 다만 Redis 장애 시 대비 로직이 필요합니다.

전략장점단점적합한 경우
낙관적 락별도 인프라 불필요, 간단충돌 시 재시도 필요충돌 빈도 낮은 경우
비관적 락충돌 시 즉시 대기lock contention, 데드락 위험무조건 성공 보장 필요 시
분산 락 (Redis)DB 부하 ↓, 빠름Redis 장애 시 fallback 필요초고트래픽, 수평 확장 필요 시
"낙관적 락이면 충분하지 않나요? 왜 분산 락까지 필요한가요?"
낙관적 락은 충돌 후 재시도 방식이라, 티켓 오픈 직후처럼 수천 명이 동시에 같은 좌석을 누르는 상황에서는 대부분의 요청이 실패하고 재시도가 폭주합니다. 이를 thundering herd 문제라 합니다. 분산 락을 사용하면 락을 획득한 한 명만 DB에 접근하므로 DB 부하가 급감하고, 나머지는 락 획득 실패 시 즉시 "이미 선택된 좌석" 응답을 받아 재시도 자체가 발생하지 않습니다.
Redis 분산 락을 쓴다고 했는데, Redis가 죽으면 어떻게 되나요?

이건 면접에서 거의 반드시 나오는 꼬리질문입니다. Redis 단일 노드가 죽으면 락 자체가 작동하지 않으므로, 두 가지 방어 전략을 세워야 합니다.

첫째, Redlock 알고리즘입니다. Redis를 5대(또는 홀수 N대) 독립적으로 운영하고, 과반수(N/2+1) 이상에서 락을 획득해야 성공으로 판정합니다. 한두 대가 죽어도 나머지에서 과반을 채울 수 있어 가용성이 높습니다. 하지만 Martin Kleppmann이 지적한 것처럼, 클럭 스큐나 GC pause 등에서 안전하지 않을 수 있다는 논쟁이 있습니다.

둘째, 더 실용적인 접근은 fallback으로 DB의 낙관적 락을 두는 것입니다. Redis 락 획득을 시도하되, Redis가 불가하면 DB 레벨의 UPDATE ... WHERE status='AVAILABLE' AND version=?로 최종 방어선을 칩니다. 즉 Redis는 성능 최적화 레이어이고, DB가 최종 일관성 보장을 담당하는 구조입니다.

추가 꼬리질문 대비: "Redlock에 대한 논쟁을 알고 있나요?" — Martin Kleppmann은 분산 환경에서 시계 동기화 문제 때문에 Redlock이 안전하지 않다고 주장했고, Antirez(Redis 창시자)는 대부분의 실용적 케이스에서 충분히 안전하다고 반박했습니다. 면접에서는 "논쟁이 있다는 것을 알고, 그래서 DB fallback을 함께 사용합니다"라고 말하면 깊이 있는 답변이 됩니다.
HELD 상태의 TTL이 만료됐는데, 마침 결제가 완료되면 어떻게 되나요?

이것은 race condition의 전형적인 사례입니다. TTL 만료 처리와 결제 완료 처리가 동시에 일어나면, 좌석이 풀린 뒤에 결제가 성공하는 비정상 상태가 발생할 수 있습니다.

해결 방법은 결제 완료 시 원자적으로 상태 전이를 검증하는 것입니다. UPDATE seat_inventory SET status='RESERVED' WHERE event_id=? AND seat_id=? AND status='HELD' AND held_by=?처럼 현재 상태가 HELD이면서 본인이 점유한 경우에만 RESERVED로 전환합니다. 만약 TTL 만료로 이미 AVAILABLE이 됐다면 이 UPDATE는 0행이 영향받으므로 실패하게 됩니다.

결제는 성공했지만 좌석 확보에 실패한 경우, 결제 취소(환불) 프로세스를 자동으로 트리거해야 합니다. 이를 보상 트랜잭션(Compensating Transaction)이라 합니다. PG사에 취소 요청을 보내고, 사용자에게 "결제는 완료되었으나 좌석이 이미 만료되어 자동 환불됩니다" 같은 알림을 보내야 합니다.

"그럼 TTL을 좀 넉넉하게 잡으면 되지 않나요?"
TTL을 늘리면 이 race condition의 확률은 줄지만, 다른 문제가 생깁니다. 사용자가 좌석을 잡아놓고 결제를 안 하면 그 시간 동안 다른 사용자가 해당 좌석을 예약할 수 없습니다. 인기 공연의 경우 HELD 상태 좌석이 많아지면 "좌석이 없는데 실제로는 다 HELD"인 상황이 벌어집니다. 그래서 TTL은 결제에 필요한 최소 시간 + 약간의 버퍼로 설정하되 (보통 7~10분), race condition은 보상 트랜잭션으로 처리하는 것이 올바른 접근입니다.
한 사용자가 여러 좌석을 동시에 예약하면 어떤 문제가 생기나요?

여러 좌석을 한 번에 HOLD하려면, 부분 성공 문제를 고려해야 합니다. 예를 들어 3석을 선택했는데 2석만 HOLD에 성공하고 1석은 이미 선택된 좌석이라면, 2석만 HOLD된 채로 남겨둘지 아니면 전부 롤백할지를 결정해야 합니다.

일반적으로 좌석 예약에서는 "All or Nothing" 방식이 맞습니다. 3석을 요청했는데 하나라도 실패하면 전부 롤백합니다. 이를 위해 DB 트랜잭션 안에서 여러 좌석의 상태를 한꺼번에 변경하거나, 분산 락을 사용한다면 모든 좌석에 대해 락을 획득한 뒤에만 상태 변경을 진행합니다.

"여러 좌석에 대해 순서대로 락을 잡으면 데드락이 발생하지 않나요?"
맞습니다. 사용자 A가 좌석 1→2 순서로 락을 잡고, 사용자 B가 좌석 2→1 순서로 잡으면 교차 대기(deadlock)가 발생합니다. 해결법은 간단합니다. 좌석 ID를 항상 오름차순으로 정렬한 뒤 순서대로 락을 잡는 것입니다. 이를 Lock Ordering이라 하며, 모든 요청이 같은 순서로 락을 획득하면 순환 대기가 발생하지 않습니다.
SELECT 시점과 UPDATE 시점 사이에 다른 사용자가 끼어들 수 있지 않나요? (TOCTOU 문제)

이것은 TOCTOU(Time of Check to Time of Use) 문제의 전형입니다. "좌석 AVAILABLE인지 확인 → HELD로 업데이트" 사이에 갭이 있으면 동시에 두 요청이 모두 AVAILABLE로 확인하고, 둘 다 HELD로 변경하려 시도할 수 있습니다.

해결의 핵심은 확인(Check)과 변경(Update)을 하나의 원자적 연산으로 만드는 것입니다. UPDATE seat_inventory SET status='HELD', held_by=?, held_until=?, version=version+1 WHERE event_id=? AND seat_id=? AND status='AVAILABLE' 이렇게 하면 WHERE 절에서 상태 확인과 UPDATE가 DB의 row-level lock으로 원자적으로 실행됩니다. 별도로 SELECT를 먼저 하지 않고, 조건부 UPDATE 한 방으로 처리하는 것이 핵심입니다.

이 패턴을 Compare-And-Swap(CAS)이라고도 합니다. UPDATE 결과의 affected rows가 1이면 성공, 0이면 이미 다른 사용자가 선점한 것입니다.

티켓 오픈 순간 수만 명이 몰리면 어떻게 대응하나요?

티켓 오픈 순간은 읽기 폭주 → 쓰기 폭주 패턴입니다. 먼저 수만 명이 좌석 목록을 동시에 조회(읽기)하고, 그중 상당수가 거의 동시에 좌석을 선택(쓰기)합니다.

읽기 부하는 비교적 쉽게 해결됩니다. 좌석 목록을 Redis나 CDN에 캐싱하고, 짧은 TTL(1~2초)로 갱신합니다. 이미 HELD된 좌석이 1~2초 지연되어 보이는 것은 사용자 경험상 허용 가능하며, 어차피 HOLD 시점에 실제 검증을 합니다.

쓰기 부하가 진짜 문제입니다. 이를 위해 여러 전략을 조합합니다. 첫째, 대기열(Queue) 시스템을 앞에 둡니다. 사용자 요청을 즉시 처리하지 않고 대기열에 넣은 뒤, 순서대로 처리합니다. 사용자에게는 "현재 대기 1,234번째입니다"를 보여줍니다. 인터파크, Yes24가 이 방식을 씁니다. 둘째, 서버 수평 확장을 하되, 좌석 단위로 분산 락을 사용해 특정 좌석의 동시 접근은 직렬화합니다. 셋째, rate limiting으로 한 사용자가 비정상적으로 빠르게 요청을 보내는 것(봇/매크로)을 차단합니다.

"대기열을 쓰면 사용자 경험이 안 좋지 않나요? 대기열 없이는 못 하나요?"
대기열 없이 수만 TPS를 직접 처리하려면 DB가 그 부하를 감당해야 합니다. 인기 좌석의 경우 하나의 row에 수천 개의 UPDATE 요청이 몰리므로, lock contention으로 DB 자체가 병목이 됩니다. 대기열은 단순히 사용자를 기다리게 하는 것이 아니라, 시스템을 보호하면서 공정한 순서를 보장하는 것입니다.
그 대기열을 메시지 큐(Kafka/RabbitMQ)로 구현하면 되는 거 아닌가요?

직관적으로는 맞는 것 같지만, 대기열과 메시지 큐는 목적이 다릅니다. 이걸 구분하는 게 면접에서 중요한 포인트입니다. 비유하면, 메시지 큐는 "주방으로 주문서를 보내는 파이프"이고, 대기열은 "식당 입구에서 번호표 받고 줄 서는 것"입니다. 줄을 서는 것과 주문을 넣는 것은 별개의 행위입니다.

대기열(Waiting Queue)의 핵심 역할은 "좌석 선택 페이지에 동시에 들어갈 수 있는 사람 수를 제한"하는 것입니다. 5만 명이 동시에 몰려도, 실제로 좌석 선택 페이지에는 동시 2,000명만 입장시키고, 나머지는 대기 화면에서 기다리게 합니다. 이건 동시 접속자 수 제어(Concurrency Throttling)이지, 예약 요청 처리가 아닙니다.

반면 메시지 큐에 예약 요청을 넣는 것은, 사용자가 이미 좌석 선택 페이지에 들어와서 특정 좌석을 클릭한 뒤의 이야기입니다. 이 두 가지를 혼동하면 설계가 꼬입니다.

"그러면 대기열은 실제로 어떻게 구현하나요?"
대기열은 보통 Redis Sorted Set으로 구현합니다. 사용자가 접속하면 ZADD waiting_queue {timestamp} {user_id}로 대기열에 넣고, score(timestamp)로 순서를 보장합니다. 클라이언트는 2~3초마다 폴링으로 "내 순서가 왔나요?"를 확인하고, 서버는 ZRANK로 현재 위치를 알려줍니다. 입장 가능하면 입장 토큰(JWT 등)을 발급하고, 이 토큰이 있어야만 좌석 선택 API에 접근할 수 있게 합니다.
"대기열에서 입장한 다음에도 결국 좌석 선택에서 동시성 문제가 발생하지 않나요?"
맞습니다. 대기열은 동시 접속자 수를 제한할 뿐, 좌석 단위의 동시성 문제를 해결하지는 않습니다. 2,000명이 입장해서 같은 좌석을 동시에 누르면 여전히 분산 락이나 CAS가 필요합니다. 하지만 핵심은 5만 명 vs 2,000명의 차이입니다. 5만 명이 동시에 DB를 때리면 시스템이 죽지만, 2,000명이면 분산 락 + CAS로 충분히 처리 가능합니다.
좌석 상태를 캐싱하면, 캐시와 DB 사이에 불일치가 생기지 않나요?

네, 발생합니다. 하지만 좌석 예약에서는 읽기 캐시의 불일치가 허용 가능한 것이 핵심입니다. 좌석 목록에서 "빈 좌석"으로 보였지만 클릭 시 이미 HELD였다면, 사용자에게 "이미 선택된 좌석입니다"만 보여주면 됩니다. 이것은 불편하지만 데이터 손상이 아닙니다.

반대로, HELD인 좌석이 빈 것처럼 보이는 것은 절대 안 됩니다 — 이건 사실 발생할 수 있는데, 캐시 갱신 지연 때문입니다. 하지만 이것도 실제로 HOLD 시도 시 DB에서 거부하므로 double booking으로 이어지지는 않습니다. 캐시는 어디까지나 "보여주기 용도"이고, 실제 상태 변경의 source of truth는 항상 DB입니다.

캐시 갱신 전략으로는 Write-Through(상태 변경 시 즉시 캐시도 갱신) 또는 이벤트 기반(상태 변경 이벤트를 발행하면 별도 consumer가 캐시를 갱신)을 사용합니다. Write-Through가 더 단순하고 지연이 짧아 좌석 예약에 적합합니다.

결제 중간에 서버가 죽으면 어떻게 되나요?

이건 분산 트랜잭션 문제입니다. 좌석 HOLD → PG 결제 → 좌석 RESERVE 이 3단계 중 어디서 장애가 나느냐에 따라 대응이 다릅니다.

HOLD 후 결제 요청 전 서버 사망: 좌석은 HELD 상태로 남지만, TTL 만료 시 자동으로 AVAILABLE이 됩니다. 사용자는 결제를 다시 시도하면 됩니다. 가장 안전한 케이스입니다.

결제 성공 후 RESERVE 처리 전 서버 사망: 이게 가장 위험합니다. 돈은 빠져나갔는데 좌석은 HELD 상태이고, TTL이 만료되면 AVAILABLE로 풀려서 다른 사람이 예약할 수 있습니다. 이를 위해 결제 완료 콜백(Webhook)을 별도로 수신하는 서비스를 두거나, 배치 잡(Scheduler)이 주기적으로 "결제 완료됐지만 RESERVED가 아닌" 건을 찾아 후처리합니다. 이것을 Reconciliation(정합성 대사)이라 합니다.

"Saga 패턴을 적용할 수 있지 않나요?"
네, 적용 가능합니다. 좌석 예약은 Choreography 기반 Saga보다는 Orchestration 기반 Saga가 적합합니다. 예약 오케스트레이터가 "좌석 HOLD → 결제 요청 → 좌석 RESERVE" 순서를 관리하고, 각 단계 실패 시 보상 트랜잭션을 실행합니다. 단, Saga는 구현 복잡도가 높으므로 단순한 시스템에서는 TTL + Reconciliation으로 충분합니다.
DB 하나로는 한계가 있을 텐데, 샤딩은 어떻게 하시겠어요?

좌석 예약 시스템에서 샤딩 키의 가장 자연스러운 후보는 event_id입니다. 같은 이벤트의 좌석들은 항상 함께 조회되고 함께 업데이트되므로, event_id 기반 샤딩을 하면 크로스 샤드 쿼리 없이 대부분의 연산이 가능합니다.

다만 주의할 점이 있습니다. 핫 이벤트 문제입니다. BTS 콘서트처럼 초인기 이벤트의 경우 하나의 샤드에 모든 트래픽이 집중됩니다. 이를 핫스팟(hotspot)이라 합니다. 이때는 해당 이벤트의 좌석을 구역(zone) 단위로 sub-sharding하거나, 아예 해당 이벤트 전용 dedicated DB 인스턴스를 할당하는 방법을 씁니다.

꼬리질문 대비: "user_id로 샤딩하면 안 되나요?" — 좌석 예약 시스템의 핵심 쿼리가 "이 이벤트의 빈 좌석 목록"이므로, user_id 샤딩을 하면 하나의 이벤트 좌석이 여러 샤드에 흩어져 조회 시 모든 샤드를 스캔해야 합니다. "내 예약 내역 조회"가 중요하다면 별도의 user_id 인덱스 테이블을 만드는 것이 낫습니다.
이 시스템에서 메시지 큐는 어디에 왜 필요한가요?

좌석 예약 시스템에서 메시지 큐(Kafka, RabbitMQ 등)가 필요한 지점은 크게 세 가지입니다.

첫째, 결제 완료 후 후속 처리(Post-Payment Processing)입니다. 결제가 성공하면 좌석 RESERVE 외에도 확인 이메일/SMS 발송, 예약 이력 저장, 포인트 적립, 영수증 생성 등 여러 작업이 필요합니다. 메시지 큐에 "결제 완료" 이벤트를 발행하면, 각 후속 처리가 독립적으로 비동기 처리됩니다.

둘째, Reconciliation(정합성 대사)입니다. 결제 성공 후 서버 장애로 RESERVE 처리가 안 된 건, PG 콜백과 내부 상태가 불일치하는 건 등을 큐에 쌓아두고 별도의 Consumer가 주기적으로 정합성을 맞춥니다.

셋째, 좌석 상태 변경 이벤트 브로드캐스트입니다. 좌석이 HELD/RESERVED/AVAILABLE로 바뀔 때 이 이벤트를 큐에 발행하면, 캐시 갱신 서비스, 실시간 좌석 현황 SSE 서버, 모니터링/분석 시스템이 각각 구독해서 처리합니다.

"결제 완료 처리를 동기로 하면 안 되나요? 굳이 비동기로 해야 하나요?"
동기로 처리하면 두 가지 문제가 있습니다. 첫째, SMS 발송 서버가 3초 걸리면 결제 API 응답도 3초 늦어집니다. 피크 시 수천 요청이 SMS 서버 응답을 기다리면서 스레드 풀이 고갈됩니다. 둘째, SMS 서버가 장애나면 결제 자체가 실패합니다. 메시지 큐를 쓰면 "결제 성공"과 "후속 처리"가 완전히 디커플링되어, 알림 서버가 죽어도 결제는 정상 완료되고, 알림은 나중에 재처리됩니다.
Kafka와 RabbitMQ 중 어떤 걸 선택하시겠어요?

좌석 예약 시스템의 특성을 기준으로 비교하면, 규모에 따라 선택이 달라집니다.

소규모~중규모(단일 영화관, 소규모 공연장)라면 RabbitMQ가 더 적합합니다. 설정이 간단하고, 메시지 라우팅이 유연하며, 운영 부담이 적습니다. 대규모(인터파크급, 수만 TPS)라면 Kafka가 맞습니다. 처리량이 압도적이고, 메시지 보존으로 Reconciliation이나 재처리가 쉽고, Consumer Group으로 여러 시스템이 독립적으로 같은 이벤트를 구독할 수 있습니다.

면접에서는 "둘 다 가능하지만, 이 시스템의 규모와 요구사항을 기준으로 선택하겠습니다"라고 말한 뒤 trade-off를 설명하면 좋은 인상을 줍니다. 정답을 하나 고르는 것보다 왜 그것을 선택했는지가 더 중요합니다.

꼬리질문 대비: "Redis Streams는요?" — Redis Streams도 경량 메시지 큐로 사용 가능합니다. 이미 Redis를 분산 락/캐시로 쓰고 있으므로 추가 인프라 없이 간단한 이벤트 처리가 가능합니다. 다만 Kafka 수준의 내구성이나 대규모 Consumer Group 관리에는 한계가 있어, 보조적인 용도(캐시 무효화 알림 등)에 적합합니다.
좌석 목록을 실시간으로 보여주면 WebSocket을 써야 하지 않나요? 그러면 서버 부하가 너무 심하지 않나요?

맞는 지적입니다. 수만 명이 WebSocket 커넥션을 동시에 유지하면 서버 리소스가 상당히 소비됩니다. 하지만 진짜 실시간이 필요한지를 먼저 따져봐야 합니다.

대부분의 좌석 예약에서는 폴링(Polling) + 짧은 캐시 TTL이면 충분합니다. 클라이언트가 2~3초마다 좌석 상태를 조회하되, 서버는 캐시에서 응답합니다. 사용자 입장에서 좌석이 2~3초 지연되어 보이는 것은 큰 문제가 아닙니다. 어차피 실제 선점은 HOLD 시점에 결정되니까요.

정말 실시간이 필요하다면 SSE(Server-Sent Events)가 WebSocket보다 나은 선택일 수 있습니다. 좌석 상태 변경은 서버→클라이언트 단방향이므로 양방향인 WebSocket은 과도합니다. SSE는 HTTP 기반이라 로드밸런서와의 호환성도 더 좋습니다.

"SSE도 커넥션은 유지해야 하잖아요?"
맞습니다. 그래서 이벤트 단위로 SSE 커넥션을 관리합니다. 좌석 선택 화면에 들어왔을 때만 커넥션을 열고, 나가면 즉시 닫습니다. 그리고 SSE 서버를 별도로 두고 수평 확장합니다. 실제 상태 변경 이벤트는 Redis Pub/Sub으로 각 SSE 서버에 브로드캐스트하면, 어떤 SSE 서버에 연결된 클라이언트든 동일한 이벤트를 받을 수 있습니다.
매크로(봇)로 좌석을 싹쓸이하는 것은 어떻게 막나요?

봇 방어는 여러 레이어에서 동시에 해야 합니다. 하나의 방어선만으로는 우회됩니다.

첫째, Rate Limiting입니다. 한 IP/사용자당 초당 요청 수를 제한합니다. Token Bucket이나 Sliding Window 알고리즘을 사용합니다. 둘째, CAPTCHA입니다. 좌석 선택 직전에 reCAPTCHA 같은 인간 검증을 넣습니다. 셋째, 디바이스 핑거프린팅으로 동일 기기에서의 다중 계정 사용을 탐지합니다. 넷째, 행동 패턴 분석으로 인간과 봇의 클릭 패턴 차이를 감지합니다. 사람은 좌석을 고르는 데 시간이 걸리지만, 봇은 오픈 즉시 밀리초 단위로 요청을 보냅니다.

실무에서는 이 중 Rate Limiting + CAPTCHA를 기본으로 하고, 규모가 커지면 행동 분석을 추가합니다. 인터파크가 대기열 시스템을 도입한 이유 중 하나도 봇이 직접 API를 때리는 것을 차단하기 위함입니다.

결제 시스템(PG)이 응답이 느려서 10분 이상 걸리면 TTL이 만료되는데, 이런 경우는요?

이건 외부 의존성의 타임아웃 문제입니다. PG사 응답이 비정상적으로 느린 경우, 우리 시스템의 TTL 정책과 충돌합니다.

첫째, 결제 요청 시 PG에 대한 타임아웃을 명확히 설정합니다 (예: 30초). 30초 안에 응답이 없으면 결제 실패로 간주하고 사용자에게 재시도를 안내합니다. 둘째, PG 응답이 늦게 도착해서 TTL 만료 후에 결제 성공 콜백이 오는 경우, 앞서 설명한 Reconciliation 배치가 이를 감지하고 환불 처리합니다. 셋째, TTL 만료 5분 전이면 좌석이 곧 풀릴 수 있다는 경고를 사용자에게 보내고, 필요하면 TTL 연장(extension)을 한 번 허용할 수도 있습니다.

추가 꼬리질문 대비: "TTL 연장을 무한히 할 수 있으면 악용되지 않나요?" — 연장은 최대 1회, 고정 시간(예: 5분)으로 제한합니다. 그리고 연장 요청 시에도 HELD 상태인지 검증해야 합니다.
이벤트 소싱(Event Sourcing)을 적용하면 좋지 않나요?

좌석 예약 시스템에 이벤트 소싱을 적용하면 감사 추적(audit trail)과 상태 복원에서 큰 장점이 있습니다. "좌석 A1이 언제 HELD되고, 누가 잡았다가, 취소되고, 다시 다른 사람이 잡았는지"를 완전한 이력으로 추적할 수 있습니다.

다만 trade-off가 있습니다. 이벤트 소싱은 현재 상태를 알기 위해 이벤트를 리플레이하거나, 별도의 Materialized View(읽기 모델)를 유지해야 합니다. 이것이 CQRS(Command Query Responsibility Segregation)와 함께 가는 이유입니다. 복잡도가 상당히 올라가므로, "이 시스템에 이벤트 소싱이 정말 필요한가?"를 먼저 물어야 합니다.

실용적으로는, 이벤트 소싱 전체를 도입하기보다 reservation_events 테이블에 상태 변경 이력을 append-only로 쌓는 "이벤트 로그 패턴"이면 대부분의 감사 요구를 충족합니다. 이건 이벤트 소싱의 장점을 일부 취하면서 복잡도는 크게 낮추는 실용적 접근입니다.

"CQRS를 쓰면 읽기/쓰기 모델이 불일치하는 건 괜찮은 건가요?"
CQRS에서 읽기 모델은 Eventually Consistent입니다. 좌석 목록 조회(읽기)가 몇 초 지연되는 것은 앞서 말씀드린 대로 허용 가능하고, 실제 HOLD/RESERVE(쓰기)는 Command 모델에서 강한 일관성으로 처리합니다. 즉, "보여주는 것"과 "실제 상태 변경"의 일관성 수준을 분리하는 것이 CQRS의 핵심이고, 좌석 예약 시스템에서는 이 분리가 자연스럽습니다.
좌석 선택 없이 "최적 좌석 자동 배정"이라면 설계가 어떻게 달라지나요?

자동 배정 방식은 동시성 제어가 훨씬 단순해집니다. 사용자가 특정 좌석을 지정하지 않으므로, 서버가 빈 좌석 중에서 알아서 배정하면 됩니다. 비유하면, 선택형은 "뷔페에서 모두가 같은 메뉴에 몰리는 것"이고, 자동 배정은 "번호표 뽑고 순서대로 배정받는 것"입니다.

구현은 큐 기반이 깔끔합니다. 빈 좌석들을 우선순위 큐에 넣고 (가운데 앞쪽이 우선순위 높음), 요청이 오면 큐에서 하나씩 뽑아 배정합니다. 큐에서 뽑는(dequeue) 연산 자체가 원자적이므로 double booking이 구조적으로 불가능합니다. Redis의 LPOP이나 ZPOPMIN을 사용하면 됩니다.

단, "연석 배정"(인접 좌석 같이 배정)이 필요하면 단순 큐로는 어렵습니다. 연석 좌석 그룹을 하나의 단위로 관리하거나, 배정 시 인접 좌석을 탐색하는 알고리즘이 추가로 필요합니다.

좌석 HOLD API의 핵심 로직을 코드로 보여주세요.

분산 락(Redis) + DB 낙관적 락 조합으로 구현합니다. Redis 락은 성능 최적화, DB CAS는 최종 안전장치입니다.

// Spring Boot + Redis 기반 예시
@Transactional
public HoldResult holdSeat(Long eventId, Long seatId, Long userId) {
    String lockKey = "seat_lock:" + eventId + ":" + seatId;

    // 1. Redis 분산 락 획득 시도 (3초 대기, 10초 자동 해제)
    boolean locked = redisLock.tryLock(lockKey, 3, 10, TimeUnit.SECONDS);
    if (!locked) {
        return HoldResult.fail("이미 다른 사용자가 선택 중입니다.");
    }

    try {
        // 2. DB CAS (Compare-And-Swap) — 최종 안전장치
        int updated = seatInventoryRepository.holdSeat(
            eventId, seatId, userId,
            LocalDateTime.now().plusMinutes(10), // TTL 10분
            SeatStatus.AVAILABLE  // WHERE status = AVAILABLE
        );

        if (updated == 0) {
            return HoldResult.fail("이미 선택된 좌석입니다.");
        }

        return HoldResult.success(seatId);
    } finally {
        redisLock.unlock(lockKey); // 3. 반드시 해제
    }
}

// Repository — Native Query
@Modifying
@Query("UPDATE seat_inventory " +
       "SET status = 'HELD', held_by = :userId, " +
       "    held_until = :heldUntil, version = version + 1 " +
       "WHERE event_id = :eventId AND seat_id = :seatId " +
       "  AND status = :requiredStatus")
int holdSeat(@Param("eventId") Long eventId,
             @Param("seatId") Long seatId,
             @Param("userId") Long userId,
             @Param("heldUntil") LocalDateTime heldUntil,
             @Param("requiredStatus") SeatStatus requiredStatus);

핵심 포인트는 세 가지입니다. 첫째, Redis 락은 빠른 거절을 위한 것이고, 둘째 DB WHERE 절이 실제 일관성을 보장하며, 셋째 finally에서 반드시 unlock해야 합니다. 락 해제를 빠뜨리면 TTL(10초)이 지나야 다른 사용자가 시도할 수 있어 UX가 나빠집니다.

TTL 만료 좌석을 복구하는 스케줄러를 어떻게 구현하시겠어요?

HELD 상태인데 TTL이 지난 좌석을 주기적으로 AVAILABLE로 되돌려야 합니다. 두 가지 방식이 있습니다.

// 방식 1: 주기적 배치 (Spring @Scheduled)
@Scheduled(fixedRate = 30_000) // 30초마다
public void releaseExpiredHolds() {
    int released = seatInventoryRepository.releaseExpired(
        LocalDateTime.now(),
        SeatStatus.HELD,
        SeatStatus.AVAILABLE
    );
    if (released > 0) {
        log.info("Released {} expired holds", released);
        // 좌석 상태 변경 이벤트 발행 (캐시 갱신 트리거)
        eventPublisher.publish(new SeatsReleasedEvent(released));
    }
}

// 방식 2: Lazy Expiration (조회 시점에 체크)
public SeatStatus getEffectiveStatus(SeatInventory seat) {
    if (seat.getStatus() == SeatStatus.HELD
        && seat.getHeldUntil().isBefore(LocalDateTime.now())) {
        // 만료된 HELD → AVAILABLE로 취급 (실제 DB 업데이트는 비동기로)
        asyncReleaser.release(seat);
        return SeatStatus.AVAILABLE;
    }
    return seat.getStatus();
}

방식 1(주기적 배치)은 단순하지만, 배치 주기(30초) 동안 만료된 좌석이 아직 HELD로 보이는 갭이 있습니다. 방식 2(Lazy Expiration)는 조회 시점에 즉시 반영되지만, 아무도 조회하지 않는 좌석은 정리되지 않습니다. 실무에서는 둘을 병행합니다. 조회 시 Lazy로 빠르게 반영하면서, 배치로 놓치는 건 없도록 정리합니다.
