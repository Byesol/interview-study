객체지향이란 무엇인가요? 절차지향과 어떻게 다른가요?

객체지향은 현실 세계의 사물을 객체라는 단위로 모델링하고, 객체들이 서로 협력하며 동작하도록 프로그램을 구성하는 패러다임입니다. 각 객체는 자신만의 상태(필드)와 행동(메서드)을 캡슐화하고 있으며, 이를 통해 변경이 한 객체 내부에 머물게 됩니다.

절차지향은 코드를 위에서 아래로 순서대로 실행하는 함수 중심의 방식입니다. 데이터와 함수가 분리되어 있어 초기에는 단순하지만, 규모가 커질수록 함수들이 공유 데이터를 뒤섞어 사이드 이펙트를 추적하기 어려워집니다. 객체지향은 데이터와 그 데이터를 처리하는 로직을 하나의 객체 안에 묶어 응집도를 높이고 변경의 파급 범위를 좁힙니다.

자주 나오는 꼬리질문: Java는 100% 객체지향인가요? → 아닙니다. int, boolean 같은 원시 타입은 객체가 아니기 때문입니다. 성능 최적화를 위해 James Gosling이 의도적으로 타협한 설계 결정입니다.
OOP 4대 특성을 설명하고, 그 중 가장 중요한 것은 무엇인지 본인의 의견을 말해주세요.

OOP의 4대 특성은 캡슐화, 상속, 다형성, 추상화입니다. 캡슐화는 내부 상태를 외부에 숨기고 공개된 메서드로만 접근하게 하는 것, 상속은 부모 클래스의 속성과 행동을 자식이 물려받아 재사용하는 것, 다형성은 동일한 메서드 호출이 객체 타입에 따라 다르게 동작하는 것, 추상화는 복잡한 내부를 숨기고 핵심 개념만 드러내는 것입니다.

저는 캡슐화가 가장 근본적이라고 생각합니다. 캡슐화가 제대로 되어야 상속 계층에서 내부 구현이 외부에 새어나가지 않고, 다형성도 안전하게 작동합니다. 캡슐화 없이 내부 상태가 외부에 노출된다면, 아무리 잘 짜인 상속 구조라도 어느 코드가 어떤 상태를 바꿨는지 추적하기 어렵습니다.

캡슐화를 실무에서 어떻게 적용하나요? getter/setter 무분별한 사용의 문제점은?

캡슐화의 핵심은 단순히 private + getter/setter를 만드는 게 아닙니다. 진짜 캡슐화는 비즈니스 의미 있는 메서드를 통해서만 상태가 바뀌게 하는 것입니다. 예를 들어 setBalance(amount) 대신 withdraw(amount)로 표현하면, 출금 한도 검사나 잔액 부족 예외 처리 같은 비즈니스 규칙을 객체 안에 응집시킬 수 있습니다.

모든 필드에 getter/setter를 열어두면 캡슐화는 형식만 남습니다. 외부에서 언제든 상태를 꺼내 변경할 수 있어 객체의 불변식(invariant)을 보장하기 어려워집니다. 실무에서는 setter를 제거하고 생성자나 팩토리 메서드로만 상태를 설정하는 불변 객체 설계를 지향합니다.

자주 나오는 꼬리질문: 불변 객체의 단점은 없나요? → 상태 변경 시마다 새 객체를 생성하므로 잦은 변경이 있는 경우 GC 부담이 생길 수 있습니다. String vs StringBuilder가 대표적인 트레이드오프 예시입니다.
SOLID 원칙 다섯 가지를 설명해주세요.

SOLID는 좋은 객체지향 설계를 위한 다섯 가지 원칙의 머리글자입니다.

원칙이름핵심 내용
S 단일 책임 (SRP) 하나의 클래스는 변경의 이유가 하나뿐이어야 합니다. 책임이 여럿이면 하나가 바뀔 때 관련 없는 기능이 영향 받습니다.
O 개방-폐쇄 (OCP) 확장에는 열려 있고, 수정에는 닫혀 있어야 합니다. 인터페이스 + 새 구현체 추가로 기존 코드를 건드리지 않고 확장합니다.
L 리스코프 치환 (LSP) 자식 클래스는 부모 클래스를 완전히 대체할 수 있어야 합니다. 부모 타입 변수로 자식을 써도 프로그램이 올바르게 동작해야 합니다.
I 인터페이스 분리 (ISP) 사용하지 않는 메서드를 구현하도록 강제해선 안 됩니다. 뚱뚱한 인터페이스 하나보다 역할별 작은 인터페이스 여러 개가 낫습니다.
D 의존관계 역전 (DIP) 고수준 모듈이 저수준 모듈에 직접 의존하면 안 됩니다. 둘 다 추상화(인터페이스)에 의존해야 합니다.
가장 좋아하는 SOLID 원칙과 이유는?

저는 OCP(개방-폐쇄 원칙)를 가장 좋아합니다. 실무에서 요구사항 변경은 피할 수 없는데, OCP를 잘 지킨 코드는 기존 로직을 건드리지 않고 새로운 기능을 추가할 수 있기 때문입니다. 기능 추가 때마다 기존 코드를 수정하면 이미 검증된 로직에 버그가 생길 위험이 항상 존재합니다.

인터페이스 하나를 잘 설계해두면 결제 수단이나 할인 정책이 추가될 때 구현체만 만들면 되는 경험을 한 뒤 OCP의 가치를 체감했습니다. Strategy 패턴이 OCP를 실현하는 대표적인 방법으로, if-else 분기 없이 새로운 전략 클래스만 추가하면 됩니다.

자주 나오는 꼬리질문: OCP를 완벽히 지키는 게 가능한가요? → 현실에서는 어렵습니다. 처음부터 모든 변경 방향을 예측해 추상화를 설계하는 것은 오버엔지니어링이 될 수 있습니다. 실제로 변경이 두세 번 반복된 지점을 추상화하는 점진적 접근이 현실적입니다.
LSP 위반의 대표적인 예시는 무엇인가요?

가장 유명한 예시는 정사각형(Square)이 직사각형(Rectangle)을 상속하는 경우입니다. Rectangle에는 너비와 높이를 독립적으로 설정하는 메서드가 있습니다. 그런데 Square를 Rectangle처럼 다루면서 setWidth(5)를 호출하면, Square는 정사각형의 성질을 유지하기 위해 높이도 5로 바꿔버립니다. Rectangle이라고 믿고 쓰던 코드에서 예상치 못한 동작이 발생하므로 LSP 위반입니다.

LSP를 검증하는 실용적인 방법은 "부모 타입의 단위 테스트를 자식 타입으로 그대로 돌렸을 때 모두 통과하는가"입니다. 통과하지 못하면 LSP 위반입니다.

원시 타입 8가지와 크기를 말해주세요. 왜 스택에 저장되나요?

원시 타입은 총 8가지입니다. 정수형으로 byte(1B), short(2B), int(4B), long(8B), 실수형으로 float(4B), double(8B), 문자형 char(2B, 유니코드), 논리형 boolean(1bit, JVM은 보통 1B 처리)입니다.

원시 타입이 스택에 저장되는 이유는 크기가 고정되어 있고 단순한 값이기 때문입니다. 스택은 접근 속도가 빠르고 메서드 종료 시 자동으로 해제되어 GC 부담이 없습니다. 단, 인스턴스 필드로 선언된 원시 타입은 객체와 함께 힙에 저장된다는 점은 꼭 기억해야 합니다.

저장 위치스택 (필드면 힙)힙 (주소는 스택)
null 가능XO
메서드 호출XO

래퍼 클래스(Wrapper Class)란 무엇이고 왜 필요한가요? 오토박싱의 주의점은?

래퍼 클래스는 원시 타입을 객체로 감싼 클래스입니다. int → Integer, double → Double, boolean → Boolean처럼 대응됩니다. 컬렉션(List, Map 등)은 제네릭에 원시 타입을 직접 쓸 수 없기 때문에 래퍼 클래스가 필요합니다. 또한 null을 표현해야 할 때, Integer.parseInt() 같은 유틸 메서드를 쓸 때도 래퍼 클래스를 사용합니다.

Java 5부터 지원되는 오토박싱(auto-boxing)은 원시 타입과 래퍼 클래스 간 변환을 컴파일러가 자동으로 해줍니다. 그러나 반복문 안에서 오토박싱이 반복 발생하면 매번 객체가 생성되어 성능 저하와 GC 부담이 커질 수 있습니다. Integer를 ==로 비교할 때 -128~127 범위는 캐시를 사용해 같은 객체를 반환하지만, 그 범위를 벗어나면 equals()로 비교해야 합니다.

// 오토박싱 주의 — 반복문에서의 성능 저하
Long sum = 0L;
for (long i = 0; i < 1_000_000; i++) {
    sum += i;  // 매 반복마다 Long 객체 생성 → 성능 저하
}
// → long sum = 0L; 으로 원시 타입 사용이 올바른 방법

// Integer == 비교 함정
Integer a = 127, b = 127;  System.out.println(a == b);   // true (캐시)
Integer c = 128, d = 128;  System.out.println(c == d);   // false (새 객체)
System.out.println(c.equals(d));                          // true (값 비교)
float vs double, 금융 계산에는 왜 둘 다 쓰면 안 되나요?

일반적으로 double을 씁니다. float은 4바이트로 정밀도가 약 7자리, double은 8바이트로 약 15자리입니다. Java의 실수 리터럴도 기본이 double이라 3.14f처럼 명시하지 않으면 double로 처리됩니다.

그러나 금융 계산에는 float도 double도 쓰면 안 됩니다. 부동소수점은 이진수로 근사값을 표현하기 때문에 0.1 + 0.2 == 0.3이 false가 됩니다. 금융처럼 정확한 소수 표현이 필요한 경우에는 반드시 BigDecimal을 사용해야 합니다. 다만 BigDecimal은 문자열 생성자로 초기화해야 정밀도가 보장됩니다.

System.out.println(0.1 + 0.2);          // 0.30000000000000004 ← 부동소수점 오차
System.out.println(0.1 + 0.2 == 0.3);  // false ← 주의!

// 금융 계산: BigDecimal 사용
BigDecimal a = new BigDecimal("0.1");   // 문자열 생성자 사용 필수
BigDecimal b = new BigDecimal("0.2");
System.out.println(a.add(b));           // 0.3 ← 정확
알고 있는 디자인 패턴을 카테고리별로 설명해주세요.

GoF 디자인 패턴은 생성·구조·행동 세 카테고리로 나뉩니다.

패턴카테고리설명
Singleton 생성 인스턴스를 하나만 보장합니다. Spring Bean이 대표적 예입니다. 상태가 없는 Service, Repository 객체를 재사용해 객체 생성 비용을 줄입니다.
Builder 생성 생성자 파라미터가 많을 때 가독성과 안전성을 높여줍니다. Lombok @Builder로 일상적으로 사용합니다.
Proxy 구조 실제 객체 앞에 대리자를 두어 부가 기능(로깅, 트랜잭션, 캐시)을 덧붙입니다. Spring AOP의 핵심 메커니즘입니다.
Strategy 행동 알고리즘을 런타임에 교체합니다. if-else 분기 대신 각 전략을 인터페이스 구현체로 분리해 OCP를 실현합니다.
Observer 행동 이벤트 발생 시 구독자에게 알림을 보냅니다. Spring의 ApplicationEvent, MQ 기반 이벤트 처리가 이 패턴입니다.
Template Method 행동 알고리즘 뼈대는 부모에서 final로 고정하고, 세부 단계만 자식이 구현합니다. JdbcTemplate, AbstractController가 대표적입니다.
Spring에서 Proxy 패턴은 어디에 쓰이나요? JDK Proxy와 CGLIB의 차이는?

Spring AOP에서 @Transactional, @Cacheable, @Async 같은 어노테이션이 붙은 Bean에 Spring이 프록시 객체를 씌웁니다. 실제 메서드 호출 전후로 트랜잭션 시작/커밋, 캐시 처리 같은 부가 로직을 끼워넣는 방식입니다.

프록시 생성 방식은 두 가지입니다. JDK Dynamic Proxy는 인터페이스를 기반으로 프록시를 생성하며, 타깃 객체가 인터페이스를 구현해야 합니다. CGLIB는 바이트코드를 조작해 클래스를 상속하는 방식으로 프록시를 만들기 때문에 인터페이스 없이도 동작합니다. Spring Boot 2.x 이후 기본은 CGLIB입니다. 단, CGLIB는 final 클래스나 final 메서드에는 적용할 수 없다는 제약이 있습니다.

자주 나오는 꼬리질문: @Transactional이 같은 클래스 내부 호출에서 안 먹히는 이유는? → 프록시를 거치지 않아서입니다. 같은 클래스에서 self-invoke하면 this를 통해 직접 호출되어 프록시 래핑이 적용되지 않습니다.
Spring에서 왜 Singleton을 쓰나요? 주의해야 할 점은?

웹 서버는 수많은 요청을 동시에 처리합니다. 요청마다 Service, Repository 객체를 새로 생성하면 매 요청마다 수십 개의 객체가 생성·소멸되어 GC 부담과 성능 저하가 심각해집니다. 반면 Service, Repository는 내부에 상태(인스턴스 변수)를 가질 필요가 없고 로직만 담고 있어, 하나의 인스턴스를 여러 스레드가 공유해도 안전합니다.

주의할 점은 Singleton Bean이 인스턴스 변수에 상태(mutable state)를 저장하면 안 된다는 것입니다. 여러 스레드가 동시에 같은 필드를 읽고 쓰면 레이스 컨디션이 발생합니다. 요청별 데이터는 메서드의 지역 변수나 ThreadLocal에 저장해야 합니다.

Sync/Async와 Blocking/Non-Blocking의 차이를 설명해주세요.

동기/비동기는 "결과를 언제 처리하느냐"에 관한 개념입니다. 동기는 요청을 보낸 후 결과가 올 때까지 기다렸다가 다음 작업을 합니다. 비동기는 요청 후 결과를 기다리지 않고 다음 작업을 먼저 수행하다가 결과가 오면 콜백/Future로 처리합니다.

블로킹/논블로킹은 "스레드가 제어권을 반환하느냐"에 관한 개념으로 동기/비동기와는 다른 축입니다. 블로킹은 작업 완료까지 해당 스레드가 대기 상태가 되어 다른 일을 못 합니다. 논블로킹은 작업 완료를 기다리지 않고 스레드가 즉시 제어권을 반환받아 다른 작업을 계속할 수 있습니다.

조합설명예시
동기 + 블로킹결과 기다리며 스레드 멈춤일반 JDBC 쿼리
동기 + 논블로킹결과 기다리지만 스레드는 계속 polling바쁜 대기(busy wait)
비동기 + 블로킹요청 후 콜백이지만 결과 수신 시 블로킹Future.get()
비동기 + 논블로킹요청 후 즉시 반환, 콜백으로 처리CompletableFuture, WebFlux
@Async의 동작 원리는? 주의해야 할 함정 두 가지를 말해주세요.

@EnableAsync를 선언하면 Spring이 @Async가 붙은 메서드를 가진 Bean에 AOP 프록시를 씌웁니다. 해당 메서드가 호출될 때 호출자의 스레드가 아닌, 별도의 Thread Pool에서 실행되도록 가로챕니다.

함정이 두 가지 있습니다. 첫째, 같은 클래스 내부에서 @Async 메서드를 호출하면 비동기로 동작하지 않습니다. 프록시를 거치지 않는 self-invoke이기 때문입니다. 둘째, 기본 Executor는 스레드를 재사용하지 않아 부하가 높을 때 스레드가 무한 생성될 수 있습니다. 실무에서는 반드시 ThreadPoolTaskExecutor를 별도로 설정해야 합니다.

// 올바른 ThreadPoolTaskExecutor 설정
@Configuration
@EnableAsync
public class AsyncConfig {

    @Bean
    public ThreadPoolTaskExecutor taskExecutor() {
        ThreadPoolTaskExecutor executor = new ThreadPoolTaskExecutor();
        executor.setCorePoolSize(5);      // 기본 스레드 수
        executor.setMaxPoolSize(20);      // 최대 스레드 수
        executor.setQueueCapacity(100);   // 큐 용량
        executor.setThreadNamePrefix("async-");
        executor.initialize();
        return executor;
    }
}
Java에서 스레드를 생성하는 방법을 비교해주세요. 실무에서는 어떤 방식을 쓰나요?

전통적인 방법으로 Thread 상속과 Runnable 구현이 있습니다. Thread를 상속하면 다른 클래스를 상속할 수 없어 Runnable이 더 유연합니다. Java 5의 Callable + Future는 작업 결과값을 반환받을 수 있게 해줬고, Java 8의 CompletableFuture는 비동기 작업을 체이닝·조합할 수 있어 훨씬 표현력이 높습니다.

실무에서는 스레드를 직접 생성하지 않고 ExecutorService(Thread Pool)를 통해 스레드를 재사용합니다. 스레드 생성 비용은 비싸기 때문에 풀에서 관리하는 것이 효율적입니다. 고수준에서는 Spring의 @Async + ThreadPoolTaskExecutor를 주로 사용하고, 반응형(reactive) 처리가 필요할 때는 WebFlux + Project Reactor를 사용합니다.
