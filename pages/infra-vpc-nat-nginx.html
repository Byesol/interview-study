VPC · NAT · Nginx

인프라 네트워크 면접 가이드 — 실전 Q&A

VPC(Virtual Private Cloud)란 무엇이고, 왜 필요한가요?

VPC는 클라우드 내에서 논리적으로 격리된 네트워크 공간입니다. 비유하자면 거대한 오피스 빌딩(AWS/GCP)에서 우리 회사만의 전용 층을 임대한 것과 같습니다. 같은 빌딩에 다른 회사(다른 계정)가 있지만, 우리 층에는 우리만 출입할 수 있습니다.

VPC가 필요한 이유는 보안 격리와 네트워크 제어 때문입니다. VPC를 사용하면 IP 대역(CIDR)을 직접 설정하고, 서브넷을 나누고, 라우팅 테이블과 보안 그룹으로 트래픽을 세밀하게 제어할 수 있습니다. 기본적으로 VPC 간에는 통신이 차단되므로, 멀티 테넌트 환경에서도 다른 고객의 리소스와 완전히 분리됩니다.

실무에서는 보통 10.0.0.0/16 같은 프라이빗 CIDR 블록을 할당하여 최대 65,536개의 IP를 사용할 수 있도록 구성합니다. 이 안에서 서브넷을 나누어 퍼블릭 영역과 프라이빗 영역을 분리하는 것이 기본 아키텍처 패턴입니다.

꼬리질문 대비: "VPC 피어링과 Transit Gateway의 차이"를 물어볼 수 있습니다. 피어링은 1:1 연결이라 VPC가 많아지면 관리가 복잡해지고, Transit Gateway는 허브-스포크 모델로 중앙에서 관리합니다.
Public Subnet과 Private Subnet은 어떻게 다르고, 왜 분리하나요?

둘의 핵심 차이는 인터넷 게이트웨이(IGW)로의 라우팅이 있느냐 없느냐입니다. Public Subnet은 라우팅 테이블에 0.0.0.0/0 → IGW 경로가 있어서 인터넷과 직접 통신할 수 있고, Private Subnet에는 이 경로가 없어 외부에서 직접 접근이 불가능합니다.

분리하는 이유는 최소 권한 원칙(Principle of Least Privilege)을 네트워크 수준에서 적용하기 위해서입니다. 비유하자면 식당에서 홀(Public)과 주방(Private)을 나누는 것과 같습니다. 손님은 홀까지만 오고, 주방에는 직원만 들어갑니다. 마찬가지로 DB, 내부 API 서버, 캐시 서버 등은 Private Subnet에 두어 인터넷에 직접 노출되지 않게 합니다.

구분Public SubnetPrivate Subnet
인터넷 접근IGW 통해 직접 가능불가 (NAT 통해 아웃바운드만)
Public IP할당 가능불필요 / 할당 안 함
배치 대상ALB, Bastion Host, NAT GWWAS, DB, Redis, 내부 서비스
보안 수준상대적으로 낮음 (외부 노출)높음 (외부 차단)

실무에서는 보통 가용영역(AZ)마다 Public/Private 서브넷 쌍을 만들어 고가용성을 확보합니다. 예를 들어 2개 AZ라면 최소 4개 서브넷(Public 2 + Private 2)이 기본 구성이 됩니다.

VPC의 일반적인 아키텍처를 설명해 주세요.

위 다이어그램은 전형적인 2-AZ VPC 구성입니다. Public Subnet에는 ALB(Application Load Balancer)와 NAT Gateway를 배치하고, Private Subnet에는 Nginx(리버스 프록시), WAS(애플리케이션 서버), DB, 캐시 서버를 배치합니다. 외부 트래픽은 IGW → ALB → Private Subnet의 서버 순으로 흐르고, Private Subnet의 아웃바운드 트래픽(패키지 업데이트, 외부 API 호출 등)은 NAT Gateway를 거쳐 나갑니다.

NAT가 무엇이고, 클라우드에서 왜 필요한가요?

NAT는 사설 IP 주소를 공인 IP 주소로 변환해주는 기술입니다. 비유하자면, 회사 내선 전화(사설 IP)로 외부에 전화를 걸 때 회사 대표번호(공인 IP)로 발신되는 것과 같습니다. 외부에서는 대표번호만 보이고 내선번호는 알 수 없습니다.

클라우드에서 NAT가 필요한 핵심 이유는, Private Subnet의 인스턴스가 인터넷에 접근해야 하는 경우가 있기 때문입니다. OS 패치, 외부 API 호출, NTP 시간 동기화, 패키지 다운로드 등이 대표적입니다. 하지만 동시에 외부에서 이 인스턴스에 직접 접근하는 것은 차단해야 합니다. NAT는 이 아웃바운드는 허용, 인바운드는 차단이라는 비대칭 통신을 가능하게 해줍니다.

꼬리질문 대비: "NAT를 안 쓰고 Private Subnet에서 외부 통신을 할 수 있는 방법은?" — VPC Endpoint(S3, DynamoDB 등 AWS 서비스 접근), PrivateLink(다른 서비스와 프라이빗 연결) 등이 있습니다. 이들은 인터넷을 거치지 않아 더 안전하고 비용도 절감됩니다.
NAT Gateway와 NAT Instance의 차이는?

AWS 기준으로, 둘 다 NAT 기능을 제공하지만 운영 방식이 크게 다릅니다.

구분NAT Gateway (관리형)NAT Instance (EC2 기반)
관리AWS가 관리 (패치, 가용성)직접 EC2 운영 · 패치
가용성AZ 내 자동 이중화직접 HA 구성 필요
대역폭최대 100 Gbps 자동 확장인스턴스 타입에 의존
비용시간당 + 데이터 처리 요금EC2 비용 (저렴할 수 있음)
보안그룹연결 불가 (NACL로 제어)보안그룹 적용 가능
포트포워딩불가가능 (iptables)

실무에서는 프로덕션 환경에서는 거의 100% NAT Gateway를 사용합니다. 관리 부담이 없고 자동 확장·이중화가 보장되기 때문입니다. 다만 비용이 걱정되는 개발/테스트 환경에서는 NAT Instance나 fck-nat 같은 경량 솔루션을 쓰기도 합니다. 면접에서는 "관리형 vs 비관리형의 trade-off를 이해하고 상황에 맞게 선택한다"는 점을 보여주는 것이 중요합니다.

NAT Gateway의 동작 원리를 설명해 주세요.

Private Subnet의 인스턴스(예: 10.0.2.10)가 외부 API(예: api.github.com)에 요청을 보낸다고 가정하겠습니다. 이때 흐름은 다음과 같습니다.

  1. 1인스턴스가 패킷을 보냅니다. 출발지: 10.0.2.10:54321, 도착지: 140.82.121.3:443
  2. 2Private Subnet의 라우팅 테이블에 0.0.0.0/0 → NAT Gateway로 설정되어 있으므로, 패킷이 NAT Gateway로 전달됩니다.
  3. 3NAT Gateway가 출발지 IP를 자신의 Elastic IP(공인 IP)로 변환하고, 내부적으로 변환 테이블(Connection Tracking Table)에 매핑을 기록합니다.
  4. 4응답이 돌아오면, NAT Gateway가 변환 테이블을 조회하여 원래 인스턴스(10.0.2.10)로 패킷을 전달합니다.

핵심은 이 과정에서 외부 서버는 NAT Gateway의 공인 IP만 알 뿐, Private Subnet 인스턴스의 사설 IP는 전혀 노출되지 않는다는 것입니다. 또한 외부에서 먼저 연결을 시도해도 변환 테이블에 매핑이 없으므로 패킷이 드롭됩니다.

Nginx란 무엇이고, Apache와 어떻게 다른가요?

Nginx는 이벤트 기반(Event-Driven) 비동기 아키텍처의 웹 서버이자 리버스 프록시입니다. Apache가 식당에서 손님 한 명당 웨이터 한 명을 배치하는 방식(프로세스/스레드 기반)이라면, Nginx는 한 명의 슈퍼 웨이터가 수십 개 테이블을 동시에 관리하는 방식입니다.

Apache는 기본적으로 prefork(프로세스)나 worker(스레드) MPM을 사용하여 각 연결마다 프로세스나 스레드를 할당합니다. 동시 접속이 수만 건으로 늘어나면 메모리 사용량이 급격히 증가합니다. 반면 Nginx는 소수의 워커 프로세스가 epoll/kqueue 같은 OS 이벤트 알림 메커니즘을 사용해 수천~수만 개의 동시 연결을 비동기로 처리합니다. 이 때문에 C10K 문제(동시 1만 연결 처리)를 해결하는 대표적인 솔루션으로 자리잡았습니다.

구분NginxApache
아키텍처이벤트 기반 비동기프로세스/스레드 기반
동시 접속 처리수만 연결도 적은 메모리연결당 프로세스/스레드
정적 파일 처리매우 빠름상대적으로 느림
설정 반영설정 파일 기반, reload.htaccess 지원 (동적)
모듈컴파일 시 포함 (동적 모듈도 지원)런타임 동적 로드

실무에서는 둘을 경쟁 관계보다는 조합해서 쓰기도 합니다. Nginx를 프론트(리버스 프록시)에 두고, 뒤에 Apache+PHP 같은 전통적 스택을 두는 구성도 흔합니다.

Nginx의 리버스 프록시 역할을 설명해 주세요.

리버스 프록시란 클라이언트와 백엔드 서버 사이에서 중개자 역할을 하는 서버입니다. 비유하자면 호텔의 컨시어지 데스크와 같습니다. 고객(클라이언트)은 컨시어지(Nginx)에게 요청하고, 컨시어지가 적절한 직원(백엔드 서버)에게 전달한 뒤 결과를 고객에게 돌려줍니다. 고객은 뒤에서 누가 처리했는지 알 필요가 없습니다.

Nginx가 리버스 프록시로서 수행하는 핵심 기능은 다음과 같습니다. 로드 밸런싱으로 여러 WAS에 트래픽을 분산하고, SSL Termination으로 HTTPS 처리를 Nginx에서 담당하여 백엔드 부담을 줄이며, 캐싱으로 정적 컨텐츠를 Nginx 레벨에서 바로 응답하고, 보안 측면에서 백엔드 서버의 IP와 구조를 외부에 숨깁니다.

# 기본 리버스 프록시 설정 예시
upstream backend {
    server 10.0.2.10:8080;
    server 10.0.2.11:8080;
}

server {
    listen 80;
    server_name api.example.com;

    location / {
        proxy_pass http://backend;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    }
}

위 설정에서 proxy_set_header X-Real-IP는 면접에서 자주 나오는 포인트입니다. Nginx가 중개하면 백엔드에는 Nginx의 IP가 찍히므로, 원래 클라이언트 IP를 전달하기 위해 이 헤더를 설정해야 합니다.

Nginx의 로드 밸런싱 알고리즘은 어떤 것들이 있나요?

Nginx는 여러 로드 밸런싱 방식을 지원하며, 상황에 따라 적절한 것을 선택해야 합니다.

알고리즘설명적합한 상황
Round Robin (기본)순서대로 돌아가며 배분서버 성능이 동일할 때
Weighted Round Robin가중치에 따라 배분서버 스펙이 다를 때
Least Connections현재 연결이 가장 적은 서버로요청 처리 시간이 불균일할 때
IP Hash클라이언트 IP로 해시하여 고정세션 유지가 필요할 때

면접관이 여기서 파고들 수 있는데요, "세션 유지가 필요하면 IP Hash를 쓰는 게 최선인가?"라는 질문이 올 수 있습니다. IP Hash는 특정 서버에 부하가 집중되는 문제가 있고, 프록시 뒤의 사용자는 같은 IP로 묶이므로, 실무에서는 Sticky Session(쿠키 기반)이나 아예 세션을 Redis 같은 외부 저장소로 분리하는 것이 더 나은 접근입니다.

Nginx의 이벤트 기반 아키텍처는 내부적으로 어떻게 동작하나요?

Nginx는 Master-Worker 프로세스 모델로 동작합니다. Master 프로세스는 설정 파일을 읽고 워커 프로세스를 관리하는 역할만 하고, 실제 요청 처리는 Worker 프로세스가 담당합니다. 보통 Worker 수는 CPU 코어 수와 동일하게 설정합니다(worker_processes auto;).

각 Worker 프로세스는 하나의 스레드에서 이벤트 루프(Event Loop)를 돌며 수천 개의 연결을 비동기로 처리합니다. 리눅스에서는 epoll, BSD에서는 kqueue를 사용합니다. 연결마다 스레드를 만들지 않기 때문에 컨텍스트 스위칭 비용이 거의 없고, 메모리 사용량도 매우 적습니다. 이것이 Nginx가 적은 리소스로 대량의 동시 연결을 처리할 수 있는 비결입니다.

다만 Nginx에서 주의할 점은, 블로킹 작업은 전체 워커를 멈출 수 있다는 것입니다. 예를 들어 디스크에서 큰 파일을 동기적으로 읽으면 해당 워커가 블로킹됩니다. 이를 방지하기 위해 aio(비동기 I/O)나 sendfile, 스레드 풀(aio threads;) 등을 활용합니다.

VPC에서 Security Group과 NACL의 차이는?

Security Group은 인스턴스 레벨의 방화벽이고, NACL(Network ACL)은 서브넷 레벨의 방화벽입니다. Security Group은 Stateful하여 인바운드를 허용하면 해당 응답(아웃바운드)은 자동으로 허용됩니다. 반면 NACL은 Stateless하여 인바운드와 아웃바운드 규칙을 각각 명시해야 합니다.

실무에서는 Security Group을 주로 사용하고, NACL은 서브넷 전체에 대한 추가 방어선(특정 IP 대역 차단 등)으로 활용합니다. 비유하자면 Security Group은 각 방의 도어락이고, NACL은 건물 출입구의 보안 게이트입니다.

Nginx에서 502 Bad Gateway가 발생하면 어떻게 트러블슈팅하나요?

502는 Nginx가 백엔드 서버로부터 유효한 응답을 받지 못했다는 의미입니다. 원인을 단계별로 확인합니다.

먼저, 백엔드 서버가 살아있는지 확인합니다. WAS가 죽었거나 포트가 열려있지 않은 경우가 가장 흔합니다. 다음으로, upstream timeout 설정을 확인합니다. 백엔드 처리 시간이 proxy_read_timeout(기본 60초)을 초과하면 502가 발생합니다. 그 다음으로, Keep-Alive 관련 race condition을 의심합니다. ALB의 idle timeout과 Nginx의 keepalive_timeout 불일치가 원인일 수 있습니다. 마지막으로, 백엔드의 max connections가 초과되었는지도 확인합니다.

디버깅 시 error.log에 나오는 upstream 관련 로그가 가장 중요하며, proxy_next_upstream 설정으로 실패 시 다른 백엔드로 재시도하게 구성하는 것이 실무 팁입니다.

Private Subnet의 인스턴스에 SSH 접속은 어떻게 하나요?

Private Subnet은 외부에서 직접 접근이 불가하므로, 몇 가지 방법이 있습니다. 전통적으로는 Public Subnet에 Bastion Host(점프 서버)를 두고, 이를 경유하여 SSH 접속합니다. 하지만 Bastion Host 자체가 보안 취약점이 될 수 있고 관리 부담이 있습니다.

최근에는 AWS SSM Session Manager가 더 선호됩니다. 별도 인바운드 포트(22)를 열 필요 없이 IAM 권한으로 접속하며, 모든 세션이 CloudTrail에 기록되어 감사(audit)에도 유리합니다. SSH 키 관리 부담도 없어지므로, 면접에서는 "Bastion Host 방식을 알지만, 보안과 운영 편의성 측면에서 SSM을 선호한다"고 답하면 좋은 인상을 줄 수 있습니다.
