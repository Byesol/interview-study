TLB & 컨텍스트 스위칭 심화 면접 가이드

가상 메모리 → 페이지 테이블 → TLB → TLB 플러시 → 컨텍스트 스위칭 비용의 전체 그림

TLB를 이해하려면 먼저 "가상 메모리"와 "페이지 테이블"을 알아야 합니다. Part 1부터 순서대로 읽어주세요.
가상 메모리(Virtual Memory)란 무엇이고, 왜 필요한가요?
비유로 먼저 이해하기
호텔을 생각해보세요. 손님(프로세스)은 자기가 "301호"에 묵는다고 알고 있습니다. 하지만 호텔 측(OS)은 내부 사정에 따라 실제 물리적인 방을 유동적으로 배정합니다. 손님은 항상 "301호"라는 가상 주소만 사용하고, 실제 어떤 방(물리 메모리)에 있는지는 프론트(OS)가 관리합니다.

가상 메모리는 각 프로세스에게 "자기만의 넓은 메모리 공간이 있다"는 환상을 주는 기법입니다. 모든 프로세스는 0번지부터 시작하는 자기만의 가상 주소 공간을 갖습니다. 프로세스 A의 주소 0x1000과 프로세스 B의 주소 0x1000은 같은 숫자이지만, 실제 물리 메모리에서는 완전히 다른 위치에 매핑됩니다.

왜 이게 필요한지 세 가지 이유가 있습니다. 첫째, 격리(Isolation)입니다. 프로세스 A가 프로세스 B의 메모리를 직접 접근할 수 없으므로 보안과 안정성이 보장됩니다. 둘째, 편의성입니다. 프로그래머는 다른 프로세스가 메모리를 얼마나 쓰는지 신경 쓸 필요 없이 0번지부터 사용하면 됩니다. 셋째, 물리 메모리보다 큰 공간을 쓸 수 있습니다. 실제 RAM이 8GB여도 프로세스에게 훨씬 큰 가상 공간을 줄 수 있고, 당장 안 쓰는 페이지는 디스크(스왑)로 내보내면 됩니다.

가상 주소를 물리 주소로 바꾸는 "페이지 테이블"은 어떻게 동작하나요?
비유로 먼저 이해하기
도서관의 색인 카드를 떠올려보세요. "제목: 해리포터 → 서가 B, 3번째 칸"처럼, 페이지 테이블은 "가상 페이지 5번 → 물리 프레임 12번"이라는 번역표입니다. CPU가 가상 주소를 내면, 이 번역표를 보고 실제 물리 주소를 찾습니다.

OS는 메모리를 페이지(Page)라는 고정 크기 블록(보통 4KB)으로 나눕니다. 가상 메모리의 블록을 "페이지", 물리 메모리의 블록을 "프레임"이라고 부릅니다. 페이지 테이블은 각 프로세스마다 하나씩 존재하며, "가상 페이지 번호 → 물리 프레임 번호"의 매핑을 저장하는 자료구조입니다.

CPU가 가상 주소 0x12345에 접근하면, 이 주소는 두 부분으로 나뉩니다. 앞부분이 가상 페이지 번호(VPN)이고, 뒷부분이 오프셋(Offset)입니다. VPN으로 페이지 테이블을 조회하면 물리 프레임 번호(PFN)를 얻고, 여기에 오프셋을 붙이면 실제 물리 주소가 완성됩니다.

여기서 큰 문제가 생깁니다. 페이지 테이블 자체가 메모리(RAM)에 저장되어 있다는 점입니다. CPU가 메모리 주소 하나를 읽으려면, 먼저 메모리에 있는 페이지 테이블을 읽어서 물리 주소를 알아내고, 그 다음에 실제 데이터를 읽어야 합니다. 메모리 접근이 2배로 늘어나는 것입니다. 이 문제를 해결하기 위해 등장한 것이 바로 TLB입니다.

TLB(Translation Lookaside Buffer)란 정확히 무엇인가요?
비유로 먼저 이해하기
페이지 테이블이 두꺼운 사전이라면, TLB는 포스트잇에 적어놓은 자주 쓰는 단어 메모입니다. 사전을 매번 펼치는 건 느리니까, 자주 찾는 단어를 포스트잇에 적어서 모니터에 붙여두는 겁니다. 포스트잇에 있으면 바로 확인(TLB 히트), 없으면 사전을 다시 펼쳐야 합니다(TLB 미스).

TLB는 CPU 칩 안에 있는 작지만 매우 빠른 하드웨어 캐시로, 최근에 사용한 "가상 페이지 번호 → 물리 프레임 번호" 매핑을 저장합니다. 페이지 테이블은 메모리(RAM)에 있어서 접근하는 데 수십~수백 나노초가 걸리지만, TLB는 CPU 내부에 있어서 1~2 나노초 만에 조회할 수 있습니다.

구분속도비고
TLB 조회~1nsCPU 내부 하드웨어
L1 캐시~1-2nsCPU 코어 내
메모리 (RAM)~100ns페이지 테이블 위치
디스크 (SSD)~100,000ns스왑 영역

TLB의 크기는 매우 작습니다. 보통 64~1024개 엔트리 정도입니다. 하지만 프로그램은 지역성(Locality)이 있어서 — 최근에 접근한 메모리 근처를 또 접근하는 경향 — 작은 TLB로도 95~99%의 히트율을 달성할 수 있습니다. 이것이 TLB가 효과적인 핵심 이유입니다.

TLB 히트와 TLB 미스일 때 각각 어떤 일이 일어나나요?

CPU가 가상 주소에 접근할 때, 가장 먼저 TLB를 확인합니다. 여기서 두 가지 경우로 나뉩니다.

TLB 히트(Hit): TLB에 해당 가상 페이지의 매핑 정보가 있는 경우입니다. 즉시 물리 프레임 번호를 얻어서 물리 주소를 완성하고 메모리에 접근합니다. 추가적인 메모리 접근 없이 끝나므로 매우 빠릅니다. 전체 메모리 접근의 95~99%가 이 경우입니다.

TLB 미스(Miss): TLB에 매핑 정보가 없는 경우입니다. 이때는 메모리에 있는 페이지 테이블을 직접 조회해야 합니다. 매핑 정보를 찾으면 그 결과를 TLB에 새로 등록(TLB가 꽉 찼으면 오래된 엔트리를 교체)하고, 그 다음에 실제 데이터를 읽습니다. 미스가 나면 메모리 접근이 2번(페이지 테이블 1번 + 실제 데이터 1번) 필요해서 느립니다.

꼬리질문 대비: "멀티레벨 페이지 테이블에서 TLB 미스가 나면?" — 64비트 시스템의 4단계 페이지 테이블에서는 미스 한 번에 메모리 접근이 4번(각 레벨마다 1번) + 실제 데이터 1번 = 총 5번이 됩니다. TLB의 중요성이 더욱 커지는 이유입니다.
TLB의 구조는 어떻게 되어 있나요? 엔트리에 어떤 정보가 들어있나요?

TLB 하나의 엔트리는 크게 세 부분으로 구성됩니다. VPN(가상 페이지 번호)은 검색 키 역할을 합니다. PFN(물리 프레임 번호)은 변환 결과입니다. 그리고 부가 비트들이 있는데, 유효 비트(Valid bit), 보호 비트(읽기/쓰기/실행 권한), 더티 비트(수정 여부), ASID(Address Space ID) 등이 포함됩니다.

TLB의 하드웨어 구조는 보통 Fully Associative 방식입니다. 이것은 검색할 때 모든 엔트리를 동시에 비교하는 것으로, 모든 엔트리에 병렬로 VPN을 대조해서 매칭되는 것을 찾습니다. 엔트리 수가 적기(64~1024개) 때문에 이 방식이 가능하고, 덕분에 1~2 클럭 사이클 만에 결과를 얻을 수 있습니다.

면접관이 여기서 더 파고들 수 있는 부분이 ASID입니다. ASID는 "이 TLB 엔트리가 어떤 프로세스의 것인지"를 태깅하는 식별자입니다. ASID가 있으면 컨텍스트 스위칭 때 TLB를 통째로 비울 필요 없이, 현재 프로세스의 ASID와 일치하는 엔트리만 히트로 인정하면 됩니다. 이것이 Part 3에서 다룰 TLB 플러시 비용을 줄이는 핵심 기법입니다.

TLB 플러시(Flush)란 무엇이고, 왜 필요한가요?
비유로 먼저 이해하기
회사에서 직원 A가 쓰던 모니터의 포스트잇(TLB)에 "거래처 전화번호"가 적혀 있습니다. 이제 직원 B가 그 자리에 앉으면? 포스트잇 내용은 직원 A의 것이라 직원 B에겐 틀린 정보입니다. "삼성전자 → 02-1234"가 직원 B에게는 "삼성전자 → 031-5678"일 수 있습니다. 그래서 자리를 바꿀 때 포스트잇을 전부 떼어내야(TLB 플러시) 합니다.

TLB 플러시는 TLB의 모든 엔트리를 무효화하는 것입니다. 컨텍스트 스위칭 때 이것이 필요한 이유는 명확합니다: 각 프로세스마다 페이지 테이블이 다르기 때문입니다.

프로세스 A에서 "가상 페이지 5 → 물리 프레임 12"라는 매핑이 TLB에 캐싱되어 있다고 합시다. 프로세스 B로 전환했는데 TLB를 그대로 두면, B가 가상 페이지 5에 접근할 때 TLB가 "물리 프레임 12"를 돌려줍니다. 하지만 이것은 프로세스 A의 물리 메모리입니다! B가 A의 데이터를 읽거나 덮어쓰게 되는 심각한 보안/안정성 문제가 발생합니다. 그래서 프로세스가 바뀔 때 TLB를 비워야 합니다.

컨텍스트 스위칭이 느린 진짜 이유를 구체적으로 설명해주세요.

많은 사람들이 컨텍스트 스위칭의 비용을 "레지스터 저장/복원"으로 알고 있는데, 이것은 빙산의 일각입니다. 진짜 비용의 대부분은 "보이지 않는 간접 비용"에서 옵니다.

비유로 전체 그림 잡기
요리사가 스테이크를 굽다가 갑자기 파스타를 만들어야 하는 상황입니다. 직접 비용은 "스테이크 재료를 치우고 파스타 재료를 꺼내는 시간"(레지스터 저장/복원)입니다. 하지만 진짜 느린 이유는: 스테이크를 위해 예열해뒀던 팬이 식고(CPU 캐시 오염), 가까이 놓아둔 양념을 치우고 파스타 양념을 찬장에서 꺼내야 하고(TLB 미스), 스테이크 레시피가 머릿속에서 사라져 다시 레시피를 봐야 하는 것(파이프라인 플러시)입니다.

1단계: 직접 비용 (작음, ~수 μs)
레지스터 값과 PC를 PCB에 저장하고, 새 프로세스의 PCB에서 복원합니다. 이것은 CPU 명령어 몇십 개면 끝나므로 매우 빠릅니다.

2단계: TLB 플러시 → TLB 콜드 스타트 (매우 큼)
TLB를 비운 직후, 새 프로세스가 메모리에 접근할 때마다 TLB 미스가 발생합니다. 미스가 나면 페이지 테이블을 읽으러 메모리에 가야 하는데, 이 과정이 100ns씩 걸립니다. 프로세스가 실행 초기에 자주 쓰는 페이지를 TLB에 다시 채우는 "워밍업" 기간 동안 수십~수백 번의 미스가 연쇄적으로 발생합니다. 이 기간에 메모리 접근 속도가 정상 대비 50~100배 느려질 수 있습니다.

3단계: CPU 캐시 오염 (매우 큼)
TLB뿐 아니라 L1/L2/L3 CPU 캐시도 오염됩니다. 프로세스 A가 캐시에 올려놓은 데이터가 프로세스 B의 데이터로 교체됩니다. 다시 A로 돌아왔을 때 캐시 미스가 대량 발생하면서 느려집니다. CPU 캐시 미스 한 번은 L3 기준 ~10ns, 메모리까지 가면 ~100ns입니다.

4단계: 파이프라인 플러시 (중간)
현대 CPU는 명령어를 미리 가져와서 파이프라인에 넣어두는데, 프로세스가 바뀌면 이 파이프라인을 비우고 새 명령어를 처음부터 채워야 합니다. 분기 예측기(Branch Predictor)의 학습 결과도 초기화됩니다.

비용 요소소요 시간발생 위치체감 영향
레지스터 저장/복원~수 μsCPU ↔ PCB낮음
TLB 플러시 자체~수십 nsCPU 내부 TLB낮음
TLB 콜드 스타트 (워밍업)~수십 μsCPU → 메모리 반복매우 높음
CPU 캐시 오염~수십 μsL1/L2/L3 캐시 미스매우 높음

정리하면, 컨텍스트 스위칭 자체(레지스터 교체)는 빠르지만, 그 이후에 벌어지는 캐시/TLB 워밍업의 간접 비용이 직접 비용의 수십 배에 달합니다. 이것이 컨텍스트 스위칭이 "느리다"고 말하는 진짜 이유입니다.

같은 프로세스의 스레드 간 컨텍스트 스위칭은 왜 가벼운가요?

핵심 한마디로: 같은 프로세스의 스레드들은 페이지 테이블을 공유하기 때문입니다.

프로세스 A의 스레드 1에서 스레드 2로 전환할 때, 가상 주소 공간이 바뀌지 않습니다. "가상 페이지 5 → 물리 프레임 12"라는 매핑이 스레드 1에서도, 스레드 2에서도 동일합니다. 따라서 TLB를 비울 필요가 전혀 없습니다. TLB에 캐싱된 내용이 그대로 유효하므로 콜드 스타트도 없습니다.

교체하는 것은 PC(Program Counter), 레지스터, 스택 포인터 정도입니다. CPU 캐시도 같은 프로세스의 데이터이므로 상당 부분이 여전히 유효합니다. 이것이 멀티스레드가 멀티프로세스보다 컨텍스트 스위칭 비용이 작다고 말하는 구체적인 이유입니다.

ASID를 사용하면 TLB 플러시를 안 해도 되나요?

네, ASID(Address Space Identifier)가 있으면 TLB 전체를 비우지 않아도 됩니다. ASID는 TLB 엔트리마다 "이건 프로세스 몇 번의 것"이라는 태그를 다는 것입니다.

프로세스 A(ASID=1)에서 프로세스 B(ASID=2)로 전환해도, TLB에 A와 B의 엔트리가 섞여서 공존할 수 있습니다. CPU가 가상 주소를 조회할 때, VPN뿐 아니라 현재 ASID와 TLB 엔트리의 ASID도 함께 비교합니다. ASID가 일치하지 않으면 미스로 처리됩니다. 이렇게 하면 다시 A로 돌아왔을 때 TLB에 A의 엔트리가 아직 남아 있을 수 있어, 콜드 스타트를 크게 줄일 수 있습니다.

다만 ASID는 비트 수가 제한되어 있습니다(보통 8~16비트). ARM은 8비트(256개), x86은 12비트(PCID로 4096개)를 지원합니다. 프로세스 수가 ASID 범위를 초과하면 ASID를 재활용해야 하고, 그 때는 TLB 플러시가 필요합니다. 하지만 일반적인 상황에서는 ASID가 TLB 플러시 비용을 크게 줄여줍니다.

용어 참고: Intel x86에서는 ASID 대신 PCID(Process Context Identifier)라는 이름을 사용합니다. Linux 커널은 4.14 버전부터 PCID를 적극 활용하고 있으며, 특히 Meltdown 취약점 패치(KPTI)에서 PCID가 성능 저하를 완화하는 데 핵심적인 역할을 했습니다.
페이지 크기가 크면(예: 2MB Huge Page) TLB 성능에 어떤 영향이 있나요?

페이지 크기가 커지면 TLB의 커버리지가 극적으로 증가합니다. 일반 4KB 페이지에서 TLB 엔트리 512개 = 커버리지 2MB입니다. 2MB Huge Page를 쓰면 같은 512개 엔트리로 1GB를 커버할 수 있습니다. 메모리를 많이 쓰는 프로그램(데이터베이스, JVM)에서 TLB 미스가 급감하므로 성능이 크게 향상됩니다.

실무에서 Java의 대용량 힙(-Xmx 수 GB 이상)이나 Redis, PostgreSQL 같은 메모리 집약 애플리케이션에서 Huge Page를 설정하면 성능 개선을 얻을 수 있습니다. Linux에서는 Transparent Huge Pages(THP)가 기본 활성화되어 있어 OS가 자동으로 관리합니다. 다만 Huge Page는 메모리 단편화 문제가 있고, 페이지 폴트 한 번에 복사해야 할 양이 커지므로 trade-off가 존재합니다.

x86에서 페이지 테이블 교체는 구체적으로 어떻게 이루어지나요?

x86 아키텍처에서는 CR3 레지스터가 현재 프로세스의 최상위 페이지 테이블(PML4)의 물리 주소를 가리킵니다. 컨텍스트 스위칭 때 OS가 CR3 레지스터의 값을 새 프로세스의 페이지 테이블 주소로 바꾸면, 그 순간부터 CPU는 새 프로세스의 가상 주소 체계를 사용합니다.

전통적으로 CR3에 새 값을 쓰는 순간 TLB가 자동으로 전체 플러시되었습니다. 하지만 최신 Intel CPU에서 PCID가 활성화되면, CR3에 "TLB를 플러시하지 마라"는 플래그를 함께 설정할 수 있어서 전체 플러시를 피할 수 있습니다. Linux 커널이 이 기능을 적극 활용하면서 컨텍스트 스위칭 비용이 상당히 줄었습니다.

시스템 콜(System Call)을 할 때도 TLB 플러시가 발생하나요?

일반적인 시스템 콜에서는 TLB 플러시가 발생하지 않습니다. 시스템 콜은 사용자 모드에서 커널 모드로 전환되지만, 같은 프로세스의 주소 공간 안에서 동작하므로 페이지 테이블이 바뀌지 않습니다. 커널 영역의 페이지도 프로세스의 페이지 테이블에 매핑되어 있기 때문입니다.

하지만 Meltdown 취약점 이후 상황이 달라졌습니다. 2018년에 발견된 Meltdown 공격은 사용자 모드에서 커널 메모리를 읽을 수 있는 취약점이었고, 이를 막기 위해 KPTI(Kernel Page Table Isolation)가 도입되었습니다. KPTI는 사용자 모드와 커널 모드의 페이지 테이블을 분리해서, 시스템 콜로 커널에 진입할 때 페이지 테이블을 교체합니다. 이때 TLB 플러시가 발생하면서 시스템 콜 성능이 크게 저하되었습니다.

이 성능 저하를 줄이기 위해 Intel의 PCID(ASID의 x86 버전)가 핵심적인 역할을 했습니다. 사용자 모드 페이지 테이블과 커널 모드 페이지 테이블에 다른 PCID를 부여하면, 전환 시 TLB를 완전히 비우지 않아도 됩니다.

"컨텍스트 스위칭이 잦으면 성능이 떨어진다"를 면접에서 어떻게 설명하면 좋을까요?

면접에서 이 질문이 나오면, 직접 비용 vs 간접 비용을 구분해서 설명하는 것이 좋습니다. 아래는 면접에서 바로 쓸 수 있는 답변 흐름입니다:

"컨텍스트 스위칭이 잦으면 성능이 떨어지는 이유는, 전환 자체의 비용보다 전환 이후의 간접 비용이 훨씬 크기 때문입니다. 전환할 때는 레지스터와 PC를 PCB에 저장/복원하는데, 이것은 수 마이크로초면 끝납니다. 하지만 그 이후가 문제입니다."

"첫째, TLB가 플러시됩니다. 프로세스마다 페이지 테이블이 다르기 때문에 이전 프로세스의 주소 변환 캐시를 무효화해야 합니다. 새 프로세스는 TLB가 비어있는 상태에서 시작하므로 메모리 접근마다 페이지 테이블까지 갔다 오는 추가 비용이 발생합니다."

"둘째, CPU 캐시가 오염됩니다. L1/L2/L3 캐시에 이전 프로세스의 데이터가 남아있고, 새 프로세스의 데이터로 교체되면서 캐시 미스가 연쇄 발생합니다. 이 간접 비용이 직접 비용의 수십 배에 달하기 때문에, 컨텍스트 스위칭이 자주 일어나면 CPU가 실제 일을 하는 시간보다 전환 후 워밍업에 쓰는 시간이 더 많아집니다."

"그래서 실무에서는 스레드 풀로 스레드 수를 제한하거나, 비동기/이벤트 기반 모델로 스레드 수를 줄이거나, 같은 프로세스 내 스레드를 활용해 TLB 플러시를 피하는 등의 방법으로 컨텍스트 스위칭을 줄입니다."
