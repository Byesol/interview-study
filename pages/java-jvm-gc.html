JVM & Garbage Collection

JVM 구조, 메모리 영역, GC 알고리즘, G1/ZGC, 튜닝

Part 1 — JVM 기본 구조 · JVM이 무엇이고, 어떤 구조로 동작하는지 면접관에게 설명할 수 있어야 합니다.
JVM이 무엇이고 왜 필요한가요?

JVM(Java Virtual Machine)은 자바 바이트코드를 실행하는 가상 머신입니다. 비유하자면, 자바 소스코드를 한국어로 쓴 책이라고 하면, 컴파일러가 이걸 '국제 공용 번역본(바이트코드)'으로 만들고, JVM은 각 나라(OS)에서 그 번역본을 현지 언어로 읽어주는 통역사입니다.

JVM이 필요한 핵심 이유는 "Write Once, Run Anywhere"입니다. 개발자가 Windows에서 작성한 코드가 Linux, Mac에서도 그대로 실행됩니다. JVM이 OS별 차이를 추상화해주기 때문입니다. 또한 JVM은 메모리 관리(GC), 보안(샌드박스), 최적화(JIT 컴파일)를 자동으로 처리해줍니다.

꼬리질문 대비

"자바가 인터프리터 언어인가요, 컴파일 언어인가요?" — 둘 다입니다. javac가 소스를 바이트코드로 컴파일하고, JVM이 바이트코드를 인터프리트하다가 핫스팟을 발견하면 JIT 컴파일로 네이티브 코드로 변환합니다.

JVM의 메모리 구조를 설명해주세요.

JVM 메모리는 크게 5가지 영역으로 구분됩니다. 비유하자면 JVM은 하나의 사무실 건물이고, 각 영역은 용도가 다른 방입니다.

Method Area는 클래스 메타데이터, static 변수, 상수 풀이 저장되는 공용 창고입니다. Heap은 new 키워드로 생성된 모든 객체가 살아있는 곳이며, GC의 주요 대상입니다. 이 두 영역은 모든 스레드가 공유합니다.

반면 Stack, PC Register, Native Method Stack은 스레드마다 독립적으로 존재합니다. Stack에는 메서드 호출 시 프레임 단위로 지역변수와 연산 결과가 쌓이고, 메서드가 끝나면 프레임이 제거됩니다. 면접관이 "스택 오버플로우가 왜 발생하나요?"라고 물으면, 재귀 호출이 너무 깊어져 스레드의 Stack 크기를 초과할 때 발생한다고 답하면 됩니다.

JIT 컴파일러란 무엇이고, 어떻게 최적화하나요?

JIT(Just-In-Time) 컴파일러는 자주 실행되는 코드(핫스팟)를 런타임에 네이티브 머신 코드로 변환하는 최적화 엔진입니다. 식당에 비유하면, 처음 주문은 레시피를 보고 만들지만(인터프리팅), 같은 메뉴가 계속 나오면 아예 레시피를 외워서 빠르게 만드는 것(JIT 컴파일)과 같습니다.

HotSpot JVM에는 C1(Client) 컴파일러와 C2(Server) 컴파일러가 있습니다. C1은 빠른 컴파일에 초점을 맞추고, C2는 시간이 더 걸리지만 더 공격적인 최적화를 수행합니다. Java 9부터는 Tiered Compilation이 기본으로, C1으로 먼저 컴파일한 뒤 핫한 코드를 C2로 재컴파일하는 전략을 사용합니다.

대표적인 JIT 최적화로는 인라이닝(메서드 호출을 본문으로 대체), 루프 언롤링, 이스케이프 분석(객체가 메서드 밖으로 나가지 않으면 스택에 할당) 등이 있습니다.

클래스 로딩 과정을 설명해주세요.

JVM의 클래스 로더는 .class 파일을 메모리에 올리는 역할을 합니다. 이 과정은 Loading → Linking → Initialization 세 단계로 진행됩니다.

Loading에서는 Bootstrap → Extension → Application 순서의 부모 위임 모델(Parent Delegation Model)로 클래스를 찾습니다. 부모 클래스 로더가 찾지 못하면 자식이 찾는 구조입니다. 이렇게 하는 이유는 핵심 클래스(java.lang.Object 등)가 악의적으로 교체되는 것을 방지하기 위함입니다. Linking에서는 바이트코드 검증, static 변수 메모리 할당, 심볼릭 레퍼런스를 다이렉트 레퍼런스로 교체합니다. Initialization에서는 static 블록과 static 변수의 초기값이 실행됩니다.

Part 2 — JVM 심화
Java 8의 Permanent Generation이 왜 Metaspace로 바뀌었나요?

Java 7까지는 클래스 메타데이터가 PermGen(Permanent Generation)이라는 Heap 내부의 고정 크기 영역에 저장되었습니다. 문제는 이 크기가 고정(-XX:MaxPermSize)이어서, 동적으로 클래스를 많이 생성하는 프레임워크(Spring, Hibernate 등)에서 java.lang.OutOfMemoryError: PermGen space가 빈번하게 발생했다는 것입니다.

Java 8부터는 PermGen을 없애고 Metaspace라는 네이티브 메모리 영역으로 대체했습니다. Metaspace는 OS의 네이티브 메모리를 사용하므로 기본적으로 크기 제한이 없고(물론 -XX:MaxMetaspaceSize로 제한 가능), 필요에 따라 자동으로 늘어납니다. 실무에서는 메모리 누수 방지를 위해 MaxMetaspaceSize를 적절히 설정하는 것이 좋습니다.

스레드 덤프와 힙 덤프의 차이점과 활용법을 설명해주세요.

스레드 덤프는 현재 JVM의 모든 스레드 상태를 스냅샷으로 찍은 것입니다. "지금 각 직원이 뭐 하고 있지?"를 확인하는 것과 같습니다. 데드락, CPU 과점유 스레드, 스레드 블로킹 문제를 진단할 때 사용합니다. jstack 명령어나 kill -3으로 생성할 수 있습니다.

힙 덤프는 Heap 메모리에 존재하는 모든 객체의 스냅샷입니다. "사무실에 어떤 물건들이 얼마나 쌓여있지?"를 확인하는 것과 같습니다. 메모리 누수(OOM)를 진단할 때 사용하며, jmap이나 -XX:+HeapDumpOnOutOfMemoryError 옵션으로 생성합니다. Eclipse MAT이나 VisualVM으로 분석합니다.

Part 1 — GC 기본 개념 · Garbage Collection의 원리와 세대별 구조를 이해합니다.
Garbage Collection이란 무엇이고, 왜 필요한가요?

Garbage Collection(GC)은 JVM이 더 이상 참조되지 않는 객체를 자동으로 찾아서 메모리를 회수하는 메커니즘입니다. C/C++에서는 개발자가 malloc/free를 직접 관리해야 하지만, 자바는 GC가 대신 해줍니다. 비유하면, 사무실에서 사용이 끝난 서류를 자동으로 파쇄해주는 청소 로봇과 같습니다.

GC가 필요한 이유는 메모리 누수 방지와 개발 생산성 향상입니다. 수동 메모리 관리에서 흔한 dangling pointer, double free 같은 버그를 원천 차단합니다. 다만 GC에도 비용이 있어서, GC가 동작할 때 애플리케이션이 잠시 멈추는 Stop-The-World(STW)가 발생합니다. 이 STW 시간을 줄이는 것이 GC 튜닝의 핵심입니다.

GC의 세대별(Generational) 구조를 설명해주세요.

JVM Heap은 Young Generation과 Old Generation으로 나뉩니다. 이런 구조를 사용하는 이유는 "대부분의 객체는 금방 죽는다"는 경험적 관찰(Weak Generational Hypothesis) 때문입니다.

Young Generation
Eden | S0 | S1
→ Promotion →
Old Generation
장기 생존 객체
Generational Heap 구조

Young Generation은 새로 생성된 객체가 들어가는 공간으로, Eden과 두 개의 Survivor(S0, S1) 영역으로 구성됩니다. Eden이 가득 차면 Minor GC가 발생하고, 살아남은 객체는 Survivor로 이동합니다. Survivor에서 일정 횟수(Age Threshold, 기본 15)를 버틴 객체가 Old Generation으로 승격(Promotion)됩니다. Old Generation이 가득 차면 Major GC(Full GC)가 발생하는데, 이것이 STW가 길어지는 주범입니다.

꼬리질문 대비

"Survivor 영역이 왜 두 개인가요?" — Minor GC 시 Eden + 활성 Survivor의 살아남은 객체를 빈 Survivor로 복사하는 방식을 씁니다. 이렇게 하면 메모리 단편화(fragmentation)가 발생하지 않고, 연속된 메모리 공간을 확보할 수 있습니다.

GC가 "살아있는 객체"를 어떻게 판별하나요?

현대 JVM은 Reachability Analysis(도달성 분석)를 사용합니다. GC Root라고 불리는 시작점에서 참조 체인을 따라가며, 도달할 수 있는 객체는 살아있고, 도달할 수 없으면 GC 대상입니다.

GC Root가 될 수 있는 것들은 스레드 스택의 지역변수, static 변수, JNI 참조, 활성 스레드 등입니다. 참조 카운팅(Reference Counting) 방식은 순환 참조 문제가 있어 자바에서는 사용하지 않습니다. 예를 들어 객체 A와 B가 서로만 참조하면 카운트가 0이 안 되지만, Reachability 방식에서는 GC Root에서 도달 불가하므로 정상적으로 수거됩니다.

Minor GC와 Major GC(Full GC)의 차이를 설명해주세요.

Minor GC는 Young Generation만 대상으로 하는 GC입니다. Eden이 가득 차면 트리거되며, 대부분의 객체가 이미 죽어있기 때문에 보통 수 ms 이내로 매우 빠릅니다. 살아남은 소수의 객체만 Survivor로 복사하면 되기 때문입니다.

Major GC(Full GC)는 Old Generation을 대상으로 하며, 경우에 따라 전체 Heap을 스캔합니다. Old Generation에는 오래 살아남은 객체가 밀집해 있어 스캔 범위가 넓고 STW 시간이 길어집니다. 실무에서 GC 튜닝의 핵심은 바로 이 Full GC의 빈도와 시간을 줄이는 것입니다.

Serial, Parallel, CMS, G1, ZGC의 차이를 설명해주세요.

각 GC 수집기를 식당의 청소 방식에 비유하면 이해가 쉽습니다.

GC방식STW적합한 환경
Serial싱글 스레드로 수집길다싱글 코어, 소규모 앱

Serial은 혼자서 청소하는 것이고, Parallel은 여러 명이 한꺼번에 청소하되 가게 문을 닫습니다. CMS는 영업 중에 청소하지만 쓰레기가 구석구석 흩어져(단편화) 결국 한번은 문을 닫아야 합니다. G1은 구역별로 나눠서 쓰레기가 가장 많은 구역(Garbage-First)부터 청소하니 효율적입니다. ZGC는 아예 손님이 거의 느끼지 못하게 청소합니다.

G1 GC의 동작 원리를 자세히 설명해주세요.

G1(Garbage-First) GC는 Heap을 동일 크기의 Region(기본 1~32MB)으로 나누고, 각 Region에 Eden, Survivor, Old, Humongous 역할을 동적으로 부여합니다. 기존의 물리적으로 고정된 세대 구분 대신, 논리적 세대를 사용하는 것이 핵심 혁신입니다.

G1 GC — Region 기반 Heap 레이아웃

G1의 수집 과정은 크게 Young GC와 Mixed GC로 나뉩니다. Young GC는 Eden Region만 수집하며, Mixed GC는 Young + 쓰레기가 많은 Old Region을 함께 수집합니다. -XX:MaxGCPauseMillis(기본 200ms)로 목표 STW 시간을 설정하면, G1이 수집할 Region 수를 자동으로 조절합니다. 또한 Remembered Set(RSet)으로 Region 간 참조를 추적하여, 전체 Heap을 스캔하지 않고도 특정 Region의 참조 관계를 파악합니다.

ZGC는 어떻게 1ms 이하의 STW를 달성하나요?

ZGC의 핵심 기술은 Colored Pointer(컬러 포인터)와 Load Barrier(로드 배리어)입니다. 64비트 포인터의 상위 비트에 객체의 상태(Marked, Remapped, Finalizable 등)를 인코딩하여, 객체를 읽는 순간에 해당 객체의 GC 상태를 즉시 파악합니다.

ZGC는 객체를 이동시키면서도 애플리케이션을 멈추지 않습니다. 객체가 이동되면 이전 주소로 접근하는 스레드가 Load Barrier를 통해 자동으로 새 주소로 리다이렉트됩니다. 이것은 마치 택배 물류센터가 이사해도, 기존 주소로 보낸 택배가 자동 전달되는 것과 같습니다. STW는 GC Root 스캔에만 발생하며, 이는 매우 짧은 시간(보통 1ms 미만)에 완료됩니다. Java 21 LTS에서 Generational ZGC가 도입되어 Young/Old 세대 구분까지 추가되었습니다.

Part 2 — GC 실무 튜닝
실무에서 GC 튜닝은 어떻게 접근하나요?

GC 튜닝의 첫 번째 원칙은 "측정 먼저, 튜닝은 나중에"입니다. -Xlog:gc* (Java 9+) 또는 -verbose:gc로 GC 로그를 활성화하고, GC Viewer나 GCEasy 같은 도구로 분석합니다.

확인해야 할 핵심 지표는 GC 빈도, STW 시간, Heap 사용률 추이입니다. Old Generation 사용량이 GC 후에도 계속 증가하면 메모리 누수를 의심해야 합니다. 보통은 Heap 크기(-Xms, -Xmx)를 적절히 설정하는 것만으로도 대부분의 문제가 해결됩니다. -Xms와 -Xmx를 동일하게 설정하면 Heap 리사이징에 의한 불필요한 GC를 방지할 수 있습니다.

실무 권장 사항으로는, Java 17+ 환경에서 특별한 이유가 없다면 G1 GC(기본값)를 그대로 사용하고, 초저지연이 필요하면 ZGC를 고려합니다. 무작정 옵션을 많이 건드리기보다, 애플리케이션 특성(처리량 vs 지연시간)에 맞는 GC를 선택하고 최소한의 옵션만 조정하는 것이 좋습니다.

GC가 애플리케이션 성능에 미치는 영향을 어떻게 모니터링하나요?

실무에서는 Prometheus + Grafana 조합으로 JVM 메트릭을 실시간 모니터링합니다. Spring Boot라면 Micrometer가 JVM 메트릭을 자동 수집하고, jvm_gc_pause_seconds, jvm_memory_used_bytes 같은 메트릭으로 GC 상태를 대시보드에서 확인합니다.

주의 깊게 볼 지표는 GC Pause 시간의 P99(99퍼센타일), GC 발생 빈도, Heap 점유율 추이입니다. Full GC가 잦아지거나 STW 시간이 급격히 늘어나면 알람을 설정해두는 것이 좋습니다. 또한 jstat -gcutil 명령어로 각 세대의 사용률을 실시간으로 확인할 수 있습니다.

JVM = 바이트코드 실행 가상 머신 (플랫폼 독립성)
GC = 자동 메모리 회수 (STW 최소화가 핵심)
G1 = Region 기반 + 예측 가능한 STW · ZGC = Colored Pointer로 1ms 이하 STW
