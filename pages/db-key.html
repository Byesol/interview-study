DB Key & 인덱스 구조

인덱스의 Key, 리프에 뭐가 저장되는지, 데이터를 어떻게 찾아가는지 그림으로 완전 이해

users 테이블에서 실제 데이터는 어떻게 저장되나요?

아래와 같이 id(PK), name, email, age 컬럼을 가진 테이블이 디스크에 저장되어 있다고 가정합니다. 이 전체 데이터가 클러스터드 인덱스(PK)의 리프 노드에 id 순서로 저장됩니다.

인덱스에서 "Key"란 무엇인가요?

Key는 B+Tree에서 데이터를 찾기 위한 정렬 기준 컬럼 값입니다. 사전에서 "가나다"가 정렬 기준이듯, 인덱스의 Key가 B+Tree의 정렬 기준입니다.

Key = 그 인덱스(B+Tree)에서 데이터를 찾기 위한 "정렬 기준 컬럼 값"
사전에서 "가나다"가 정렬 기준이듯, 인덱스의 Key가 B+Tree의 정렬 기준입니다.
클러스터드 인덱스의 리프 노드에는 무엇이 저장되나요?

PK(id) 기준 B+Tree의 리프 노드에는 행 데이터 전체가 저장됩니다. 추가 조회 없이 바로 반환할 수 있습니다.

세컨더리 인덱스의 리프 노드에는 무엇이 저장되며, 왜 테이블 룩업이 필요한가요?

CREATE INDEX idx_email ON users(email)을 만들면 email 순서로 정렬된 별도의 B+Tree가 생깁니다. 이 트리의 리프에는 email 값과 PK(id)만 들어있습니다. name, age 같은 다른 컬럼은 없습니다!

커버링 인덱스란 무엇이며, 테이블 룩업을 피하려면 어떻게 해야 하나요?

쿼리에서 필요한 컬럼이 인덱스 Key에 이미 다 포함되어 있으면, 클러스터드 인덱스까지 갈 필요 없이 세컨더리 인덱스 리프에서 바로 끝납니다.

커버링 인덱스 = "인덱스만으로 쿼리 완성"
SELECT하는 컬럼이 인덱스 Key에 전부 포함되면, 테이블 룩업이 불필요합니다. EXPLAIN에서 Extra: Using index로 확인 가능합니다.
클러스터드 인덱스, 세컨더리 인덱스, 커버링 인덱스를 한 줄로 정리해주세요.
Key = B+Tree의 정렬 기준 컬럼 값. PK면 id, 이메일 인덱스면 email, 복합이면 (A, B)가 Key.
클러스터드 인덱스의 리프 = 행 데이터 전체. 테이블 자체가 PK 순서로 정렬된 B+Tree. 추가 조회 없이 바로 반환.
세컨더리 인덱스의 리프 = Key값 + PK값만. 다른 컬럼 데이터는 없음. PK를 들고 클러스터드 인덱스를 한 번 더 조회해야 함 = 테이블 룩업.
테이블 룩업이 발생하는 이유? 세컨더리 인덱스 리프에 "필요한 컬럼이 없으니까". 거기엔 Key + PK밖에 없으니까.
커버링 인덱스 = 테이블 룩업 스킵. SELECT하는 컬럼이 전부 인덱스 안에 있으면 클러스터드 인덱스까지 안 가도 됨.
