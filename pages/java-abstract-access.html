Java abstract & 접근제어자

abstract class vs interface, 접근제어자, 다형성, 실무 설계

abstract class(추상 클래스)란 무엇인가요?

추상 클래스는 직접 인스턴스를 생성할 수 없는 미완성 클래스입니다. abstract 키워드가 붙은 메서드(추상 메서드)를 하나 이상 포함할 수 있으며, 이를 상속받은 자식 클래스가 반드시 구현해야 합니다. 반대로 구현이 완성된 일반 메서드, 필드, 생성자도 함께 가질 수 있다는 점이 인터페이스와의 핵심 차이입니다.

비유하자면, 추상 클래스는 "반쯤 지어진 집의 설계도"입니다. 공통 뼈대(공통 필드·메서드)는 미리 지어져 있고, 각 방(추상 메서드)의 내부 인테리어만 입주자(자식 클래스)가 완성하는 구조입니다.

// abstract class 기본 구조
abstract class Animal {
    String name;  // 일반 필드 가능

    Animal(String name) { this.name = name; }  // 생성자 가능

    abstract void sound();   // 추상 메서드 → 자식이 반드시 구현

    void breathe() {         // 일반 메서드도 가능
        System.out.println("숨 쉰다");
    }
}

class Dog extends Animal {
    Dog(String name) { super(name); }
    @Override
    void sound() { System.out.println("멍멍"); }  // 반드시 구현
}
자주 나오는 꼬리질문: 추상 메서드가 하나도 없어도 abstract class로 선언할 수 있나요? → 가능합니다. 이 경우 인스턴스 생성을 막는 용도로만 사용할 수 있습니다.
추상 클래스와 인터페이스의 차이는 무엇인가요?

가장 핵심적인 차이는 목적입니다. 추상 클래스는 "is-a" 관계의 공통 부모로, 상태(필드)와 공통 구현을 공유합니다. 인터페이스는 "can-do" 관계로, 어떤 기능을 할 수 있는지 규약을 정의합니다. Java는 단일 상속만 허용하므로 추상 클래스는 하나만 상속할 수 있지만, 인터페이스는 여러 개를 구현할 수 있습니다.

또 다른 차이는 상태 보유 여부입니다. 추상 클래스는 일반 인스턴스 필드를 가질 수 있지만, 인터페이스의 필드는 암묵적으로 public static final(상수)입니다.

자주 나오는 꼬리질문: Java 8 이후 인터페이스에 default 메서드가 생겼는데, 그러면 추상 클래스가 필요 없어진 것 아닌가요? → 아닙니다. 인스턴스 필드(상태)를 공유해야 할 때, 생성자 로직이 필요할 때, protected 멤버가 필요할 때는 여전히 추상 클래스가 필요합니다.
인터페이스의 default 메서드란 무엇이고 왜 도입되었나요?

default 메서드는 Java 8에서 도입된 기능으로, 인터페이스 안에 구현 코드를 가진 메서드를 선언할 수 있게 해줍니다. 기존에는 인터페이스에 메서드를 추가하면 이를 구현한 모든 클래스를 수정해야 했습니다. Java 8에서 Stream, forEach 등을 Collection 인터페이스에 추가할 때, 기존 코드를 깨뜨리지 않고 하위 호환성을 유지하기 위해 default 메서드가 탄생했습니다.

주의할 점은 다이아몬드 문제입니다. 두 인터페이스가 같은 이름의 default 메서드를 가질 경우, 구현 클래스에서 반드시 명시적으로 오버라이딩해야 합니다.

// Java 8 - 인터페이스에 default 메서드 추가
interface Flyable {
    void fly();

    default void takeOff() {   // 구현 코드 포함 가능
        System.out.println("이륙 준비 중...");
        fly();
    }
}

// 다이아몬드 문제: 두 인터페이스가 같은 default 메서드를 가질 때
interface A { default void hello() { System.out.println("A"); } }
interface B { default void hello() { System.out.println("B"); } }

class C implements A, B {
    @Override
    public void hello() { A.super.hello(); }  // 반드시 명시
}
인터페이스의 private 메서드(Java 9)는 무엇인가요?

Java 9부터 인터페이스 내부에서 private 메서드를 정의할 수 있게 되었습니다. 주된 목적은 default 메서드들 사이에서 공통 로직을 추출하는 헬퍼(helper) 메서드를 만들기 위해서입니다. 외부에 노출하지 않아야 하는 내부 구현을 캡슐화할 수 있어, 인터페이스의 응집도가 높아집니다.

한 가지 제약이 있습니다. private 메서드는 당연히 구현 클래스에서 상속되거나 오버라이딩될 수 없습니다. 오직 같은 인터페이스 내부의 default 메서드 또는 다른 private 메서드에서만 호출할 수 있습니다.

// Java 9 - 인터페이스 private 메서드
interface Logger {
    default void logInfo(String msg)  { log("[INFO]", msg); }
    default void logError(String msg) { log("[ERROR]", msg); }

    private void log(String level, String msg) {   // 내부 헬퍼
        System.out.println(level + " " + msg);
    }
}
protected 접근제어자는 정확히 어떤 범위를 허용하나요?

protected는 같은 패키지 내 모든 클래스, 그리고 다른 패키지라도 해당 클래스를 상속받은 자식 클래스에서 접근할 수 있습니다. public보다는 좁고 default(package-private)보다는 넓은 범위입니다. 상속 관계를 통해서만 외부 패키지의 접근을 허용한다는 점이 핵심입니다.

실수하기 쉬운 부분이 있습니다. 다른 패키지의 자식 클래스에서 부모 타입 참조 변수를 통해서는 protected 멤버에 접근할 수 없습니다. 반드시 자식 클래스 자신의 참조(this, super, 혹은 자식 타입 변수)를 통해야 합니다.

// com.base 패키지
package com.base;
public class Parent {
    protected String data = "보호된 데이터";
}

// com.child 패키지 (다른 패키지)
package com.child;
import com.base.Parent;

public class Child extends Parent {
    void test() {
        System.out.println(this.data);   // 자식 클래스 내부 - 접근 가능

        Parent p = new Parent();
        System.out.println(p.data);     // 컴파일 에러 - 다른 패키지 + 부모 참조
    }
}
abstract 메서드와 일반 메서드 오버라이딩의 차이는?

abstract 메서드는 자식 클래스가 반드시 구현해야 합니다. 구현하지 않으면 자식 클래스도 abstract가 되어야 합니다. 반면 일반 메서드의 오버라이딩은 선택 사항입니다. 자식 클래스가 원하지 않으면 부모의 구현을 그대로 사용할 수 있습니다.

또한 abstract 메서드에는 private을 붙일 수 없습니다. private이면 자식 클래스에서 보이지 않아 오버라이딩 자체가 불가능하기 때문입니다. static도 마찬가지로 abstract와 함께 쓸 수 없습니다. 반면 final 키워드를 일반 메서드에 붙이면 오버라이딩 자체를 금지할 수 있습니다.

// abstract는 private, static, final과 함께 쓸 수 없다
abstract class Base {
    abstract void mustOverride();      // 올바른 사용
    // private abstract void x();     // 컴파일 에러
    // static abstract void y();      // 컴파일 에러
    // final abstract void z();       // 컴파일 에러

    final void cannotOverride() { }   // 오버라이딩 금지
}
인터페이스 default 메서드와 추상 클래스의 일반 메서드, 무엇이 다른가요?

언뜻 비슷해 보이지만 결정적인 차이가 있습니다. 추상 클래스의 일반 메서드는 상태(인스턴스 필드)에 접근할 수 있습니다. 예를 들어 this.name처럼 객체 상태를 읽거나 변경할 수 있죠. 반면 인터페이스의 default 메서드는 인터페이스 자체에 인스턴스 필드가 없으므로, 상태를 직접 보유할 수 없습니다. 오직 다른 인터페이스 메서드를 호출하거나 파라미터로 받은 값만 다룰 수 있습니다.

또한 추상 클래스는 protected, package-private 접근 수준의 메서드를 가질 수 있어 내부 구현을 세밀하게 숨길 수 있습니다. 인터페이스의 default 메서드는 암묵적으로 public이라 외부에 노출됩니다.

자주 나오는 꼬리질문: 그러면 어떤 기준으로 추상 클래스 vs 인터페이스를 고르나요? → 공유할 상태(필드)가 있다면 추상 클래스, 여러 클래스에 공통 행동(타입)을 부여하고 싶다면 인터페이스를 선택합니다. 실무에서는 인터페이스를 기본으로 선택하고, 상태 공유가 필요할 때 추상 클래스를 고려합니다.
접근 범위를 시각적으로 비교해주세요. private · default · protected · public

접근제어자 네 가지를 좁은 범위 → 넓은 범위 순으로 이해하면 됩니다. private은 선언된 클래스 내부만, default(package-private)는 같은 패키지, protected는 같은 패키지 + 다른 패키지의 자식 클래스, public은 어디서든 접근 가능합니다. 캡슐화 원칙에 따라 항상 가장 좁은 범위부터 시작하는 것이 좋습니다.

접근제어자같은 클래스같은 패키지자식 클래스(외부 패키지)전체

추상 클래스의 생성자는 왜 존재하나요? 어디서 사용되나요?

추상 클래스는 직접 인스턴스화할 수 없지만, 자식 클래스의 생성자에서 super()로 호출되기 위해 생성자가 존재합니다. 자식 객체가 생성될 때 부모(추상 클래스)의 공통 필드도 초기화해야 하기 때문입니다. 즉 추상 클래스의 생성자는 간접적으로 실행됩니다.

이것이 인터페이스와 추상 클래스의 또 다른 차이입니다. 인터페이스는 생성자가 없기 때문에 초기화 로직을 담을 수 없습니다. 반면 추상 클래스는 생성자에 공통 유효성 검사나 의존성 주입 코드를 넣어 자식 클래스가 재사용하게 할 수 있습니다.

abstract class Shape {
    final String color;   // 공통 상태

    Shape(String color) {
        if (color == null) throw new IllegalArgumentException("색상은 필수");
        this.color = color;    // 공통 초기화 로직
    }

    abstract double area();
}

class Circle extends Shape {
    double radius;

    Circle(String color, double radius) {
        super(color);   // 추상 클래스 생성자 호출
        this.radius = radius;
    }

    @Override
    double area() { return Math.PI * radius * radius; }
}
실무에서 추상 클래스를 쓰는 대표적인 패턴은?

실무에서 추상 클래스가 가장 빛나는 곳은 Template Method 패턴입니다. 알고리즘의 전체 뼈대를 추상 클래스에서 final 메서드로 정의하고, 세부 단계만 추상 메서드로 두어 자식 클래스가 채우게 합니다. 예를 들어 데이터 처리 파이프라인(읽기 → 변환 → 쓰기)에서 "변환" 단계만 자식마다 달라지는 경우가 전형적입니다.

Spring의 AbstractController, JdbcTemplate 등도 이 패턴을 사용합니다. 공통 트랜잭션 처리나 예외 변환 로직은 추상 클래스가 담당하고, 실제 SQL이나 비즈니스 로직만 구현체가 맡는 구조입니다.

// Template Method 패턴
abstract class DataProcessor {
    // 전체 흐름은 고정 (final로 오버라이딩 방지)
    public final void process() {
        List<String> data = readData();    // 공통 단계
        List<String> result = transform(data); // 자식마다 다른 단계
        writeData(result);               // 공통 단계
    }

    protected abstract List<String> transform(List<String> data);

    private List<String> readData() { /* 공통 구현 */ return List.of(); }
    private void writeData(List<String> d) { /* 공통 구현 */ }
}
default 메서드를 남용하면 어떤 문제가 생기나요?

default 메서드를 과도하게 추가하면 인터페이스가 비대해져 ISP(인터페이스 분리 원칙)를 위반하게 됩니다. 인터페이스는 본래 "계약"이라는 명확한 역할을 가져야 하는데, 구현 코드가 쌓이다 보면 사실상 추상 클래스처럼 변해 설계 의도가 흐려집니다.

또한 앞서 말한 다이아몬드 문제가 발생할 수 있습니다. 라이브러리 작성자 입장에서는 기존 인터페이스에 default 메서드를 추가할 때, 해당 인터페이스를 구현하는 모든 클래스에서 오버라이딩 충돌이 발생하지 않는지 신중하게 고려해야 합니다. 원칙적으로 default 메서드는 하위 호환성 유지 목적에 한정해서 사용하는 것이 좋습니다.

접근제어자를 설계할 때 가장 중요한 원칙은 무엇인가요?

최소 권한 원칙(Principle of Least Privilege)입니다. 처음에 가장 좁은 범위(private)로 시작하고, 필요한 경우에만 단계적으로 넓히는 것이 좋습니다. 반대로 처음부터 public으로 열어두면 외부 코드가 내부 구현에 의존하게 되어, 나중에 내부를 바꿀 때 사이드 이펙트가 걷잡을 수 없이 커집니다.

실무에서 자주 하는 실수는 getter/setter를 무분별하게 생성하는 것입니다. 모든 필드에 getter/setter를 만들면 캡슐화가 사실상 무너집니다. 비즈니스 의미 있는 메서드(예: withdraw())로 상태 변경을 표현하고, setter는 꼭 필요한 경우에만 만드는 것이 좋습니다.

추상 클래스를 상속한 클래스가 추상 메서드를 구현하지 않으면 어떻게 되나요?

해당 자식 클래스도 abstract로 선언되어야 합니다. 컴파일러가 구현하지 않은 추상 메서드가 남아 있는데 concrete class라고 선언하면 컴파일 오류를 냅니다. 이 체인은 어딘가에서 반드시 모든 추상 메서드를 구현한 concrete class가 등장해야 끝납니다.

private 생성자는 언제 쓰나요?

크게 두 가지 경우입니다. 첫째, Singleton 패턴에서 외부에서 인스턴스를 직접 생성하지 못하게 막을 때 사용합니다. 둘째, 유틸리티 클래스(예: Math, Collections)처럼 오직 static 메서드만 모아둔 클래스에서 인스턴스화 자체를 금지할 때 사용합니다. private 생성자가 있으면 상속도 불가능해진다는 부수적인 효과도 있습니다.

static 메서드를 인터페이스에 선언할 수 있나요? (Java 8+)

Java 8부터 가능합니다. 인터페이스의 static 메서드는 default 메서드와 달리, 구현 클래스나 자식 인터페이스로 상속되지 않습니다. 오직 인터페이스명으로만 호출할 수 있습니다(Validator.of()처럼). 주로 해당 인터페이스와 관련된 팩토리 메서드나 유틸리티 메서드를 인터페이스 안에 캡슐화할 때 사용합니다.

sealed class(Java 17)가 abstract class와 다른 점은?

sealed class는 Java 17에 정식 도입된 기능으로, 상속 가능한 하위 클래스를 명시적으로 제한합니다. 추상 클래스는 누구나 상속할 수 있지만, sealed class는 permits 키워드로 허용된 클래스만 상속할 수 있습니다. 이는 도메인 모델을 닫힌 계층 구조로 표현할 때 유용하며, Java의 패턴 매칭(switch 표현식)과 함께 사용하면 컴파일러가 모든 케이스를 처리했는지 검증해줍니다.

sealed class Shape permits Circle, Rectangle, Triangle { }
final class Circle    extends Shape { }
final class Rectangle extends Shape { }
final class Triangle  extends Shape { }

// switch로 모든 하위 타입을 커버 → 컴파일러가 완전성 보장
double area = switch (shape) {
    case Circle c    -> Math.PI * c.r * c.r;
    case Rectangle r -> r.w * r.h;
    case Triangle t  -> 0.5 * t.base * t.height;
};
