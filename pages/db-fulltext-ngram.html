Full-Text Index & ngram Parser

LIKE '%검색어%' 는 왜 느린가? Full-Text Index는 어떻게 다른가? ngram은 뭔가?

LIKE '%검색어%'는 왜 느리며, Full-Text Index는 어떻게 이 문제를 해결하나요?

LIKE '%검색어%'는 B+Tree 인덱스를 활용하지 못하고 모든 행의 컬럼을 처음부터 끝까지 읽는 Full Table Scan이 발생합니다. Full-Text Index는 역인덱스(Inverted Index) 구조로 토큰으로 바로 매칭하여 매우 빠릅니다.

Full-Text Index의 역인덱스(Inverted Index) 구조와 B+Tree 인덱스의 차이는 무엇인가요?

B+Tree 인덱스는 "행 → 데이터" 방향이지만, Full-Text 인덱스는 "단어(토큰) → 어떤 문서에 있는지"로 반대(역) 방향입니다. 그래서 역인덱스라고 부릅니다.

한국어/중국어/일본어 검색에 ngram Parser가 필요한 이유는 무엇인가요?

MySQL 기본 Full-Text Parser는 공백/구두점으로 단어를 분리합니다. 영어는 단어 사이에 공백이 있으니 잘 됩니다. 하지만 한국어는 공백 없이 이어 쓰는 경우가 많아 분리가 불가능합니다.

ngram 토큰화는 어떻게 동작하며 ngram_token_size 값에 따라 어떻게 달라지나요?

ngram_token_size의 기본값은 2 (bigram)입니다. 설정 범위는 1~10이며, 서버 시작 시에만 변경 가능합니다 (read-only). 글자를 N개씩 슬라이딩하여 토큰을 생성합니다.

Full-Text Index with ngram은 어떻게 생성하고 어떤 쿼리 문법으로 검색하나요?
-- ① ngram_token_size 설정 (서버 시작 시, my.cnf) [mysqld] ngram_token_size=2 -- 기본값 2, 범위 1~10, read-only 변수 -- ② 테이블 생성 시 FULLTEXT 인덱스 추가 CREATE TABLE articles ( id INT AUTO_INCREMENT PRIMARY KEY, title VARCHAR(200), body TEXT, FULLTEXT (title, body) WITH PARSER ngram -- ← ngram 파서 지정! ) ENGINE=InnoDB CHARACTER SET utf8mb4; -- ③ 기존 테이블에 추가 ALTER TABLE articles ADD FULLTEXT INDEX ft_idx (title, body) WITH PARSER ngram; -- ④ 검색 쿼리 (3가지 모드) -- 자연어 모드 (기본) SELECT * FROM articles WHERE MATCH(title, body) AGAINST('코딩테스트'); -- 불리언 모드 (연산자 사용 가능) SELECT * FROM articles WHERE MATCH(title, body) AGAINST('+코딩 -파이썬' IN BOOLEAN MODE); -- 쿼리 확장 모드 (연관 단어도 검색) SELECT * FROM articles WHERE MATCH(title, body) AGAINST('코딩' WITH QUERY EXPANSION);
ngram 관련 주요 설정 변수와 각 역할은 무엇인가요?
설정 변수기본값설명
ngram_token_size 2 ngram 토큰 크기 (1~10). read-only, 서버 시작 시만 변경 가능. 한국어는 2가 적합
innodb_ft_min_token_size 3 InnoDB 기본 파서 최소 단어 길이. ngram 파서에서는 무시됨
innodb_ft_max_token_size 84 InnoDB 기본 파서 최대 단어 길이. ngram 파서에서는 무시됨
ft_min_word_len 4 MyISAM 기본 파서 최소 길이. ngram 파서에서는 무시됨
innodb_ft_enable_stopword ON 불용어(stopword) 필터링 활성화 여부
핵심: ngram 파서를 쓰면 innodb_ft_min_token_size, innodb_ft_max_token_size 등은 무시됩니다. 오직 ngram_token_size만 토큰 크기를 결정합니다.
ngram 파서의 주의사항과 특이 동작에는 어떤 것들이 있나요?
상황동작
공백 처리 ngram은 공백을 제거한 후 토큰화합니다. "코딩 테스트" → "코딩테스트" → "코딩","딩테","테스","스트"
Stopword 기본 파서는 stopword와 "같은" 토큰을 제거하지만, ngram은 stopword를 "포함하는" 토큰을 제거합니다
와일드카드 검색 접두어가 token_size보다 짧으면 모든 행 반환. 길면 ngram phrase로 변환되어 와일드카드 무시
BOOLEAN MODE "코딩테스트" → '"코딩 딩테 테스 스트"' (ngram phrase)로 변환. "코딩"만 있고 "딩테"가 없으면 매칭 안 됨
token_size 변경 read-only 변수이므로 서버 재시작 필요. 변경 후 인덱스 재생성(REBUILD) 필수
Full-Text Index & ngram에 대해 면접 핵심 포인트를 정리해주세요.
① Full-Text Index = 역인덱스(Inverted Index) — B+Tree와 완전히 다른 구조. "단어 → 문서ID" 매핑.
② LIKE '%검색어%' 가 느린 이유 — B+Tree 인덱스를 활용 못하고 Full Table Scan. Full-Text Index는 토큰으로 바로 매칭.
③ 한국어에 ngram이 필요한 이유 — 기본 파서는 공백 기반 분리. 한국어는 공백 없이 이어붙여 쓰므로 N글자씩 잘라야 함.
④ ngram_token_size 기본값 = 2 — 한국어 2글자(bigram)에 적합. read-only 변수, 서버 시작 시만 변경 가능 (1~10).
⑤ 검색 쿼리 — MATCH(col) AGAINST('검색어') 사용. 3가지 모드: NATURAL LANGUAGE, BOOLEAN, QUERY EXPANSION.
⑥ 한계 — 형태소 분석 없음(은/는/이/가 구분 못함), Elasticsearch 같은 전문 검색 엔진 대비 기능 제한적.
