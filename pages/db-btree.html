B+Tree & 인덱스 구조

MySQL InnoDB 기준으로 B+Tree가 인덱스에서 어떻게 사용되는지 시각적으로 이해하기

InnoDB의 페이지(Page)란 무엇이며, 리프 페이지 내부는 어떻게 구성되나요?

페이지는 InnoDB가 디스크에서 데이터를 읽고 쓰는 최소 단위입니다. 기본 크기는 16KB이며, B+Tree의 각 노드는 하나의 페이지에 대응됩니다. 디스크에서 데이터를 읽을 때 행 하나가 아니라 페이지 단위로 통째로 메모리(Buffer Pool)에 올립니다.

InnoDB 기본 페이지 크기: innodb_page_size = 16384 (16KB)

📄 리프 페이지 내부 구조 (Page 7 예시)
Page Header prev: Page6 next: Page8
Record 1 Key: 15 Data: 홍길동
Record 2 Key: 18 Data: 김철수
Record 3 Key: 20 Data: 이영희
Page Trailer Checksum
핵심: 리프 페이지의 Header에 이전/다음 페이지 번호가 있어서 연결 리스트를 형성합니다. 이것이 바로 B+Tree 리프 노드의 Linked List입니다.

B+Tree 노드와 InnoDB 페이지는 어떻게 대응되며, 트리 높이가 중요한 이유는 무엇인가요?

B+Tree의 각 노드는 InnoDB의 페이지 하나에 대응됩니다. 따라서 트리 높이 = 디스크 I/O 횟수가 됩니다.

  • 내부 노드 N개
  • 리프 노드 M개
⟷ 1 : 1 대응

  • 루트 페이지 1개
  • 내부 페이지 N개
  • 리프 페이지 M개
왜 트리 높이가 중요한가?
높이 3인 B+Tree에서 특정 키를 찾으려면 루트 → 내부 → 리프, 총 3번의 페이지 읽기(디스크 I/O)가 필요합니다. InnoDB는 내부 노드에 데이터를 저장하지 않으므로 한 페이지에 약 1,000개 이상의 키를 담을 수 있고, 높이 3이면 약 10억 건의 데이터를 커버할 수 있습니다.
B+Tree에서 범위 탐색(WHERE id BETWEEN 10 AND 30)이 효율적인 이유는 무엇인가요?

핵심 포인트: 시작 키를 찾기 위해 트리를 타고 내려가는 것은 딱 한 번이고, 이후에는 리프 노드의 연결 리스트를 따라 순차적으로 스캔합니다. 트리를 다시 올라갈 필요가 없으므로 디스크 I/O가 최소화됩니다.
B+Tree 인덱스 구조에 대해 면접에서 한 줄로 요약한다면?
페이지 = InnoDB의 디스크 I/O 최소 단위 (기본 16KB)
B+Tree 노드 = 페이지 하나에 대응 → 트리 높이 = 디스크 I/O 횟수
내부 노드는 키+포인터만 → 한 페이지에 ~1,000개 키 → 높이 3으로 10억 건 커버
리프 노드는 키+데이터 저장 + 연결 리스트 → 범위 탐색이 순차 스캔으로 해결
클러스터드 인덱스의 리프 = 실제 행 데이터 / 세컨더리 인덱스의 리프 = PK 값
