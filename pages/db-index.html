인덱스 완전 정복

Key, B+Tree, 클러스터드/세컨더리, 복합 인덱스, 커버링 인덱스까지

🔑 Part 1. Key & 저장 구조 — 인덱스란 무엇인가, 리프에 뭐가 들어있나
users 테이블에서 실제 데이터는 어떻게 저장되나요?

아래와 같이 id(PK), name, email, age 컬럼을 가진 테이블이 디스크에 저장되어 있다고 가정합니다. 이 전체 데이터가 클러스터드 인덱스(PK)의 리프 노드에 id 순서로 저장됩니다.

인덱스에서 "Key"란 무엇인가요?

Key는 B+Tree에서 데이터를 찾기 위한 정렬 기준 컬럼 값입니다. 사전에서 "가나다"가 정렬 기준이듯, 인덱스의 Key가 B+Tree의 정렬 기준입니다. PK든, 일반 인덱스든, 복합 인덱스든 모두 각자의 Key를 가집니다.

PK만 Key인가요? 모든 인덱스는 Key를 가지나요?
네, 모든 인덱스는 "키(Key)"를 가집니다. PK는 클러스터드 인덱스의 Key이고, 우리가 만드는 일반 인덱스도 각자의 Key를 가집니다. 복합 인덱스면 Key가 (컬럼A, 컬럼B, 컬럼C)처럼 여러 컬럼의 조합이 됩니다.
Key = 그 인덱스(B+Tree)에서 데이터를 찾기 위한 "정렬 기준 컬럼 값"
인덱스 = 별도의 B+Tree. 그 B+Tree의 정렬 기준이 되는 컬럼이 바로 "Key"입니다.
클러스터드 인덱스의 리프 노드에는 무엇이 저장되나요?

PK(id) 기준 B+Tree의 리프 노드에는 행 데이터 전체가 저장됩니다. 추가 조회 없이 바로 반환할 수 있습니다.

세컨더리 인덱스의 리프 노드에는 무엇이 저장되며, 왜 테이블 룩업이 필요한가요?

CREATE INDEX idx_email ON users(email)을 만들면 email 순서로 정렬된 별도의 B+Tree가 생깁니다. 이 트리의 리프에는 email 값과 PK(id)만 들어있습니다. name, age 같은 다른 컬럼은 없습니다!

커버링 인덱스란 무엇이며, 테이블 룩업을 피하려면 어떻게 해야 하나요?

쿼리에서 필요한 컬럼이 인덱스 Key에 이미 다 포함되어 있으면, 클러스터드 인덱스까지 갈 필요 없이 세컨더리 인덱스 리프에서 바로 끝납니다.

커버링 인덱스 = "인덱스만으로 쿼리 완성"
SELECT하는 컬럼이 인덱스 Key에 전부 포함되면, 테이블 룩업이 불필요합니다. EXPLAIN에서 Extra: Using index로 확인 가능합니다.
🌳 Part 2. B+Tree 구조 — InnoDB 페이지, 노드-페이지 대응, 범위 스캔
InnoDB의 페이지(Page)란 무엇이며, 리프 페이지 내부는 어떻게 구성되나요?

페이지는 InnoDB가 디스크에서 데이터를 읽고 쓰는 최소 단위입니다. 기본 크기는 16KB이며, B+Tree의 각 노드는 하나의 페이지에 대응됩니다. 디스크에서 데이터를 읽을 때 행 하나가 아니라 페이지 단위로 통째로 메모리(Buffer Pool)에 올립니다.

핵심: 리프 페이지의 Header에 이전/다음 페이지 번호가 있어서 연결 리스트를 형성합니다. 이것이 바로 B+Tree 리프 노드의 Linked List입니다.
B+Tree 인덱스 구조는 어떻게 생겼으며, 리프 노드는 어떻게 연결되어 있나요?

아래는 PK(id) 기준 클러스터드 인덱스의 B+Tree 구조입니다. 내부 노드는 키+포인터만, 리프 노드에 실제 데이터가 저장됩니다. 리프끼리는 양방향 연결 리스트로 연결됩니다.

루트 노드 (키만)
내부 노드 (키+포인터)
리프 노드 (키+데이터)
실제 행 데이터
B+Tree 노드와 InnoDB 페이지는 어떻게 대응되며, 트리 높이가 중요한 이유는 무엇인가요?

B+Tree의 각 노드는 InnoDB의 페이지 하나에 대응됩니다. 따라서 트리 높이 = 디스크 I/O 횟수가 됩니다.

🌳 B+Tree 노드

  • 루트 노드 1개
  • 내부 노드 N개
  • 리프 노드 M개
⟷ 1 : 1 대응

📄 InnoDB 페이지

  • 루트 페이지 1개
  • 내부 페이지 N개
  • 리프 페이지 M개
왜 트리 높이가 중요한가?
높이 3인 B+Tree에서 특정 키를 찾으려면 루트 → 내부 → 리프, 총 3번의 페이지 읽기(디스크 I/O)가 필요합니다. InnoDB는 내부 노드에 데이터를 저장하지 않으므로 한 페이지에 약 1,000개 이상의 키를 담을 수 있고, 높이 3이면 약 10억 건의 데이터를 커버할 수 있습니다.
B+Tree에서 범위 탐색(WHERE id BETWEEN 10 AND 30)이 효율적인 이유는 무엇인가요?

B+Tree에서 범위 탐색이 효율적인 이유를 단계별로 볼 수 있습니다.

복합 인덱스 (name, age)의 정렬 구조는 어떻게 되며 컬럼 순서가 왜 중요한가요?

복합 인덱스 (name, age)가 어떻게 정렬되는지 직접 봐야 이해됩니다. 전화번호부가 성(姓) → 이름 순으로 정렬된 것과 같습니다.

복합 인덱스 (A, B, C)에서 거꾸로 검색하면 왜 인덱스를 타지 못하나요? (Leftmost Prefix Rule)

복합 인덱스 (A, B, C)는 왼쪽부터 순서대로 사용됩니다. 전화번호부에서 "성"을 모르면 "이름"만으로는 찾을 수 없는 것과 같습니다.

-- 인덱스: INDEX idx_abc (A, B, C)
쿼리 WHERE 조건인덱스 사용이유
WHERE A = 1사용첫 번째 컬럼 A 사용
WHERE A = 1 AND B = 2사용A → B 순서대로 사용
WHERE A = 1 AND B = 2 AND C = 3완전 사용A → B → C 전부 사용
WHERE A = 1 AND C = 3A만 사용B를 건너뛰었으므로 C는 인덱스 활용 불가
WHERE B = 2미사용첫 번째 컬럼 A가 없음 → 인덱스 전혀 못 탐
WHERE B = 2 AND C = 3미사용A가 없으면 B, C도 정렬이 보장되지 않음
WHERE C = 3미사용마지막 컬럼만 단독 사용 불가
WHERE A = 1 ORDER BY B사용A로 필터 후 B는 이미 정렬되어 있어 filesort 불필요
ORDER BY B, C미사용A 없이 B, C 정렬은 보장되지 않음
핵심 원리 — 왜 거꾸로면 안 타는가?
복합 인덱스 (A, B, C)의 B+Tree는 A로 먼저 정렬 → 같은 A 그룹 내에서 B로 정렬 → 같은 A,B 그룹 내에서 C로 정렬합니다. 따라서 A를 모르면 B의 정렬 순서를 알 수 없고, A,B를 모르면 C의 정렬 순서를 알 수 없습니다. 이것이 Leftmost Prefix Rule입니다.

✦ Part 4. 면접 정리 — Key·B+Tree·클러스터드·세컨더리·커버링 한 줄 요약
Key, 클러스터드 인덱스, 세컨더리 인덱스, 커버링 인덱스를 한 줄씩 정리해주세요.
Key = B+Tree의 정렬 기준 컬럼 값. PK면 id, 이메일 인덱스면 email, 복합이면 (A, B)가 Key.
클러스터드 인덱스의 리프 = 행 데이터 전체. 테이블 자체가 PK 순서로 정렬된 B+Tree. 추가 조회 없이 바로 반환.
세컨더리 인덱스의 리프 = Key값 + PK값만. 다른 컬럼 데이터는 없음. PK를 들고 클러스터드 인덱스를 한 번 더 조회해야 함 = 테이블 룩업.
테이블 룩업이 발생하는 이유? 세컨더리 인덱스 리프에 "필요한 컬럼이 없으니까". 거기엔 Key + PK밖에 없으니까.
커버링 인덱스 = 테이블 룩업 스킵. SELECT하는 컬럼이 전부 인덱스 안에 있으면 클러스터드 인덱스까지 안 가도 됨. EXPLAIN Extra: Using index.
B+Tree 인덱스 구조를 면접에서 한 줄로 요약한다면?
페이지 = InnoDB의 디스크 I/O 최소 단위 (기본 16KB). B+Tree 노드 = 페이지 하나에 1:1 대응.
트리 높이 = 디스크 I/O 횟수. 내부 노드는 키+포인터만 → 한 페이지에 ~1,000개 키 → 높이 3으로 10억 건 커버.
리프 노드는 키+데이터 저장 + 양방향 연결 리스트 → 범위 탐색이 순차 스캔으로 해결 (트리를 다시 올라갈 필요 없음).
범위 탐색: 시작 키를 찾기 위해 트리를 한 번 내려간 뒤, Linked List를 따라 순차 스캔 → 디스크 I/O 최소화.
클러스터드 인덱스의 리프 = 실제 행 데이터 / 세컨더리 인덱스의 리프 = PK 값만.
인덱스 전체 개념을 면접 포인트로 정리해주세요.
① 인덱스 = B+Tree — 모든 인덱스는 각자의 Key를 기준으로 정렬된 별도의 B+Tree
② 클러스터드 — 리프 = 행 데이터 전체 / 세컨더리 — 리프 = Key + PK (→ 테이블 룩업 필요)
③ 복합 인덱스 — (A, B, C)면 A→B→C 순서로 정렬. 왼쪽부터만 인덱스를 탈 수 있음 (Leftmost Prefix Rule)
④ 커버링 인덱스 — 인덱스만으로 쿼리 결과를 반환 → 테이블 룩업 제거 → I/O 절약
⑤ 거꾸로(B만, C만) 검색하면? — 첫 번째 컬럼 없이는 정렬 순서를 알 수 없어 인덱스를 못 탐
⑥ 복합 인덱스 컬럼 순서 전략 — 카디널리티 높은 컬럼 앞에 + WHERE 자주 쓰는 컬럼 앞에
