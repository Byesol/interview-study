인덱스 완전 정복

PK·Key·클러스터드·논클러스터드·복합·커버링 인덱스와 쿼리 동작 원리

인덱스의 "Key"란 무엇이며, PK만 Key인가요?
Q. PK가 인덱스의 Key인가? 모든 인덱스는 Key인가?
네, 모든 인덱스는 "키(Key)"를 가집니다. 여기서 Key란 B+Tree에서 정렬·탐색 기준이 되는 컬럼 값입니다.
PK는 클러스터드 인덱스의 Key이고, 우리가 만드는 일반 인덱스도 각자의 Key를 가집니다.
복합 인덱스면 Key가 (컬럼A, 컬럼B, 컬럼C) 처럼 여러 컬럼의 조합이 됩니다.
결론: 인덱스 = 별도의 B+Tree. 그 B+Tree의 정렬 기준이 되는 컬럼이 바로 "Key"입니다. PK든, 일반 인덱스든, 복합 인덱스든 모두 각자의 Key를 가집니다.
클러스터드 인덱스와 세컨더리(논클러스터드) 인덱스의 차이는 무엇인가요?

가장 중요한 차이: 리프 노드에 무엇이 저장되는가?

구분클러스터드 인덱스세컨더리(논클러스터드) 인덱스
개수테이블당 1개테이블당 여러 개
KeyPK (없으면 UNIQUE NOT NULL, 그것도 없으면 InnoDB가 내부 RowID 생성)CREATE INDEX로 지정한 컬럼
리프 노드행 데이터 전체 (name, email, age...)인덱스 Key + PK 값
데이터 정렬테이블 물리적 정렬 = PK 순별도 B+Tree (인덱스 Key 순)
조회 과정리프에서 바로 데이터 반환리프에서 PK 확인 → 클러스터드 인덱스에서 데이터 조회 (테이블 룩업)
복합 인덱스 (name, age)의 정렬 구조는 어떻게 되며 컬럼 순서가 왜 중요한가요?

복합 인덱스 (name, age)가 어떻게 정렬되는지 직접 봐야 이해됩니다. 전화번호부가 성(姓) → 이름 순으로 정렬된 것과 같습니다.

복합 인덱스 (A, B, C)에서 거꾸로 검색하면 왜 인덱스를 타지 못하나요? (Leftmost Prefix Rule)

복합 인덱스 (A, B, C)는 왼쪽부터 순서대로 사용됩니다. 전화번호부에서 "성"을 모르면 "이름"만으로는 찾을 수 없는 것과 같습니다.

-- 인덱스: INDEX idx_abc (A, B, C)
쿼리 WHERE 조건인덱스 사용이유
WHERE A = 1 ✅ 사용 첫 번째 컬럼 A 사용
WHERE A = 1 AND B = 2 ✅ 사용 A → B 순서대로 사용
WHERE A = 1 AND B = 2 AND C = 3 ✅ 완전 사용 A → B → C 전부 사용
WHERE A = 1 AND C = 3 ⚠️ A만 사용 B를 건너뛰었으므로 C는 인덱스 활용 불가
WHERE B = 2 ❌ 미사용 첫 번째 컬럼 A가 없음 → 인덱스 전혀 못 탐
WHERE B = 2 AND C = 3 ❌ 미사용 A가 없으면 B, C도 정렬이 보장되지 않음
WHERE C = 3 ❌ 미사용 마지막 컬럼만 단독 사용 불가
WHERE A = 1 ORDER BY B ✅ 사용 A로 필터 후 B는 이미 정렬되어 있어 filesort 불필요
ORDER BY B, C ❌ 미사용 A 없이 B, C 정렬은 보장되지 않음
핵심 원리 — 왜 거꾸로면 안 타는가?
복합 인덱스 (A, B, C)의 B+Tree는 A로 먼저 정렬 → 같은 A 그룹 내에서 B로 정렬 → 같은 A,B 그룹 내에서 C로 정렬합니다. 따라서 A를 모르면 B의 정렬 순서를 알 수 없고, A,B를 모르면 C의 정렬 순서를 알 수 없습니다. 이것이 Leftmost Prefix Rule입니다.
실제 쿼리에서 세컨더리 인덱스가 타지는 전체 흐름을 단계별로 설명해주세요.
SELECT name, age
FROM users
WHERE email = 'hello@test.com'

클러스터드, 세컨더리, 복합, 커버링, 유니크 인덱스의 종류와 특징을 정리해주세요.
종류 Key 리프 노드 저장 내용 핵심 특징
클러스터드 PK 행 데이터 전체 테이블당 1개, 물리적 정렬
세컨더리 지정 컬럼 인덱스 Key + PK 여러 개 가능, 테이블 룩업 필요
복합 인덱스 (컬럼A, B, C..) 복합 Key + PK Leftmost Prefix Rule 적용
커버링 인덱스 쿼리 필요 컬럼 전부 포함 Key에 모든 정보 있음 테이블 룩업 제거, Using index
유니크 인덱스 UNIQUE 컬럼 Key + PK 중복 불허, 동등 검색 최적화
참고: "커버링 인덱스"는 별도의 인덱스 종류가 아니라, 쿼리에 필요한 모든 컬럼이 인덱스에 포함된 상황을 말합니다. 복합 인덱스가 커버링 인덱스가 될 수도, 단일 컬럼 인덱스가 커버링이 될 수도 있습니다.
인덱스 전체 개념을 면접 한 줄로 요약해주세요.
① 인덱스 = B+Tree — 모든 인덱스는 각자의 Key를 기준으로 정렬된 별도의 B+Tree
② 클러스터드 — 리프 = 행 데이터 전체 / 세컨더리 — 리프 = Key + PK (→ 테이블 룩업 필요)
③ 복합 인덱스 — (A, B, C)면 A→B→C 순서로 정렬. 왼쪽부터만 인덱스를 탈 수 있음
④ 커버링 인덱스 — 인덱스만으로 쿼리 결과를 반환 → 테이블 룩업 제거 → I/O 절약
⑤ 거꾸로(B만, C만) 검색하면? — 첫 번째 컬럼 없이는 정렬 순서를 알 수 없어 인덱스를 못 탐
⑥ 복합 인덱스 컬럼 순서 전략 — 카디널리티 높은 컬럼 앞에 + WHERE 자주 쓰는 컬럼 앞에
