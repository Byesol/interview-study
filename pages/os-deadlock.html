Deadlock (교착상태)

발생 조건, 예방, 회피, 탐지, 실무 사례까지

Deadlock(교착상태)이란 무엇인가요?

Deadlock은 두 개 이상의 프로세스(또는 스레드)가 서로 상대방이 점유한 자원을 기다리면서 무한히 대기하는 상태입니다. 어떤 프로세스도 더 이상 진행하지 못하고 영원히 멈춥니다.

비유하자면, 좁은 일방통행 다리 양쪽 끝에서 차 두 대가 마주보고 서 있는 상황입니다. 각자 상대방이 먼저 비켜주길 기다리지만, 아무도 움직일 수 없습니다.

꼬리질문 대비: "Deadlock과 무한루프의 차이는?" — 무한루프는 CPU를 계속 사용하면서 반복 실행되지만, Deadlock은 CPU를 전혀 사용하지 않고 자원을 기다리며 멈춰있는 상태입니다.
Deadlock 발생의 4가지 필요조건(Coffman 조건)은 무엇인가요?

Deadlock은 다음 4가지 조건이 동시에 성립할 때만 발생합니다. 하나라도 깨지면 Deadlock은 불가능합니다. 이 조건들을 Coffman(1971)이 정리했습니다.

조건설명예시
1. 상호배제
(Mutual Exclusion)		 자원은 한 번에 하나의 프로세스만 사용할 수 있다 프린터, DB의 레코드 락
2. 점유대기
(Hold and Wait)		 자원을 하나 이상 보유한 상태에서 다른 자원을 추가로 기다린다 R1 잡은 채로 R2 대기
3. 비선점
(No Preemption)		 프로세스가 자원을 스스로 반납하기 전까지 강제로 빼앗을 수 없다 DB 락은 트랜잭션 완료 전 해제 불가
4. 순환대기
(Circular Wait)		 프로세스들이 원형으로 서로의 자원을 기다린다 A→R2대기, B→R1대기 → 사이클 형성
암기 팁: 상·점·비·순 — 상호배제, 점유대기, 비선점, 순환대기. 4가지 모두 동시 성립 시에만 Deadlock 발생. 예방(Prevention)은 이 중 하나를 제거하는 전략입니다.
꼬리질문 대비: "4가지 중 실무에서 가장 깨기 쉬운 조건은?" — 순환대기입니다. 모든 스레드/트랜잭션이 동일한 순서로 자원을 요청하도록 코딩 컨벤션을 정하는 것만으로 순환대기를 원천 차단할 수 있습니다.
Deadlock vs Starvation vs Livelock의 차이를 설명해주세요.

세 가지 모두 프로세스가 원하는 작업을 진행하지 못하는 상황이지만 원인과 상태가 다릅니다.

개념상태CPU 사용진행 가능성비유
Deadlock 완전히 멈춤, 자원 대기 없음 (대기 상태) 외부 개입 없이 불가 두 차가 좁은 다리서 교착
Starvation 대기 중, 다른 프로세스는 진행 없음 (대기 상태) 이론상 가능하나 실제 불가 항상 새치기 당하는 손님
Livelock 실행 중이지만 진전 없음 있음 (바쁘게 동작) 외부 개입 없이 불가 복도에서 계속 같은 방향으로 피하는 두 사람

Starvation(기아)은 우선순위가 낮은 프로세스가 계속 높은 우선순위 프로세스에게 밀려 자원을 영영 받지 못하는 상태입니다. Aging(시간이 지날수록 우선순위를 높여주는 기법)으로 해결합니다.

Livelock은 두 프로세스가 서로 배려하다가 오히려 아무것도 못하는 상황입니다. 예를 들어 두 스레드가 락 충돌 시 서로 양보하는 로직이 완벽하게 동기화되어, 둘 다 계속 양보만 반복하며 작업을 진행하지 못하는 경우입니다. CPU는 사용하지만 실질적 진전이 없다는 점이 Deadlock과의 핵심 차이입니다.

꼬리질문 대비: "Livelock 해결 방법은?" — 랜덤 백오프(Random Backoff)가 대표적입니다. Ethernet 충돌 해결(CSMA/CD)이나 분산 시스템의 재시도 로직에서 랜덤한 대기 시간을 추가하는 방식으로 Livelock을 깨뜨립니다.
자원 할당 그래프(Resource Allocation Graph)란 무엇인가요?

자원 할당 그래프는 프로세스와 자원 간의 관계를 방향 그래프로 표현한 것입니다. Deadlock 유무를 시각적으로 판단하는 데 사용됩니다.

그래프의 구성 요소는 두 가지입니다. 노드(Node)는 프로세스(원으로 표시)와 자원(사각형으로 표시, 내부 점은 자원 인스턴스 수)으로 구성됩니다. 간선(Edge)은 두 종류가 있습니다. 요청 간선(Request Edge, P→R)은 프로세스가 자원을 요청 중임을 나타내고, 할당 간선(Assignment Edge, R→P)은 자원이 프로세스에 할당된 상태를 나타냅니다.

Deadlock 판별 규칙: 자원 인스턴스가 하나씩인 경우, 그래프에 사이클이 존재하면 반드시 Deadlock. 인스턴스가 여러 개인 경우 사이클이 있어도 Deadlock이 아닐 수 있습니다 — 다른 인스턴스로 할당 가능하면 진행이 되기 때문입니다.
Deadlock 예방(Prevention)이란? 각 조건을 어떻게 제거하나요?

Deadlock 예방은 4가지 필요조건 중 하나를 시스템 설계 단계에서 원천적으로 불성립하게 만드는 전략입니다. 가장 보수적인 접근으로, Deadlock이 절대 발생하지 않습니다. 단, 자원 활용률이 낮아지는 트레이드오프가 있습니다.

제거할 조건방법단점
상호배제 제거 자원을 공유 가능하게 만듦 (읽기 전용 파일 등) 프린터·DB락 같이 본질적으로 배타적인 자원에는 적용 불가
점유대기 제거 프로세스 시작 시 필요한 모든 자원을 한꺼번에 요청, 없으면 대기 자원 이용률 저하, 기아(Starvation) 가능성
비선점 제거 자원을 기다릴 때 현재 보유 자원을 모두 반납 작업 중단·재시작 비용 발생, 일부 자원(프린터 등)엔 부적합
순환대기 제거 모든 자원에 번호를 부여하고, 오름차순 순서로만 요청 허용 자원 번호 관리 필요, 유연성 감소
실무에서 가장 현실적인 예방: 순환대기 제거입니다. "항상 락을 A → B 순서로 획득한다"는 코딩 규약만으로 순환대기를 예방할 수 있습니다. 오버헤드가 거의 없어 DB, Java 멀티스레드 코드에서 널리 사용됩니다.
Deadlock 회피(Avoidance)란? 은행원 알고리즘(Banker's Algorithm)을 설명해주세요.

Deadlock 회피는 자원 요청이 들어올 때마다 이 요청을 허용하면 안전 상태(Safe State)가 유지되는지 검사하고, 불안전 상태가 될 것 같으면 요청을 거부하는 전략입니다. 예방보다 유연하지만, 가용 자원과 최대 요구량을 미리 알아야 한다는 전제가 있습니다.

안전 상태(Safe State)란 모든 프로세스를 데드락 없이 종료시킬 수 있는 순서(안전 순서열, Safe Sequence)가 존재하는 상태입니다. 안전 상태이면 Deadlock이 발생하지 않습니다.

은행원 알고리즘은 Dijkstra가 고안한 회피 알고리즘입니다. 은행이 대출 요청을 받을 때 모든 고객이 상환 가능한지 시뮬레이션하는 것에서 이름이 유래했습니다.

동작 원리:
1. 프로세스가 자원 요청 시, 요청을 임시로 허용해 본다
2. 자원을 할당한 후 상태가 안전 상태인지 검사 (안전 순서열이 존재하는지)
3. 안전 상태 → 요청 허용 / 불안전 상태 → 요청 거부, 이전 상태로 복구

안전 순서열 탐색 방법: 현재 가용 자원(Available)으로 완료 가능한 프로세스를 찾고, 그 프로세스의 자원을 회수해 가용량을 늘리는 과정을 반복합니다. 모든 프로세스를 종료시킬 수 있으면 안전 상태입니다.

은행원 알고리즘의 한계: 프로세스 수와 최대 자원 요구량을 사전에 알아야 하고, 자원 요청마다 안전 순서열 탐색(O(n²·m)) 비용이 발생합니다. 실제 운영 시스템에서는 거의 사용하지 않고, CS 이론 시험과 면접에서 자주 출제됩니다.
꼬리질문 대비: "Safe State와 Deadlock-Free State는 같은 건가요?" — 아닙니다. 안전 상태이면 Deadlock이 없습니다. 하지만 불안전 상태(Unsafe State)라고 해서 반드시 Deadlock이 발생하는 것은 아닙니다. 불안전 상태는 Deadlock 가능성이 있는 상태입니다.
Deadlock 탐지(Detection)와 Wait-for Graph는 무엇인가요?

Deadlock 탐지는 Deadlock의 발생을 허용하되, 주기적으로 시스템을 검사해 Deadlock이 생겼는지 찾아내는 전략입니다. 예방·회피보다 자원 활용률이 높지만, Deadlock 발생 후 복구 비용이 발생합니다.

Wait-for Graph는 자원 할당 그래프에서 자원 노드를 제거하고 프로세스 간의 대기 관계만 표현한 그래프입니다. "A → B" 간선은 A가 B가 보유한 자원을 기다린다는 의미입니다.

Wait-for Graph에서 사이클이 감지되면 Deadlock이 발생한 것입니다(자원 인스턴스가 하나씩인 경우). 탐지 알고리즘은 DFS/BFS로 사이클을 찾습니다. 탐지 주기가 너무 잦으면 오버헤드가 크고, 너무 드물면 Deadlock이 오래 지속됩니다. 실무에서는 CPU 사용률이 급격히 낮아질 때 탐지를 트리거하는 방식을 쓰기도 합니다.

Deadlock 복구(Recovery) 방법에는 무엇이 있나요?

Deadlock이 탐지된 후 이를 해결하는 방법은 크게 프로세스 종료와 자원 선점 두 가지입니다.

1. 프로세스 종료(Process Termination)

두 가지 방식이 있습니다. 첫째, Deadlock에 관련된 프로세스를 모두 종료하는 방법입니다. 가장 확실하지만, 진행 중이던 모든 작업이 손실됩니다. 둘째, 프로세스를 하나씩 종료하면서 Deadlock이 해결될 때까지 반복하는 방법입니다. 어느 프로세스를 먼저 종료할지 선택 기준이 필요합니다.

종료 프로세스 선택 기준: 우선순위가 낮은 것 / 작업 진행률이 낮은 것 / 사용한 자원이 많은 것 / 종료 후 재시작 비용이 낮은 것 / 대화형보다 배치 작업 우선

2. 자원 선점(Resource Preemption)

Deadlock에 연루된 프로세스로부터 자원을 강제로 빼앗아 다른 프로세스에 할당합니다. 세 가지 고려사항이 있습니다.

고려사항내용
선점 대상 선택어느 프로세스의 자원을 빼앗을지 결정 (비용 최소화 기준)
롤백(Rollback)선점당한 프로세스를 안전 상태(체크포인트)로 되돌려야 함
기아 방지같은 프로세스만 반복 선점되지 않도록 횟수 제한
꼬리질문 대비: "OS 수준에서 Deadlock을 어떻게 처리하나요?" — 현대 범용 OS(Linux, Windows)는 대부분 Deadlock을 무시합니다(Ostrich Algorithm). Deadlock은 드물게 발생하고 완전한 예방·회피는 오버헤드가 크기 때문에, 발생하면 사용자가 프로세스를 강제 종료하도록 맡깁니다. 대신 DB나 Java 같은 응용 프로그램 레벨에서 자체적으로 처리합니다.
MySQL InnoDB에서 Deadlock은 어떻게 발생하고 어떻게 확인하나요?

MySQL InnoDB는 트랜잭션마다 레코드 락(Row Lock)을 사용합니다. 두 트랜잭션이 서로 다른 레코드를 잠근 채 상대방의 레코드를 기다리면 Deadlock이 발생합니다.

전형적인 MySQL Deadlock 시나리오:

-- 트랜잭션 A
START TRANSACTION;
UPDATE orders SET status = 'PROCESSING' WHERE id = 1;  -- 레코드 1 락 획득
-- (이 시점에 스레드 B가 레코드 2 락 획득)
UPDATE orders SET status = 'PROCESSING' WHERE id = 2;  -- 레코드 2 대기 → DEADLOCK

-- 트랜잭션 B (동시 실행)
START TRANSACTION;
UPDATE orders SET status = 'DONE' WHERE id = 2;    -- 레코드 2 락 획득
UPDATE orders SET status = 'DONE' WHERE id = 1;    -- 레코드 1 대기 → DEADLOCK

InnoDB는 Wait-for Graph를 내부적으로 유지하며 Deadlock을 자동 탐지합니다. 탐지되면 비용이 낮은(변경 레코드 수가 적은) 트랜잭션을 자동으로 롤백하고 ERROR 1213: Deadlock found를 반환합니다.

Deadlock 확인 방법:

-- 가장 최근 Deadlock 정보 확인
SHOW ENGINE INNODB STATUS\G

-- 출력에서 LATEST DETECTED DEADLOCK 섹션을 확인
-- 어떤 트랜잭션이 어떤 락을 잡고 기다렸는지 상세히 나옴

-- 현재 락 대기 현황 확인
SELECT * FROM performance_schema.data_lock_waits;
SELECT * FROM information_schema.INNODB_LOCK_WAITS;
innodb_deadlock_detect: MySQL 8.0부터 innodb_deadlock_detect=OFF로 탐지를 끄고, 대신 innodb_lock_wait_timeout으로 타임아웃을 설정하는 방법도 있습니다. 고부하 시스템에서 탐지 오버헤드를 줄이기 위한 최적화입니다.
꼬리질문 대비: "MySQL에서 Deadlock을 애플리케이션 레벨에서 어떻게 처리하나요?" — ERROR 1213을 catch해서 트랜잭션을 재시도(Retry)합니다. Spring에서는 @Retryable을 활용하거나, 트랜잭션 어드바이저에 재시도 로직을 넣습니다. 단, 무한 재시도는 금지하고 최대 횟수(예: 3회)와 지수 백오프를 적용합니다.
Java에서 Deadlock이 발생하는 코드 패턴과 탐지 방법을 설명해주세요.

Java에서 Deadlock은 주로 synchronized 블록이나 ReentrantLock을 여러 스레드가 다른 순서로 획득할 때 발생합니다.

Deadlock 발생 예시:

public class DeadlockExample {
    private final Object lockA = new Object();
    private final Object lockB = new Object();

    public void methodA() {
        synchronized (lockA) {          // 스레드1: A 먼저 획득
            Thread.sleep(100);
            synchronized (lockB) {      // 스레드1: B 기다림 → DEADLOCK
                // 작업
            }
        }
    }

    public void methodB() {
        synchronized (lockB) {          // 스레드2: B 먼저 획득
            Thread.sleep(100);
            synchronized (lockA) {      // 스레드2: A 기다림 → DEADLOCK
                // 작업
            }
        }
    }
}

#   which is held by "Thread-0"
# "Thread-0": waiting to lock monitor 0x...  (object 0x..., a java.lang.Object)
#   which is held by "Thread-1"
프로그래밍으로 탐지: ThreadMXBean을 사용하면 코드 내에서 Deadlock을 감지할 수 있습니다.
ManagementFactory.getThreadMXBean().findDeadlockedThreads() — null이 아니면 Deadlock 발생.
꼬리질문 대비: "ReentrantLock을 쓰면 Deadlock을 피할 수 있나요?" — Deadlock 자체를 피할 수는 없지만, tryLock(timeout)을 사용하면 지정 시간 내에 락을 얻지 못하면 포기하고 보유 락을 반납하게 할 수 있습니다. 이렇게 하면 Deadlock 상태에서 자동으로 빠져나올 수 있습니다(비선점 조건 위반).
Deadlock 방지를 위한 실무 코딩 패턴을 설명해주세요.

실무에서 Deadlock을 예방하는 핵심 패턴은 4가지입니다.

1. 락 획득 순서 통일 (가장 중요)

// BAD: 스레드마다 락 순서가 달라 순환대기 가능
// Thread 1: lock(accountA) → lock(accountB)
// Thread 2: lock(accountB) → lock(accountA)

// GOOD: 계좌 ID 순서로 항상 작은 번호 먼저 락
public void transfer(Account from, Account to, int amount) {
    Account first  = from.getId() < to.getId() ? from : to;
    Account second = from.getId() < to.getId() ? to   : from;
    synchronized (first) {
        synchronized (second) {
            from.debit(amount);
            to.credit(amount);
        }
    }
}

2. 타임아웃 사용 (ReentrantLock.tryLock)

private final ReentrantLock lock = new ReentrantLock();

public boolean doWork() throws InterruptedException {
    if (lock.tryLock(500, TimeUnit.MILLISECONDS)) {
        try {
            // 작업 수행
            return true;
        } finally {
            lock.unlock();   // finally에서 반드시 해제
        }
    }
    return false;  // 타임아웃 → 재시도 또는 실패 처리
}

3. 락 범위 최소화

// BAD: 락을 오래 보유 → 충돌 가능성 증가
synchronized (this) {
    data = fetchFromDB();   // DB 호출을 락 안에서 (느림)
    process(data);
}

// GOOD: 공유 상태 변경 시에만 락 사용
Data data = fetchFromDB(); // 락 밖에서 조회
synchronized (this) {
    sharedState = process(data); // 필요한 최소 범위만
}

4. 한 번에 모든 락 획득 (점유대기 제거)

필요한 모든 자원을 원자적으로 획득하거나 아예 얻지 않는 전략. 부분 획득 후 대기하는 상황을 없앱니다. Java의 StampedLock이나 DB의 SELECT ... FOR UPDATE SKIP LOCKED가 이 원칙을 활용합니다.
꼬리질문 대비: "락 없이 동시성을 처리하는 방법은?" — Lock-Free 알고리즘입니다. CAS(Compare-And-Swap) 연산을 사용하는 AtomicInteger, AtomicReference 같은 java.util.concurrent.atomic 패키지가 대표적입니다. 락 없이 원자적 연산을 보장해 Deadlock 자체가 불가능합니다.
분산 시스템에서의 Deadlock은 어떻게 다른가요?

분산 시스템에서의 Deadlock은 여러 노드(서버)에 걸쳐 발생하기 때문에 단일 시스템보다 탐지와 복구가 훨씬 복잡합니다.

분산 Deadlock의 특징:

구분단일 시스템분산 시스템
탐지 방법jstack, OS 내부 Wait-for Graph글로벌 Wait-for Graph 수집 필요
복잡도낮음높음 (네트워크 지연, 메시지 순서 문제)
대표 해결책순서 통일, tryLock 타임아웃분산 락 타임아웃, 우선순위 기반 중단

분산 Deadlock 처리 전략:

타임아웃 기반 해결 (가장 실용적): 분산 환경에서 Wait-for Graph를 글로벌하게 수집하는 것은 복잡하고 비용이 크기 때문에, 실무에서는 락 타임아웃을 설정해 일정 시간 내에 락을 얻지 못하면 트랜잭션을 롤백하는 방식을 주로 사용합니다. Redis 분산 락(Redlock)의 TTL, DB 락 타임아웃이 이 방식입니다.

Redis 분산 락 타임아웃 예시:

// Redisson을 사용한 분산 락 (타임아웃 적용)
RLock lock = redissonClient.getLock("order:lock:123");
boolean acquired = lock.tryLock(
    3,     // 최대 3초 대기
    10,    // 락 보유 최대 10초 (TTL)
    TimeUnit.SECONDS
);
if (!acquired) {
    throw new LockAcquisitionException("락 획득 실패");
}
try {
    // 임계 영역
} finally {
    lock.unlock();
}
꼬리질문 대비: "마이크로서비스 아키텍처에서 Saga 패턴이 Deadlock과 어떤 관련이 있나요?" — Saga 패턴은 분산 트랜잭션을 여러 로컬 트랜잭션으로 분리하고, 실패 시 보상 트랜잭션(Compensating Transaction)으로 롤백합니다. 각 서비스가 독립적으로 락을 잡고 바로 해제하기 때문에 분산 Deadlock 가능성이 크게 줄어듭니다.
