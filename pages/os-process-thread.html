프로세스 & 스레드 면접 가이드

프로세스 · 스레드 · 컨텍스트 스위칭 · PCB · 실무 연결

프로세스란 무엇인가요?

프로세스는 실행 중인 프로그램입니다. 디스크에 저장된 프로그램 파일은 그냥 코드 덩어리일 뿐이고, 이걸 더블클릭해서 메모리에 올려 CPU가 실행하기 시작하면 그때부터 "프로세스"라고 부릅니다. 비유하자면, 레시피(프로그램)를 보고 실제로 요리를 시작하는 행위(프로세스)와 같습니다.

OS는 각 프로세스에게 독립된 메모리 공간을 할당합니다. 이 메모리는 크게 Code 영역(실행 코드), Data 영역(전역 변수), Heap(동적 할당), Stack(함수 호출/지역 변수) 네 가지로 나뉩니다. 각 프로세스는 자기만의 주소 공간을 갖기 때문에, 한 프로세스가 죽어도 다른 프로세스에 영향을 주지 않습니다.

꼬리질문 대비: "프로세스 간 통신(IPC)은 어떻게 하나요?" — 독립된 메모리 공간 때문에 직접 데이터를 공유할 수 없고, 파이프, 소켓, 공유 메모리, 메시지 큐 같은 IPC 기법을 사용해야 합니다.
스레드란 무엇이고, 프로세스와 어떻게 다른가요?

스레드는 프로세스 내에서 실행되는 흐름의 단위입니다. 하나의 프로세스 안에 여러 스레드가 존재할 수 있습니다. 비유하자면, 프로세스가 하나의 공장이라면 스레드는 그 공장 안에서 일하는 각각의 작업자입니다. 작업자들은 같은 공장(메모리 공간)을 공유하면서 각자 맡은 일을 합니다.

핵심적인 차이는 자원 공유 범위입니다. 같은 프로세스의 스레드들은 Code, Data, Heap 영역을 공유하고, 각 스레드마다 독립적인 Stack과 PC(Program Counter), 레지스터만 따로 가집니다. 반면 프로세스 간에는 메모리 전체가 격리되어 있습니다.

구분프로세스스레드
메모리독립된 주소 공간Code/Data/Heap 공유, Stack만 독립
생성 비용무거움 (메모리 할당, PCB 생성)가벼움 (Stack, 레지스터만 할당)
통신IPC 필요 (오버헤드 큼)공유 메모리로 직접 통신 (빠름)
안정성한 프로세스 죽어도 다른 프로세스 무관한 스레드 문제 → 전체 프로세스 영향
컨텍스트 스위칭비용 큼 (TLB 플러시 등)비용 작음 (같은 주소 공간)

이 차이 때문에 실무에서는 성능이 중요한 작업엔 멀티스레드를, 안정성이 중요한 작업엔 멀티프로세스를 선택합니다. 웹 서버의 경우 Nginx는 멀티프로세스(워커 프로세스), Spring Boot의 Tomcat은 멀티스레드 방식을 사용하는 것이 대표적인 예입니다.

PCB(Process Control Block)란 무엇이고, 왜 필요한가요?

PCB는 OS가 프로세스를 관리하기 위해 유지하는 자료구조입니다. 비유하자면, 병원에서 환자마다 차트를 만들어 상태를 기록하듯이, OS도 프로세스마다 PCB라는 "신상카드"를 만들어 커널 메모리에 보관합니다. 프로세스가 생성되면 PCB가 만들어지고, 종료되면 PCB가 삭제됩니다.

PCB가 필요한 핵심 이유는 컨텍스트 스위칭 때문입니다. CPU가 프로세스 A에서 B로 전환할 때, A의 현재 상태(PC, 레지스터 등)를 A의 PCB에 저장하고, B의 PCB에서 이전 상태를 복원합니다. PCB가 없으면 프로세스를 중단했다가 재개할 수 없습니다. Linux에서는 이를 task_struct라는 구조체로 구현합니다.

꼬리질문 대비: "스레드에도 TCB(Thread Control Block)가 있나요?" — 네, 있습니다. 다만 스레드는 프로세스의 메모리를 공유하므로, TCB에는 스레드 고유의 PC, 레지스터, 스택 포인터 정도만 저장되어 PCB보다 훨씬 가볍습니다.
프로세스의 상태 변화(State Transition)를 설명해주세요.

프로세스는 생성부터 종료까지 5가지 상태를 거칩니다. New(생성됨) → Ready(준비 큐에서 CPU 할당 대기) → Running(CPU에서 실행 중) → Waiting/Blocked(I/O 등 이벤트 대기) → Terminated(종료)의 흐름입니다.

면접에서 중요한 포인트는 Running → Ready 전환(Preemption)입니다. 이것은 타임 슬라이스가 만료되었을 때 OS 스케줄러가 강제로 CPU를 뺏는 것으로, 여기서 컨텍스트 스위칭이 발생합니다. 또한 Running → Waiting 전환은 프로세스 스스로가 I/O를 요청하면서 CPU를 자발적으로 반납하는 것이고, Waiting → Ready 전환은 I/O가 완료되어 인터럽트가 발생했을 때 일어납니다.

컨텍스트 스위칭이란 무엇이고, 구체적으로 어떤 과정이 일어나나요?

컨텍스트 스위칭은 CPU가 현재 실행 중인 프로세스(또는 스레드)를 멈추고 다른 프로세스로 전환하는 작업입니다. 비유하자면, 책을 읽다가 다른 책으로 바꿀 때 현재 페이지에 책갈피를 꽂고(상태 저장), 다른 책의 책갈피가 있는 페이지를 여는 것(상태 복원)과 같습니다.

구체적인 과정은 다음과 같습니다:

  1. 1인터럽트/시스템 콜 발생 — 타이머 인터럽트, I/O 요청, 또는 더 높은 우선순위 프로세스 도착
  2. 2현재 프로세스 상태 저장 — PC, 레지스터, 스택 포인터 등을 현재 프로세스의 PCB에 저장
  3. 3스케줄러가 다음 프로세스 선택 — Ready 큐에서 다음에 실행할 프로세스를 결정
  4. 4선택된 프로세스의 상태 복원 — 해당 프로세스의 PCB에서 PC, 레지스터 등을 CPU에 로드
  5. 5메모리 주소 공간 전환 — 페이지 테이블 교체, TLB(Translation Lookaside Buffer) 플러시
  6. 6새 프로세스 실행 재개 — 복원된 PC 위치부터 실행 계속

여기서 핵심은 5번 단계(TLB 플러시)입니다. TLB는 가상 주소 → 물리 주소 변환을 캐싱하는 하드웨어인데, 프로세스가 바뀌면 주소 공간이 바뀌므로 기존 TLB 내용이 무효화됩니다. TLB 미스가 연쇄적으로 발생하면서 메모리 접근이 느려지는 것이 컨텍스트 스위칭 비용의 상당 부분을 차지합니다.

꼬리질문 대비: "스레드 간 컨텍스트 스위칭은 어떻게 다른가요?" — 같은 프로세스의 스레드끼리는 주소 공간이 동일하므로 TLB 플러시가 필요 없습니다. PC와 레지스터, 스택 포인터만 교체하면 되기 때문에 프로세스 간 스위칭보다 훨씬 빠릅니다.
컨텍스트 스위칭의 비용(오버헤드)에는 어떤 것들이 있나요?

컨텍스트 스위칭 비용은 크게 직접 비용과 간접 비용으로 나뉩니다.

직접 비용은 PCB에 상태를 저장하고 복원하는 시간입니다. 레지스터 값 저장/복원, 커널 모드 전환 등이 여기에 해당합니다. 이것만 보면 수 마이크로초(μs) 수준으로 작아 보이지만, 문제는 간접 비용입니다.

간접 비용의 핵심은 캐시 오염(Cache Pollution)입니다. 프로세스 A가 열심히 CPU 캐시(L1, L2, L3)와 TLB를 워밍업 해놨는데, 프로세스 B로 전환하면 B의 데이터로 캐시가 교체됩니다. 다시 A로 돌아왔을 때 캐시 미스가 대량 발생하면서 "콜드 스타트" 상태가 됩니다. 이 간접 비용이 직접 비용보다 수십 배 클 수 있습니다. 실측으로 프로세스 간 스위칭은 보통 수십 μs, 스레드 간 스위칭은 수 μs 수준으로 알려져 있습니다.

멀티프로세스 vs 멀티스레드, 어떤 기준으로 선택하나요?

선택 기준은 안정성 vs 성능의 트레이드오프입니다.

멀티프로세스는 한 프로세스가 죽어도 다른 프로세스에 영향이 없어서 안정성이 높습니다. 대신 프로세스 생성 비용이 크고, 데이터를 공유하려면 IPC를 써야 해서 오버헤드가 있습니다. 크롬 브라우저가 탭마다 별도 프로세스를 쓰는 이유가 바로 이것입니다 — 한 탭이 크래시되어도 브라우저 전체가 죽지 않습니다.

멀티스레드는 메모리를 공유하므로 통신이 빠르고 컨텍스트 스위칭 비용이 작습니다. 하지만 하나의 스레드에서 발생한 메모리 오류(예: 잘못된 포인터 접근)가 전체 프로세스를 죽일 수 있고, 동기화 문제(Race Condition, Deadlock)를 주의해야 합니다. Spring Boot의 Tomcat이 요청을 스레드 풀로 처리하는 것이 대표적인 예입니다.

실무에서의 판단 기준을 정리하면, I/O 바운드 작업이 많고 데이터 공유가 빈번하면 멀티스레드가 유리하고, 각 작업 단위의 격리가 중요하거나 CPU 바운드 작업이면 멀티프로세스가 유리합니다. 물론 현대 시스템에서는 둘을 혼합해서 쓰는 경우가 대부분입니다.

사용자 수준 스레드(User-Level Thread)와 커널 수준 스레드(Kernel-Level Thread)의 차이를 설명해주세요.

사용자 수준 스레드는 커널이 그 존재를 모르고, 사용자 공간의 라이브러리가 스레드 관리를 합니다. 스레드 전환 시 커널 모드로 전환할 필요가 없어 빠르지만, 한 스레드가 블로킹 I/O를 호출하면 같은 프로세스의 모든 스레드가 멈추는 치명적인 단점이 있습니다. 커널 입장에서는 프로세스 하나가 통째로 블록된 것으로 보이기 때문입니다.

커널 수준 스레드는 OS가 직접 스레드를 인식하고 스케줄링합니다. 한 스레드가 블록되어도 같은 프로세스의 다른 스레드가 실행될 수 있지만, 스레드 전환 시 커널 모드 전환이 필요하므로 오버헤드가 더 큽니다.

현대 OS(Linux, Windows)는 대부분 1:1 모델(사용자 스레드 1개 = 커널 스레드 1개)을 사용합니다. Java의 스레드가 대표적으로, Java 스레드 하나가 OS 커널 스레드 하나에 매핑됩니다. 다만 Java 21에서 도입된 Virtual Thread는 M:N 모델로, 수백만 개의 경량 스레드를 소수의 커널 스레드 위에서 돌리는 새로운 접근입니다.

꼬리질문 대비: "Go의 고루틴(Goroutine)은 어떤 모델인가요?" — Go의 고루틴은 M:N 모델입니다. Go 런타임 스케줄러가 수천~수만 개의 고루틴을 소수의 OS 스레드 위에서 관리합니다. Java의 Virtual Thread와 유사한 철학입니다.
크롬 브라우저가 탭을 많이 열면 느려지는 이유를 OS 관점에서 설명해주세요.

크롬은 멀티프로세스 아키텍처를 채택하고 있습니다. 각 탭, 확장 프로그램, GPU 처리, 유틸리티 기능이 별도의 프로세스로 동작합니다. 안정성 면에서는 훌륭하지만, 탭을 많이 열수록 OS 관점에서 여러 문제가 겹칩니다.

첫째, 메모리 소비입니다. 각 프로세스마다 독립된 메모리 공간을 가지므로, 탭 30개를 열면 30개의 프로세스가 각각 수백 MB를 차지합니다. 물리 메모리(RAM)가 부족해지면 OS가 스와핑(Swapping)을 시작하는데, 디스크에 메모리 페이지를 쓰고 읽는 작업이 발생하면서 급격하게 느려집니다. SSD라도 RAM 대비 수백~수천 배 느립니다.

둘째, 컨텍스트 스위칭 오버헤드입니다. 프로세스가 많아지면 CPU 스케줄러가 이들 사이를 빈번하게 전환해야 합니다. 프로세스 간 스위칭은 앞서 말한 TLB 플러시와 캐시 오염이 발생하므로, 프로세스 수가 늘어날수록 실제 작업에 쓰는 CPU 시간 비율이 줄어듭니다.

셋째, 각 탭의 렌더링 프로세스 내부에서도 다수의 스레드(메인 스레드, 컴포지터 스레드, 래스터 스레드 등)가 돌아가므로 OS가 관리해야 할 스레드 수가 폭발적으로 증가합니다.

IntelliJ(JetBrains IDE)가 무거운 이유를 프로세스/스레드 관점에서 설명해주세요.

IntelliJ는 크롬과 다른 이유로 무거워집니다. IntelliJ는 단일 프로세스, 멀티스레드 구조입니다. 하나의 JVM 프로세스 위에서 수십~수백 개의 스레드가 동시에 동작합니다.

첫째, JVM의 힙 메모리 소비가 큽니다. IntelliJ는 프로젝트의 전체 코드를 파싱해서 AST(Abstract Syntax Tree)와 인덱스를 메모리에 올려둡니다. 코드 자동완성, 리팩토링, 에러 감지를 실시간으로 하려면 이 인덱스가 반드시 필요한데, 대규모 프로젝트일수록 인덱스 크기가 커집니다. 기본 힙 사이즈가 부족하면 GC(Garbage Collection)가 빈번해지면서 "STW(Stop-The-World)" 일시 정지가 자주 발생합니다.

둘째, 백그라운드 스레드의 경합입니다. 코드 분석 스레드, 인덱싱 스레드, 빌드 스레드, 파일 시스템 워처 스레드 등이 동시에 돌아가면서 CPU와 메모리를 놓고 경쟁합니다. 특히 대규모 프로젝트에서 인덱싱이 돌 때 UI 스레드도 영향을 받아 입력이 버벅거리는 현상이 생깁니다.

셋째, 플러그인도 같은 JVM 프로세스 안에서 동작하므로, 무거운 플러그인이 메모리 누수를 일으키면 IDE 전체가 느려집니다. 실무에서 IntelliJ가 느리다면 -Xmx 값을 올리거나, 불필요한 플러그인을 비활성화하고, Invalidate Caches로 인덱스를 재생성하는 것이 일반적인 트러블슈팅 방법입니다.

꼬리질문 대비: "크롬은 멀티프로세스인데 IntelliJ는 왜 멀티스레드인가요?" — 크롬은 신뢰할 수 없는 웹 콘텐츠를 실행하므로 탭 간 격리(보안+안정성)가 최우선입니다. IntelliJ는 자체 코드만 실행하므로 격리보다 스레드 간 빠른 데이터 공유(코드 인덱스, AST 등)가 더 중요합니다.
크롬은 왜 멀티프로세스 구조를 선택했나요? 멀티스레드 대비 장단점은?

크롬이 멀티프로세스를 선택한 핵심 이유는 보안과 안정성입니다. 웹 브라우저는 인터넷에서 받은 임의의 HTML, JavaScript, 웹어셈블리를 실행해야 합니다. 이것은 "신뢰할 수 없는 코드"를 실행하는 것과 같습니다.

만약 멀티스레드 구조였다면, 악성 JavaScript가 공유 메모리를 통해 다른 탭의 데이터(쿠키, 로그인 세션 등)에 접근할 수 있는 보안 취약점이 생깁니다. 멀티프로세스는 OS 수준의 메모리 격리를 제공하므로, 한 탭의 렌더러 프로세스가 다른 탭의 메모리를 읽는 것이 원천적으로 차단됩니다. 이것이 크롬의 샌드박스(Sandbox) 보안 모델의 기반입니다.

안정성 면에서도, Flash 같은 플러그인이 크래시되어도 브라우저 전체가 죽지 않습니다. 그 탭만 "앗, 이런!" 페이지를 보여주고 나머지는 정상 동작합니다. 반면 과거의 IE(Internet Explorer)는 단일 프로세스 구조여서 한 탭의 문제로 브라우저 전체가 멈추곤 했습니다.

대신 메모리 사용량이 크다는 것이 가장 큰 단점입니다. 크롬은 이를 완화하기 위해 "탭 폐기(Tab Discarding)" 기능으로 오래 안 쓴 탭의 메모리를 해제하고, Site Isolation을 통해 같은 사이트의 iframe은 프로세스를 공유하는 등의 최적화를 하고 있습니다.

서버 개발에서 스레드 풀(Thread Pool)을 사용하는 이유는 무엇인가요?

스레드 풀을 사용하는 핵심 이유는 스레드 생성/소멸 비용을 줄이기 위해서입니다. 요청이 들어올 때마다 새 스레드를 만들고 응답 후 스레드를 죽이면, 스레드 생성 시 커널 자원 할당(스택 메모리, TCB 등)과 소멸 시 자원 회수가 반복되면서 오버헤드가 발생합니다.

스레드 풀은 미리 일정 수의 스레드를 만들어 두고, 작업이 들어오면 대기 중인 스레드에게 할당하고, 작업이 끝나면 스레드를 죽이지 않고 풀에 반환합니다. 비유하자면, 택시를 탈 때마다 새 차를 사는 게 아니라, 택시 승강장에 대기 중인 택시를 배정받는 것과 같습니다.

또 하나의 중요한 이유는 동시 실행 스레드 수를 제한할 수 있다는 것입니다. 제한 없이 스레드를 만들면 수천 개의 요청이 동시에 들어왔을 때 컨텍스트 스위칭 폭증, 메모리 고갈로 서버가 죽을 수 있습니다. Tomcat의 기본 스레드 풀은 200개가 최대인데, 이 한도를 넘는 요청은 큐에서 대기하도록 해서 시스템의 안정성을 지킵니다.

Java에서는 ExecutorService, Spring에서는 @Async + TaskExecutor, Tomcat은 내장 스레드 풀을 통해 이를 구현합니다.

fork()로 프로세스를 생성하면 메모리가 두 배로 늘어나나요?

아닙니다. Linux에서 fork()는 COW(Copy-On-Write) 방식을 사용합니다. 자식 프로세스를 생성할 때 부모의 메모리를 실제로 복사하지 않고, 부모와 자식이 같은 물리 메모리 페이지를 "읽기 전용"으로 공유합니다. 둘 중 하나가 페이지에 쓰기를 시도하는 순간에만 해당 페이지만 복사합니다.

이렇게 하면 fork() 자체는 매우 빠르게 완료됩니다. 실제로 복사되는 것은 페이지 테이블과 PCB 정도이고, 메모리 페이지는 실제 수정이 일어날 때만 점진적으로 복사됩니다. 만약 자식 프로세스가 바로 exec()를 호출해서 다른 프로그램으로 교체되면, 부모의 메모리를 한 번도 복사하지 않으므로 매우 효율적입니다.

실무 연결: Redis가 RDB 스냅샷을 찍을 때 fork()를 사용하는 이유가 바로 이 COW 때문입니다. 자식 프로세스가 현재 메모리 상태를 디스크에 저장하는 동안, 부모 프로세스는 계속 클라이언트 요청을 처리할 수 있습니다.
프로세스 간 통신(IPC) 방법에는 어떤 것들이 있고, 각각 언제 쓰나요?

프로세스는 메모리가 격리되어 있으므로 데이터를 주고받으려면 OS가 제공하는 IPC 메커니즘을 사용해야 합니다. 주요 방식은 다음과 같습니다.

파이프(Pipe)는 부모-자식 프로세스 간 단방향 통신에 적합합니다. 셸에서 ls | grep txt 같은 명령이 파이프를 사용하는 예시입니다. 공유 메모리(Shared Memory)는 가장 빠른 IPC 방식으로, 두 프로세스가 같은 물리 메모리 영역을 매핑해서 사용합니다. 다만 동기화를 직접 관리해야 합니다. 메시지 큐는 커널이 관리하는 큐에 메시지를 넣고 빼는 방식으로 비동기 통신에 적합합니다. 소켓은 네트워크를 넘어서도 통신할 수 있어 가장 범용적이지만, 오버헤드도 가장 큽니다.

IPC 방식속도특징대표 사용처
공유 메모리가장 빠름동기화 직접 관리DB 공유 버퍼
파이프빠름단방향, 부모-자식셸 파이프라인
메시지 큐보통비동기, 커널 관리비동기 작업 처리
소켓느림네트워크 가능MSA 서비스 간 통신

실무에서 마이크로서비스 아키텍처(MSA)의 서비스 간 통신은 대부분 소켓 기반(HTTP, gRPC)이고, 같은 머신의 프로세스 간에는 Unix 도메인 소켓이나 공유 메모리를 사용합니다.

멀티스레드 환경에서 발생할 수 있는 동기화 문제에는 어떤 것들이 있나요?

대표적으로 Race Condition, Deadlock, Starvation 세 가지가 있습니다.

Race Condition은 여러 스레드가 공유 데이터에 동시에 접근할 때, 실행 순서에 따라 결과가 달라지는 문제입니다. 예를 들어 두 스레드가 동시에 count++를 실행하면, read-modify-write가 원자적이지 않아 값이 1만 증가할 수 있습니다. 이를 방지하려면 Mutex, Semaphore, synchronized, Atomic 변수 등을 사용합니다.

Deadlock은 두 스레드가 서로가 가진 자원을 기다리면서 영원히 멈추는 상태입니다. 발생 조건은 상호배제, 점유대기, 비선점, 순환대기 네 가지(Coffman 조건)가 동시에 성립할 때이고, 하나라도 깨면 데드락을 예방할 수 있습니다. 실무에서 가장 흔한 방법은 자원 획득 순서를 정하는 것(순환대기 방지)입니다.

Starvation은 특정 스레드가 우선순위가 낮아서 영원히 자원을 할당받지 못하는 상태입니다. 공정한 스케줄링이나 에이징(Aging) 기법으로 완화합니다.

컨텍스트 스위칭을 최소화하는 방법은 무엇이 있나요?

실무에서 컨텍스트 스위칭을 줄이는 접근법은 크게 세 가지입니다.

첫째, 비동기/이벤트 기반 프로그래밍입니다. Node.js의 이벤트 루프, Java의 NIO, Spring WebFlux 같은 리액티브 프로그래밍은 소수의 스레드로 많은 I/O 작업을 처리합니다. 스레드 수가 줄면 컨텍스트 스위칭도 줄어듭니다.

둘째, CPU 친화도(Affinity) 설정입니다. 특정 스레드를 특정 CPU 코어에 고정하면 캐시 오염을 줄일 수 있습니다. 고성능 네트워크 서버나 게임 서버에서 사용하는 기법입니다. Linux에서는 taskset 명령이나 pthread_setaffinity_np()로 설정합니다.

셋째, 경량 스레드(코루틴/파이버) 사용입니다. Go의 고루틴, Java 21의 Virtual Thread, Kotlin의 코루틴은 OS 스레드 전환 없이 사용자 수준에서 작업을 전환하므로, OS 레벨 컨텍스트 스위칭이 발생하지 않습니다. 특히 I/O 바운드 서버에서 극적인 효과가 있습니다.

좀비 프로세스와 고아 프로세스란 무엇인가요?

좀비 프로세스(Zombie Process)는 자식 프로세스가 종료되었지만, 부모가 아직 wait()을 호출하지 않아서 PCB가 커널에 남아있는 상태입니다. 비유하자면, 퇴사한 직원인데 아직 사원증이 회수되지 않은 상태입니다. 프로세스 자체는 CPU나 메모리를 쓰지 않지만, PCB가 커널 메모리를 차지하고 PID를 점유하고 있어 PID가 고갈될 수 있습니다.

고아 프로세스(Orphan Process)는 부모 프로세스가 먼저 종료되어 부모가 없어진 자식 프로세스입니다. Linux에서는 이런 고아 프로세스를 init(PID 1) 또는 systemd가 양부모로 입양하여, 자식이 종료되었을 때 wait()을 호출해 줍니다. 따라서 고아 프로세스 자체는 큰 문제가 아니지만, 좀비 프로세스가 대량으로 쌓이면 시스템에 문제가 될 수 있습니다.

실무 팁: ps aux | grep Z로 좀비 프로세스를 확인할 수 있습니다. 좀비를 없애려면 부모 프로세스가 wait()을 호출하도록 수정하거나, 부모를 종료시켜 init이 처리하게 합니다.
