📦 Part 1. 트랜잭션 기본 개념 — 트랜잭션이란? ACID? 커밋과 롤백?
트랜잭션이란 무엇인가요?

트랜잭션은 하나의 논리적 작업 단위를 구성하는 일련의 연산입니다. 이 연산들은 모두 성공(Commit)하거나 모두 실패(Rollback)해야 하는 "All or Nothing" 특성을 가집니다. 예를 들어 계좌 이체에서 "A 계좌 출금"과 "B 계좌 입금"은 반드시 함께 성공하거나 함께 실패해야 합니다. 하나만 성공하면 돈이 사라지거나 복제되는 문제가 발생하기 때문입니다.

ACID 속성을 각각 설명해주세요.

Atomicity(원자성) — 트랜잭션 내 모든 연산은 전부 성공하거나 전부 취소됩니다. 중간 상태로 멈추는 것을 허용하지 않습니다.

Consistency(일관성) — 트랜잭션 수행 전후에 데이터베이스는 항상 일관된 상태를 유지해야 합니다. 제약 조건(PK, FK, CHECK 등)이 위반되면 트랜잭션이 실패합니다.

Isolation(격리성) — 동시에 실행되는 트랜잭션들이 서로에게 영향을 주지 않아야 합니다. 격리 수준(Isolation Level)에 따라 허용 범위가 달라집니다.

Durability(지속성) — 커밋된 트랜잭션의 결과는 시스템 장애가 발생하더라도 영구적으로 보존됩니다. InnoDB는 Redo Log(WAL)를 통해 이를 보장합니다.

Commit과 Rollback의 차이는?

Commit은 트랜잭션 내 모든 변경사항을 데이터베이스에 영구 반영하는 것입니다. 커밋 이후에는 되돌릴 수 없습니다. Rollback은 트랜잭션 내 모든 변경사항을 취소하고 트랜잭션 시작 전 상태로 되돌리는 것입니다. InnoDB에서는 Undo Log에 기록된 이전 버전 데이터를 이용해 롤백을 수행합니다.

MySQL에서 autocommit이란?

MySQL은 기본적으로 autocommit=1로 설정되어 있어서 각 SQL 문장이 자동으로 하나의 트랜잭션으로 처리되고 즉시 커밋됩니다. 여러 문장을 하나의 트랜잭션으로 묶으려면 START TRANSACTION 또는 BEGIN을 명시적으로 선언하고, 작업 후 COMMIT 또는 ROLLBACK을 호출해야 합니다.

🔒 Part 2. 격리 수준 (Isolation Level) — 동시성 문제, 4가지 격리 수준, 각 수준에서 발생하는 문제
동시성 문제 3가지(Dirty Read, Non-Repeatable Read, Phantom Read)를 설명해주세요.

Dirty Read — 다른 트랜잭션이 아직 커밋하지 않은 데이터를 읽는 것. 그 트랜잭션이 롤백하면 읽은 데이터는 존재하지 않는 값이 됩니다.

Non-Repeatable Read — 같은 트랜잭션 내에서 같은 행을 두 번 읽었는데, 그 사이에 다른 트랜잭션이 수정·커밋해서 값이 달라지는 현상입니다.

Phantom Read — 같은 조건으로 두 번 조회했는데, 그 사이에 다른 트랜잭션이 새 행을 삽입·삭제해서 결과 행 수가 달라지는 현상입니다.

REPEATABLE READ에서 Read View 생성 시점이 READ COMMITTED와 어떻게 다른가요?

REPEATABLE READ는 트랜잭션 내 첫 번째 SELECT 시점에 Read View를 한 번만 생성하고 트랜잭션 끝까지 재사용합니다. 같은 쿼리를 여러 번 실행해도 항상 동일한 결과입니다.

READ COMMITTED는 매 SELECT마다 새로운 Read View를 생성합니다. 다른 트랜잭션이 커밋하면 다음 SELECT에서 바로 보입니다. 이것이 Non-Repeatable Read가 발생하는 이유입니다.

🌿 Part 3. Spring @Transactional 기본 — 동작 원리, 프록시, 주의사항
Spring의 @Transactional은 어떻게 동작하나요?

Spring은 AOP(관점 지향 프로그래밍) 기반의 프록시 패턴으로 트랜잭션을 관리합니다. @Transactional이 붙은 메서드를 호출하면 실제 객체가 아닌 프록시 객체가 호출을 가로채서, 메서드 실행 전에 트랜잭션을 시작(DB 커넥션 획득 + BEGIN)하고, 메서드가 정상 완료되면 커밋, 예외가 발생하면 롤백합니다. 내부적으로는 TransactionManager가 커넥션 풀에서 커넥션을 가져와 ThreadLocal에 바인딩하여, 같은 트랜잭션 내 모든 DB 작업이 동일한 커넥션을 사용하도록 보장합니다.

@Transactional은 왜 같은 클래스 내부 호출(self-invocation)에서 동작하지 않나요?

@Transactional은 프록시 기반으로 동작합니다. 외부에서 빈을 호출하면 프록시를 거치지만, 같은 클래스 내에서 this.method()로 호출하면 프록시를 거치지 않고 실제 객체를 직접 호출하기 때문에 트랜잭션 AOP가 적용되지 않습니다. 해결 방법으로는 메서드를 별도 클래스로 분리하거나, AopContext.currentProxy()를 사용하거나, AspectJ weaving 모드로 전환하는 방법이 있습니다.

@Transactional의 롤백 규칙은 어떻게 되나요?

기본적으로 Unchecked Exception(RuntimeException, Error)이 발생하면 롤백하고, Checked Exception이 발생하면 롤백하지 않고 커밋합니다. Checked Exception에도 롤백하려면 @Transactional(rollbackFor = Exception.class)를 지정해야 합니다. 이것은 Spring의 규칙이며, 실무에서 혼동하기 쉬운 포인트입니다.

🔀 Part 4. 트랜잭션 전파 레벨 (Propagation) — 7가지 전파 레벨, 각각 언제 쓰는지
트랜잭션 전파(Propagation)란 무엇인가요?

이미 트랜잭션이 진행 중인 상태에서 또 다른 @Transactional 메서드가 호출되었을 때, 기존 트랜잭션에 참여할지, 새로 만들지, 트랜잭션 없이 실행할지를 결정하는 정책입니다.

7가지 전파 레벨을 설명해주세요.
전파 레벨기존 트랜잭션 있을 때기존 트랜잭션 없을 때사용 예시
REQUIRED (기본값) 기존에 참여 새로 생성 대부분의 서비스 메서드
REQUIRES_NEW 기존 일시중단 + 새로 생성 새로 생성 독립적 로깅, 감사 기록 (메인 롤백되어도 로그는 저장)
SUPPORTS 기존에 참여 트랜잭션 없이 실행 있으면 쓰고, 없어도 되는 조회
NOT_SUPPORTED 기존 일시중단 + 트랜잭션 없이 실행 트랜잭션 없이 실행 트랜잭션이 불필요한 외부 API 호출
MANDATORY 기존에 참여 예외 발생! 반드시 트랜잭션 내에서 호출되어야 하는 메서드
NEVER 예외 발생! 트랜잭션 없이 실행 트랜잭션 밖에서만 실행해야 하는 작업
NESTED 중첩 트랜잭션 (Savepoint) 새로 생성 부분 롤백이 필요한 경우 (JPA에서는 미지원)
REQUIRED 주의점: A 메서드(REQUIRED)가 B 메서드(REQUIRED)를 호출하면 같은 트랜잭션입니다. B에서 예외가 발생하면 A에서 try-catch로 잡아도 전체 트랜잭션이 rollback-only로 마킹되어 커밋 시 UnexpectedRollbackException이 발생합니다.
REQUIRED와 REQUIRES_NEW의 핵심 차이는?

REQUIRED는 기존 트랜잭션에 "합류"합니다. 하나라도 실패하면 전체가 롤백됩니다. REQUIRES_NEW는 기존 트랜잭션을 일시 중단하고 완전히 독립적인 새 트랜잭션을 만듭니다. 새 트랜잭션이 롤백되어도 기존 트랜잭션에 영향을 주지 않고, 반대도 마찬가지입니다. 단, REQUIRES_NEW는 새 DB 커넥션을 추가로 사용하므로 커넥션 풀 고갈에 주의해야 합니다.

📖 Part 5. readOnly = true — 왜 쓰는지, 어떤 최적화가 되는지, 주의사항
@Transactional(readOnly = true)는 무엇이고, 왜 쓰나요?

해당 트랜잭션이 읽기 전용임을 선언하는 힌트입니다. 쓰는 이유는 크게 세 가지입니다.

① Hibernate/JPA 레벨 최적화 — 영속성 컨텍스트에서 Dirty Checking(변경 감지)을 수행하지 않습니다. 엔티티의 스냅샷을 저장하지 않아 메모리가 절약되고, flush도 발생하지 않습니다. 조회 대상 엔티티가 많을수록 효과가 큽니다.

② DB 레벨 최적화 — JDBC 드라이버에 Connection.setReadOnly(true)를 호출하고, MySQL에서는 SET TRANSACTION READ ONLY를 실행합니다. InnoDB는 읽기 전용 트랜잭션에서 트랜잭션 ID 할당, 행에 TRX_ID 기록 등의 오버헤드를 생략할 수 있습니다.

③ Read/Write 분리 라우팅 — DataSource 라우팅 설정 시 readOnly=true인 트랜잭션을 Read Replica로 보내서 Primary 노드의 부하를 줄일 수 있습니다.

readOnly = true인데 쓰기(INSERT/UPDATE)를 하면 어떻게 되나요?

반드시 예외가 발생하는 것은 아닙니다. readOnly는 "힌트"일 뿐이며, 실제 동작은 JDBC 드라이버와 DB에 따라 다릅니다. MySQL의 경우 SET TRANSACTION READ ONLY 상태에서 DML을 실행하면 에러가 발생합니다. 하지만 Hibernate 레벨에서는 flush를 하지 않을 뿐 예외를 던지지 않을 수도 있어서, 변경이 조용히 무시되는 위험한 상황이 발생할 수 있습니다. 따라서 읽기 전용 메서드에만 정확히 사용해야 합니다.

readOnly = true의 주의사항이나 단점은?

① 불필요한 트랜잭션 오버헤드: 단순 단건 조회에 readOnly=true를 붙이면, 실제 DB에 SET TRANSACTION READ ONLY → SELECT → COMMIT 순으로 총 3번의 네트워크 왕복이 발생합니다. 트랜잭션 없이 autocommit으로 실행하면 1번이면 되므로, 단순 조회에는 오히려 성능이 떨어질 수 있습니다.

② DB 커넥션 점유: 트랜잭션이 열리면 커넥션을 점유합니다. 오래 걸리는 조회에 readOnly를 걸면 커넥션 풀이 고갈될 수 있습니다.

③ 전파 레벨 기본값 주의: readOnly=true의 기본 전파 레벨은 REQUIRED입니다. 상위에 readOnly=false 트랜잭션이 있으면 그 트랜잭션에 참여하게 되어 readOnly 설정이 무시됩니다.

🔌 Part 6. 트랜잭션과 DB 커넥션 — 커넥션 풀, 트랜잭션 생명주기와 커넥션의 관계
트랜잭션과 DB 커넥션은 어떤 관계인가요?

하나의 트랜잭션은 하나의 DB 커넥션을 점유합니다. Spring에서 @Transactional이 시작되면 TransactionManager가 커넥션 풀(HikariCP 등)에서 커넥션을 하나 가져와 ThreadLocal에 바인딩합니다. 트랜잭션이 커밋되거나 롤백될 때까지 이 커넥션은 반환되지 않습니다. 같은 트랜잭션 내의 모든 Repository 호출은 이 동일한 커넥션을 사용합니다.

트랜잭션이 길어지면 왜 문제인가요?

트랜잭션이 길면 그만큼 DB 커넥션을 오래 점유합니다. 커넥션 풀은 유한한 자원(보통 10~50개)이므로, 여러 트랜잭션이 동시에 오래 걸리면 다른 요청이 커넥션을 획득하지 못해 커넥션 풀 고갈(Connection Pool Starvation)이 발생합니다. 또한 InnoDB 관점에서는 오래 열린 트랜잭션 때문에 Undo Log가 정리되지 않아 테이블스페이스가 비대해지고, Lock 경합도 증가합니다.

트랜잭션 안에서 외부 API 호출을 하면 안 되는 이유는?

외부 API 호출은 네트워크 지연으로 수 초~수십 초가 걸릴 수 있습니다. 그 동안 DB 커넥션이 아무 일도 안 하면서 점유됩니다. 이는 커넥션 풀 고갈의 대표적 원인입니다. 해결 방법으로는 외부 API 호출을 트랜잭션 밖으로 빼거나, 전파 레벨을 NOT_SUPPORTED로 설정하여 API 호출 구간만 트랜잭션 없이 실행하는 방법이 있습니다.

REQUIRES_NEW가 커넥션 풀에 미치는 영향은?

REQUIRES_NEW는 기존 트랜잭션의 커넥션을 그대로 유지한 채 새로운 커넥션을 추가로 획득합니다. 즉 하나의 스레드가 동시에 2개의 커넥션을 점유하게 됩니다. 중첩이 깊어지면 그만큼 커넥션이 더 필요합니다. 동시 요청이 많으면 커넥션 풀이 빠르게 고갈될 수 있으므로, REQUIRES_NEW는 꼭 필요한 경우에만 사용해야 합니다.

⚡ Part 7. 실무 심화 질문 — 프록시 문제, 테스트, 분산 트랜잭션 등
@Transactional을 Service 레이어에 거는 이유는?

트랜잭션은 비즈니스 로직의 원자성을 보장하는 단위이므로, 비즈니스 로직이 있는 Service 레이어에 거는 것이 적절합니다. Controller에 걸면 HTTP 요청/응답 처리까지 트랜잭션에 포함되어 커넥션 점유가 길어지고, Repository에 걸면 여러 Repository 호출을 하나의 트랜잭션으로 묶을 수 없습니다.

클래스 레벨 @Transactional(readOnly = true) + 메서드 레벨 @Transactional 전략은?

실무에서 권장되는 패턴입니다. 클래스 레벨에 readOnly = true를 기본으로 선언하고, 쓰기가 필요한 메서드에만 @Transactional을 오버라이드합니다. 이렇게 하면 대부분의 조회 메서드가 자동으로 readOnly 최적화를 받고, 개발자가 실수로 readOnly를 빠뜨리는 것을 방지할 수 있습니다.

@Service
@Transactional(readOnly = true)  // 클래스 기본: 읽기 전용
public class UserService {

    public User getUser(Long id) { ... }         // readOnly = true 상속
    public List<User> getUsers() { ... }          // readOnly = true 상속

    @Transactional  // 오버라이드: readOnly = false
    public void createUser(UserDto dto) { ... }

    @Transactional
    public void updateUser(Long id, UserDto dto) { ... }
}
Spring 테스트에서 @Transactional을 쓰면 왜 자동 롤백되나요?

Spring TestContext 프레임워크는 테스트 메서드에 @Transactional이 있으면 테스트 완료 후 자동으로 롤백하도록 기본 설정되어 있습니다. 이는 각 테스트가 DB를 더럽히지 않아 테스트 간 격리를 보장하기 위함입니다. 롤백을 원하지 않으면 @Commit 또는 @Rollback(false)를 추가하면 됩니다. 하지만 테스트의 @Transactional 때문에 실제 서비스의 트랜잭션 전파가 가려질 수 있어 주의가 필요합니다.

분산 트랜잭션은 무엇이고, 왜 어렵나요?

여러 DB나 서비스에 걸쳐 하나의 트랜잭션을 보장해야 하는 상황입니다. 전통적으로 2PC(Two-Phase Commit) 프로토콜을 사용하지만, 네트워크 지연과 장애에 취약하고 성능 오버헤드가 큽니다. 마이크로서비스 환경에서는 각 서비스가 독립 DB를 가지므로 2PC가 현실적으로 어렵고, 대신 SAGA 패턴(각 서비스가 로컬 트랜잭션 + 보상 트랜잭션으로 일관성 유지)이나 이벤트 기반 최종 일관성(Eventual Consistency) 방식을 사용합니다.

낙관적 락(Optimistic Lock)과 비관적 락(Pessimistic Lock)의 차이는?

비관적 락은 데이터를 읽을 때 Lock을 걸어서 다른 트랜잭션의 접근을 원천 차단합니다. SELECT ... FOR UPDATE가 대표적이며, 충돌이 빈번한 환경에 적합하지만 동시성이 떨어집니다.

낙관적 락은 Lock을 걸지 않고, 커밋 시점에 버전(@Version)을 비교하여 충돌을 감지합니다. 충돌이 발생하면 OptimisticLockException을 던집니다. 충돌이 적은 환경에서 동시성이 높지만, 충돌 시 재시도 로직이 필요합니다. JPA에서는 엔티티에 @Version 필드를 추가하여 간단히 구현할 수 있습니다.

✦ 면접 한 줄 정리
트랜잭션 = All or Nothing. ACID 속성을 보장하는 논리적 작업 단위.
격리 수준 = 동시성 ↔ 일관성 트레이드오프. MySQL 기본은 REPEATABLE READ (MVCC + Next-Key Lock).
@Transactional = AOP 프록시 기반. 메서드 전후로 BEGIN/COMMIT/ROLLBACK을 자동 관리.
전파 레벨 = 기존 트랜잭션과의 관계. REQUIRED(합류), REQUIRES_NEW(독립), NOT_SUPPORTED(없이 실행) 등.
readOnly=true = Dirty Checking 생략(JPA) + DB 읽기 최적화 + Read Replica 라우팅. 단순 단건 조회에는 오히려 오버헤드.
트랜잭션 = DB 커넥션 점유. 길어지면 커넥션 풀 고갈. 외부 API 호출은 트랜잭션 밖으로!
