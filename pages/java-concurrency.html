Part 1 — 기본 동기화 · 동시성 문제의 근본 원인과 synchronized, volatile의 동작 원리를 이해합니다.
동시성 문제란 무엇이고, Race Condition이 왜 발생하나요?

동시성 문제(Concurrency Problem)는 여러 스레드가 공유 자원에 동시에 접근하면서 예상과 다른 결과가 발생하는 것입니다. 가장 대표적인 것이 Race Condition(경쟁 조건)입니다.

예를 들어 두 스레드가 count++를 동시에 실행한다고 가정합시다. count++는 실제로 세 단계 — 읽기 → 더하기 → 쓰기 — 로 나뉩니다. 스레드 A가 count를 0으로 읽은 직후, 스레드 B도 count를 0으로 읽고 1을 쓰면, 이후 스레드 A도 1을 씁니다. 두 번 증가했는데 결과는 1이 되는 것입니다.

// Race Condition 예시 (잘못된 코드)
class Counter {
    private int count = 0;

    public void increment() {
        count++; // 원자적 연산이 아님! read-modify-write 3단계
    }
}

동시성 문제는 크게 세 가지 원인에서 발생합니다. 첫째 원자성(Atomicity) 위반 — 읽고 쓰는 과정이 분리되어 중간에 다른 스레드가 끼어듦. 둘째 가시성(Visibility) 문제 — 한 스레드의 변경이 다른 스레드의 캐시에는 반영되지 않음. 셋째 순서(Ordering) 문제 — 컴파일러/CPU가 성능을 위해 명령어 순서를 재배치(reordering)함.

꼬리질문 대비

"스레드 안전(Thread-Safe)하다는 말의 의미는?" — 멀티스레드 환경에서 여러 스레드가 동시에 접근해도 항상 올바른 결과를 보장한다는 의미입니다. Immutable 객체, synchronized 메서드, Atomic 클래스, Concurrent 컬렉션 등으로 달성할 수 있습니다.

synchronized 키워드의 사용 방법과 동작 원리를 설명해주세요.

synchronized는 자바에서 가장 기본적인 동기화 메커니즘으로, 한 번에 하나의 스레드만 임계 구역(Critical Section)에 진입할 수 있도록 보장합니다.

// 1. 인스턴스 메서드 레벨 — this 객체의 모니터 락 사용
public synchronized void increment() {
    count++;
}

// 2. 블록 레벨 — 더 세밀한 제어, 락 범위 최소화
public void increment() {
    // 락 없이 실행 가능한 코드
    synchronized(this) {
        count++; // 락이 필요한 최소 범위만
    }
}

// 3. static 메서드 — Class 객체의 모니터 락 사용
public static synchronized void staticMethod() {
    // Counter.class 객체의 락
}

// 4. 별도 락 객체 사용 — 인스턴스 노출 방지
private final Object lock = new Object();
public void safeMethod() {
    synchronized(lock) {
        count++;
    }
}

동작 원리는 Monitor Lock(모니터 락) 입니다. 자바의 모든 객체는 하나의 모니터를 가지고 있습니다. synchronized 블록에 진입하려면 해당 객체의 모니터 락을 획득해야 하고, 블록을 나가면 락을 반환합니다. 락을 얻지 못한 스레드는 BLOCKED 상태로 대기합니다.

synchronized는 원자성과 가시성을 모두 보장합니다. 락을 획득할 때 CPU 캐시를 메인 메모리와 동기화하고, 락을 반환할 때 변경사항을 메인 메모리에 플러시합니다.

synchronized의 내부 동작 — Monitor Lock과 Object Header를 설명해주세요.

JVM은 성능을 위해 락을 단계적으로 운용합니다. Biased Locking(편향 락): 특정 스레드만 계속 접근한다면 그 스레드 ID를 Mark Word에 기록하고, CAS 없이 진입합니다. Thin Lock(경량 락): 경합이 발생하면 CAS(Compare-And-Swap) 연산으로 스택에 락 레코드를 생성합니다. Fat Lock(중량 락): 경합이 심해지면 OS의 Mutex(운영체제 뮤텍스)를 사용합니다. 이 단계에서 컨텍스트 스위칭이 발생하며 성능 비용이 큽니다.

synchronized를 쓸 때 과도하게 사용하면 불필요한 락 경합으로 성능 저하가 생깁니다. 락의 범위를 최소화하고, 필요 없는 곳에는 synchronized를 붙이지 않는 것이 중요합니다.

volatile 키워드란 무엇이고, 가시성(Visibility) 문제를 어떻게 해결하나요?

멀티코어 CPU에서는 각 코어가 L1/L2 캐시를 별도로 가집니다. 스레드 A가 변수를 변경해도 그 값이 캐시에만 있고 메인 메모리에 반영되지 않으면, 스레드 B는 오래된 값을 읽는 가시성(Visibility) 문제가 발생합니다.

// 가시성 문제 예시
class StopTask implements Runnable {
    private boolean running = true; // volatile 없음

    public void run() {
        while (running) { // 스레드B의 캐시에서 읽어 영원히 true일 수 있음!
            doWork();
        }
    }
    public void stop() { running = false; } // 스레드A가 변경
}

// volatile로 해결
private volatile boolean running = true;

volatile을 선언하면 두 가지가 보장됩니다. 첫째, 해당 변수에 대한 읽기/쓰기는 항상 메인 메모리를 직접 사용합니다 — CPU 캐시를 건너뜁니다. 둘째, Memory Barrier(메모리 장벽)를 삽입하여 volatile 변수 접근 전후의 명령어 순서 재배치를 방지합니다. volatile 쓰기는 이전 모든 연산이 완료된 후 실행되고, volatile 읽기는 이후 모든 연산보다 먼저 실행됩니다.

주의: volatile은 가시성만 보장하고 원자성은 보장하지 않습니다. volatile int count; count++;는 여전히 Race Condition이 있습니다. count++는 읽기-수정-쓰기가 세 단계이기 때문입니다.

happens-before 관계란 무엇인가요?

happens-before는 Java Memory Model(JMM)에서 정의하는 개념으로, "연산 A가 연산 B보다 happens-before 관계에 있다면, A의 결과가 B에서 반드시 보인다"는 것을 보장합니다. 프로그래머에게 동시성 보장을 추론할 수 있는 규칙을 제공합니다.

주요 happens-before 규칙은 다음과 같습니다.

// 1. 프로그램 순서 규칙: 같은 스레드 내 앞 연산 → 뒤 연산
int a = 1; // hb a를 읽는 코드

// 2. 모니터 락 규칙: unlock hb 동일 락의 lock
synchronized(obj) { x = 1; } // unlock
synchronized(obj) { int v = x; } // lock, x는 반드시 1

// 3. volatile 변수 규칙: volatile 쓰기 hb volatile 읽기
volatile boolean flag = false;
// 스레드A: flag = true;  // hb
// 스레드B: if (flag) ... // flag는 반드시 true

// 4. 스레드 시작/종료 규칙
// thread.start() hb 해당 스레드의 run()
// run() 내 모든 연산 hb thread.join() 이후

happens-before가 중요한 이유는 컴파일러와 CPU가 성능을 위해 명령어를 재배치(reordering)하는데, happens-before 관계가 있는 경우에는 이 재배치가 허용되지 않기 때문입니다. synchronized와 volatile은 모두 happens-before 관계를 형성합니다.

Part 2 — Lock & Atomic · ReentrantLock, ReadWriteLock, Atomic 클래스와 CAS 연산 원리를 이해합니다.
ReentrantLock vs synchronized — 언제 어떤 것을 써야 하나요?

ReentrantLock은 java.util.concurrent.locks 패키지의 명시적 락으로, synchronized보다 더 세밀한 제어가 가능합니다.

import java.util.concurrent.locks.*;

class SafeCounter {
    private final ReentrantLock lock = new ReentrantLock();
    private int count = 0;

    public void increment() {
        lock.lock(); // 명시적 락 획득
        try {
            count++;
        } finally {
            lock.unlock(); // 반드시 finally에서 해제!
        }
    }

    // 타임아웃 시도 — 일정 시간 내 락 획득 실패 시 포기
    public boolean tryIncrement() throws InterruptedException {
        if (lock.tryLock(100, TimeUnit.MILLISECONDS)) {
            try {
                count++;
                return true;
            } finally {
                lock.unlock();
            }
        }
        return false; // 100ms 안에 못 얻으면 false 반환
    }
}

일반적으로는 synchronized를 먼저 선택합니다. 코드가 간결하고 JVM이 자동으로 최적화(Biased Locking 등)를 해주기 때문입니다. ReentrantLock은 타임아웃 락, 공정 락, 여러 Condition이 필요한 복잡한 상황에서만 사용합니다.

꼬리질문 대비

"공정 락(Fair Lock)이란?" — new ReentrantLock(true)로 생성하면, 대기 중인 스레드 중 가장 오래 기다린 스레드에게 락을 줍니다. 기아(Starvation)를 방지하지만, 처리량(Throughput)이 낮아질 수 있습니다.

ReadWriteLock — 읽기/쓰기 분리 락을 설명해주세요.

일반 synchronized는 읽기와 쓰기를 구분하지 않아, 여러 스레드가 동시에 읽어도 직렬화됩니다. ReadWriteLock은 "읽기끼리는 동시에 가능하고, 쓰기는 단독으로"라는 원칙으로 성능을 높입니다.

class Cache {
    private final ReadWriteLock rwLock = new ReentrantReadWriteLock();
    private final Lock readLock  = rwLock.readLock();
    private final Lock writeLock = rwLock.writeLock();
    private Map<String, String> data = new HashMap<>();

    public String get(String key) {
        readLock.lock(); // 여러 스레드 동시 읽기 가능
        try {
            return data.get(key);
        } finally { readLock.unlock(); }
    }

    public void put(String key, String val) {
        writeLock.lock(); // 쓰기 시 단독 락
        try {
            data.put(key, val);
        } finally { writeLock.unlock(); }
    }
}
상황readLockwriteLock
readLock 보유 중다른 스레드 readLock 획득 가능다른 스레드 writeLock 불가
writeLock 보유 중다른 스레드 readLock 불가다른 스레드 writeLock 불가

읽기가 압도적으로 많은 상황(캐시, 설정 조회 등)에서 효과적입니다. 쓰기 빈도가 높으면 오히려 오버헤드가 될 수 있습니다.

StampedLock이란 무엇이고, ReadWriteLock과 어떻게 다른가요?

StampedLock은 Java 8에서 도입된 더 발전된 형태의 읽기-쓰기 락으로, 낙관적 읽기(Optimistic Read)를 지원합니다.

class Point {
    private final StampedLock sl = new StampedLock();
    private double x, y;

    public double distanceFromOrigin() {
        // 낙관적 읽기: 락을 걸지 않고 stamp만 받음
        long stamp = sl.tryOptimisticRead();
        double curX = x, curY = y;

        // stamp 유효성 확인: 그 사이 쓰기가 있었다면 false
        if (!sl.validate(stamp)) {
            stamp = sl.readLock(); // 쓰기가 있었으면 실제 읽기 락으로 재시도
            try {
                curX = x; curY = y;
            } finally {
                sl.unlockRead(stamp);
            }
        }
        return Math.sqrt(curX * curX + curY * curY);
    }
}

낙관적 읽기는 실제로 락을 획득하지 않고 읽은 뒤, validate()로 그 사이에 쓰기가 없었는지 확인합니다. 쓰기가 없었다면 락 비용 없이 읽기 성공입니다. 단, StampedLock은 재진입(Reentrant)이 불가하고, Condition을 지원하지 않으므로 단순 읽기 최적화 목적으로만 사용해야 합니다.

Atomic 클래스와 CAS(Compare-And-Swap) 연산 원리를 설명해주세요.

Atomic 클래스(AtomicInteger, AtomicLong, AtomicReference 등)는 락 없이 스레드 안전한 연산을 제공합니다. 내부적으로 CAS(Compare-And-Swap) 명령어를 사용합니다.

import java.util.concurrent.atomic.*;

AtomicInteger counter = new AtomicInteger(0);

counter.incrementAndGet();           // ++counter (원자적)
counter.addAndGet(5);               // counter += 5 (원자적)
counter.compareAndSet(10, 20);      // 값이 10이면 20으로 변경

// AtomicReference — 객체 참조를 원자적으로 교체
AtomicReference<String> ref = new AtomicReference<>("old");
ref.compareAndSet("old", "new"); // old이면 new로

CAS 원리: CAS는 CPU 레벨의 원자적 명령어입니다. "현재 메모리의 값이 내가 예상한 값(expected)과 같으면 새 값(update)으로 바꾸고 true를 반환, 다르면 false 반환"합니다. 이를 하드웨어가 보장하므로 OS 레벨의 락이 필요 없습니다.

// CAS의 의사 코드 (실제로는 CPU 명령어 하나로 원자적 실행)
boolean compareAndSwap(int* addr, int expected, int update) {
    if (*addr == expected) {
        *addr = update;
        return true;
    }
    return false;
}

Atomic 클래스의 장점은 락 없이도 스레드 안전하여 synchronized보다 경합이 적을 때 성능이 훨씬 좋습니다. 단, 경합이 극심하면 CAS 실패 후 재시도(Spin) 비용이 커질 수 있습니다.

ABA 문제(ABA Problem)란 무엇인가요?

ABA 문제는 CAS 연산의 잠재적 결함으로, 값이 A → B → A로 변했을 때 CAS는 여전히 A라고 보고 성공을 반환하는 문제입니다.

예시 시나리오: 스레드 T1이 값 A를 읽고 잠시 멈춥니다. 그 사이 스레드 T2가 A→B→A로 두 번 변경합니다. T1이 깨어나 CAS(expected=A, update=C)를 실행하면 현재 값이 A이므로 성공합니다. 하지만 T1은 중간에 변경이 있었음을 모릅니다. 연결 리스트 같은 구조에서 ABA가 발생하면 Node를 잘못 재사용하여 데이터가 손상될 수 있습니다.

// 해결책: AtomicStampedReference — 값 + 버전 번호(stamp)를 함께 관리
AtomicStampedReference<Integer> ref =
    new AtomicStampedReference<>(100, 0); // 초기값, 초기 stamp

int[] stampHolder = new int[1];
Integer current = ref.get(stampHolder); // 값과 stamp 함께 읽기
int stamp = stampHolder[0];

// stamp도 같아야만 성공 — A→B→A라도 stamp가 달라서 실패
ref.compareAndSet(current, 200, stamp, stamp + 1);

실무에서 ABA 문제가 발생할 가능성이 있는 자료구조를 직접 구현할 때는 AtomicStampedReference 또는 AtomicMarkableReference를 사용합니다. 대부분의 경우 표준 라이브러리의 Concurrent 컬렉션이 이런 문제를 내부적으로 처리하므로, 직접 CAS 기반 알고리즘을 구현하는 경우에만 주의하면 됩니다.

Part 3 — Thread Pool & Executor · 스레드를 직접 생성하면 안 되는 이유와 ThreadPoolExecutor, CompletableFuture를 이해합니다.
Thread를 직접 생성하면 안 되는 이유는 무엇인가요?

스레드는 생성 비용이 비쌉니다. OS 커널 레벨에서 스택 메모리(기본 수백KB~1MB)를 할당하고, 커널 스레드와 매핑하는 시스템 콜이 필요합니다. 요청마다 스레드를 새로 만들면 다음 문제가 생깁니다.

// 절대 이렇게 하면 안 됨!
void handleRequest(Request req) {
    new Thread(() -> process(req)).start(); // 요청마다 스레드 생성 → OOM 위험
}

문제점 세 가지: 첫째 생성/소멸 오버헤드 — 스레드 생성은 무겁고, 즉시 소멸하면 낭비입니다. 둘째 무제한 스레드 생성 — 트래픽이 폭증하면 수천 개의 스레드가 생성되어 OOM(Out-of-Memory) 또는 컨텍스트 스위칭 폭발로 서버가 다운됩니다. 셋째 스레드 수 제어 불가 — 시스템 자원을 예측할 수 없게 사용합니다.

Thread Pool은 미리 생성한 스레드를 재사용하여 이 문제를 해결합니다. 스레드를 빌렸다가 반납하는 풀장과 같습니다.

ThreadPoolExecutor의 구조와 동작 방식을 설명해주세요.

ThreadPoolExecutor는 Java 스레드 풀의 핵심 클래스입니다. 주요 파라미터를 이해하는 것이 중요합니다.

new ThreadPoolExecutor(

    50,                         // maximumPoolSize: 최대 스레드 수

    new LinkedBlockingQueue<>(100), // workQueue: 작업 대기 큐
    new ThreadPoolExecutor.CallerRunsPolicy() // 거부 정책

Executors 클래스는 세 가지 편의 메서드를 제공합니다.

// 고정 스레드 수 — corePoolSize = max = N, 큐 = 무제한 LinkedBlockingQueue
ExecutorService fixed = Executors.newFixedThreadPool(10);

ExecutorService single = Executors.newSingleThreadExecutor();
OOM 위험: newFixedThreadPool과 newSingleThreadExecutor는 workQueue로 용량 제한이 없는 LinkedBlockingQueue를 사용합니다. 작업 처리보다 적재가 빠르면 큐가 무한 증가하여 OOM이 발생합니다. newCachedThreadPool은 max 스레드 수가 Integer.MAX_VALUE여서 요청 폭증 시 수만 개의 스레드가 생성되어 OOM이 발생합니다.

실무에서는 반드시 ThreadPoolExecutor를 직접 생성하여 큐 용량과 최대 스레드 수를 명시적으로 설정해야 합니다. Naver, Kakao 등 국내 대형 서비스에서도 이 원칙을 강조합니다.

CompletableFuture의 사용법과 주요 메서드를 설명해주세요.

CompletableFuture는 Java 8에서 도입된 비동기 프로그래밍 API로, Future의 단점(블로킹 get, 콜백 불가)을 해결합니다.

// 비동기 작업 실행
CompletableFuture<String> future =
    CompletableFuture.supplyAsync(() -> fetchUserFromDB(1L));

// 결과 변환 (thenApply — 동기적으로 변환)
CompletableFuture<String> upper = future.thenApply(String::toUpperCase);

// 다른 비동기 작업 연결 (thenCompose — flatMap과 유사)
CompletableFuture<Order> order = future
    .thenCompose(userId -> fetchOrderAsync(userId));

// 두 작업 병렬 실행 후 결합
CompletableFuture<User> userFuture  = CompletableFuture.supplyAsync(this::getUser);
CompletableFuture<Order> orderFuture = CompletableFuture.supplyAsync(this::getOrder);

CompletableFuture<String> combined = userFuture.thenCombine(
    orderFuture,
    (user, order) -> user.name() + " / " + order.id()
);

// 예외 처리
future.exceptionally(ex -> "default");
future.handle((result, ex) -> ex != null ? "error" : result);

// 여러 Future 모두 완료 대기
CompletableFuture.allOf(userFuture, orderFuture).join();

기본적으로 supplyAsync는 ForkJoinPool.commonPool()을 사용합니다. 실무에서는 별도의 스레드 풀을 명시적으로 전달하여 다른 작업과 풀을 공유하지 않도록 해야 합니다.

Spring의 @Async와 ThreadPool 설정 방법을 설명해주세요.

@Async는 메서드를 비동기로 실행하게 만드는 Spring 어노테이션입니다. 내부적으로 AOP 프록시가 메서드 호출을 가로채 스레드 풀에 작업을 위임합니다.

@Configuration
@EnableAsync // 메인 클래스나 설정 클래스에 추가 필수
public class AsyncConfig implements AsyncConfigurer {

    @Override
    public Executor getAsyncExecutor() {
        ThreadPoolTaskExecutor executor = new ThreadPoolTaskExecutor();
        executor.setCorePoolSize(10);
        executor.setMaxPoolSize(50);
        executor.setQueueCapacity(200); // 반드시 용량 제한!
        executor.setThreadNamePrefix("async-");
        executor.setRejectedExecutionHandler(
            new ThreadPoolExecutor.CallerRunsPolicy()
        );
        executor.initialize();
        return executor;
    }
}

@Service
public class EmailService {

    @Async
    public CompletableFuture<String> sendEmail(String to) {
        // 비동기로 실행됨
        sendActualEmail(to);
        return CompletableFuture.completedFuture("sent");
    }
}
주의: @Async는 같은 클래스 내에서 호출하면 동작하지 않습니다(Self-Invocation). AOP 프록시를 거쳐야 하므로, 반드시 다른 빈에서 호출해야 합니다. 또한 @Async 메서드가 예외를 던지면 호출자가 알 수 없으므로, 반드시 AsyncUncaughtExceptionHandler나 CompletableFuture로 예외 처리를 해야 합니다.

ConcurrentHashMap의 내부 구조(Java 8 이후)를 설명해주세요.

Java 8 이전의 ConcurrentHashMap은 Segment 단위로 락을 분할했지만, Java 8부터는 더 세밀한 방식으로 바뀌었습니다. 핵심 아이디어는 버킷(배열 인덱스) 단위로 락을 거는 것입니다.

ConcurrentHashMap — 버킷별 synchronized 락
// Java 8 ConcurrentHashMap put 핵심 로직 (단순화)
final V putVal(K key, V value, boolean onlyIfAbsent) {
    int hash = spread(key.hashCode());
    for (Node<K,V>[] tab = table;;) {
        Node<K,V> f; int n, i;
        if (tab == null) { tab = initTable(); } // 초기화는 CAS 사용
        else if ((f = tabAt(tab, i = (n-1) & hash)) == null) {
            // 버킷이 비어있으면 CAS로 Node 삽입 (락 없이)
            if (casTabAt(tab, i, null, new Node<>(hash,key,value)))
                break;
        } else {
            // 버킷에 기존 Node가 있으면 해당 버킷의 head에 synchronized
            synchronized (f) { // 이 버킷에만 락 — 다른 버킷은 자유
                // 연결리스트 또는 트리 탐색 및 삽입
            }
        }
    }
}

핵심: 빈 버킷에 첫 삽입은 CAS로 락 없이, 충돌이 있는 버킷은 해당 버킷 head 노드에만 synchronized. 서로 다른 버킷에 대한 쓰기는 동시에 가능합니다. 또한 연결리스트 길이가 8을 초과하면 Red-Black Tree로 변환하여 조회 성능을 O(n)에서 O(log n)으로 향상합니다.

"ConcurrentHashMap에서 size()의 정확도는?" — size()는 정확한 값을 보장하지 않습니다. 여러 버킷의 카운터를 합산하는 시점에 다른 스레드가 수정할 수 있기 때문입니다. 정확한 크기가 필요하다면 별도의 AtomicInteger로 관리하거나, LongAdder를 사용하는 것이 권장됩니다.

CopyOnWriteArrayList란 무엇이고, 언제 사용하나요?

CopyOnWriteArrayList는 쓰기 시 내부 배열 전체를 복사(Copy-On-Write)하여 스레드 안전성을 확보하는 리스트입니다.

// 내부 동작 단순화
public boolean add(E e) {
    synchronized(lock) {
        Object[] elements = getArray();
        Object[] newElements = Arrays.copyOf(elements, elements.length + 1);
        newElements[elements.length] = e;
        setArray(newElements); // 새 배열로 원자적 교체 (volatile 참조)
        return true;
    }
}

// 읽기는 락 없이 (현재 참조하는 배열 스냅샷을 읽음)
public E get(int index) {
    return elementAt(getArray(), index); // 락 없음!
}

장점: 읽기가 락 없이 빠릅니다. 이터레이션 중 ConcurrentModificationException이 발생하지 않습니다(이터레이터가 쓰기 당시의 스냅샷을 봅니다). 단점: 쓰기 시 배열 전체 복사 비용이 큽니다 — O(n). 쓰기가 빈번하면 GC 압박이 심해집니다.

적합한 상황: 이벤트 리스너 목록, 설정 목록처럼 읽기가 압도적으로 많고 쓰기가 드문 경우에 사용합니다.

BlockingQueue — LinkedBlockingQueue와 ArrayBlockingQueue의 차이를 설명해주세요.

BlockingQueue는 큐가 비어있으면 꺼내려는 스레드가 블로킹되고, 가득 차면 넣으려는 스레드가 블로킹되는 스레드 안전 큐입니다. 생산자-소비자 패턴에서 핵심 자료구조입니다.

BlockingQueue<Task> queue = new LinkedBlockingQueue<>(1000);

// 생산자 스레드
queue.put(task);     // 꽉 차면 블로킹
queue.offer(task, 500, TimeUnit.MILLISECONDS); // 타임아웃 후 실패 반환

// 소비자 스레드
Task t = queue.take();   // 비어있으면 블로킹
Task t2 = queue.poll(1, TimeUnit.SECONDS); // 타임아웃 후 null 반환

ThreadLocal이란 무엇이고, 어떤 주의점이 있나요?

ThreadLocal은 스레드마다 독립적인 변수 사본을 제공합니다. 각 스레드가 자신만의 독립 공간에 값을 저장하므로 동기화 없이 스레드 안전합니다.

public class UserContext {
    private static final ThreadLocal<User> currentUser =
        new ThreadLocal<>();

    public static void setUser(User user) { currentUser.set(user); }
    public static User getUser() { return currentUser.get(); }
    public static void clear() { currentUser.remove(); } // 반드시 호출!
}

// Spring Security의 SecurityContextHolder도 ThreadLocal 사용
// JPA의 EntityManager(트랜잭션 컨텍스트)도 ThreadLocal로 관리

내부 구조: 각 스레드(Thread 객체)는 ThreadLocalMap을 필드로 가집니다. ThreadLocal을 키로, 저장된 값을 value로 가지는 맵입니다. ThreadLocal.get()은 현재 스레드의 ThreadLocalMap을 조회합니다.

주의점: 스레드 풀 환경에서 반드시 remove()를 호출해야 합니다. 스레드 풀에서는 스레드가 재사용되므로, 이전 요청의 데이터가 다음 요청에서 보일 수 있습니다(메모리 누수 및 데이터 오염). ThreadLocalMap은 ThreadLocal 키를 WeakReference로 저장하지만 값(value)은 강참조여서, ThreadLocal 인스턴스가 GC되어도 값은 남아 메모리 누수가 됩니다. Spring의 인터셉터나 필터의 afterCompletion에서 remove()를 호출하는 것이 모범 사례입니다.
synchronized = Monitor Lock — 원자성 + 가시성 보장. 범위 최소화가 핵심
volatile = 가시성만 보장 (원자성 없음). 단순 플래그, DCL 싱글톤에만 사용
