== 연산자와 equals() 메서드의 차이는 무엇인가요?

==는 참조(주소) 동일성을 비교하고, equals()는 값(내용) 동등성을 비교합니다. 기본형(primitive)에서 ==는 값 자체를 비교하지만, 참조형(Reference Type)에서는 두 변수가 힙의 동일한 객체를 가리키는지를 비교합니다.

String a = new String("hello");
String b = new String("hello");

a == b;         // false  → 힙의 서로 다른 객체를 가리킴
a.equals(b);    // true   → 문자열 내용이 같음

String c = "hello";
String d = "hello";
c == d;         // true   → String Pool의 같은 객체 (intern)

Object의 기본 equals() 구현은 ==와 동일하게 참조를 비교합니다. 따라서 값 동등성이 필요한 클래스는 반드시 equals()를 재정의해야 합니다. String, Integer, LocalDate 등 Java 표준 라이브러리 클래스들은 이미 재정의되어 있습니다.

꼬리질문 대비: "null에 대해 equals()를 호출하면?" — a.equals(null)은 false를 반환해야 하며(NPE 없이), null.equals(a)는 NPE가 발생합니다. Objects.equals(a, b)를 사용하면 null 안전하게 비교할 수 있습니다.
equals()를 재정의해야 하는 이유와 올바른 구현 조건은 무엇인가요?

비즈니스 객체는 같은 내용을 가진 두 인스턴스를 "같다"고 판단해야 하는 경우가 많습니다. 예를 들어 User(id=1)이 두 번 조회되어 다른 객체로 생성되었더라도, ID가 같으면 논리적으로 같은 사용자입니다. Object의 기본 equals()는 참조를 비교하므로 이를 값 기반으로 재정의해야 합니다.

equals() 계약(Contract) — 반드시 지켜야 할 5가지 속성:

속성의미
반사성 (Reflexive)x.equals(x)는 항상 true
대칭성 (Symmetric)x.equals(y)이면 y.equals(x)도 true
추이성 (Transitive)x.equals(y)이고 y.equals(z)이면 x.equals(z)도 true
일관성 (Consistent)객체가 변경되지 않으면 여러 번 호출해도 항상 같은 결과
null 비교x.equals(null)은 항상 false (NPE 발생 금지)
public class User {
    private final long id;
    private final String name;

    @Override
    public boolean equals(Object o) {
        if (this == o) return true;           // 1. 같은 참조
        if (!(o instanceof User)) return false; // 2. 타입 확인 (null도 처리)
        User user = (User) o;
        return id == user.id;                   // 3. 핵심 필드만 비교
    }
}
주의: 상속 관계에서 equals() 재정의는 추이성(Transitivity)을 깨뜨릴 수 있습니다. 조슈아 블로크(Effective Java)는 상속 대신 컴포지션(Composition)을 사용하라고 권장합니다.
hashCode()를 equals()와 함께 재정의해야 하는 이유는 무엇인가요?

Java 명세에 따르면 equals()로 같다고 판단된 두 객체는 반드시 같은 hashCode()를 가져야 합니다. 이것이 equals-hashCode 계약입니다. 이 계약을 지키지 않으면 HashMap, HashSet 같은 해시 기반 컬렉션이 오동작합니다.

// hashCode()를 재정의하지 않은 경우
User u1 = new User(1L, "Alice");
User u2 = new User(1L, "Alice");

u1.equals(u2);  // true (재정의했으므로)

Set<User> set = new HashSet<>();
set.add(u1);
set.contains(u2);  // false!! → hashCode가 달라 다른 버킷을 탐색

Map<User, String> map = new HashMap<>();
map.put(u1, "value");
map.get(u2);  // null!! → 같은 키로 조회해도 찾지 못함

올바른 hashCode() 구현:

@Override
public int hashCode() {
    return Objects.hash(id);  // equals에서 사용한 필드만 포함
}

// 또는 직접 구현 (31을 곱하는 이유: 홀수 소수, XOR 연산 최적화)
@Override
public int hashCode() {
    int result = Long.hashCode(id);
    result = 31 * result + (name != null ? name.hashCode() : 0);
    return result;
}
equals-hashCode 계약 요약:
- equals()가 true이면 → hashCode()는 반드시 같아야 함
- hashCode()가 같다고 해서 → equals()가 true일 필요는 없음 (해시 충돌 허용)
- 즉, equals() true → hashCode() 같음은 필수. 역방향은 불필요.
꼬리질문 대비: "hashCode()가 항상 같은 값(예: 1)을 반환하면 어떻게 되나요?" — 계약은 지켜지지만 성능이 최악입니다. HashMap이 O(1)에서 O(n)으로 저하됩니다. 모든 객체가 하나의 버킷에 쌓여 연결 리스트(또는 트리)로 탐색하게 되기 때문입니다.
HashMap에서 equals()와 hashCode()가 어떻게 동작하는지 설명해주세요.

HashMap은 키-값 쌍을 버킷(bucket) 배열에 저장합니다. 조회 시 두 단계를 거칩니다.

put/get 동작 순서:

  1. 1키의 hashCode()를 호출해 해시값을 얻는다
  2. 2해시값을 버킷 배열 인덱스로 변환 (hash ^ (hash >>> 16)) & (n-1)
  3. 3해당 버킷에서 equals()로 키를 하나씩 비교해 정확한 항목을 찾는다
결론: hashCode()가 다르면 equals() 비교 자체가 일어나지 않습니다 (다른 버킷을 탐색). hashCode()가 같으면 그때 equals()로 정밀 비교합니다. 그래서 "hashCode가 같으면 equals도 같아야 한다"는 계약이 필수입니다.
String이 불변(immutable)인 이유는 무엇인가요?

Java의 String은 한 번 생성되면 내용을 변경할 수 없는 불변 객체입니다. String에 concat()이나 + 연산을 하면 기존 객체를 수정하는 게 아니라 새로운 String 객체가 생성됩니다.

불변으로 설계된 이유는 크게 4가지입니다.

이유설명
1. String Pool 공유 가능 불변이기 때문에 여러 변수가 같은 객체를 안전하게 참조할 수 있음. 변경 가능하면 한 쪽에서 수정 시 다른 참조자에게 영향을 미침
2. 스레드 안전(Thread-Safe) 불변 객체는 상태가 변하지 않아 동기화 없이 여러 스레드가 안전하게 공유 가능
3. 해시코드 캐싱 hashCode()를 최초 계산 후 캐싱 가능. HashMap의 키로 자주 사용되는 String 성능에 중요
4. 보안 클래스 로딩, 네트워크 연결 등에 String이 파라미터로 사용될 때 변경 불가능하면 보안 위협 차단
꼬리질문 대비: "Reflection으로 String 내부 char 배열을 바꿀 수 있지 않나요?" — 기술적으로 가능하지만, 이는 불변성 계약을 위반하는 행위로 String Pool을 오염시켜 예측 불가능한 버그를 유발합니다. 실무에서 절대로 해서는 안 됩니다. Java 9부터 String 내부가 byte[]로 변경(Compact Strings)되어 메모리가 더 효율적으로 됐습니다.
String Pool(Intern Pool)이란 무엇인가요?

String Pool은 JVM이 힙 영역 내 특별 영역(Java 7+ 기준 힙 내부)에 문자열 리터럴을 캐싱해두는 공간입니다. 같은 문자열 리터럴이 여러 번 사용될 때 동일한 객체를 재사용해 메모리를 절약합니다.

String a = "hello";         // Pool에 "hello" 저장
String b = "hello";         // Pool에서 같은 객체 재사용
String c = new String("hello"); // 힙에 새 객체 생성 (Pool 사용 안 함)
String d = c.intern();     // Pool에서 "hello" 반환 → a와 같은 객체

a == b;  // true  (같은 Pool 객체)
a == c;  // false (c는 힙에 별도 생성)
a == d;  // true  (intern()으로 Pool 객체 반환)
Java 6 vs Java 7+ 차이: Java 6까지 String Pool은 Permanent Generation(PermGen)에 위치해 크기가 고정되어 OutOfMemoryError가 발생하기 쉬웠습니다. Java 7부터 힙(Heap)으로 이동해 GC 대상이 되고 크기 제한이 유연해졌습니다.
String vs StringBuilder vs StringBuffer의 차이는 무엇인가요?

세 클래스 모두 문자열을 다루지만 가변성과 스레드 안전성에서 차이가 있습니다.

클래스가변성스레드 안전성능사용 시기
String 불변 (Immutable) 안전 반복 연결 시 느림 문자열이 변경 없이 사용될 때
StringBuilder 가변 (Mutable) 안전하지 않음 가장 빠름 단일 스레드에서 문자열 반복 조합
StringBuffer 가변 (Mutable) 안전 (synchronized) 중간 멀티스레드에서 공유 문자열 조합 
// BAD: 루프에서 String + 연산 → 매 반복마다 새 객체 생성 O(n²)
String result = "";
for (int i = 0; i < 10000; i++) {
    result += i;  // 매번 새로운 String 객체 생성!
}

// GOOD: StringBuilder 사용 → O(n)
StringBuilder sb = new StringBuilder();
for (int i = 0; i < 10000; i++) {
    sb.append(i);  // 내부 버퍼에 추가, 새 객체 생성 없음
}
String result = sb.toString();
Java 컴파일러 최적화: 단순한 "a" + "b" + "c"는 컴파일러가 자동으로 StringBuilder로 변환합니다. 하지만 루프 내 문자열 연결은 매 반복마다 새 StringBuilder를 생성하므로, 루프 밖에서 명시적으로 StringBuilder를 선언해야 합니다.
꼬리질문 대비: "실무에서 StringBuffer를 언제 쓰나요?" — 거의 쓰지 않습니다. 멀티스레드 환경에서 문자열을 공유해야 한다면 StringBuffer보다 스레드 로컬(ThreadLocal)이나 동기화 설계를 재검토하는 것이 나을 때가 많습니다.
new String("abc")와 "abc"의 차이, 그리고 String의 hashCode 캐싱을 설명해주세요.

"abc"는 String Pool에서 객체를 찾아 반환하고, new String("abc")는 힙에 무조건 새 객체를 생성합니다. Pool에 있든 없든 관계없이 별도 객체를 만들기 때문에 불필요한 메모리 낭비가 생깁니다.

String s1 = "abc";                  // Pool 사용
String s2 = new String("abc");     // 힙에 새 객체
String s3 = s2.intern();           // Pool에서 "abc" 반환

s1 == s2;       // false
s1 == s3;       // true
s1.equals(s2);  // true (내용은 같음)

String의 hashCode 캐싱:

String은 내부에 private int hash; 필드를 가지며, 최초 hashCode() 호출 시 계산한 값을 이 필드에 저장합니다. 이후 호출에서는 저장된 값을 바로 반환합니다. 불변 객체이기 때문에 값이 변하지 않으므로 이 캐싱이 안전합니다.

// String.java 내부 (간략화)
public int hashCode() {
    int h = hash;
    if (h == 0 && !hashIsZero) {
        h = isLatin1() ? StringLatin1.hashCode(value)
                       : StringUTF16.hashCode(value);
        if (h == 0) hashIsZero = true;
        else hash = h;  // 캐싱
    }
    return h;
}
HashMap의 키로 String이 선호되는 이유: 불변성(키가 변경되지 않음) + hashCode 캐싱(매 조회마다 재계산 불필요)의 조합으로 HashMap에서 최고의 성능을 냅니다.
Generic이란 무엇이고, 왜 사용하나요?

Generic은 클래스, 인터페이스, 메서드를 정의할 때 타입을 파라미터로 지정하는 기능입니다. Java 5에서 도입되었으며, 컴파일 시점에 타입을 검사해 런타임 타입 오류를 방지합니다.

Generic 도입 전에는 컬렉션에서 꺼낼 때마다 형변환(Casting)이 필요하고, 잘못된 타입을 넣어도 컴파일러가 잡아주지 못해 런타임에 ClassCastException이 발생했습니다.

// Generic 없는 코드 (Java 5 이전)
List list = new ArrayList();
list.add("hello");
list.add(42);                      // 컴파일 통과! 런타임에 문제
String s = (String) list.get(1);  // ClassCastException!

// Generic 사용 (타입 안전)
List<String> list = new ArrayList<>();
list.add("hello");
list.add(42);         // 컴파일 오류! → 실수를 즉시 발견
String s = list.get(0); // 형변환 불필요

Generic의 3가지 장점: 타입 안전성(컴파일 시 오류 감지), 코드 재사용성(하나의 클래스로 다양한 타입 처리), 가독성(형변환 코드 제거로 깔끔해짐).

타입 소거(Type Erasure)란 무엇인가요?

타입 소거는 Java 컴파일러가 Generic 타입 정보를 컴파일 시에만 사용하고, 바이트코드에서는 제거(소거)하는 것입니다. 런타임에는 Generic 타입 정보가 존재하지 않습니다. Java 5 이전 코드와의 하위 호환성을 위해 이 방식을 선택했습니다.

// 컴파일 전 (소스 코드)
List<String> strings = new ArrayList<String>();
strings.add("hello");
String s = strings.get(0);

// 컴파일 후 바이트코드 (타입 소거됨)
List strings = new ArrayList();  // <String> 사라짐
strings.add("hello");
String s = (String) strings.get(0); // 컴파일러가 자동 형변환 삽입

타입 소거로 인해 발생하는 제약사항:

제약사항이유대안
new T() 불가런타임에 T가 뭔지 모름팩토리 메서드, Class<T> 파라미터 전달
T[] arr 생성 불가배열은 런타임 타입 검사를 함List<T> 사용
o instanceof T 불가런타임에 T 정보 없음Class<T>.isInstance(o) 사용
오버로딩 제약소거 후 시그니처가 같아짐메서드 이름 다르게 지정
꼬리질문 대비: "런타임에 Generic 타입을 알아내는 방법이 있나요?" — Reified Generics 패턴을 사용합니다. 익명 클래스를 이용해 슈퍼타입에서 타입 파라미터를 읽어낼 수 있습니다 (Jackson의 TypeReference, Spring의 ParameterizedTypeReference가 이 방식 사용). Kotlin의 reified 키워드는 인라인 함수에서 타입 소거 없이 타입을 다룰 수 있습니다.
와일드카드(?, extends, super)와 PECS 원칙을 설명해주세요.

와일드카드는 Generic 타입 파라미터에 유연성을 주기 위한 문법입니다. ?는 "알 수 없는 타입"을 의미합니다.

문법의미가능불가능
List<?> 어떤 타입이든 OK (비한정 와일드카드) 읽기 (Object로) null 외 쓰기
List<? extends T> T 또는 T의 하위 타입 (상한 경계) 읽기 (T로) 쓰기
List<? super T> T 또는 T의 상위 타입 (하한 경계) 쓰기 (T 추가) 타입 안전 읽기

PECS (Producer Extends, Consumer Super)

Joshua Bloch가 정리한 원칙으로, 컬렉션의 역할에 따라 와일드카드를 결정합니다.

// Producer: 데이터를 꺼내서(생산) 사용 → extends
public void printAll(List<? extends Number> list) {
    for (Number n : list) System.out.println(n); // 읽기 OK
    // list.add(1.0);  → 컴파일 오류 (어떤 서브타입인지 모르므로)
}

// Consumer: 데이터를 받아서(소비) 저장 → super
public void addNumbers(List<? super Integer> list) {
    list.add(1);   // 쓰기 OK (Integer는 어떤 상위 타입에도 들어갈 수 있음)
    list.add(2);
}

// Collections.copy의 실제 시그니처
public static <T> void copy(List<? super T> dest,    // Consumer (super)
                            List<? extends T> src) { // Producer (extends)
    for (T t : src) dest.add(t);
}
PECS 암기법: 데이터를 꺼내 생산(produce)하면 extends, 데이터를 받아 소비(consume)하면 super. 읽기만 하면 extends, 쓰기만 하면 super, 둘 다 하면 구체적 타입 파라미터 T.
Generic의 제한사항 — primitive 불가, instanceof 불가 등을 설명해주세요.

타입 소거와 Java 타입 시스템의 특성상 Generic에는 여러 제한사항이 있습니다.

// 1. 기본형(primitive) 타입 파라미터 불가
List<int> list = new ArrayList<>(); // 컴파일 오류!
List<Integer> list = new ArrayList<>(); // OK (오토박싱 활용)
// 이유: Generic은 Object 기반이고 int는 Object가 아님
// Java 10+ Valhalla 프로젝트에서 해결 예정

// 2. instanceof 검사 불가
if (obj instanceof List<String>) { } // 컴파일 오류!
if (obj instanceof List<?>) { }    // OK (비한정 와일드카드)

// 3. Generic 타입의 배열 생성 불가
T[] arr = new T[10]; // 컴파일 오류!
// 이유: 배열은 공변(covariant)이고 런타임 타입 검사를 하는데
// 타입 소거로 T가 없어지면 타입 안전성을 보장 못 함
@SuppressWarnings("unchecked")
T[] arr = (T[]) new Object[10]; // 우회 (비권장)

// 4. Static 멤버에 타입 파라미터 사용 불가
class Box<T> {
    private static T instance; // 컴파일 오류!
    // static은 클래스 수준이지만 T는 인스턴스 수준
}

// 5. Generic 예외 클래스 불가
class MyException<T> extends Exception { } // 컴파일 오류!
// catch 블록에서 타입 소거로 구별 불가
꼬리질문 대비: "Generic 메서드와 Generic 클래스의 차이는?" — Generic 클래스는 클래스 전체에서 타입 파라미터를 공유합니다(class Box<T>). Generic 메서드는 해당 메서드에서만 독립적인 타입 파라미터를 선언합니다(public <T> T identity(T t)). static 메서드는 Generic 클래스의 T를 쓸 수 없으므로 Generic 메서드로 별도 선언해야 합니다.
불변 객체(Immutable Object)란 무엇이고, 어떤 장점이 있나요?

불변 객체는 한 번 생성된 후 상태(필드 값)가 절대 변경되지 않는 객체입니다. String, Integer, LocalDate, BigDecimal이 대표적입니다.

불변 객체의 장점:

장점설명
스레드 안전 상태가 변하지 않으므로 동기화 없이 여러 스레드가 공유 가능. synchronized 불필요
방어적 복사 불필요 공유해도 변경될 우려가 없으므로 복사 없이 참조를 전달해도 안전
HashMap 키로 안전 키 객체가 변경되어 hashCode가 달라지는 문제 없음
실패 원자성 보장 생성이 실패해도 기존 상태를 오염시키지 않음
캐싱 가능 값이 변하지 않으므로 안전하게 캐싱, String Pool이 대표적 예
Effective Java 원칙: "클래스는 꼭 필요한 경우가 아니라면 불변으로 만들라." 불변으로 만들 수 없다면 변경 가능성을 최소화하라.
꼬리질문 대비: "불변 객체의 단점은?" — 값을 변경할 때마다 새 객체를 생성해야 하므로 객체 생성 비용이 증가합니다. 특히 String 연결을 반복할 때가 전형적인 예입니다. 이 경우 중간 단계 객체를 재사용하는 가변 동반 클래스(StringBuilder)를 사용합니다.
불변 객체를 만드는 방법을 코드로 설명해주세요.

불변 객체를 만들기 위한 핵심 규칙은 다음과 같습니다.

public final class ImmutableOrder {   // 1. 클래스를 final로 (상속 금지)

    private final long orderId;         // 2. 모든 필드를 private final
    private final String customerName;
    private final List<String> items;  // 3. 가변 객체 필드는 방어적 복사 필요

    public ImmutableOrder(long orderId, String customerName,
                          List<String> items) {
        this.orderId = orderId;
        this.customerName = customerName;
        // 4. 생성자에서 방어적 복사 (외부 변경 차단)
        this.items = List.copyOf(items);  // Java 10+ 불변 리스트
    }

    public long getOrderId() { return orderId; }
    public String getCustomerName() { return customerName; }

    // 5. getter에서도 방어적 복사 (내부 상태 노출 차단)
    public List<String> getItems() {
        return Collections.unmodifiableList(items); // 또는 List.copyOf(items)
    }

    // 6. setter 없음
    // 7. 상태 변경 대신 새 객체 반환
    public ImmutableOrder withCustomerName(String newName) {
        return new ImmutableOrder(orderId, newName, items);
    }
}
방어적 복사의 타이밍: 생성자에서 받은 가변 객체(Date, List 등)는 즉시 복사해야 합니다. TOCTOU(Time-Of-Check-Time-Of-Use) 취약점을 막기 위해 타입 검사 후 복사가 아니라 복사 후 타입 검사 순서를 지켜야 합니다.
Checked Exception vs Unchecked Exception의 차이와 언제 사용해야 하나요?

Java 예외 계층 구조에서 Throwable은 Error와 Exception으로 나뉩니다. Exception 중 RuntimeException을 상속하면 Unchecked, 그 외는 Checked입니다.

구분Checked ExceptionUnchecked (RuntimeException)
컴파일러 강제try-catch 또는 throws 선언 필수강제 아님
예시IOException, SQLException, ParseExceptionNullPointerException, IllegalArgumentException
사용 시기호출자가 복구 가능한 외부 상황 (파일 없음, 네트워크 오류)프로그래밍 오류 (null 전달, 잘못된 인덱스)
단점API가 복잡해짐, 과도하면 catch 블록 남용예외 놓칠 수 있음
실무 추세: Spring, JPA 등 현대 프레임워크는 대부분 Unchecked Exception을 선호합니다. Checked Exception은 호출자에게 복구 가능한 경우에만 한정해 사용하고, 복구 불가능한 오류는 RuntimeException으로 wrapping해 던집니다.
꼬리질문 대비: "Error는 catch하면 안 되나요?" — OutOfMemoryError, StackOverflowError 같은 Error는 JVM 수준의 심각한 문제로, 일반적으로 catch해서 복구를 시도하면 안 됩니다. 다만 특수한 경우(특정 OOM 후 리소스 해제 등)에 catch하되 재던지거나 종료하는 패턴을 쓰기도 합니다.
try-with-resources는 무엇이고, finally와 어떻게 다른가요?

try-with-resources는 Java 7에서 도입된 문법으로, AutoCloseable을 구현한 자원을 자동으로 닫아주는 기능입니다. 예외가 발생하든 안 하든 반드시 close()가 호출됩니다.

// 기존 finally 방식 (Java 6 이하) — 문제: close()에서도 예외 발생 시 복잡
Connection conn = null;
try {
    conn = dataSource.getConnection();
    // ... 작업 ...
} finally {
    if (conn != null) {
        try { conn.close(); }
        catch (SQLException e) { /* 삼킴 */ }
    }
}

// try-with-resources (Java 7+) — 깔끔하고 안전
try (Connection conn = dataSource.getConnection();
     PreparedStatement ps = conn.prepareStatement(sql)) {
    // ... 작업 ...
}  // 블록 종료 시 ps.close(), conn.close() 자동 호출 (역순)

// AutoCloseable 직접 구현
public class ManagedResource implements AutoCloseable {
    @Override
    public void close() {
        // 자원 해제 로직
    }
}
억제된 예외(Suppressed Exceptions): try 블록과 close()에서 동시에 예외 발생 시, close()의 예외는 억제되고 try 블록의 예외가 전파됩니다. 억제된 예외는 Throwable.getSuppressed()로 확인할 수 있습니다. 기존 finally에서는 close() 예외가 원본 예외를 덮어쓰는 버그가 있었습니다.
커스텀 예외를 만드는 기준은 무엇인가요?

커스텀 예외는 표준 예외로 표현할 수 없는 도메인 특유의 에러 상황을 명확히 표현하고, 호출자가 특정 예외를 구분해 처리해야 할 때 만듭니다.

커스텀 예외를 만들어야 하는 경우:

상황예시
도메인 특유의 비즈니스 오류주문 불가 상태, 잔액 부족, 중복 예약
에러 코드 체계가 필요한 경우API 응답에 에러 코드 포함 필요
계층별 예외 변환이 필요한 경우Repository 예외 → Service 예외로 wrapping
// 기본 커스텀 예외 구조
public class OrderException extends RuntimeException {

    private final ErrorCode errorCode;

    public OrderException(ErrorCode errorCode) {
        super(errorCode.getMessage());
        this.errorCode = errorCode;
    }

    public OrderException(ErrorCode errorCode, Throwable cause) {
        super(errorCode.getMessage(), cause); // 원인 예외 체인
        this.errorCode = errorCode;
    }

    public ErrorCode getErrorCode() { return errorCode; }
}

public enum ErrorCode {
    ORDER_NOT_FOUND("ORDER_001", "주문을 찾을 수 없습니다"),
    INSUFFICIENT_STOCK("ORDER_002", "재고가 부족합니다");

    private final String code;
    private final String message;
    // 생성자, getter 생략
}
피해야 할 안티패턴:
- 예외를 단순히 메시지 전달 용도로만 사용하고 실제 처리 없이 catch 후 rethrow
- 너무 세분화된 예외 (ExceptionA, ExceptionB, ExceptionC...)로 예외 계층이 과도하게 복잡
- 예외를 흐름 제어(flow control)용으로 사용 — 정상 로직 분기에 예외를 쓰면 성능 저하와 가독성 문제
꼬리질문 대비: "예외 메시지에 민감한 정보를 담아도 되나요?" — 안 됩니다. 예외 메시지는 로그에 남고 스택 트레이스와 함께 외부에 노출될 수 있습니다. 비밀번호, 카드번호, 개인정보는 절대 예외 메시지에 포함하지 않습니다. Spring의 @ControllerAdvice에서 클라이언트 응답 시 내부 메시지를 필터링해야 합니다.
